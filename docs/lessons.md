## L-031 — "Remove the hardcoded label cap" failed twice because the cap lived in TWO rendering engines with DIFFERENT footprint models; the shared authority was real but one engine fed it labels the wrong way

**Rule:** When a user says "I changed the code but the cap is still there," and a "single authority" for the cap already exists, do NOT assume the authority is unwired — assume one *render path* feeds it non-uniformly. Here the x-axis label budget [`maxXAxisLabels`](../client/src/lib/charts/xAxisLabelCap.ts) was correctly wired into the **recharts** surfaces (which rotate labels −45° → ~28px footprint → many labels), but the **visx** path (Explore/generate-charts, a *different charting engine*) called the same authority with `rotationDeg: 0` and rendered labels **horizontal** → a ~10-char date footprint ≈74px → ~11 labels even fullscreen. Same authority, opposite outcome, because the *projection model* (rotated height-bound vs horizontal text-width-bound) differed per engine. The visible "10–11 everywhere" was actually FOUR compounding causes: (1) magic bounds `ABS_MAX=60`/`DEFAULT=10` with no data-count input; (2) the visx horizontal footprint; (3) a first-paint `useEffect` width=0 → fixed-10 flash; (4) per-surface param drift. Fixing only the recharts path (the one easy to eyeball) left the disease.

**Why:** this is the L-020 pattern (a max limit has many enforcement points — enumerate ALL before editing) crossed with L-019 (a single authority only fixes the bug if its INPUTS are built uniformly on every path). The trap that hid it: the author verifies on the surface they know (chat = recharts, which already mostly worked) and never opens the *other engine's* path (Explore = visx). A budget function being "centralized" is necessary but not sufficient — if two callers pass it geometrically different inputs (`rotationDeg`, `labels` vs `dataPointCount`), the "single authority" silently produces two different answers. Also: the fix is 100% client-side (Vite HMR) for the on-screen charts, so the L-009 "stale server" reflex does NOT explain a persisting client-render symptom — only the PPTX/PDF export edit ([`chartSsr.ts`](../server/lib/exports/chartSsr.ts)) needs a server restart.

**How to apply:** (1) For "the cap is still there after I changed it," grep for EVERY caller of the cap authority and classify by *render engine* (recharts vs visx vs ECharts here) — each may have its own footprint/projection model. (2) Make the authority's contract explicit enough that callers can't diverge: add the real ceiling (`dataPointCount` — you can't label more buckets than exist) and one entry point (`xAxisTickBudget`) that decides BOTH count and rotation, so no caller re-derives geometry. (3) When "remove a magic cap," replace the number with a *derived* bound (no-overlap density ∧ data count), keeping only a high pathological-DOM backstop — not a UX target. (4) Verify on the engine the user actually looks at, not the one you find first; a deterministic density table (count at tile/modal/fullscreen widths) proves it without a fragile browser count. See [[L-020]], [[L-019]], [[L-024]] (single-authority + uniform inputs), [[L-009]] (client HMR vs server staleness).

## L-030 — "Test scripts keep running indefinitely / servers left hanging" — the leak is wedged `node:test` orphans, SIGTERM can't kill them, and the sandboxed Bash tool can't either; only an out-of-sandbox hook + a process-group watchdog can

**Rule:** When "code-started servers keep hanging," don't assume dev servers — *measure*. Here the ports (3000/3002/8001) were empty; the resource hog was **60+ orphaned `node --import tsx --test` processes** (one at 96.9% CPU for ~63h). A test wedges two ways: a **sync infinite loop** (which `--test-timeout` can't interrupt — the event loop is blocked, the timer never fires) or an **open handle** (timer/socket/duckdb left open → `node:test` hangs on a non-empty event loop). They run as **detached, no-controlling-terminal (TTY `??`) grandchildren** of the agent's Bash tool, so they reparent to launchd and survive the turn AND the session, accumulating for days. The fix is three layers, all default-on: (1) [`runTests.mjs`](../server/scripts/runTests.mjs) gets `--test-force-exit` (open-handle class) + a **detached process-group** with signal forwarding + a **wall-clock watchdog** that SIGKILLs the *group* (sync-loop class) → exit 124; (2) [`.claude/hooks/kill-stray-servers.sh`](../.claude/hooks/kill-stray-servers.sh) wired to **Stop** (`--tests-only`, every turn) and **SessionEnd** (`--all`); (3) `npm run kill:strays` as a manual hatch. See [`docs/conventions/managed-dev-processes.md`](conventions/managed-dev-processes.md).

**Why:** two non-obvious facts made this resistant to the "just kill them" reflex. (a) **SIGTERM does nothing to a CPU-bound loop** — a busy `while(true){}` never returns to the event loop to run its signal handler, so it ignores everything except SIGKILL; the watchdog and the hook both go straight to `-9`. (b) **The Bash tool is sandboxed and gets `EPERM`** signalling processes it didn't spawn — so the agent literally *cannot* reap cross-session orphans from inside its own shell (my first `kill -9` loop reported `killed=0` until I disabled the Bash-tool sandbox). A **hook runs outside that sandbox**, which is why the durable cleanup HAS to be a hook, not agent discipline. The clean signal for "code-started vs you-started-it-in-a-terminal" is the **controlling TTY**: terminal servers show a real tty (ttysNNN), code-spawned ones show `??`; gating on `$tty=="??"` spares the user's three-terminal dev loop and, unlike ppid-ancestry, survives the reparent-to-launchd that orphans undergo. Also: killing a `node:test` parent doesn't kill its per-file worker subprocess — you must kill the **process group** (hence `detached:true` + `kill(-pid)`), or the worker keeps spinning.

**How to apply:** (1) For "X keeps leaking/hanging," `ps -axo pid=,tty=,command=` and *count by pattern* before theorizing — the culprit may be a different process class than the user named. (2) A self-cleanup that must reap things the agent's shell spawned in PRIOR sessions belongs in a **hook** (out-of-sandbox), never in a Bash-tool command (sandboxed → EPERM). (3) `TTY=="??"` is the durable "not started by a human in a terminal" test; prefer it over ppid-ancestry, which breaks once a process reparents to init. (4) To bound a runner that can wedge on a sync loop, a per-test timeout is insufficient — add a wall-clock watchdog that SIGKILLs the whole **process group**. (5) `--test-force-exit` (Node ≥20.14) fixes the open-handle hang without failing any passing test; keep the riskier per-test `--test-timeout` opt-in so it can't fail slow live-integration tests. See [[L-013]] (write-out is my job) for why the backstop is automated, not remembered.

## L-029 — "A quick answer should still show a chart + pivot of all performers" — the fast path returns BEFORE the depth-budget machinery, so output-shaping never reaches it; and a single-winner answer can't visualize itself

**Rule:** The quick-answer fast path ([`quickAnswerPath.ts`](../server/lib/agents/runtime/quickAnswerPath.ts)) returns its `AgentLoopResult` *above* `classifyQueryIntent`/`ctx.depthBudget` (the [query-intent authority](../server/lib/agents/runtime/queryIntentAuthority.ts), invariant #12). So everything the FULL loop does to a minimal-depth answer — including the deterministic single fallback chart in [`visualPlanner`](../server/lib/agents/runtime/visualPlanner.ts) — simply **never runs** for a fast-path answer. To add a chart there you enrich the fast path itself; it won't inherit anything from the loop. Two sub-traps: (1) the **pivot was already there** (`derivePivotDefaultsFromExecution` re-queries *base*, so even a 1-row answer's pivot shows all entities) — the only gap was the chart; (2) a single-winner answer (`who is THE top performer` → `limit 1` → one row) **can't chart itself** ([`chartFromTable`](../server/lib/agents/runtime/chartFromTable.ts) returns null for 1 row), so re-execute a *leaderboard frame* (same groupBy/measure, drop the limit, sort by the measure, cap under the 60-cardinality ceiling) and chart THAT, leaving the concise answer table untouched.

**Why:** Adding a chart+pivot to a "minimal" ask looks like it reopens the "simple question → plethora" bug (invariant #12), but it doesn't: one chart + one pivot that **visualize the same answer data** are not the speculative *extra* charts / recommendations / dashboard-offer / follow-up fan-out that invariant guards — it's exactly parity with what the full-loop minimal path already emits (one deterministic chart). The load-bearing detail when re-deriving the leaderboard sort: the executor's output column name is computed by `outputAliasForAgg` ([queryPlanDuckdbExecutor.ts](../server/lib/queryPlanDuckdbExecutor.ts)) with a branchy rule (`count`-on-id, `countIf`→`matching`, `sumIf`, `perDimension`, plain `${col}_${op}`). Re-implementing that to name the `sort.column` is fragile — instead **pin** the alias back onto the aggregation (`agg.alias` wins in every executor) so the sorted column is guaranteed to exist.

**How to apply:** (1) Before "make the answer also do X," check whether the question takes a *fast path* — a fast-path return bypasses every loop-stage gate, so the change belongs in the fast path, not the loop. (2) When an answer is a single value but the user wants "all of them," the answer frame ≠ the visualization frame — re-query a broader frame for the chart; don't try to chart the one-row answer. (3) To make a re-derived plan's `sort`/filter reference an executor output column, pin an explicit `alias` rather than re-computing the default name. (4) Keep the new behavior a flag with a kill switch ([`QUICK_ANSWER_CHART_ENABLED`](../server/lib/featureFlags.ts), default ON) and best-effort try/catch so a re-exec failure never breaks the 2-second path. See [[L-019]] (uniform inputs) and the dual-engine alias rule.

## L-028 — "Why is a daily/weekly thing a bar not a line?" — the line-vs-bar decision was scattered with non-uniform inputs, and temporal FACET columns masquerade as categoricals in `summary.columns`

**Rule:** A temporal facet column (`Day · Date`, `Week · Date` = `<Grain> · <SourceDate>`) is materialized into `summary.columns` as `type:"string"` and is **absent from `summary.dateColumns`** (via `temporalFacetMetadataForDateColumns` in [fileParser.ts](../server/lib/fileParser.ts)). So any builder that classifies columns by `dateColumns`/`numericColumns` treats it as a category. The dashboard feature-sweep did exactly that — its dimension loop guarded only raw date columns (`dateCols.has(t)`), never the facet keys — and charted `Compliance Visit (avg) by Day · Date` as a **bar**, while [`chartFromTable`](../server/lib/agents/runtime/chartFromTable.ts) (which checked `isTemporalFacetColumnKey`) charted the same facet as a **line**. The fix: one leaf authority [`chartTypeAuthority.ts`](../server/lib/chartTypeAuthority.ts) (`isTemporalChartX`/`resolveChartType`) consulted by every builder with the SAME inputs — `chartFromTable`, the feature sweep (facet → line), the `build_chart` tool (coerce temporal-x bar → line at construction), and the verifier (predicate unified, stays flag-only).

**Why:** this is [[L-019]] again — a "single authority" only fixes a bug if its INPUTS are uniform across every path, and here there wasn't even one authority: line-vs-bar was re-derived inline in ≥4 places, each checking a different subset of (raw date col / facet key / period axis). Two compounding traps: (1) the verifier *detected* `BAR_ON_TEMPORAL_X` but, per the single-flow policy (invariant #6), only emits a `flow_decision` — it never rewrites, so detection ≠ fix; the deterministic FIX has to live at construction. (2) flipping the sweep's facet charts from bar→line surfaced a LATENT duplicate-line bug: the trend branch's line and a same-grain loop line now share `chartAxisSignature` (`type|x|y|series`), so `coveredX` had to be updated AFTER the trend push (it was snapshotted from `mergedCharts` before the loop). Also: `tryBuildChart` had `aggregate:"mean"` welded to `type==="bar"`; decoupling it (optional override) is what lets a temporal-facet line carry a per-period mean for a rate metric.

**How to apply:** (1) For "wrong chart TYPE," suspect a scattered decision before the renderer — grep for every `? "line" : "bar"` and route them through `resolveChartType`. (2) Remember facet columns live in `summary.columns` as `type:"string"`; a `dateColumns`-only guard will let them through as categoricals — use `isTemporalFacetColumnKey` / `isTemporalChartX`. (3) Coercing a TOOL ARGUMENT at construction (build_chart bar→line) is in-policy normalization, NOT a single-flow violation — that rule governs *flow* (replan/rewrite) decisions, not a tool sanitizing its own inputs. (4) When you flip a chart's type across builders, re-check dedup: a signature that includes `type` means two builders that previously differed only by type now collide. (5) Aggregate is metric-aware for a time line: rate/%/score → `mean` (reuse `RATE_METRIC_RX`), counts → `sum`; keep the lower-bound `<2 buckets` skip (a one-point line is degenerate) but drop the upper-bound top-N bucketing (a time axis is not a leaderboard).

## L-027 — "Why is the average 0 when the column has values?" — a `HH:MM:SS` column is being read as text, not a quantity; fix the TYPE upstream, don't patch the aggregation

**Rule:** When an aggregate is structurally 0/empty and the narrator faithfully reports it ("none of its rows had a numeric value"), the narrative is *correct* — the bug is the INPUT. A "Working Hrs" = `03:31:57` column is a **duration** (elapsed time = a quantity) that the type detector classified as **time-of-day** (`classifyAsTimeOfDay` fires on any ≥95% `HH:MM:SS` column regardless of name), so it stayed text, never entered `numericColumns`, and every coercion (`toNumber`/`numericValue`/DuckDB `TRY_CAST … AS DOUBLE`) dropped it to NaN/NULL → 0. The fix is upstream typing, not the three aggregation sites: detect durations, convert the cells to **decimal hours at ingest**, and carry a `duration` annotation — exactly the **currency** pattern (store a number, annotate, format at the leaves). Once the cell is a number, every aggregation path and DuckDB work **unchanged** — the single most valuable property to confirm before touching hot paths.

**Why:** durations and time-of-day both look like `HH:MM:SS` but answer different questions, so one classifier can't serve both. A duration is averaged (convert in place — nothing else reads the string); a time-of-day is *compared chronologically* against quoted `'09:30:00'` literals, so it must stay a string — averaging it needs type-aware coercion (`timeOfDayToSeconds`) at the aggregation boundary + clock-formatting of the ephemeral result. Two different mechanisms, picked by the constraint (does anything else read the raw value?), not by surface similarity. Disambiguator that's robust: a value ≥ 24h CAN'T be a clock, and the strong signal must require a colon-string — else a plain count of `30` ([[L-019]] input-contract trap) gets mis-flagged as a 30-hour duration.

**How to apply:** (1) For "0 but the data's there," check the column's TYPE/`numericColumns` membership before suspecting the math. (2) Mirror an existing annotation pattern (currency) instead of inventing one; the `duration` field is ONE Zod object (client re-exports the server schema), so [[L-021]] didn't bite — but I still grepped every column-meta `z.object` to confirm. (3) Convert on the FULL dataset inside `applyUploadPipelineWithProfile` via a per-column side-channel tally (the cells are numbers by the time `createDataSummary` re-reads), and `reset` the tally per run so incremental-refresh doesn't inherit a prior dataset's. (4) When a fix can't cover all engines cleanly (time-of-day avg only wired into `run_breakdown_ranking`, not pivots/DuckDB), scope it to the common question shape and DOCUMENT the gap rather than ship surprising half-formatted output. (5) Existing uploads need a re-upload — the conversion runs at ingest only; say so.

## L-026 — "Is the chart insight from the same place as the other smart stuff?" — no: per-chart `keyInsight` and the answer-envelope `likelyDrivers` are TWO engines; converge them on discipline, not by merging

**Rule:** When a user says one output "isn't as smart as the other things we built," first establish whether they are even the same code path. Here the per-chart insight (`generateChartInsights` → `keyInsight`, a single prose blob capped at 2200 chars) is a *different, older* engine from the answer envelope's gated causal lane (`likelyDrivers`, hedged + deterministically verified + chip-rendered in `AnswerCard`). They are NOT the same place — so "make it as smart" means converging the cruder one onto the *discipline* of the smart one (a tight HEADLINE, a clearly-hedged WHY, a concrete DO), reusing its primitives, **not** routing charts through the envelope generator (a chart has no 1:1 envelope). The fix: cut the cap 2200 → 550, drop the verbose "SHAPE" lane, and gate the chart's WHY with the SAME `hasHedge`/`STAT_NUMBER_RE` rail the envelope uses — by *exporting* those primitives, not copying them.

**Why:** the two compounding causes were both real (the [[L-018]] "decoration = weak generation AND buried rendering" pattern): the text was a 3–5-sentence wall (generation) AND it was rendered as one flat markdown blob with no scannable structure (rendering). Fixing only one end leaves it still feeling unhelpful. Carrying structure as a new persisted schema field would have hit the [[L-021]] five-object strip trap; instead the three lanes ride inside the existing `keyInsight` string via a `WHY:`/`DO:` marker wire-format parsed identically on both tiers (one shared `server/shared/chartInsightLanes.ts`, re-exported to the client like `chartSort`) — zero schema change, full back-compat for already-persisted blobs. The parser is MARKER-based, not newline-based, on purpose: `normalizeInsightText` collapses the model's newlines, so a line-based split would silently lose the lanes.

**How to apply:** (a) before "make X as good as Y," diff their generators — they may not share one; the cheap win is exporting Y's gate so X reuses it ([[L-022]]: ship the deterministic gate before the prompt opens the speculation permission). (b) To add structure to an output without a schema migration, encode it in an existing string field with a marker convention + a single shared parser, and keep the parser robust to whatever whitespace normalization runs downstream. (c) Keep the WHY in `keyInsight`, never the domain-pack-gated `businessCommentary` — the latter is absent on non-FMCG datasets (the W-DX2 invariant), so relocating the "why" there silently drops it for, e.g., Titanic.

## L-025 — A user pointing at "these cards" on the dashboard can mean the read-only Executive-Summary BAND, not the grid tiles — and a feature "missing" after a v1→v2 migration is usually present on the frozen v1 path only

**Rule:** Two traps converged in one report. (1) When the user said "I can't move/delete these cards," the obvious target was the react-grid-layout tiles — but the cards they meant were the **Executive-Summary band** ([DashboardSummaryBand.tsx](../client/src/pages/Dashboard/Components/DashboardSummaryBand.tsx)), a *separate, read-only* component with no edit affordances at all. The grid's drag/resize/delete machinery was provably correct (single shared `DashboardEditModeProvider`, CSS imported, grab-handle on every tile) — so "it doesn't work" was never a grid bug; the band simply never had the feature. **Confirm WHICH component renders the thing the user is pointing at before diagnosing.** (2) The chart "Sort by" control and click-to-fullscreen were both "missing" because the **Chart v1→v2 convergence** routes most charts through `<PremiumChart>`, and both features lived only on the v1 path (`InteractiveChartCard` gated the control on `!!localV1`; click-to-expand lived in `ChartRenderer`, not `PremiumChart`). The feature wasn't deleted — the *render route* moved out from under it.

**Why:** this is the L-019 pattern ("a centralized decision re-opens the bug when its INPUT is path-dependent") applied to a migration. When a render path forks (v1 legacy ↔ v2 premium), every interactive affordance attached to ONE branch silently disappears for charts that take the other branch — and which branch a chart takes is a per-mark feature flag, so it's invisible in the spec. The user's own hunch ("maybe on a different route that isn't making it to the final app response") was exactly right.

**How to apply:** (a) when a dashboard/chart feature is "missing," enumerate EVERY component that can render that surface (grid tile vs summary band; `ChartRenderer` vs `PremiumChart` vs the two fullscreen modals) and check each — don't assume one path. (b) Put shared affordances behind a spec-shape-agnostic predicate: `chartSupportsSort` now accepts v1 OR v2 ([useChartSort.ts](../client/src/lib/charts/useChartSort.ts)), and reordering reads `source.rows` for v2 / `data` for v1 through the same `applyChartSort` authority. (c) For editing AI-generated structured content (the answer envelope), edit the RAW persisted arrays (not the capped/sorted display) so indices map 1:1, and conform edits to the *dashboard* envelope object, not the message one ([[L-021]]).

## L-024 — A big "new" feature is often an existing engine pointed at a new entry point — find it before building; and capture the replay recipe BEFORE you swap the data

**Rule:** Before designing a large feature from scratch, hunt the tree for an engine that already does ~70% of the hard part. "Incremental data refresh" (attach new data → regenerate the whole dashboard/insights) turned out to BE the existing automation capture→replay system ([buildRecipeFromChat](../server/lib/automations/buildRecipeFromChat.ts) + [replayLoop.service.ts](../server/lib/automations/replayLoop.service.ts)) — it already re-runs a recipe of saved plan steps against new data and re-narrates live. The whole WR0–WR8 feature was a re-target: extract `replayRecipe(RecipeSource, {mode})` out of `replayAutomation` (keeping the loader thin so 49/49 automation tests stayed green), add an `overwrite` mode, and wrap it with ingest + dashboard-in-place. Reusing the proven core (incl. the `AutomationReplayBanner` + `AutomationRemapDialog` client components verbatim) is why a feature this size shipped with the risk concentrated in ONE wave (the rebind, WR5).

**Why:** two ordering traps make the difference between correct and silently-wrong. (1) **Capture the recipe from the PRE-ingest chat.** The recipe's `expectedSchema` + plan-step column refs must be the OLD schema, because the column-mapping (`computeAutomationColumnRemap`) maps OLD→new; if you build the recipe after swapping the data, it compares new-vs-new, detects no drift, and the old column names inside the plan steps fail at tool-exec time. (2) **A title-only chart rebind is an L-010 collision waiting to happen** — two charts that share a title but differ in breakdown will swap data on rebind; key by the axis-aware `chartIdentityKey` and only fall back to title when it's unambiguous among the fresh charts.

**How to apply:** when re-targeting a shared core, keep the old entry point a thin loader over the extracted core (so the existing test suite is your regression guard), and run that full suite — the shared replay core is used by production automations, so a regression there silently corrupts every replay (L-017). Also: on a destructive swap (refresh ingest), snapshot the pre-state in memory and roll the WHOLE session back on any downstream failure, so a half-done regenerate never leaves the user worse than before.

## L-013 — Name-based "is this a date?" inference must be backstopped by a VALUE-magnitude guard

**Rule:** Detecting a date axis from the column NAME alone is fragile: `Week`/`Month`/`Year`/`Day` columns are ordinal counters whose VALUES (1..52, 1..12, 2025, 1..31) are not calendar instants. Running such a value through `new Date(n)` renders epoch-relative garbage ("1 Jan 1970"). The robust fix is a magnitude guard at the format layer: on a date-named axis, a finite numeric value below `DATE_EPOCH_MS_MIN` (1e8 ms ≈ 1970-04-26) is an ordinal → render as a plain number, never a date. Real calendar values arrive as ISO strings (`Number(v)` is NaN) or full epoch-millis (≥ ~1e12), both of which still format as dates. See [client/src/lib/charts/format.ts](../client/src/lib/charts/format.ts) `formatChartValue`. This generalises the [[L-011]] "Day" token removal to every ordinal.

**Why:** removing just the `day` token from `DATE_RE` fixed the reported "Day = 1-Jan-1970" but left `Week`/`Month`/`Year` numeric columns exposed to the identical bug. The name heuristic is the wrong layer to be authoritative — the value's magnitude is.

**How to apply:** Two adjacent completions shipped alongside this guard, both extending earlier waves — (1) **best+worst leaderboard** for high-cardinality breakdown dims ([dashboardFeatureSweep.ts](../server/lib/agents/runtime/dashboardFeatureSweep.ts) `bucketTopAndBottom`): exhaustive breadth now keeps top-K AND bottom-K groups (worst no longer buried in "Other") so "top & worst at every level" holds for people-level dims like TSO_TSE Name; (2) **durable `investigatedSubQuestions`** — the "Investigated · N charts" badge is now persisted (AgentLoopResult → dataAnalyzer → messageSchema `.max(16)` → chatStream → `MessageBubble` archived `ThinkingPanel`), closing the [[L-010]] rank-3 "live-SSE-only, vanishes on reload" gap. When you "fix" a name-based heuristic, ask whether the deeper invariant is a value property, and whether a sibling surface (persistence/reload) needs the same data.

## L-012 — A literal control byte in source hides the file from ripgrep

**Rule:** A source file containing a literal control byte (NUL U+0000, SOH U+0001, Unit Separator U+001F, …) is classified as *binary* by `file(1)`, and **ripgrep — hence the Grep tool, the shell `grep`/`rg` wrappers, and most editor search — silently skips it**. A "no matches" result for such a file is a false negative, not proof the symbol is absent.

**Why:** During the 2026-06 duplication audit, four chart files (`server/lib/chartGenerator.ts`, `server/lib/agents/runtime/tools/marketBasketTool.ts`, `client/src/lib/charts/dataEngine.ts`, `client/src/lib/charts/visxRenderers/BarRenderer.tsx`) embedded literal control bytes as composite-Map-key delimiters. The audit's ripgrep-based finders silently skipped them — missing ~10 real duplications — and a verifier mis-read an invisible NUL as a space and reported a non-existent "bug". Any dead-code / "no caller" deletion verified with ripgrep over such files would be unsafe.

**How to apply:** (1) When a `grep`/Grep "no match" is load-bearing (e.g. before deleting "dead" code), confirm with `LC_ALL=C command /usr/bin/grep -a` (forces text mode, bypasses the shell's rg wrapper) or `Read` the file directly; `file <path>` reveals binary classification. (2) Don't embed literal control bytes in source — write the unicode escape and route composite keys through the shared `KEY_SEP`/`compositeKey` (convention: [composite-key-separators](conventions/composite-key-separators.md)). Fixed across the Dup1–Dup7 waves; see [duplication-audit-deferrals](decisions/duplication-audit-deferrals.md) for what was intentionally left.

## L-001 — Don't restore the legacy orchestrator fallback

**Rule:** `dataAnalyzer.answerQuestion` MUST throw when `AGENTIC_LOOP_ENABLED` is false. No fallback.

**Why:** The handler-based orchestrator chain was deleted in commit `9422bed7` (2026-04-26). Reintroducing a fallback re-creates the silent-divergence failure mode where two pipelines drift, masking agent regressions.

**How to apply:** If a test or scenario seems to need a "non-agentic" path, the right fix is `AGENTIC_ALLOW_NO_RAG=true` for that path, not a code-level fallback. See [docs/plans/agentic_only_rag_chat.md](plans/agentic_only_rag_chat.md).

## L-002 — Audit summaries are NOT ground truth

**Rule:** Before acting on an audit / inventory / "system overview" that flags a bug, re-verify against the live code with grep + Read. Audit text can lag the tree by weeks.

**Why:** The 2026-05-06 audit pass corrected three drifted sections in CLAUDE.md that had accumulated over 2-3 weeks of waves. The audit caught real drift, but if Claude had implemented changes based on the pre-audit text, work would have been wasted. Similarly, A1's "race" finding turned out partially false-positive (pivot cache was already correct via FA2).

**How to apply:** Whenever the user's question references "the audit says X" or "the system overview claims Y", verify X/Y against the actual file at the cited line numbers before proposing changes.

## L-003 — The unified `withSessionWriteLock` is the per-session write mutex

**Rule:** All RMW writes to a `ChatDocument` must acquire `withSessionWriteLock(sessionId, fn)` from [server/lib/sessionWriteLock.ts](../server/lib/sessionWriteLock.ts) (Wave A2).

**Why:** Pre-A2 there were three independent in-process mutex maps (`sessionPersistChain`, `sessionPatchChain`, `activeFilterLocks`) that serialized within their own call sites but did NOT coordinate with each other. ## L-007 — Temporal chart/filter values are CANONICAL keys; format only at the render layer. Never down-convert a coarse grain.

**Rule:** Chart x-values and temporal-facet filter values must stay canonical, sortable period keys (`2023-Q1`, `2023-01`, `2023`, `2023-H1`, `2023-W12`, `2023-MM-DD`). Human labels (`Q1 2023`, `Jan 2023`) are produced ONLY at the render layer (tick/axis formatter), never baked into the data. And the bucketing grain must match the data's actual grain — a quarter is NOT a month.

**Why:** The server baked `MMM-yy` display strings (`"Jan-23"`) into chart x-values via [chartGenerator.applyTemporalXAxisLabels](../server/lib/chartGenerator.ts) + [aggregateData](../server/lib/chartGenerator.ts) (`displayLabel || key`). The user corrected: "this data has no month column, only quarters — don't convert that into monthly data." The codebase already had the right primitives ([compareTemporalOrLexicalLabels](../client/src/lib/temporalAxisSort.ts) parses `YYYY-Qn`; [normalizeDateToPeriod](../server/lib/dateUtils.ts) emits `2023-Q1` + `Q1 2023`); the bug was discarding them.

**How to apply:** Server aggregation/labelling emits `normalizedKey` (canonical), and temporal-facet columns pass through verbatim (never re-bucketed). When adding a new chart/temporal surface, sort by the canonical key and format for display separately — and drive the grain from the data, never from day-gap spacing.

## L-008 — Windowed/time-series admin metrics must come from the per-turn event log (`past_analyses`), never from session-lifetime counts attributed to `createdAt`.

**Rule:** Any "per day" / "in this window" / "active users" KPI must be aggregated from an event log whose rows carry the event's OWN timestamp + actor. For this product that is [`past_analyses`](../server/models/pastAnalysis.model.ts) (one doc per completed turn: `createdAt`, `userId`, `sessionId`, per-turn `charts`, `feedback`/`feedbackDetails`). Do NOT derive them from `getAllSessions()` lifetime counts.

**Why:** The superadmin dashboard's old session-metrics aggregator ([metricsAggregator.ts](../server/lib/admin/metricsAggregator.ts) — since removed/replaced by the per-turn path described under "How to apply") summed each session's *all-time* `ARRAY_LENGTH(messages/charts)` and attributed the whole total to the session's `createdAt` day. So a session created before the window contributed 0 (even if heavily used during it), a session created in-window dumped its entire lifetime into one day, `messages` double-counted user+assistant entries, and active-users/DAU/WAU/MAU came from each session's single `lastUpdatedAt` (a user active 30 days with one long-lived session counted as active on ONE day; DAU was "last populated day", not today). Feedback ([aggregateFeedbackMetrics](../server/lib/admin/metricsAggregator.ts) + [aggregateFeedbackCountsBySession](../server/models/pastAnalysis.model.ts)) counted only the root `feedback` field, silently dropping every chart-level vote (those live ONLY in `feedbackDetails[]`). Net: values were wrong on essentially every superadmin surface. The user reported "values on all pages of superadmin seem wrong in most cases."

**How to apply:** Activity/feedback KPIs come from [`aggregatePastAnalysisMetrics`](../server/lib/admin/metricsAggregator.ts) (single cross-partition query, projects `ARRAY_LENGTH(c.charts)` so chart bodies aren't fetched) → pure, unit-tested [`summarizePastAnalysisRows`](../server/lib/admin/metricsAggregator.ts). `sessionsCreated` is the ONE metric correctly keyed on the session's `createdAt` ([`aggregateSessionsCreatedByDay`](../server/lib/admin/metricsAggregator.ts)). Vote-counting is one leaf helper, [`countTurnVotes`](../server/lib/admin/feedbackVotes.ts), shared by the landing KPIs and the per-session badges (counts `feedbackDetails`, falls back to root for legacy docs). `activeUsers.window` = distinct users across the requested window (was hardwired to MAU). Cache-hit replays don't write a fresh `past_analyses` doc, so [`serveCachedExactAnswer`](../server/services/chat/chatStream.service.ts) emits an `analysis.cache_hit` usage event (one per user per analysis); the controller synthesizes a chartless/voteless turn row from each and folds it into the SAME `summarizePastAnalysisRows` pass (so Questions + active-user counts stay honest and a same-day cache hit by an already-active user isn't double-counted), with a dedicated "Cache hits" KPI breaking out the subset.

## L-009 — Server dev is NOT hot-reloaded; client (Vite) is. A server-side code change is invisible until the Node process restarts.

**Rule:** When a code change "does almost nothing" / "isn't reflected," the FIRST hypothesis is a stale running process, not a logic bug. The server [`dev`](../server/package.json) script ran `tsx index.ts` (no `watch`) — `tsx` reads source ONCE at boot and never reloads. The client runs Vite (`npm run dev`) which auto-HMRs. So after editing both layers and refreshing the browser, the CLIENT changes appear and the SERVER changes silently do not — which reads exactly like "my change was overridden / it's a race." It is neither.

**Why:** This bit a 10-wave dashboard fix: the user re-ran "build pjp adherence dashboard" and saw ONLY the client pivot/axis fixes (W1–W3, Vite HMR) while every server-side change (valid-universe scoping W5–W7, anti-hedge W8, investigation filter W9 + the lazy applicability hook) appeared to do nothing. The code was committed (`c2f848f6`), correct (inference verified against the real workbook → gate `PJP Planned Type ∈ {Market Working}`), and propagated correctly (the `ctx.summary` mutation reaches the planner + coverage gate in order). The ONLY problem was the live server (`tsx index.ts`, no watch) still executing pre-edit code.

**How to apply:** Fixed by changing the server [`dev`](../server/package.json) script to `tsx watch index.ts` so the Node server auto-restarts on any file in its import graph — bringing it to parity with the client. After this, a server edit reloads like a client edit (one in-flight SSE request dies on restart, which is normal dev behaviour). When diagnosing "change not reflected": (1) check `lsof -nP -iTCP:3002` + `ps -o etime` for the process start time vs. when the edit landed; (2) `git status`/`git log` to confirm the edit is actually on disk; (3) only then suspect logic. `esbuild`'s `npm run build` passing proves the whole import graph parses, so a clean restart won't crash. The three-terminal restart protocol still lives in [.cursor/rules/restart-servers.mdc](../.cursor/rules/restart-servers.mdc).

## L-010 — "Investigated follow-ups don't show in the response" had TWO root causes, and a red herring: the prior fix was applied to a flag-gated DEAD path.

**Rule:** When the spawned "Investigating further" follow-up pass investigates sub-questions, its CHARTS and FINDINGS must reach the ONE final response. Two distinct, real gaps broke this — with different fixes. Do not conflate them, and do not edit `runDeepInvestigation`/`investigateNode` for this: `DEEP_INVESTIGATION_ENABLED` is absent from `server.env`, so that orchestrator never runs. The active path is `runAgentTurn` → [`runSpawnedFollowUpPass`](../server/lib/agents/runtime/spawnedFollowUpPass.ts) → [`runSubInvestigation`](../server/lib/agents/runtime/investigationOrchestrator.ts) (the shared helper). A prior session spent its effort aggregating a collected-charts accumulator inside `runDeepInvestigation` — pure dead code for this symptom.

**Why (findings-not-in-text):** [`runSpawnedFollowUpPass`](../server/lib/agents/runtime/spawnedFollowUpPass.ts) returns `investigated[] = [{question, answer, chartCount}]`, but [agentLoop.service.ts](../server/lib/agents/runtime/agentLoop.service.ts) consumed it ONLY as a log count (`investigated: pass.investigated.length`). The sub-turns' raw tool findings DID reach the shared blackboard (`addFinding` fires per successful tool step), but with a label built from tool+args — carrying NO "this came from an Investigating-further chip" signal. So the single narrator had nothing to attribute and never mentioned the investigation.

**Why (charts-on-dashboard-but-not-chat):** Server-side, chat-response charts and dashboard tiles draw from the SAME `mergedCharts` array, so they can't diverge there (an adversarial-verification pass refuted both the finalize-dedup and the synthesis-weave as the chart cause). **How to apply:** (1) Weave `pass.investigated[]` into the shared blackboard as provenance-tagged `notable` findings (`label: "Investigated follow-up: <question>"`) BEFORE the narrator runs — this surfaces them in the answer prose AND flows into the already-persisted, already-rendered [`investigationSummary`](../server/lib/agents/runtime/buildInvestigationSummary.ts) digest (no new schema/client needed; `markQuestionActioned` moves the question from "open" to a "finding"). (2) Make chart persistence axis-aware via [`chartIdentityKey`](../server/shared/schema.ts) (`type::title::x::y::seriesColumn`, derived from fields that SURVIVE the data-strip → backward-compatible) in both `chat.model.ts` dedup sites and the `chatStream.service.ts` re-hydration. Covered by [chartIdentityKey.test.ts](../server/tests/chartIdentityKey.test.ts).

## L-011 — "Not going full fledged" (≈6 charts, columns ignored, top/worst only for Cluster/ASM) AND "Day = 1-Jan-1970" were one classification problem + one missing engine.

**Rule:** Comprehensive per-dimension coverage must NOT depend on the LLM analysis-brief enumerating every column. Two root causes:
1. **Day → 1-Jan-1970** is a NAME-based format heuristic, not a data-type fact. [client/src/lib/charts/format.ts](../client/src/lib/charts/format.ts) `inferFormatHint` ran the column name through `DATE_RE`, which matched the bare word `day`; `formatChartValue` then routed `avg(Day)=15` through `formatDateSmart` → `new Date(15)` → "1 Jan 1970". The same fragility exists for ordinal `Week`/`Month`/`Year` columns (a magnitude guard in `formatDateSmart` is the deeper follow-up).
2. **Breadth/ignored-columns** is structural: the deterministic breadth engine ([dashboardFeatureSweep.ts](../server/lib/agents/runtime/dashboardFeatureSweep.ts) `enumerateMissingDashboardCharts`) (a) only ran when `analysisBrief.requestsDashboard`, and [`shouldBuildAnalysisBrief`](../server/lib/agents/runtime/analysisBrief.ts) is FALSE for plain analytical questions → no brief, no sweep, breadth = whatever the LLM planner + 2-chart visual planner happened to build ≈ 6; and (b) drew dimensions ONLY from the brief's `segmentationDimensions`/`candidateDriverDimensions` → any column the LLM omitted (Android/iOS, TSO_TSE Name) was never charted, and high-card name columns were hard-skipped.

[agentLoop.service.ts](../server/lib/agents/runtime/agentLoop.service.ts) runs it on analysis turns, ties the chart ceiling to `AGENT_MAX_FINAL_CHARTS_PER_TURN` (40), and writes a `computeDimensionLeaders` best/worst FINDING per dimension so the narrator states top AND worst in prose. Tests: [exhaustiveBreadthSweep.test.ts](../server/tests/exhaustiveBreadthSweep.test.ts), [format.vitest.test.ts](../client/src/lib/charts/format.vitest.test.ts). ## L-013 — Doc-freshness was asymmetric: read-in is auto-guaranteed, write-out was a manual `/wave-commit` + a Stop nudge that only fired on `Wave` commits — so the common "uncommitted work" case drifted silently.

**Rule:** The two edges of a session must get equal treatment. The **read-in** edge is genuinely automatic and unstaleable — a SessionStart hook ([.claude/hooks/session-warmup.sh](../.claude/hooks/session-warmup.sh)) auto-injects the live-recomputed orient pack ([generate-bootstrap.ts](../server/scripts/generate-bootstrap.ts)) incl. the invariant-firewall verdict, so a fresh session cannot start with a stale view of the *machine-checkable* facts. The **write-out** edge was NOT: [`/wave-commit`](../.claude/skills/wave-commit/SKILL.md) is purely manual, and the Stop hook ([.claude/hooks/check-docs-fresh.sh](../.claude/hooks/check-docs-fresh.sh)) only nudged when HEAD's subject started with `Wave ` AND STATE/WAVES were untouched — so if work was left uncommitted (the normal case) or committed under a non-`Wave` subject, the hook was completely silent. The commit-time hard gates ([.githooks/pre-commit](../.githooks/pre-commit), CI) catch invariant/doc-ref/registry drift but do NOT check that STATE.md/WAVES.md were updated. Net: narrative-doc freshness depended entirely on Claude remembering — the failure that once left STATE.md 62 commits behind (and was 9 behind with 34 files uncommitted when this was found).

**Why:** The user's standing requirement is "never get a stale idea about the codebase, and always keep the docs updated by the end of your final response." The read-in side already delivers the first half mechanically; the second half had no forcing function, only discipline + a nudge that fired in the wrong cases.

**How to apply:** Gave the Stop hook a second, freshness-gated trigger so it fires in the common case: **recent uncommitted product source** (`server`/`client`/`python-service`/`api`/`shared`, modified within `DOC_NUDGE_FRESH_MIN`, default 45 min) present while STATE.md/WAVES.md weren't updated → print the reminder. The freshness gate (mtime within the window) keeps days-old WIP and pure read-only / conversational turns silent — without it the perpetually-dirty tree would nag every turn. Still **advisory (always `exit 0`)**: a blocking Stop hook would trap conversational turns, and the chosen design was "reliable nudge," not "hard block." The genuine guarantee on narrative docs remains Claude discipline; the hook makes forgetting loud instead of silent. The five enforcement touchpoints (3 hard gates + 2 advisory edges) are documented in [cold-start-generated-indexes.md](./conventions/cold-start-generated-indexes.md). Composes with the CLAUDE.md #8 "trust generated, verify prose" rule that covers the residual prose-staleness the orient pack can't gate.

## L-014 — A multi-file codemod that introduces a helper can collide with locals of the same name (`const X = X(...)`); guard for it and let `tsc` be the oracle.

**Rule:** When a codemod replaces an inline expression with a call to a new helper (here `errorMessage(e)` replacing `e instanceof Error ? e.message : String(e)` across 125 sites), any site that assigned the original expression to a **local variable of the same name** (`const errorMessage = …`) becomes `const errorMessage = errorMessage(error)` — a use-before-declaration self-reference (TS2448/TS7022). The codemod ran clean on 113 sites and broke 12.

**Why:** The replacement is purely textual; it can't see that the assignment target shadows the imported helper. Twelve files (`database.config.ts`, `chat.model.ts`, `chatStream.service.ts`, …) named their local `errorMessage`, so the rewrite shadowed the import with a self-referential `const`.

**How to apply:** (1) Run `tsc --noEmit` immediately after any mass codemod — it pinpointed all 24 errors (2 per collision) precisely. (2) Fix collisions deterministically with a second script (revert just the colliding lines to the inline form, or rename the local), don't hand-edit 12 files. (3) Then strip imports that became unused by counting **call sites** (`errorMessage(`), not the bare identifier (which the reverted inline form still contains). The helper + codemod live in [`server/utils/errorMessage.ts`](../server/utils/errorMessage.ts); recorded in ADR [`expert-audit-remediation.md`](./decisions/expert-audit-remediation.md).

## L-015 — A "dead code / dead dependency" audit finding from a JS-import grep can be a false positive; confirm against ALL reference kinds (CSS, dynamic import, config) before deleting.

**Rule:** The expert audit flagged `react-resizable` as a dead dependency ("zero `from 'react-resizable'` imports"). It is **not** dead — it is imported as CSS (`import 'react-resizable/css/styles.css'` in [`DashboardTiles.tsx`](../client/src/pages/Dashboard/Components/DashboardTiles.tsx)) and provides the `.react-resizable-handle-*` styles that `react-grid-layout`'s resize handles depend on. Deleting it would have silently broken dashboard tile resizing. (The genuinely-dead `stratifiedSample.ts` / `dataProvenance.ts` — confirmed zero references of any kind — were correctly removed.)

**Why:** The audit (and most "find dead code" greps) match ES module `import … from` statements only. CSS side-effect imports, `await import()`, string-keyed dynamic requires, and config/build references don't match — so a "zero importers" claim is necessary-but-not-sufficient evidence of deadness.

**How to apply:** Before acting on any "delete this / it's unused" finding, grep the bare module/symbol name across **all** file types (TS, TSX, CSS, JSON, MJS), including CSS side-effect imports (`import '…/css'`), `await import`, and config files. ## L-016 — A large `tsconfig` strictness migration (`noUncheckedIndexedAccess`, 698 errors) is safely parallelizable across disjoint file-groups; and the dev shell is **zsh**, which does NOT word-split unquoted `$var`.

**Rule (migration):** Enabling a strict-mode flag tree-wide surfaces hundreds of errors at once (here `noUncheckedIndexedAccess: true` → 698 across 95 files). Don't hand-fix serially and don't leave the tree red. Instead: (1) enable the flag, capture the errors, and **measure the blast radius first** (`tsc | grep -c 'error TS'`); (2) partition the offending files into **disjoint** groups (bin-pack by error count); (3) fan out one agent per group (a Workflow) — disjoint files mean concurrent edits never conflict, no worktrees needed; (4) instruct agents to fix **behavior-preserving** only (`!` solely where presence is provable — bounds-checked loop index, regex group the pattern guarantees, length-guarded read — else the fallback the code already implied); (5) verify **centrally** afterward (`tsc` to 0 + the hot-path test suite), and **revert the flag if you can't reach green** so the tree is never left broken. See ADR [`expert-audit-remediation.md`](./decisions/expert-audit-remediation.md).

**Rule (shell):** This environment's shell is **zsh**, not bash. Unquoted parameter expansion does **not** field-split in zsh — `node --test $FILES` passes the whole string as ONE argument (cost three failed runs: node reported "Could not find '…a.test.ts b.test.ts…'"). Use a zsh **array** (`files=(a b c); cmd $files`) or force splitting with `${=FILES}`. The same gotcha bites `grep --include=*.ts` (zsh globs the unquoted `*.ts` → "no matches found"); quote it: `--include='*.ts'`.

**Why:** The migration pattern turned a multi-day serial slog into one ~7-min parallel pass (12 agents, 698 fixes) with a clean central gate; the shell gotcha silently produced empty/garbled commands that looked like "tests didn't run" rather than erroring obviously.

**How to apply:** For any future strict-flag adoption (`noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, …) use the measure→partition→fan-out→verify-centrally→revert-if-red loop. For any multi-file shell command, build a zsh array or quote globs.

## L-017 — Targeted verification silently misses **source-inspection tests** that brittly pin code shape; a legitimate refactor breaks them and only the FULL suite catches it.

**Rule:** Several suites in this repo assert against the *source text* of a file (read via `repoFile()`/`readFileSync`, not behaviour) — e.g. `wi4Wiring*.test.ts` pins the exact React import `import { useEffect, useMemo, useRef, useState } from "react"`; the telemetry/`insightRegen` server tests pin `app.use('/api', telemetryRoutes)`; `registriesManifest.test.ts` parses `routes/index.ts` for `app.use('/api…', router)` to count route modules. A perfectly correct refactor that changes the *shape* (FE-4 prepending `memo,` to the import; API-7 replacing explicit `app.use` mounts with a `mount('<sub>', router)` helper that loops `/api` + `/api/v1`) makes these tests red even though runtime behaviour is identical — AND it silently breaks the **registry generator** (`extractRoutes()` returned 0 groups → `registries.generated.md` would show no routes). Targeted per-file test runs during a batch never exercised these files, so the regressions sat undetected across several batches until a full `npm test` surfaced 7 of them at once.

**Why:** "Verified green" via a handful of targeted files is a false signal when a change has wide source-shape blast radius (a shared import convention, a route-mount pattern, anything a generator/source-test parses). The full suite is the only thing that exercises the brittle source-inspection assertions and the registry-extraction tooling together.

**How to apply:** (1) After any cross-cutting refactor that changes an **import convention, a mount/registration pattern, or anything a generator parses**, run the FULL `npm test` (accept the known end-of-run Cosmos-handle stall — TAP `ok`/`not ok` lines stream before it; grep `^not ok`), not just targeted files, before declaring the batch done. (2) When a source-inspection test breaks on a *shape* change that preserves behaviour, fix the **test/parser to tolerate the new shape** (widen the regex: `\{\s*(?:memo\s*,\s*)?useEffect…`; teach `extractRoutes()` the `mount(...)` form) rather than contorting the source back. (3) Prefer behaviour assertions over source-text assertions when writing new tests; if you must pin source, pin the *minimum* invariant (the symbol is imported / the router is mounted), not the exact surrounding punctuation.

## L-018 — "This output feels like decoration, not help" is usually TWO compounding causes (weak generation **and** buried rendering); fix both ends or it still feels decorative.

**Rule:** The dashboard exec-summary complaint ("key insights / recommendations don't give meaningful insights … decoration rather than help") had two independent root causes that compounded: *generation* produced ungrounded, analytical-only recommendation text (`recommendations[]` was contractually "analytical next-steps", `soWhat` could be vacuous, findings shipped without a magnitude), AND *rendering/IA* buried the prescriptive content — the always-visible [`DashboardSummaryBand`](../client/src/pages/Dashboard/Components/DashboardSummaryBand.tsx) led with bare finding headlines while "so what" (implications) and "what to do" (recommendations) were absent from the band and collapsed two clicks deep in the drawer. Fixing only the prompt, or only the layout, would have left it still feeling decorative.

**Why:** A "this isn't useful" complaint about an LLM surface conflates two separable failures — *is the text decision-grade?* (generation: grounded, specific, quantified) and *is the useful part actually surfaced, in the right order?* (rendering/IA). They have different fixes (prompt+schema vs component+selector) and different owners; diagnosing only one is half a fix.

**How to apply:** (1) For any "output X feels like decoration/filler" report, explicitly split GENERATION from RENDERING and check both before deciding scope — map the pipeline end to end (a `Workflow` of parallel readers is good for this). (2) When the genuine product fork is *what kind of content the user wants* (here: analytical next-steps vs bold business decisions), `AskUserQuestion` once up front — it changes the prompt contract AND the schema AND the rendering, so guessing wrong is expensive. (3) Process gotcha: isolating a `tsc --noEmit` error by `git stash`-ing your changes gives a **false negative** when an incremental `.tsbuildinfo` cache exists (tsc skips unchanged files and reuses cached results) — `find . -name '*.tsbuildinfo' -delete` first for a deterministic typecheck.

## L-019 — A "single authority" only fixes a bug if its INPUTS are built uniformly across every path; centralizing the decision while leaving the input contract path-dependent silently re-opens the same bug.

**Rule:** The temporal-grain authority ([`resolveTrendGrain`](../server/lib/temporalGrainAuthority.ts), ADR [centralized-temporal-grain](decisions/centralized-temporal-grain.md)) was introduced precisely to stop "single month of daily data → one Month dot." It worked on the in-memory upload path but the bug persisted on the columnar / large-file / metadata-reload path. Root cause: the authority enumerates its candidate time-axes ONLY from `summary.columns` (every caller passes `summary.columns.map(c => c.name)`), and the two ingest builders disagreed on what goes there — [`createDataSummary`](../server/lib/fileParser.ts) merges the derived facet columns (`Day · Date`, …) INTO `summary.columns`, while [`metadataService.convertToDataSummary`](../server/lib/metadataService.ts) listed them only in `summary.temporalFacetColumns`. So on the columnar path the authority never saw a daily candidate → `source:"none"` → raw-date fallback → `chartGenerator` month-buckets it → one dot. Fix: merge facets into `columns` on the columnar path too (unconditionally — they're virtual, inlined via `facetColumnInlineDuckDbExpr`), making the two paths byte-equivalent for enumeration; plus a `bucketCount` hardening so an all-null materialized facet with a real span still resolves via span.

**Why:** Centralization removes duplicated *logic*, but a pure decider is only as good as the *inputs* it's handed. If two ingest paths (or two reload paths) construct that input differently, the "one authority" is quietly starved on one of them — and because the unit tests exercised the authority in isolation (it correctly returns `Day` given the right `availableColumns`), the gap hid behind green tests. The integration seam — does each real ingest path actually put the candidate into `summary.columns`? — was untested.

**How to apply:** (1) When you centralize a decision, also pin the *input contract* and assert it on EVERY producer path, not just the happy one — here, an integration test ([`convertToDataSummaryFacetColumns.test.ts`](../server/tests/convertToDataSummaryFacetColumns.test.ts)) that runs the actual `convertToDataSummary` and feeds its `summary.columns` to `resolveTrendGrain`, rather than a unit test that hand-builds `availableColumns`. (2) For any "works sometimes / inconsistent" bug, suspect path-divergence in the shared input first: diff what the in-memory vs columnar (vs reload) ingest produces for the same dataset. (3) An authority returning a safe-looking fallback (`none` → coarsest grain → "one honest point") can MASK an upstream starvation — when a fallback fires, log WHY (no candidate vs genuine single-period), so "starved of candidates" is distinguishable from "genuinely one period."

## L-020 — When asked to "remove a max limit," the visible notification is ONE of many enforcement points; enumerate ALL of them (and classify user-vs-machine) before editing, or you fix the symptom and leave the disease.

**Rule:** User pointed at a bottom-right toast ("your added context was long… a later step used a shortened version") and said *don't put this max limit, and don't show this*. The obvious grep (`applyCap` + the toast string) found **3** sites — all that emit the toast. But the user's "Give Additional Context" (`permanentContext`, + derived `verbatimNotes`/`interpretedConstraints`) was actually clipped at **eleven** sites: the planner/reflector (`context.ts` `maxUserChars`), the mode classifier, the chart-insight generator, the deck planner, and the upload-time profile all did a **raw `.slice()` with NO toast**. Fixing only the toast-emitting caps would have killed the notification while the user's context kept getting silently shortened — strictly worse (the loss is now invisible). A fan-out finder pass (4 parallel locators by value-name / by `.slice` / by `*_CAP` const / by data-flow, + a completeness critic that classified each hit user-vs-machine-vs-authored) surfaced the silent eight.

**Why:** A "limit" in a mature codebase is rarely one constant — it's re-implemented per call site (each prompt builder re-caps the same field its own way), and only the sites wired to a telemetry sink are user-visible. The visible symptom (toast) correlates with the *reported* caps, not the *binding* ones. Grepping for the symptom finds the reported subset and feels complete.

**How to apply:** (1) For "remove this limit / cap / truncation," treat the user-visible signal as a lead, not the boundary — enumerate every enforcement of the same logical value: `applyCap`, `.slice(0, N)`, `*_CAP`/`*_MAX` consts, `.substring`, Zod `.max()`, and write-time derivations, across BOTH services. (2) Classify each hit by ownership before editing — **user-provided** (remove the cap), **machine/authored** (keep it; it's model-window safety), **adjacent-but-different field** (e.g. the user's *question* vs their *added context* — leave it unless asked). A blanket "remove all caps" would blow the context window on RAG; a narrow "remove the toast" leaves silent truncation. (3) Confirm the full-fidelity carrier: here the raw `permanentContext` is stored/indexed uncapped, so honoring it in every *read* path satisfies "in full" without having to also re-plumb the derived-distillation field. (4) Removing a notification means deleting its emit too, not just its UI — a consumerless SSE row is dead code.

## L-021 — The answer envelope is declared in FIVE separate Zod `z.object`s, not one — adding a field to the persisted schema silently drops it on the dashboard mirror (a distinct object that strips unknown keys).

**Rule:** Adding `likelyDrivers` to the answer envelope meant editing FIVE declarations, not one: `narratorOutputSchema` ([narratorAgent.ts](../server/lib/agents/runtime/narratorAgent.ts)), `finalAnswerEnvelopeSchema` ([synthesis.ts](../server/lib/agents/runtime/agentLoop/synthesis.ts)), `messageAnswerEnvelopeSchema` AND `dashboardAnswerEnvelopeSchema` (TWO separate `z.object`s in [charts.ts](../server/shared/schema/charts.ts)), and — via `export *` — the client mirror. The dashboard one is a *separate* schema (not a reuse of the message one), and Zod **strips unknown keys** by default, so a field added only to the message schema would validate fine yet vanish at the chat→dashboard boundary — "scope = everywhere" would silently fail on every dashboard with green tests.

**Why:** Mature persistence layers grow parallel schemas for adjacent surfaces (message vs dashboard mirror vs synth fallback vs producer) that *look* identical but are independent objects. A new field looks "added" once you edit the obvious one; the others quietly drop it. Unit tests that parse one schema never catch the gap.

**How to apply:** (1) When adding an envelope/persisted field, grep for ALL `*Schema = z.object` (and `export *` re-exports) that model the same logical shape; edit every one. (2) Define the field ONCE as a shared exported constant and import it into each declaration — drift becomes impossible. (3) Pin it with a **forward-parity** test that pushes the SAME object through every schema (see [`likelyDriversSchemaRoundTrip.test.ts`](../server/tests/likelyDriversSchemaRoundTrip.test.ts)) plus a back-compat omit case — the L-019 "assert the input/output contract on every path" rule applied to schemas.

## L-022 — A hedged "why" section is a license to speculate — ship the deterministic gate BEFORE the prompt opens the permission, and keep the new lexicon a CONTENT vocabulary, not an intent one.

**Rule:** Letting the model state plausible causes (incl. world knowledge like "women-and-children-first") is exactly the FMCG-hallucination risk the product had guarded against. The order mattered: the deterministic [`verifierCausalCheck`](../server/lib/agents/runtime/verifierCausalCheck.ts) rail (hedge present + no statistic-number + `basis="data"` names a real column) shipped in W-SR2 **before** the `ANSWER_ENVELOPE_CONTRACT` edit (W-CP1) told the model it *may* speculate — so there was never a window where speculation was permitted with no gate. The confidence↔basis coupling is clamped **structurally at schema parse** (normalize DOWN, never reject), and a `sanitizeLikelyDrivers` belt runs at emit so a bad driver can't persist even if the model and the LLM verifier both miss. Crucially, the `CAUSAL_HEDGE_TERMS` lexicon lives in [`sharedPrompts.ts`](../server/lib/agents/runtime/sharedPrompts.ts) as a *content* vocabulary that polices generated prose — NOT in `queryIntentAuthority`, because it never reclassifies a question, so invariant #12's machine check stays green.

**Why:** "Loosen a guard for a good feature" is where regressions enter. A prompt-only loosening (telling the model it may speculate) without a deterministic gate is a net weakening of the only causal guard. And a hedge-detector regex that *looks* like an intent classifier would draw reviewer fire if it lived outside the intent authority — even though it's policing wording, not routing.

**How to apply:** (1) When opening a previously-banned capability, land the enforcement rail (deterministic + structural) in an EARLIER, independently-shippable wave than the prompt/contract change that grants the permission. (2) Prefer structural normalization (a parse-time `transform` that clamps) over rejection — it never nukes a whole answer for one over-claim. (3) Quarantine the risky output in its own labeled, hedged field with a standing disclaimer, never interleaved with measured facts; verify unverified consumer surfaces (dashboards) read only the already-verified persisted value. (4) A regex that polices PROSE is a content vocabulary (put it with the prompts); a regex that ROUTES a question is an intent vocabulary (it belongs only in the intent authority — invariant #12).

## L-023 — "Persist a new default everywhere without breaking anything" = bake the default at BUILD time + have the client TRUST the stored field; never recompute the default on every render.

**Rule:** The chart-sort feature added an "auto axis-order" default and had to ship "without breaking" existing saved dashboards/messages. The safe shape: the server resolves the default ONCE when a chart is built ([`processChartData`](../server/lib/chartGenerator.ts)) and **bakes** it into `spec.sort`, ordering the data to match. The client ([`useChartSort`](../client/src/lib/charts/useChartSort.ts)) seeds from `spec.sort` and re-orders only when it is present or the user acts — it NEVER recomputes the auto-default on render. So pre-existing specs (no `sort` field, data already ordered) render exactly as saved; only freshly built charts get the new behavior. If the client had auto-detected the default on every render, it would have retroactively re-ordered every historical chart.

**Why:** A "smart default" that runs at render time silently rewrites every existing artifact the first time it's viewed — invisible in tests that only build fresh fixtures, loud in production. Baking at creation makes the choice an explicit, persisted, inspectable value (also what makes the toolbar open on the right option and what gets persisted on "add to dashboard"). Two concrete traps this surfaced: (1) baking MUTATES the spec object, which broke a sibling test that **shared one `base` fixture** across cases (an earlier call's baked sort masked a later case's `sortDirection`) — give each case a fresh spec. (2) Reassigning an outer `let outcome` from inside a `mutateChatDocument` callback trips TS closure-narrowing (`'"ok"' and '"no-message"' have no overlap`) — resolve the branch from the pre-fetched snapshot instead.

**How to apply:** (1) For any new default-bearing field, resolve+persist it at the WRITE/BUILD boundary; make the read paths trust the stored value and fall back to a *cosmetic* default, not a recomputed one. (2) Put the resolve/order logic in ONE pure shared module imported by both tiers (here [`server/shared/chartSort.ts`](../server/shared/chartSort.ts), re-exported to the client) and fold any pre-existing duplicate (the old `temporalAxisSort.ts` became a re-export) — don't add a third copy. (3) Before naming a new schema field/const, grep the schema: a parallel grammar may already own the name (`chartSortSpecSchema` was taken by the ChartSpecV2 encoding-sort, so the v1 field became `barSortSpecSchema`). (4) When a "value-sort + cap" feature exists, preserve its bottom-N/top-N semantics: VALUE sort = order-then-slice; only CATEGORY sort needs select-top-N-by-value-then-reorder.

## L-024 — Extending a "single authority" with a NEW grain only stays global if you ride its existing dynamic seam and gate on a uniformly-built input — and the upstream parser may be silently dropping the data the feature needs.

**Rule:** Adding sub-day (hour / hour-of-day / minute) bucketing to the temporal stack worked because it rode seams that already existed: the `TemporalFacetGrain` *type* is separate from the `GRAINS` *materialization* array, so adding to the type + the inline expr ([`facetColumnInlineDuckDbExpr`](../server/lib/temporalFacetColumns.ts)) gives on-the-fly buckets with NO new `Hour · X` columns written at ingest (the user's "dynamic, not pre-determined buckets"); and the inline-expr / `dateAggregationPeriod` path the executor already uses for raw-date groupBy carried hour/minute for free. The decision stayed in the one authority ([`resolveTrendGrain`](../server/lib/temporalGrainAuthority.ts)), so every route (chat · quick-answer · dashboard · pivot) inherited it. The gate — a per-column `dateRange.temporalResolution: 'day' | 'sub_day'` — had to be set on EVERY ingest path (`createDataSummary` AND `deriveDateRangeFromRows`, the latter covering the columnar/Snowflake/reload path), or a stripped-metadata frame would mis-gate ([[L-019]] in action).

**Why:** The first three exploration agents reported "hour grain is structurally impossible" and "sub_day is an explicit gap" — true at the grain layer, but they MISSED that the time-of-day *predicate* + *duration* machinery already existed ([`durationColumns.ts`](../server/lib/durationColumns.ts) DUR1, `formatTimeOfDayBlock` CMP1, `datetime_concat`). So the real gap was narrower than "build intraday from scratch" — it was just *bucketing*. Separately, the load-bearing bug was upstream and invisible: [`parseFlexibleDate`](../server/lib/dateUtils.ts) had only a `T`-separated ISO branch, so `"2026-06-22 14:30"` (space-separated, the common Excel/CSV shape) returned **null** — the time was never recognized, and the column's `dateRange` was computed off the rows that DID parse, so intraday detail was lost before any grain logic ran. A feature that buckets by hour is dead on arrival if ingest quietly throws the hours away.

**How to apply:** (1) Before extending an authority, separate its *vocabulary* type from its *materialization* list — add to the type + the on-the-fly compute path, never to the pre-materialization list, when the new values must be dynamic/unbounded. (2) A new gating input must be built identically on every ingest path; add a shared helper (here `accumulateIntraday`/`intradayResolution` in dateUtils) and call it from each, then write a uniformity test asserting all paths agree on one fixture. (3) Trace the data from the PARSER up, not just the decision layer down — confirm the raw value survives ingest (`parseFlexibleDate`, canonical storage) before trusting the grain layer can use it; store intraday timestamps as naive wall-clock, not `toISOString()` (UTC), so `EXTRACT(hour…)` returns the typed hour on non-UTC hosts. (4) When agents report "impossible / explicit gap," verify the gap's SCOPE against code — adjacent capabilities (predicates, durations) may already exist and shrink the build.

## L-032 — A "single authority" (depthBudget) only suppresses output-padding if EVERY amplifier reads it; the loudest amplifier — the feature sweep — was gated on the wrong predicate, so a pointed trend question still fanned out 16 charts.

**Rule:** Invariant #12 says `queryIntentAuthority` decides depthBudget ONCE and every output-shaping gate reads it. The visual-planner gate honoured it (skip extras on `minimal`), but the **dashboard feature sweep** — the engine that fans out one "outcome by <dim>" chart per categorical column, and the single biggest source of charts — had its OWN gate in [`agentLoop.service.ts`](../server/lib/agents/runtime/agentLoop.service.ts) that read `!minimalDepth` instead of the authority's documented contract ("`standard` → full envelope but NO breadth augmentation unless explicitly asked"). With `EXHAUSTIVE_BREADTH_ENABLED=true` + a 40-chart ceiling, a `standard`-classified question ("what is the daily trend in total visited OL's?") swept every dimension → ~16 charts of a *different* metric. The fix: extract the decision into a PURE, unit-tested predicate [`shouldRunFeatureSweep`](../server/lib/agents/runtime/dashboardFeatureSweep.ts) that fires only on explicit-dashboard OR `full` depth OR an explicit breadth request — the breadth opt-in being a NEW [`signals.breadth`](../server/lib/agents/runtime/queryIntentAuthority.ts) on the authority (so "all columns / every level / full fledged" still sweeps). For the *positive* half, a trend ask's ONLY acceptable extras are the SAME measure re-aggregated at coarser grains ([`buildTrendTemporalCompanions`](../server/lib/agents/runtime/visualPlanner.ts), span-gated: >2 weeks → weekly, >2 months → monthly), which REPLACES the LLM proposer for trend turns so no cross-dimension drift sneaks back in.

**Why:** This is [[L-019]]/[[L-020]] again, two waves later: centralizing the *decision* doesn't fix the bug if a single enforcement point reads a *different* predicate than the authority's contract. "Are we editing the wrong files?" — yes, partially: the prior fixes touched the visual planner + dashboard offer + spawned follow-ups (all correct) but never re-pointed the feature-sweep gate, the one that actually produced the plethora. The user shouldn't scroll 16 charts for a one-line question. Separately, the per-chart "Key Insight" WHY was *speculating* ("the zero days may be non-working") about dips that are deterministically **Sundays** — the insight prompt only ever saw x/y values, never the weekday. New pure helper [`deriveWeekdayPattern`](../server/lib/insightGenerator/weekdayPattern.ts) detects a recurring off-day FROM THE DATA (no hardcoded "Sunday", no calendar config) and feeds it as ground truth to BOTH the chart-insight prompt and the narrator ([[L-022]]: ship the deterministic gate before the prompt opens the WHY permission). A latent crash also surfaced but was deliberately NOT fixed in this scope: [`narratorAgent.ts`](../server/lib/agents/runtime/narratorAgent.ts) does `epistemicNotes.length` then `.map()` — a bare string has truthy `.length` but no `.map`, so a malformed brief crashes the narrator. It is unreachable in production (the `AnalysisBrief` schema enforces `string[]`), and coercing it flips which agent (narrator vs synthesizer-fallback) answers two live-LLM e2e tests — i.e. an out-of-scope fix that perturbs unrelated CI signal. Flagged for a dedicated change.

**How to apply:** (1) When a "single authority" exists, find EVERY consumer and assert each reads the authority's *contract*, not a near-synonym (`!minimal` ≠ "standard suppresses breadth"); extract each gate into a pure predicate with a focused unit test (mirrors `classifyDashboardIntent` / `shouldRunFeatureSweep`) so the contract is checkable in isolation. (2) Suppressing padding is only half — define the POSITIVE minimum a pointed ask deserves (here: same-measure coarser-grain companions, span-gated) and make it deterministic so it can't drift. (3) Ground a hedged "why" with a deterministic fact computed BEFORE the prompt (weekday off-day), derived from data not config, returning null when the pattern is absent so unrelated charts aren't polluted. (4) Any `x.length ? x.map(...)` is a latent crash when `x` can be a string — coerce shape at the boundary; a malformed sub-field must never take down the whole narrator.
