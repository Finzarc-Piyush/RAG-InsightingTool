  **Why this design.**
    - **Why a separate scanner module instead of widening `semanticModelReferences.ts`.** The chat-doc walker takes `ReadonlyArray<unknown>` (charts) and returns `{ chartCount, totalOccurrences }`. The dashboard walker takes `ReadonlyArray<unknown>` (dashboards) and returns a dashboard-level + tile-level aggregate — different inputs, different output shape. Folding both into one function would mean a discriminated return type and a kind parameter; two pure modules read more directly. The chat-doc walker is also potentially consumed by other call sites (a future blob-stored-chartReferences mode would re-use it); the dashboard walker is unique to the references-endpoint and benefits from staying isolated.
    - **Why prefer `sheets[].charts[]` and fall back to `dashboard.charts[]` only when sheets is absent/empty.** Modern dashboards (post the DR series) keep `dashboard.charts[]` in sync as the union of all sheet charts — see [`dashboard.model.ts`](../server/models/dashboard.model.ts) `createDashboardFromSpec`, `patchDashboard`, `addChartToDashboard`. Walking both arrays would double-count every tile. Pre-sheets dashboards (legacy) only populated `dashboard.charts[]` and had `sheets` absent or empty; falling back preserves coverage for those rows. A dedicated test (`a dashboard whose only matching tile is on the legacy charts[] fallback contributes once`) pins the sheets-prefer-over-legacy invariant explicitly.
    - **Why "tile" rather than "chart" in the dashboard-side counts.** A single underlying ChartSpec object can be referenced from multiple sheets (the chat-flow agent re-uses chart specs across the Summary and Evidence sheets). For the destructive-op warning the admin needs to know "how many positions on dashboards will lose data," not "how many distinct chart objects exist" — each position is a tile, and removing the metric breaks each position independently. "Tile" also matches the dashboard surface's UX terminology (a sheet is a grid of tiles).
    - **Why a new `_dashboardListerForUser` injectable instead of overloading `_detailFetcher`.** The dashboards container is a different Cosmos resource (partitioned by username, not sessionId). The W61-references-endpoint and W61-audit-history-api re-use `_detailFetcher` because they read the same chat-doc; this wave reads a different document type from a different container, so the injectable boundary follows the container boundary. Introducing a dedicated lister codifies the **"one injectable per Cosmos container"** convention: chat docs use `_detailFetcher`; dashboards use `_dashboardListerForUser`; future shared-dashboards / past-analyses / shared-analyses reference-walk waves get their own injectables rather than reaching into existing ones. This keeps the test-fake surface proportional to actual read paths and prevents one injectable from accumulating multi-container fetch logic.
    - **Why `ReadonlyArray<unknown>` as the lister return type, not `Dashboard[]`.** The scanner's defensive guards (`typeof === "object"`, `Array.isArray`) apply uniformly — a malformed Cosmos row that doesn't match the runtime shape is silently skipped by the scanner rather than parsed eagerly here. The production `getUserDashboards` returns `Dashboard[]` (via an unsafe runtime cast inside the function), but the typed return at the lister-injectable boundary would force eager parsing that could throw on a malformed row mid-iteration. `ReadonlyArray<unknown>` defers the discrimination to the scanner where it's already handled defensively.
    - **Why the empty-username short-circuit.** A chat doc with no `username` (system-test sessions, or pre-username-field-introduction docs) used to slip through and would call `getUserDashboards("")` which would query the empty-string partition (returning nothing, but still costing a Cosmos round-trip and a `console.error` from the eventual catch). The explicit short-circuit returns `[]` immediately, saves the round-trip, and makes the no-username case visible in the test surface (a dedicated test asserts `listerCalled === false` for the empty-username branch).
    - **Why we widen the existing envelope rather than introduce a new endpoint.** Single round-trip per delete confirmation is load-bearing UX — the W61-delete-client modal opens, fires one fetch, renders the warning. A separate `/admin/semantic-models/:sessionId/dashboard-references?entry=<name>` endpoint would force two round-trips per delete and would also force the client to combine two responses into a single warning. The widened envelope keeps the API surface flat and the client logic linear. The trade-off — every references query now also hits the dashboards container — is acceptable because (a) most sessions have <10 dashboards owned by the user, (b) the dashboards container is partitioned by username so the query is single-partition, and (c) the W61-delete-client modal only opens on an explicit user action, not on every page render.
    - **Why two dashboard fields (`dashboardCount` + `dashboardTileCount`), not just one.** The admin needs both to size the blast radius. `dashboardCount` answers "how many distinct dashboards will be affected" (1 dashboard with 10 tiles is one rebuild; 10 dashboards with 1 tile each is ten owner conversations). `dashboardTileCount` answers "how many positions need to be fixed" (the absolute work surface). Surfacing only one would hide the bipolar signal: a single dashboard with many tiles vs many dashboards with few tiles look identical with one count. The combined-clause copy `"M tiles across K dashboards"` makes both visible in one sentence.
    - **Why we skip `dashboardTotalOccurrences` (the analog to in-chat `totalOccurrences`).** The in-chat subhead surfaces field-position-occurrence-density inside a chart (a metric bound to both x AND tooltip on a single chart contributes 2 occurrences). For dashboards, the actionable unit at the admin's level is the tile (the dashboard position) not the field-position-within-a-tile. If the admin needs that detail, they'd open the dashboard view; surfacing it in the confirmation copy would be more noise than signal. The scanner therefore doesn't compute `dashboardTotalOccurrences` and the envelope doesn't expose it; YAGNI applies.
    - **Why the verb agreement follows the LAST enumerated count, not the first or a fixed count.** `"1 chart and 2 tiles ... reference it"` reads naturally (the verb agrees with `2 tiles`, the noun nearest to the verb). `"1 chart and 2 tiles ... references it"` reads as ungrammatical (third-person-singular `references` agreeing with the distant `1 chart` is jarring). Mid-Atlantic-English convention is the verb agrees with the closest subject. The same rule produces the right output for the dashboard-only case (`"5 tiles across 2 dashboards ... reference it"` — last count is `2 dashboards`, plural) and the singular cases (`"1 chart and 1 tile across 1 dashboard ... references it"` — last count is `1 dashboard`, singular).
    - **Why we widen `buildDeleteReferencesWarning` in place rather than introduce a new helper.** The modal calls one helper per render, and the helper's return shape (`{ headline, subhead } | null`) is stable — the consumer is unchanged. A new helper would mean two helpers with overlapping concerns and the modal would need a branching layer to pick between them. Widening preserves the single-source-of-truth invariant for the copy.
    - **Why the chart-side subhead is preserved (not unified into a combined subhead).** The chart-side subhead surfaces when `totalOccurrences > chartCount` — exactly the case where one chart uses the entry in multiple field positions (heavy-usage signal). The dashboard side doesn't have an equivalent signal because we don't track field-positions-within-tiles. Combining them would require us to either (a) compute the tile-side analog (rejected for YAGNI), or (b) drop the chart-side signal (regression). The cleanest design keeps the chart-side subhead and accepts that dashboards have a flatter signal model.
    - **Why we update the existing W61-references-endpoint test file in place rather than create a new wave-specific endpoint test file.** The existing file's 4 200-path tests assert the envelope shape via `assert.deepEqual`; the envelope widened, so those assertions had to change either way. Adding the two new fields to the existing tests + a dedicated W61-references-dashboards test file for the dashboard-specific scenarios is the cleanest split: the existing file pins the chart-side semantics (still load-bearing); the new file pins the dashboard-side semantics. The alternative — leaving the existing tests asserting partial bodies — would have been an anti-pattern (tests that don't assert the full envelope are weaker).

  **Conventions added.**
    - **One injectable per Cosmos container.** [`docs/conventions/injectable-per-cosmos-container.md`](../docs/conventions/injectable-per-cosmos-container.md) (new). Each Cosmos container that the controller reads from gets its own `__set*ForTesting` injectable. Chat docs use `_detailFetcher` (existing); dashboards use `_dashboardListerForUser` (new this wave); future shared-dashboards / past-analyses / shared-analyses reads get their own injectables rather than overloading existing ones. The convention is now load-bearing because the W61-references-endpoint pattern (re-use the injectable when reading the same container) was an explicit choice that needed a sibling for the cross-container case. Without this convention a future contributor would either fold dashboard reads into `_detailFetcher` (mixing two responsibilities) or skip the injectable entirely (untestable handler).

  **Why this design.**
    - **Why `<Dialog>` not `<AlertDialog>` for the modal.** AlertDialog is for "confirm this destructive action" UX (single primary message + Cancel/Confirm buttons). Dialog is Radix's canonical form-modal — supports multiple inputs and a primary submit button. The W61-delete-client used AlertDialog because the body was a single warning paragraph; this wave's body has 4-7 inputs depending on the kind. Using the wrong primitive would lose semantic correctness (screen readers announce the two differently) and would also lose the multi-field form-validation patterns Dialog supports.
    - **Why `NameAlreadyExistsError` instance not error-message string-matching.** HTTP 409 carries structured JSON `{ error: "name_already_exists", sessionId, kind, name }`. Parsing the body in the helper and re-throwing a typed Error subclass lets the host distinguish via `instanceof` rather than fragile regex on `err.message`. The pattern keeps the host's error-handling branches type-narrowed — `instanceof NameAlreadyExistsError` proves `err.kind` and `err.entryName` are defined, no `?? "unknown"` fallbacks needed. Future i18n changes to the error message don't break the branch logic.
    - **Why the modal owns the draft (parent owns the mutation).** Mirrors W61-delete-client. Draft state is mount-scoped — it lives for the duration of the modal and is GC'd on close; parent ownership would force a `clearDraft` step on every close + a stale-draft concern when admin opens for one kind then re-opens for another quickly. The mutation needs to update parent `data` (the model) on success, which the modal can't do on its own; parent-owned mutation + signalling close via `onOpenChange(false)` is the proven shape from delete.
    - **Why `addOpen: AdminSemanticModelEntryKind | null` not `{ open: boolean, kind: AdminSemanticModelEntryKind | null }`.** The non-null IS the open signal. Two-prop shape would invite the invalid state `open=true && kind=null` that the modal would have to defend against. Single-discriminant shape is the same pattern as W61-delete-client's `pendingDelete: { kind, name } | null`.
    - **Why validate at submit (not on every keystroke).** Keystroke validation would show a red border on the name field the moment the admin types `M` (PascalCase rejected) before they've finished typing `Monthly_Revenue` (also rejected, but the admin's mid-keystroke). Submit-time validation matches the W61-edit-text pattern (validate-on-blur is roughly equivalent to validate-on-submit-in-a-modal). The exception: name-collision after submit, which is rendered immediately because the admin has finished typing and seen the failure.
    - **Why run every validator at submit (not short-circuit on first failure).** If the admin submits a form with three invalid fields, they should see all three errors at once — fixing one and re-submitting only to discover the next is a worse UX than seeing all three the first time. Hence the `results = [validateField(...), ...]; results.every(Boolean)` pattern rather than chained `&&`.
    - **Why `+ Add` buttons per-card, not a single page-level dropdown.** Locality. An admin reading the Dimensions table thinks "I need to add a dimension here". A page-top dropdown forces a kind-pick step before the form opens; per-card buttons skip that step. The button label echoes the card heading (`Metrics` card → `+ Add metric` button) for visual scan-ability.
    - **Why `<Button variant="outline" size="sm">` for the Add buttons.** Add is non-destructive but also non-primary at the page level (the page's primary surface is the existing entries — the admin is reviewing, not authoring). Outline reads as "secondary action" — the same visual weight as the SourceFilterChips it sits next to. Default-variant would steal visual focus from the table content; ghost-variant would under-signal the action's significance.
    - **Why we mirror `validateName` server-side regex exactly.** Saves a round-trip — typing a PascalCase identifier surfaces inline immediately rather than after a 400 round-trip. The server's `safeParse` is authoritative; drift just degrades UX (one extra round-trip) without correctness damage.
    - **Why hierarchy levels are a textarea (one per line) not a tag-input or chip-list.** The tag-input libraries (react-select multi, downshift, etc.) are heavier and require additional dependencies; the textarea-with-parser shape is good enough for this density (most hierarchies have 2-4 levels). A future W61-hierarchy-edit wave could swap in a richer reorderable UI with drag-and-drop; defer until use case demands.
    - **Why the kind-change handler in DimensionFields resets `temporalGrain` to `__auto__`.** Switching `kind` from `"temporal"` to e.g. `"categorical"` would leave a stale grain value in the draft. On switch BACK to `"temporal"`, the prior grain would re-surface — confusing because the admin's intent during the categorical detour was unrelated. Resetting to `__auto__` on each kind change keeps the form state aligned with the visible UI.
    - **Why we don't pre-fill the `references[]` field on metric add.** The schema's `.default([])` populates an empty array if the field is absent, which is the right default for a brand-new metric (the admin doesn't yet know which columns the expression touches; the W58 compiler's orphan-reference lint surfaces the gap on first execute). A future W61-add-references-field could add a tag-input for the columns; for now empty is fine.
    - **Why parent `addError` + `addNameCollision` are two separate slots.** The collision is field-scoped (rendered under the name field by `NameField`); the generic error is form-scoped (rendered at the bottom of the dialog body). Combining them would force the modal to either render the typed-409 message twice (once under name, once at the bottom) or surface generic errors in an incorrect location.

  **Why this design.**
    - **Why a dedicated POST rather than re-using W61-save's PATCH.** The PATCH path replaces the wholesale model — this forces the client to (a) round-trip the entire model body (potentially ~5–50 KB depending on session) for what's logically a single-entry append, and (b) re-implement name-uniqueness validation locally before sending (the server's `safeParse` would accept duplicate names because the schema doesn't enforce within-collection uniqueness). The typed POST takes just the new entry as the body, the server validates uniqueness inside the lock against the authoritative state (not a stale client view), and the response carries the full new model so the client mutation success handler is identical to save / revert / delete. The four-endpoint family now reads as a coherent set: save = wholesale replace, add = append one, delete = remove one, revert = restore prior.
    - **Why 409 Conflict for `name_already_exists`, not 404.** HTTP 409 semantically means "the request conflicts with the current state of the resource" — exactly the case here (we're trying to create an entry that already exists by name within the same kind). The W61-revert / W61-delete endpoints use 404 for `*_not_found` because the resource is absent; for `name_already_exists` the conflict semantic is collision, not absence. The client can distinguish 409 from 4xx generally and render a "this name is already taken" inline message rather than the generic "something went wrong" copy. The 409 envelope includes `{ sessionId, kind, name }` echo so the client doesn't need to track which request triggered the collision.
    - **Why cross-kind name collisions are explicitly allowed.** A metric named "revenue" and a dimension named "revenue" are different resources — the planner addresses them via `{ kind, name }` not `{ name }` alone, and the SemanticModel schema doesn't forbid the collision (each kind has its own collection in the model). Requiring `:kind` in the URL gives the uniqueness check the right scope: same-kind collisions are rejected with 409, cross-kind collisions are allowed. A dedicated test pins this behaviour (`metric "revenue"` exists; `POST /entries/dimension` with `body.name === "revenue"` → 200, both coexist in the model).
    - **Why we skip the source-bumper on add.** Same precedent as W61-delete-server: survivors are byte-identical to their prior selves (we're appending, not editing existing entries), so the bumper's content-hash diff would match them as unchanged and preserve their prior `source`. Running it would be a no-op for the existing entries. For the new entry, the bumper's lookup against `priorByName` returns undefined (the new name doesn't yet exist in the prior model — we just verified that with the uniqueness check) so the bumper's "new entry" branch returns the entry unchanged with its client-sent `source` preserved. The schema's `.default("user")` on each source field populates new entries by default; if the client sends `"domain"` (e.g., importing from a domain pack) it's preserved verbatim. Skipping the bumper call entirely is clearer about intent — the documentation comment explains the no-op equivalence so a future reader doesn't waste time wondering why W61-add doesn't bump while W61-save does.
    - **Why audit-write BEFORE the append (not after).** Same precedent as W61-save / W61-audit-revert / W61-delete-server. The buffer's role is to remember what was just changed; if the audit-write happened after the mutation, a crash between the two would lose the snapshot. Inside `withSessionWriteLock` the pair is atomic at the per-session level (single-instance correctness per invariant #9). A future "undo this add via revert" walks the audit log, finds the pre-add snapshot, and restores it — wiping the just-added entry. The audit-write-before-mutation pattern is the same shape across all four W61 endpoints; the consistency makes the four-endpoint family interchangeable from the audit-trail perspective.
    - **Why bump `version` on add (we're not changing existing entries).** W64's compiled-query cache keys on `semanticModel.version`. Adding a new metric — even one with a snake_case name that no existing chart references — could affect compilation of future queries: the planner might suggest the new metric in subsequent turns, the W58 compiler might emit lint warnings for orphan references against the new entry's `references[]` columns, etc. Bumping makes the cache miss explicit so the next planner round consults the fresh model. The cost of an unnecessary cache miss (one extra compilation) is much smaller than the cost of stale compilation against an out-of-date model.
    - **Why we reuse the existing `_detailFetcher` / `_updater` injectables.** Same Cosmos doc read+write path as save / revert / delete — fetch the chat document by sessionId, mutate `semanticModel` + `semanticModelAuditLog` + `lastUpdatedAt`, write the whole doc back. A new pair of add-specific injectables would just be the same functions under different names and would double the test-fake surface for no behavioural benefit.
    - **Why the kind-appropriate body schema is picked via ternary, not a `Record<kind, schema>` lookup.** A `Record<"metric"|"dimension"|"hierarchy", z.ZodSchema>` would type-check fine, but the three zod schemas have different inferred entry types (`SemanticMetric` vs `SemanticDimension` vs `SemanticHierarchy`) — the ternary preserves the per-kind type narrowing inside the `if (parsedKind.data === "metric")` blocks where we cast `parsedBody.data` to the concrete kind type. A `Record` lookup would resolve to a union type that the `.some((m) => m.name === entry.name)` line would have to type-narrow again. The ternary is uglier but reads more directly as "pick the right schema for this kind".

  **Why this design.**
    - **Why a controlled `<AlertDialog>` rather than `window.confirm`.** The downstream-references count is async — the modal needs to render `"Checking for downstream chart references…"` while the round-trip is in flight. `window.confirm` is synchronous and can't show intermediate states; the W61-audit-revert path uses `window.confirm` because its only context is the entry already in hand, but the delete path is references-count-first. The async dependency dictates the UI primitive.
    - **Why the modal owns the references fetch, not the parent.** References state is mount-scoped — it lives for the duration of the modal and is garbage-collected on close. Keeping the fetch / loading / error state inside the component lets React's effect cleanup handle the teardown automatically. Putting the references in the parent would force a `clearReferences` step on every close + a stale-fetch guard when the admin re-opens for a different entry quickly. The audit-history-tab's references-fetch-in-parent shape is appropriate THERE because the parent re-fetches on every `data.lastUpdatedAt` bump (save / revert grow the buffer); for delete-confirm the references count is per-modal-instance and doesn't outlive it.
    - **Why the parent still owns the delete mutation.** Success updates the parent's `data` (the model). The modal can't update parent state on its own; it would have to call back to a parent callback regardless. Putting the mutation in the parent + signalling close via `onOpenChange(false)` is the simpler shape. The parent's `deleting` flag disables both modal buttons while the DELETE is in flight; the `e.preventDefault()` on the action button keeps Radix from closing the modal early so the admin sees the result.
    - **Why a trailing Actions column rather than an inline button next to each entry name.** Visual scan-ability. An "Actions" column reads as a destructive-op landing strip; an inline delete button next to the entry name would compete with the SourceBadge chip and the EditableText label inputs for the eye. The trailing column is the same convention Radix uses for table actions. The `<th><span className="sr-only">Actions</span></th>` keeps the column accessible without consuming a visual header slot (the icon-only buttons in the body cells provide the visual cue).
    - **Why `Trash2` (lucide) rather than `X` or a text-only "Delete" button.** `Trash2` reads as "delete this entry"; `X` reads as "close this modal / dismiss this card". The icon-only button (no text label) keeps the trailing column narrow so the existing edit cells don't get squeezed; the `aria-label="Delete metric ${m.name}"` provides screen-reader coverage. Lucide is already a project dependency from W61-audit-history-tab's `ChevronDown`.
    - **Why we reuse `PatchSemanticModelResponse` as the delete return type.** The server returns the W61-save envelope (`{ sessionId, lastUpdatedAt, model }`) byte-identical across save / revert / delete per the W61 envelope-reuse convention. Reusing the type keeps the host's success handler shape symmetric across all three operations: same `setData({ ...data, lastUpdatedAt: res.lastUpdatedAt, model: res.model })` lands on every successful mutation, and the audit-history-tab's `useEffect` keyed on `lastUpdatedAt` re-fetches the buffer automatically.
    - **Why a separate sibling component for the modal, not inlined in the host.** The host was at 1,131 LOC post-W61-detail-extract. Inlining the modal (~270 LOC) would push it to ~1,400 LOC. With W61-add-entry projected at +200 LOC, the host would approach the 1,500-LOC sub-component-extract threshold. Extracting the modal preemptively keeps headroom; the W61-detail-extract precedent established `client/src/pages/Admin/components/` as the canonical landing zone for these sibling-file extractions.
    - **Why a `pending` prop shaped as `{ kind, name } | null` rather than separate `open` / `kind` / `name` props.** The non-null discriminant IS the open signal; splitting it into three props would invite the inconsistent state `open=true && kind=null` which the modal would have to defend against. The single-discriminant prop makes the consumer-side "open the modal for this entry" idiom one assignment (`setPendingDelete({ kind: "metric", name: m.name })`) and the close one assignment (`setPendingDelete(null)`).
    - **Why `e.preventDefault()` on `<AlertDialogAction>`.** Radix's default behaviour closes the dialog when the action is clicked, but we want the modal to stay open with `deleting=true` so the destructive button can show `"Deleting…"` and the parent can surface a success / failure outcome inline. Without `preventDefault`, a transient network failure would close the modal and lose the error visibility — the admin would see the page state unchanged with no breadcrumb explaining why.
    - **Why the audit-log reassurance copy renders even on the references-fetch-error branch.** The audit log is load-bearing regardless of whether the references precheck succeeded — the admin can recover from a delete via revert. Suppressing the reassurance on the error branch would leave the admin to wonder if the failed precheck also implies a broken audit-log path; explicit reassurance side-steps that doubt.
    - **Why the references-warning subhead is suppressed when `totalOccurrences === chartCount`.** Every reference is in a distinct chart, so the chart count IS the occurrence count and the subhead would be redundant — `"3 charts that reference it"` + `"3 references total"` reads as a doubled assertion. The subhead surfaces precisely when one chart uses the entry multiple times (e.g. the entry is bound to both `x` and `tooltip` channels): `"1 chart that references it"` + `"2 references total across the affected chart"` tells the admin "heavy usage in one chart" without forcing them to compute the divergence. The pure-helper test pins the boundary so a future refactor that flips the comparison surfaces immediately.
    - **Why local interface mirror for `AdminSemanticModelEntryKind` rather than importing from the server.** Continues the W61 cross-runtime-boundary convention: client interfaces that mirror server shapes are declared locally in [`client/src/lib/api/admin.ts`](../client/src/lib/api/admin.ts) rather than imported from the server. The runtime drift between mirror and server would surface as a field-undefined access at the call site rather than a compile error — the W61 trail accepts that tradeoff because the shapes are small and stable.

  **Why this design.**
    - **Why path params for `:kind` and `:name`, not query / body.** Both are part of the resource identity — "which entry are we deleting?" DELETE has no body in idiomatic HTTP; query params are for filters, not identifiers. Path identifies the resource; the three-segment shape (`/entries/:kind/:name`) reads naturally as a hierarchical resource path.
    - **Why explicit `:kind`, not auto-scanning all three collections for the name.** Name collisions across kinds are legal — the semantic model schema doesn't forbid a metric and a dimension sharing a name. A fuzzy "find this name anywhere and delete it" would silently delete the wrong thing when a collision exists. Requiring `:kind` in the URL makes the operation unambiguous; a dedicated test (`kind=dimension` + `name=alpha` when only a metric named `alpha` exists → 404 `entry_not_found`) pins the no-fuzzy contract.
    - **Why audit-write BEFORE delete (not after).** The buffer's role is to remember what was just lost. If the audit-write happened after the delete, a crash between the two would lose forensics. Inside `withSessionWriteLock` the pair is atomic at the per-session level (single-instance correctness only; multi-instance horizontal scaling still needs Cosmos `ifMatch` ETag). Mirrors the W61-audit-log + W61-audit-revert precedent.
    - **Why bump `version` even though we removed something.** The version is W64's compiled-query cache key. Removing a metric invalidates any cached compilation that referenced it; bumping makes the cache miss explicit and ensures the next planner round consults the fresh model.
    - **Why we do NOT run `bumpMetricsSource` / `bumpDimensionsSource` / `bumpHierarchiesSource` on the survivors.** The surviving entries are byte-identical to their prior selves; the bumper's content-hash diff would match them as unchanged and preserve their prior source. Running it would be a no-op for unchanged entries; skipping is semantically equivalent and clearer about intent. A success-path test pins this: a metric with `source: "user"` and another with `source: "domain"` keep those sources verbatim after a third metric is deleted (no spurious bump-to-user on the survivors).
    - **Why the audit-log write happens unconditionally on success.** A delete IS a state change; preserving the just-deleted entry in the buffer is required for "undo this delete via revert" to work. The W61-audit-revert wave's revert endpoint walks the buffer and restores `priorModel`; this wave's delete entry produces an audit entry whose `priorModel.metrics` (or `.dimensions` / `.hierarchies`) includes the deleted entry, so a future revert call can put it back.
    - **Why the `DeleteResult` discriminated union sentinel.** Throwing from inside the `withSessionWriteLock` callback would mark prior chained callers as failed (the lock serialises per-session writes); returning a sentinel from the callback and mapping it to an HTTP status in the outer handler keeps the per-session error semantics clean. Same pattern as `PatchResult` (W61-save) and `RevertResult` (W61-audit-revert).
    - **Why `removed` is set then `void removed`.** The early `return { kind: "entry_not_found" }` in each branch guarantees the flag is true on every success path; asserting it would be dead code. Keeping the variable (with the `void` to silence the unused-variable lint) makes the per-branch shape read consistently — a future fourth kind would set `removed = true` in the same way. The cost is one line; the benefit is symmetric branches.

  **Out of scope.**
    - **Client UI** (Delete button, confirmation modal) — deferred to **W61-delete-client**. The future wave will add the per-row `<Button>` on `MetricRow` / `DimensionRow` / `HierarchyRow`, the confirmation modal that fetches the W61-references-endpoint count for the "removing this will break N charts" copy, and the API helper `deleteSemanticModelEntry(sessionId, kind, name)` that calls this endpoint.
    - **Client API helper** (`deleteSemanticModelEntry` in [`client/src/lib/api/admin.ts`](../client/src/lib/api/admin.ts)) — bundled into W61-delete-client so that wave is fully self-contained (server here, client helper + UI there).
    - **Bulk delete** (`DELETE /admin/semantic-models/:sessionId/entries?names=a,b,c`) — defer until the future "audit all unused entries" UI surfaces a need; the per-entry endpoint covers the common case.
    - **Cascade delete** (e.g. deleting a dimension also removes any metric whose `references[]` includes that dimension's column) — the W58 compiler's orphan-column lint is the right place to surface this, not the delete endpoint; this endpoint deliberately keeps a narrow contract.
    - **Soft delete** (mark `deleted: true` instead of removing) — the audit log already serves as soft-delete recovery via revert (the deleted entry's snapshot stays in the buffer, recoverable via W61-audit-revert), so hard delete is the simpler model.
    - **Pre-delete reference scan inside the handler** (the handler could call `countSemanticModelReferences` itself and refuse to delete if `chartCount > 0`) — that policy choice belongs to the client confirmation flow, not the server. The server's job is to perform the requested operation; the client's job is to ask the user "are you sure?" with the reference count. Keeping the policy on the client matches how the admin save / revert paths work (no server-side "are you sure?" — the destructive op is performed if requested).
    - **Per-entry permission check beyond admin gate** (e.g. metric author can't delete metrics created by another admin) — admin endpoints are gated by `SUPERADMIN_EMAILS`; finer-grained per-entry ACLs are unrelated to this wave's scope.
