- **2026-05-20** — **Wave W61-source-badge · per-row source chip in the admin viewer.** Closes the visibility gap that W61-source-bump's server-side stamp opened: the previous wave ensures the `source` field flips from `"auto"` to `"user"` on every admin edit, but the detail page had no visual treatment for the field at all — the admin's correction was correct in the doc and read correctly by the planner but invisible to the human reviewing what they'd already fixed. This wave adds a small coloured chip per row keyed on each entry's `source`: auto (muted secondary) recedes; user (primary tint) pops; future domain-pack imports (gold accent) read as "imported authoritative knowledge". The chip is also the discoverability hook for the W61-source-bump semantics — a hover-tooltip on each chip explains what the source value means and where it comes from. Pure-client wave: zero server code touched, zero schema change.

  **What landed.**
    - New pure module [`client/src/pages/Admin/lib/semanticModelSourceBadge.ts`](../client/src/pages/Admin/lib/semanticModelSourceBadge.ts) exports the `SemanticEntrySource = "auto" | "user" | "domain"` local alias plus three accessor functions: `getSourceBadgeLabel(source)` returns the user-facing display string (`"Auto"` / `"User"` / `"Domain"` — capitalised; the lowercase enum values are the planner-facing identifier, the labels are the human-facing rendering), `getSourceBadgeVariant(source)` returns the `BadgeProps["variant"]` to feed the existing UI primitive (`secondary` / `default` / `gold` — mapped against the UX-2 Badge palette from [`client/src/components/ui/badge.tsx`](../client/src/components/ui/badge.tsx)), and `getSourceBadgeTooltip(source)` returns the hover-tooltip text (each tooltip names the upstream system: `"Auto-inferred from the dataset by the W57 inference pipeline."` for `"auto"`, `"Manually edited by an admin via this viewer."` for `"user"`, `"Imported from a domain pack (e.g. kpi-and-metric-glossary)."` for `"domain"`). The three mappings are declared as three `Readonly<Record<SemanticEntrySource, …>>` lookups so adding a fourth source value to the schema would TypeScript-error here first (the inferred `Record` key set would no longer match the union), then bubble up to the helper functions.
    - The `SemanticEntrySource` alias is intentionally a **local** alias rather than re-exported from `@/shared/schema`. Considered: import `SemanticMetric["source"]` and re-use that union. Rejected — a future schema split (e.g. `metric_source` widened to add `"computed"` while `dimension_source` stays the trio) would silently widen the chip's accepted inputs and a TypeScript-permissive Record lookup would still compile. The local alias forces a deliberate sync whenever the underlying enums diverge.
    - [`AdminSemanticModelDetail.tsx`](../client/src/pages/Admin/AdminSemanticModelDetail.tsx) gains a new `SourceBadge` sub-component sibling to `ExposedToggle` / `EditableText` / `EditableSelect`. Wraps the existing UX-2 `Badge` primitive with `variant={getSourceBadgeVariant(source)}` + `title={getSourceBadgeTooltip(source)}` (native HTML `title=` attribute — Tooltip-provider mount considered but rejected: at this density a Radix Tooltip would need a `<TooltipProvider>` mounted at the page root + four extra DOM nodes per chip, with no UX win over the browser's native hover handling). Renders the label as a single child `{getSourceBadgeLabel(source)}`. Tuned smaller than the canonical Badge (`px-1.5 py-0 h-4 text-[10px] font-medium`) so the chip reads as row-metadata next to the snake-case identifier rather than competing with it.
    - The chip is wired into each row's name cell sibling to the snake-case identifier via a `<div className="flex items-center gap-2">` wrapper: `MetricRow` (`<SourceBadge source={m.source} />` next to `{m.name}`), `DimensionRow` (sibling to `{d.name}`), `HierarchyRow` (sibling to `{h.name}` — even though the W61 polish trail doesn't ship hierarchy edits yet, W61-source-bump's `bumpHierarchiesSource` already supports the user-bump and hierarchies imported from a domain pack will eventually carry `source: "domain"`, so the chip surfaces consistently across all three entity types).
    - Tooltips anchor on concrete identifiers: the auto tooltip references the W57 pipeline by name (a future admin reading the chip can grep `W57` in the codebase to find the inference module); the domain tooltip names the `kpi-and-metric-glossary` pack as a concrete example. This is deliberate — generic descriptions ("This was created automatically") would force the admin to ask Claude or the team what "auto" means; the anchor lets them self-serve.
    - 18 new behavioural tests in [`semanticModelSourceBadgeW61SourceBadge.test.ts`](../client/src/pages/Admin/lib/semanticModelSourceBadgeW61SourceBadge.test.ts) covering: per-source label / variant / tooltip mappings (9 tests, three per accessor); coverage pin (`ALL_SOURCES` const fed through every accessor in a forEach loop to ensure non-empty outputs); distinctness pins (labels / variants / tooltips all distinct across the three sources — two sharing a variant would defeat the chip's scan-at-a-glance purpose); load-bearing distinctness pins (user ≠ auto explicitly — load-bearing for the W61-source-bump payoff so the admin sees their edit visually flip; domain ≠ user explicitly — pack imports shouldn't visually equate with admin overrides); content anchors (auto tooltip mentions `W57`; domain tooltip mentions `kpi-and-metric-glossary`). Test file appended to [`server/package.json`](../server/package.json)'s explicit test list per invariant #4 (right after `semanticModelEditValidationW61EditEnums.test.ts`).

  **Why this design.**
    - **Inline chip beside the entry name rather than a new table column.** Considered: add a `Source` column to each of the three tables. Rejected — the source field is sparse signal density (3 distinct values across the whole dataset) and a dedicated column would burn horizontal space that the existing 4–5-column tables already feel tight in; the chip beside the name reads as a row badge ("this entry is X") rather than a sortable column. The flex-row layout means the chip floats next to the identifier without disturbing the existing description / label stack below.
    - **Native `title=` tooltip rather than the Radix Tooltip primitive.** A Tooltip primitive would need a single `<TooltipProvider>` wrapping the page root (currently absent) plus a `<Tooltip>` / `<TooltipTrigger>` / `<TooltipContent>` triple per chip. At three chips per row times tens of metrics + dimensions + hierarchies that's hundreds of extra DOM nodes for hover text the browser already handles natively via `title=`. Considered: build the Tooltip provider once and reuse for other chips in the page. Deferred — no other chip in the page needs hover text today, and if a future wave wants richer tooltip content (multi-line, icons, links) the chip's `title=` can be replaced with a `<Tooltip>` wrapper at that wave's cost.
    - **Three frozen `Readonly<Record<…>>` lookups rather than a `switch`.** Considered: `switch (source) { case "auto": return ...; ... }` for each accessor. Rejected — the three-source enum is small enough that a switch is no clearer than a Record lookup, and the Record lookup gives the TypeScript type system a single source-of-truth shape (`Record<SemanticEntrySource, T>`) that a fourth-source widening would error against in one place rather than four switches. The `Readonly<Record<…>>` annotation prevents accidental run-time mutation in a future wave.
    - **Local `SemanticEntrySource` alias rather than `SemanticMetric["source"]` indexed access.** See above — a future split (e.g. `metric_source` adds `"computed"`) shouldn't silently widen the chip's inputs. The local alias forces a deliberate sync.
    - **Chip on hierarchies even though the polish trail doesn't yet edit them.** W61-source-bump's `bumpHierarchiesSource` already supports the user-bump on PATCH and a future `W61-edit-hierarchies` wave will land hierarchy-edit affordances; not wiring the chip into `HierarchyRow` this wave would leave a visible inconsistency (metrics + dimensions show source, hierarchies don't) for no behavioural reason. Adds 3 lines of JSX; preserves visual parity.
    - **`as const` lookup objects rather than module-scope `const SOURCE_LABELS = {...} as const` exports.** Considered: export the lookup objects directly so a consumer could read `SOURCE_LABELS["user"]`. Rejected — the indirection through `getSourceBadgeLabel(source)` is a small reabsorbed boilerplate cost but gives the helper module a stable API surface (a future wave can change the lookup implementation without breaking call sites). For three call sites in one file the indirection is borderline; for any future consumer it pays off.
    - **Capitalised labels ("Auto" not "auto").** The lowercase identifiers are the planner-facing canonical form (matches the zod enum exactly); the chip is human-facing UI where title-case reads as polished. Mirrors the W61-edit-enums option-list pattern (`{ value: "categorical", label: "Categorical" }`).

- **2026-05-20** — **Wave W61-source-bump · auto-mark edited entries as `source: "user"` on PATCH.** Closes the W61 source-weighting gap nobody noticed for three waves. The W57 inference pipeline writes every metric / dimension / hierarchy with `source: "auto"`; W61-save's PATCH path then preserved whatever `source` the client sent (almost always `"auto"` because the client mirrors the prior model field-by-field and admin edits land on content fields, not on `source` itself). Net effect across W61-save / W61-edit-text / W61-edit-enums: every admin correction was **invisible** to the planner's source-weighting in [`buildSemanticCatalogPromptBlock`](../server/lib/semantic/prompt.ts) — manually-corrected entries should outrank inferred ones when the planner dedupes near-identical metrics, but the `source` field still claimed `"auto"` after every edit. This wave wires the missing diff-and-bump on the W61-save controller's RMW path.

  **What landed.**
    - New pure module [`server/lib/semantic/semanticModelSourceBump.ts`](../server/lib/semantic/semanticModelSourceBump.ts) exports `bumpMetricsSource(next, prior)` / `bumpDimensionsSource(next, prior)` / `bumpHierarchiesSource(next, prior)`, each a per-entry content-hash diff that returns a new array (input arrays untouched — pure). Internal `contentHashMetric` / `contentHashDimension` / `contentHashHierarchy` helpers project each entry to its content fields (everything except `source`) and `JSON.stringify` the projection. The explicit projection is load-bearing: `JSON.stringify(entry)` would also serialise `source`, which would force a non-equal hash on a no-op save where the only thing the client changed is the source field — defeating the point. Three rules in `bumpEntries` (the generic helper): (i) **new** entry (no `name` match in prior) → preserve client-sent `source`; (ii) **unchanged** entry (content hashes equal) → preserve prior `source` (auto stays auto, domain stays domain, user stays user); (iii) **changed** entry → bump to `"user"` regardless of prior source. The `USER_SOURCE = "user" as const` constant keeps the bumped literal type-narrow so `bumpEntries<T>`'s generic signature flows through cleanly.
    - [`adminSemanticModelController.ts`](../server/controllers/adminSemanticModelController.ts) wires the three bumpers into the W61-save PATCH path inside `withSessionWriteLock` (invariant #9), between `parsed.data` construction and the version stamp. Each bumper compares the parsed body's array against `doc.semanticModel.<arrayName>` loaded inside the lock — so the diff sees the actual stored state at write time, not a client-sent prior view that could be stale (an admin who opens a detail page, then a second admin saves over the model, then the first admin clicks save: the lock-load gives the first admin's bump the most-recent prior, not the stale view in their UI).
    - Hierarchies get the bump treatment too even though the trilogy doesn't yet ship a hierarchy edit affordance — preparing for **W61-edit-hierarchies**. The hierarchy content hash projects `label` / `levels` / `description` (skipping `source` and `name`); the W56 schema doesn't have an `exposed` field on `SemanticHierarchy`.
    - 19 new behavioural tests in [`semanticModelSourceBumpW61SourceBump.test.ts`](../server/tests/semanticModelSourceBumpW61SourceBump.test.ts) covering: unchanged auto / domain / user preservation; metric label / expression / exposed / format-with-stale-currencyCode-clear / new-metric paths; mixed array with all four shapes (unchanged + changed + preserved-user + new) in one call; prior-user-edited round-trip preservation (no silent demotion); dimension kind / temporalGrain / description change-bumps; hierarchy levels reorder; empty-arrays clean-handle; prior-entries-deleted clean-drop (no orphan re-insertion); input-array no-mutation contract. Test file appended to [`server/package.json`](../server/package.json)'s explicit test list per invariant #4 (right after `adminSemanticModelPatchW61Save.test.ts`).
    - The 9 pre-existing end-to-end tests in [`adminSemanticModelPatchW61Save.test.ts`](../server/tests/adminSemanticModelPatchW61Save.test.ts) still pass byte-identical: their `VALID_PATCH_BODY` toggles `exposed` (a content change → bumps to `"user"` which matches the client-sent `"user"`) — same observable output regardless of whether `source` was preserved or bumped, so no test regression and no test update needed.

  **Why this design.**
    - **Server-side bump rather than client-side.** Considered: have the client set `source: "user"` on every PATCH body. Rejected — the client has no per-entry diff state (the EditableText / EditableSelect components fire `onSave` with a full new model built from the prior + the one edited cell), so the client can't tell which entries were touched without re-comparing against the original load. Even if it could, the source-of-truth for "what was the prior version" is the server's loaded doc inside the lock, not the client's last-seen view. The server-side bump means the client stays simple (`patchMetric` / `patchDimension` just funnel through `persistModel`) and the diff sees the authoritative state.
    - **Per-entry diff via content-hash projection rather than `lodash.isEqual` or zod-shape equality.** Considered: `JSON.stringify(entry) === JSON.stringify(prior)`. Rejected — that includes `source` in the hash, which means a no-op save where the client mistakenly sent a different `source` (an old client bug, say) would still bump. The projection skips `source` so source-only differences are NOT treated as content changes. Considered: deep-equal helpers. Rejected — adds a runtime dependency for one call site; JSON.stringify on a fixed-shape projection is byte-deterministic for zod-validated objects (key order follows declaration order in the schema, which is stable across runs).
    - **Bump rule: change → "user" regardless of prior source.** A prior-domain entry that the admin edits has been overridden — semantically it's no longer "from the pack"; it's "from the admin who deviated from the pack". Bumping to `"user"` makes the planner's weighting reflect that. If a future feature wants to preserve domain-provenance on minor admin tweaks (e.g. typo fixes), the cleanest path is a separate `domainOverride` flag, not a fuzzy preserve rule here.
    - **New entries preserve client-sent source.** The W61 trilogy doesn't ship an "add metric" flow yet, but **W61-add-delete** will. When the admin adds a metric, the client default is `source: "user"` (the schema's `.default("user")` — wait, actually the schema defaults to `"auto"`; the client would need to override to `"user"` explicitly). Either way the server's "preserve client-sent" rule lets the future add-flow choose `"user"` (manual add) or `"domain"` (import from pack) without server-side surgery.
    - **Inside the lock, not before.** The diff has to see the actual stored prior, not a stale view. The W61-save controller's RMW pattern already loads `doc` inside `withSessionWriteLock(sessionId, ...)`; slotting the bump immediately after that load + before the `nextModel` build is the minimal change that gets the right prior.

Also re-runs the 9 W61-save controller tests to verify the source-bump path doesn't regress the existing end-to-end assertions. 28/28 pass on the combined run (~530 ms). Test file appended to [`server/package.json`](../server/package.json)'s explicit list per invariant #4.

The bumper's generic `<T extends { name: string; source: "auto" | "user" | "domain" }>` constraint flows through cleanly against `SemanticMetric` / `SemanticDimension` / `SemanticHierarchy` (all three have those two fields).

  **Out of scope.**
    - **Cache invalidation downstream of the source bump.** Same scope-shape as the deferred **W61-cache-invalidate**: when an admin save bumps a metric's source from `"auto"` to `"user"`, the W57 inference cache + the planner-side `buildSemanticCatalogPromptBlock` may still serve the old manifest (the cache is keyed on `semanticModel.version`, which DOES bump every save, so the cache will see the new version and re-render — but a separate dedicated wave will close this loop end-to-end with an explicit invalidator).
    - **UI surfacing of the new "user" badge.** The detail page renders `source` somewhere on each row but the visual treatment doesn't distinguish "auto" from "user" today. A follow-on **W61-source-badge** could add a colored chip ("auto" muted, "user" primary, "domain" accent) so the admin can see which entries they've already corrected.
    - **Audit log of the source transition.** A future **W61-audit-log** wave could write `{prior_source, new_source, at, by}` to a per-entry audit array so an admin can see the source-change history without restoring the prior model. Out of scope here.
    - **Domain-override flag.** A separate `domainOverride: boolean` to mark "this was a domain entry that the admin tweaked but didn't fully re-author" would let the planner weight differently than full user-authored. Premature — wait for a real use case.
    - **Pre-W61-save reconstruction.** Sessions that were edited via direct Cosmos mutation before W61-save shipped won't have correct `source` values; those are stuck at `"auto"`. A one-off backfill script could diff against the W57 inference-time snapshot, but that snapshot wasn't preserved (W57 wrote directly into the doc). Accept the drift; the bump kicks in for all future edits.

- **2026-05-20** — **Wave W61-edit-enums · enum-select editing for `format` / `kind` / `temporalGrain` + `currencyCode`.** Second polish wave on top of the W61 trilogy + W61-edit-text foundation: admins can now correct enum-shaped inference gaps on the semantic model inline. The most common case the data team needs is a temporal column whose `kind` was inferred as `categorical` because the values are string month names ("Jan", "Feb", …) rather than ISO dates — two clicks now flip it to `temporal` + `temporalGrain: "month"`. Similarly a percent metric inferred as `format: "number"` (because the dataset stores `1.0–100.0` not `0.0–1.0`) flips with one select. Currency-code text edit appears conditionally when `format === "currency"` so the planner manifest's `currency (INR)` hint renders correctly. The trilogy → edit-text → edit-enums sequence now closes every common-admin correction except adds / deletes / hierarchy reordering.

  **What landed.**
    - [`AdminSemanticModelDetail.tsx`](../client/src/pages/Admin/AdminSemanticModelDetail.tsx) gains a new `EditableSelect<T extends string>` sub-component sibling to the `EditableText` from W61-edit-text. Unlike `EditableText` there's no draft / validation step — every option is by-construction valid because the option list is byte-locked to the zod enum at the source. Wraps Radix `<Select>` with `onValueChange={(v) => onSave(v as T)}` firing the parent's optimistic-update-and-PATCH flow directly. Standard Radix trigger / content / item composition; `h-8 text-sm` styling matches the row density of `EditableText`. The `value` prop is typed `T | undefined`; Radix accepts `undefined` and renders the placeholder, used for cases where the underlying field is `.optional()` and unset.
    - Three frozen option-list constants declared with the `as const satisfies ReadonlyArray<{ value: …; label: string }>` pattern so a typo against the zod enum becomes a TypeScript error at the declaration site rather than a runtime "Save failed: invalid_semantic_model" with a confusing zod issue message: `METRIC_FORMAT_OPTIONS` (5 entries — `number` / `percent` / `currency` / `ratio` / `duration`), `DIMENSION_KIND_OPTIONS` (4 entries — `categorical` / `temporal` / `numeric_binned` / `geo`), `TEMPORAL_GRAIN_OPTIONS` (6 entries — `__auto__` sentinel plus `day` / `week` / `month` / `quarter` / `year`). The `__auto__` constant is exported alongside the option list so the parent handler's sentinel check (`nextGrain === TEMPORAL_GRAIN_AUTO ? undefined : ...`) stays in lockstep.
    - `MetricRow` widens with two new handler props: `onEditFormat(next)` and `onEditCurrencyCode(next)`. The format cell becomes an `EditableSelect` over `METRIC_FORMAT_OPTIONS` plus a conditional `EditableText` (monospace, `placeholder="USD / INR / EUR"`) for `currencyCode` rendered only when `m.format === "currency"`. The pre-existing read-only `decimals` hint is preserved as a sibling line below.
    - `DimensionRow` widens with `onEditKind(next)` and `onEditTemporalGrain(next)`. The kind cell becomes an `EditableSelect` over `DIMENSION_KIND_OPTIONS` plus a conditional `EditableSelect` over `TEMPORAL_GRAIN_OPTIONS` rendered only when `d.kind === "temporal"`. The temporal-grain select's `value` prop maps `undefined` → `TEMPORAL_GRAIN_AUTO` so Radix has a concrete string to display ("Auto (let agent derive)").
    - Three new parent handlers funnel through the existing `patchMetric` / `patchDimension` narrow handlers from W61-edit-text: `handleMetricFormatChange(metricName, nextFormat)` (clears `currencyCode` when `nextFormat !== "currency"` — a stale `"USD"` value on a `format: "percent"` metric would mislead the planner manifest's `currency (USD)` hint), `handleDimensionKindChange(dimensionName, nextKind)` (clears `temporalGrain` when `nextKind !== "temporal"` — a `"month"` grain on a `categorical` dimension is semantically nonsensical), and `handleTemporalGrainChange(dimensionName, nextGrain)` (maps the `__auto__` sentinel back to `undefined` so the agent's `temporalFacetColumns` derivation continues to derive the grain from data when the admin hasn't explicitly pinned one).
    - [`semanticModelEditValidation.ts`](../client/src/pages/Admin/lib/semanticModelEditValidation.ts) gains `validateCurrencyCode(value)`: returns `null` for empty / whitespace-only (optional field — the empty string is a legitimate "not set yet" state while the admin is mid-edit), validates `/^[A-Z]{3}$/` after trim otherwise. Does NOT enforce the format-coupling (i.e. "if format=currency then currencyCode required") because `EditableText` validates fields in isolation; the server's `semanticModelSchema.safeParse` is authoritative on cross-field constraints and rejects a `currency` format paired with an empty `currencyCode` at PATCH time with a zod issue path that surfaces in the existing "Save failed: {message}" banner.

  **Why this design.**
    - **`as const satisfies ReadonlyArray<{ value: T; label: string }>` for the option lists.** Considered: a runtime `if (!OPTIONS.find(o => o.value === schemaValue)) throw` cross-check at app boot. Rejected — TypeScript's `satisfies` keyword does exactly this at compile time without runtime cost, makes the typo a red squiggle in the IDE, and the `as const` keeps the literal-narrowing so the `value: T` flows through `EditableSelect`'s `onSave: (next: T) => void` without a runtime cast.
    - **Save-on-select with no validation step (vs `EditableText`'s blur-then-validate).** A select's value is constrained to the option list by construction; there's no analogue to "admin typed an invalid expression". Skipping the draft/validation dance keeps the component small (~30 LOC) and lets the parent's optimistic-update-and-PATCH flow fire the moment the admin picks an option.
    - **`__auto__` sentinel for the temporal-grain undefined case.** Radix `<Select>` rejects `value={""}` (empty string is reserved for the "no selection / show placeholder" state) and doesn't have a built-in "clear selection" affordance. The sentinel keeps Radix happy AND lets the admin explicitly opt back into auto-derivation after pinning a grain by mistake. The mapping is contained to a 3-line handler at the save boundary; the schema sees only `undefined | "day" | "week" | ...` so no zod surprise.
    - **Clear `currencyCode` on format-change-away-from-currency / `temporalGrain` on kind-change-away-from-temporal.** Considered: leave the orphan fields in place so toggling currency-then-back-to-currency preserves the prior code. Rejected — the orphan fields surface in the planner manifest (W59's `formatMetricCatalog` reads currencyCode unconditionally when it's defined) and would render `Net Sales (USD)` on a metric that's now `format: "percent"`. The "preserve on round-trip" UX win is small (admin can type the 3-char code in 2 seconds); the misleading-manifest cost is the planner downstream of every wrong-shape correction.
    - **`validateCurrencyCode` empty-accepts even though the schema makes it required-with-format-currency.** The field is `.optional()` at the schema level; the cross-field constraint is enforced by the server at PATCH time. Making the client validator reject empty would block the admin's natural "I'm switching to currency, give me a second to type the code" workflow — they'd hit a red border before typing the first letter. The server's authoritative rejection (with a clear zod issue path) is the right place for that check.
    - **Conditional render rather than always-rendered-but-disabled.** A disabled `currencyCode` input on a non-currency metric would clutter the cell visually + need explanatory text ("only applies when format = currency"). The conditional render is cleaner and the admin's mental model is "if I see the currency-code field, I need to fill it" which is correct.

  **Why this design.**
    - **Always-editable inputs over click-to-edit popovers.** Considered: each cell renders as styled read-only text by default; clicking it swaps in an input with `autoFocus`; blur or commit returns to read-only. Rejected — the dance adds state (`editingFieldId`), keyboard-trap risk (Escape during the swap), and an extra click per correction. Admins fixing inference gaps spend their time tabbing through many fields per session; always-editable lets them flow without a context switch per cell. The visual distinction "this is editable" comes from the input's own styling (border, focus ring) which already signals affordance.
    - **Save-on-blur over save-on-Enter or per-row Save button.** Considered: an explicit Save button per row that batches edits + a discard-changes button. Rejected — the parent's `persistModel` already debounces by routing through `withSessionWriteLock(sessionId, ...)` (invariant #9), so multiple sequential field saves serialize cleanly without a batch button. Save-on-Enter alone leaves a "ghost dirty" state if the admin tabs out without explicitly committing. Save-on-blur matches Google Docs / Notion semantics and pairs with `isMeaningfulChange` to skip no-op PATCHes.
    - **Client-side validators that mirror but don't share zod schemas.** Considered: import `semanticMetricSchema.shape.label` and re-derive the bounds at runtime. Rejected — the validators are 96 LOC of pure-function bounds checks; if the schema bounds change, a drift here just routes the save to the server's authoritative rejection (a 400 with the zod issues array surfacing in the existing "Save failed: {message}" banner). The validators' job is to keep obviously-bad inputs from round-tripping at all, not to be a second source of truth.
    - **Keyword-guard via padded uppercase scan instead of regex.** Considered: `/\b(SELECT|FROM|JOIN|...)\b/i` over the trimmed string. Rejected — JavaScript `\b` matches at underscores which would let `_FROM_` slip through; a literal regex with `(?<![A-Za-z0-9_])` lookbehind works but the engine's lookbehind support varies by runtime. The padded-uppercase form (`" ${trimmed.toUpperCase()} "`.includes(" FROM ")) is byte-exact, runs in O(n) per keyword, and avoids regex edge cases.
    - **`isMeaningfulChange` based on trimmed equality.** Considered: deep equality on raw strings. Rejected — admins frequently add or trim trailing spaces while typing; firing a PATCH on every whitespace-only edit would burn write throughput and pollute the W64 cache key (version bump per save). The trimmed-equality form means leading/trailing whitespace is normalised by the save itself (the handler does `draft.trim()` before calling `onSave`).
    - **`emptyToUndef` helper for the description's `.optional()` shape.** A literal empty string is a valid `z.string().max(1000).optional()` value (`.optional()` only adds `undefined` to the union; it doesn't strip empty strings), but persisting `""` instead of `undefined` makes the doc shape drift: a freshly-inferred model would carry `description: undefined` for missing descriptions and a manually-cleared one would carry `""`, which are semantically the same but serialise to different bytes (and a future "no description = use the auto-inferred one" feature would break in subtle ways). The helper normalises both into `undefined`.

    - **`sessionStorage`, not `localStorage`.** Considered: `localStorage` would also survive tab close, which sounds like a longer breadcrumb. Rejected — a future operator who manually edits sessionStorage to test something could land a v2 payload in a v1 slot; the payload-level `version: 1` check catches that.     - **Per-entry validation rather than per-payload all-or-nothing.** Considered: if any entry in the payload fails validation, drop the whole tile (or the whole payload). Rejected — a single corrupt slot (e.g., from a third-party browser extension that mutated sessionStorage) shouldn't punish the user's other 2 valid slots in the same tile. Per-entry filter is the more conservative degradation.
    - **`opts.storage !== undefined ? opts.storage : default` rather than `opts.storage ?? default`.** Considered: the `??` form is shorter. Rejected — callers MUST be able to opt out of persistence explicitly even in a browser context (e.g., a special preview mode where filter state shouldn't leak across reloads), and `??` would treat an explicit `null` the same as `undefined`. Rejected — dashboards remount in practice when the route changes (wouter handles the switch); capturing the scope at mount mirrors the existing `insightRegenCache`'s empty-deps lifetime and avoids the surprise of mid-mount store replacement.

Rejected — the persistence contract is "dies with the tab"; a localStorage fallback would silently widen that to "survives tab close", which is a UX surprise.   **Why this design.**
    - **Cohen's conventions on |r|, not on R².** Considered: bucket on R² since R² is what the prose already carries and what the WQ1 classifier reads. Rejected — Cohen's published thresholds for correlation effect sizes are stated on |r|, not R²; bucketing on |r| matches the academic convention and means the same r value buckets identically whether the tool is reporting r or R². R² is monotonic in |r| so the bucket transitions still fire at the right places, but pinning to the |r| boundaries gives clean numerical values (0.1 / 0.3 / 0.5) instead of R² boundaries (0.01 / 0.09 / 0.25).
    - **Bucket via `null` on out-of-range r rather than throwing or defaulting.** Considered: throw on `|r| > 1` (it shouldn't happen with the Pearson formula but defense in depth) or default to `"negligible"`. Rejected — `null` lets the caller decide whether to skip emitting the bucket entirely; a default would silently mislabel pathological inputs. The truthy guard at the call site (`if (bucket) { ... }`) means a `null` bucket cleanly degrades to "no `effectMagnitude` emitted", which is the conservative behaviour.
    - **Helper lives in `correlationMath.ts`, not at the tool call site.** Considered: inline the 4-line ladder inside the WV4 block of `registerTools.ts`. Rejected — `correlationMath.ts` is the natural neighbour to `calculateCorrelations` (same module, same purity contract, same test path through `correlationMath`-only tests). Future callers (a hypothetical `run_correlation_pair` tool, a deck-planner correlation block) can reuse the helper without copy-pasting the threshold ladder.
    - **No change to `analyzeCorrelations` itself.** The bucket is derived at the tool boundary, not pushed into the `CorrelationResult` interface. Adding `effectBucket?: ...` to `CorrelationResult` was considered and rejected — the result struct is consumed by chart rendering / insight prose / fallback paths that don't need the bucket; the bucket is only load-bearing on the WV4 evidence emit. Keeping the field at the call site avoids a wide-blast schema migration.
    - **Sign-invariance pinned explicitly.** A negative r (anti-correlation) and its positive twin describe equally strong relationships — they just point opposite directions. The `Math.abs()` inside the helper is load-bearing for this; pinning it in a sign-invariance test (`-0.05 → "negligible"`, `-1 → "large"`) makes a future refactor that drops the abs() break loudly.

  **Why this design.**
    - **A new field on `FindingEvidence` rather than overloading the pre-existing `magnitude?: number`.** Considered: repurpose the unused `magnitude: number` field that's been sitting on `FindingEvidence` since WQ1 ("carried through for completeness") to hold the categorical bucket. Rejected — the field is a number, not a string union, and changing its type would silently break any caller that's been reading `evidence.magnitude` as a numeric (a grep showed no current readers, but the WQ1 doc string explicitly reserves it for "headline effect-size magnitude post-normalisation"). A new field keeps the existing semantic slot intact and lets a future wave wire the numeric channel in without colliding.
    - **`negligible` as a LOW hard-fail signal, not a tier downgrade.** Considered: have `negligible` only weaken HIGH-tier findings to MEDIUM (the conservative read). Rejected — the "significant but trivial" anti-pattern is precisely the failure mode this wave exists to catch. A finding that lands in MEDIUM with `n = 5000, p < 0.001, effect = negligible` would still earn the narrator's MEDIUM-tier verbosity budget (3 sentences) plus a soft hedge phrase — that's exactly the overclaim shape the verifier's WV3 short-circuit is trying to stop. Pushing all the way to LOW means the narrator's 2-sentence budget + the explicit "treat as directional" hedge are forced, which is the right framing for a finding that's statistically real but practically immaterial.
    - **`medium` effect contributes to HIGH, not just MEDIUM.** Considered: only `large` clears the HIGH gate; `medium` lives in MEDIUM. Rejected — Cohen's standard cutoffs for "medium" (d ≥ 0.5) and Cramér's V (≥ 0.3) already represent a substantively meaningful effect; demanding `large` (d ≥ 0.8 / V ≥ 0.5) would make the HIGH bar unreachable for many real-world findings even with strong p / n / CI. The HIGH bar stays achievable at `medium` effect; the `large` reason is additive when present.
    - **`small` effect drops to MEDIUM, not LOW.** Considered: treat `small` as a downgrade signal symmetric with the `negligible` hard-fail. Rejected — `small` effect (d ∈ [0.2, 0.5)) is the boundary case where the finding may still be actionable in the right context (high-leverage decisions on large populations where a small mean shift = meaningful dollar impact). Letting it land in MEDIUM with an explicit `"small effect size"` reason gives the narrator the option to hedge appropriately without forcing the floor.
    - **Optional trailing `effect = X` token in WV5 regexes rather than required.** Considered: pin the new token as required to lock in the contract. Rejected — the WV5 tests run the **real** significance-test math against real data, so the emitted bucket depends on what `bucketCohensD` / `bucketCramersV` produce. The 30-vs-30 +10 dataset in the welch_t fixture emits `large` today, but a future threshold tweak could shift it to `medium`. Making the token optional keeps the test invariant on "the suffix is well-formed", not on a specific bucket value that depends on indirect math.
