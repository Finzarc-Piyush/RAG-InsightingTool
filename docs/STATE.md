Does NOT enforce the format-coupling because `EditableText` validates fields in isolation; server's `safeParse` is authoritative on cross-field constraints. **Why `as const satisfies` for option lists** — TypeScript compile-time enum-drift catch (option-list typo = red squiggle, not runtime 400). **Why save-on-select with no validation step** — select values are constrained to the option list by construction; no analogue to "admin typed an invalid expression". **Why `__auto__` sentinel** — Radix rejects empty-string values + has no "clear selection" affordance; the sentinel lets the admin explicitly opt back into auto-derivation after pinning a grain by mistake. **Why clear orphan fields on enum-change** — the orphan surfaces in the planner manifest (W59's `formatMetricCatalog` reads `currencyCode` unconditionally when defined); the "preserve on round-trip" UX win is small (2-second retype), the misleading-manifest cost cascades downstream of every wrong-shape correction. **Why empty-accept on currencyCode validator** — server's authoritative cross-field rejection is the right place for "currency requires a code"; blocking the admin's mid-edit "I'm switching to currency, give me a second" would burn UX. **Why conditional render not always-disabled** — disabled inputs need explanatory text and clutter the cell; conditional matches the admin's mental model. 12 new behavioural tests in [`semanticModelEditValidationW61EditEnums.test.ts`](../client/src/pages/Admin/lib/semanticModelEditValidationW61EditEnums.test.ts): empty / whitespace / valid ISO-4217 (USD / INR / EUR) / lowercase / 2-char / 4-char / digits / mixed / leading-trailing-whitespace-tolerated / special-char paths for `validateCurrencyCode`. Enum option lists don't get standalone tests — `as const satisfies` is the compile-time contract. Test file appended to [`server/package.json`](../server/package.json)'s explicit list per invariant #4 (right after `semanticModelEditValidationW61EditText.test.ts`). All 12 pass; ~114 ms wave runtime. - **W61-edit-text** (2026-05-20) · Inline text editing for metric / dimension `label` / `description` / `expression` fields. First polish wave on top of the just-closed W61 trilogy: admins can now correct the most common inference gaps (an `expression` that says `AVG` when it should be `SUM`, a `label` that's still the raw snake-case column name, a missing-or-wrong `description`) directly in the per-session viewer with save-on-blur semantics. New pure module [`semanticModelEditValidation.ts`](../client/src/pages/Admin/lib/semanticModelEditValidation.ts) ships `validateLabel` / `validateDescription` / `validateExpression` / `isMeaningfulChange`. The expression validator rejects semicolons (multi-statement guard), SQL comments (`--` / `/*` could mask paren imbalance from the W58 compiler), and top-level `SELECT` / `FROM` / `JOIN` / `WHERE` / `GROUP BY` / `ORDER BY` / `UNION` keywords via a padded-uppercase scan (so `SUM(selector_count)` is NOT rejected — whitespace boundaries matter). Bounds mirror `semanticMetricSchema` (label ≤ 120, description ≤ 1000, expression ≤ 2000); the server's `semanticModelSchema.safeParse` is still authoritative — these are the cheap obvious-broken filter. [`AdminSemanticModelDetail.tsx`](../client/src/pages/Admin/AdminSemanticModelDetail.tsx) gains an `EditableText` sub-component sibling to `ExposedToggle` from W61-save: controlled `<Input>` / `<Textarea>` with save-on-blur, `Enter` blurs single-line, `Escape` discards, red border on validation fail, prop-driven draft sync via `useEffect([value])` so server reconcile lands. `MetricRow` widens with `onEditLabel` / `onEditDescription` / `onEditExpression`; `DimensionRow` widens with `onEditLabel` / `onEditDescription` (no expression — dimensions don't have one). Parent gains `patchMetric(metricName, patch)` / `patchDimension(dimensionName, patch)` narrow handlers that mapover the model and call the existing W61-save `persistModel` (optimistic update → PATCH → reconcile-with-server-stamps → roll-back-on-error). Small `emptyToUndef(s)` helper trims input and returns `undefined` for empty values so the round-trip matches the zod schema's `.optional()` shape (empty strings would re-emit and drift doc shape). **Why always-editable inputs over click-to-edit popovers** — admins fixing inference gaps flow through many fields per session; the dance adds state + an extra click per cell with no UX win. **Why save-on-blur over per-row Save button** — `withSessionWriteLock(sessionId, ...)` (invariant #9) already serializes sequential saves; a batch button adds dirty-state UX with no concurrency benefit. **Why padded-uppercase scan over regex `\b` boundaries** — JavaScript `\b` matches at underscores, which would let `_FROM_` slip through; the literal scan is byte-exact. **Why `isMeaningfulChange` via trimmed equality** — admins frequently add/trim trailing spaces; firing a PATCH on whitespace-only edits would burn write throughput AND pollute the W64 compiled-query cache key (every save bumps `version`). 25 new behavioural tests in [`semanticModelEditValidationW61EditText.test.ts`](../client/src/pages/Admin/lib/semanticModelEditValidationW61EditText.test.ts) using `node:test` (matches the WI6-persist client-side convention): empty / whitespace / 120 / 121 char paths for `validateLabel`; empty / 1000 / 1001 for `validateDescription`; empty / whitespace / valid SUM / NULLIF-ratio / semicolon / `--` / `/*` / SELECT / JOIN / WHERE / 2000 / 2001 / `SELECTOR` substring-non-rejection paths for `validateExpression`; identical / differing / whitespace-only / empty-vs-whitespace paths for `isMeaningfulChange`.
