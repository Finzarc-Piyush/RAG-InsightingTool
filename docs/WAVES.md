  **Why this design.**
    - **Context, not props.** `PremiumChart` is a generic dispatcher with a fixed `VisxRendererProps` shape (`{ spec, data, width, height, ariaLabel? }`). Threading `sourceTileId` through every renderer would (a) widen the dispatcher's prop shape, (b) require touching every renderer in this single wave, (c) re-litigate the "what props are universal" question that PremiumChart already settled. A small React context that the tile body wraps the dispatcher in keeps the dispatcher's prop shape stable and makes BarRenderer's wiring a 3-line addition. Future WD2-wiring-rest waves each get a similarly thin addition (`useDashboardTileContext` + `dispatchCrossFilter` + a one-line onClick branch).
    - **`useDashboardTileContext` returns `null` outside a provider â€” no fallback shim.** The existing ChartGrid context's `useChartGrid()` returns a no-op shim (`{ filter: null, setFilter: () => {}, toggleFilter: () => {}, applyFilter: rows => rows, inGrid: false }`) because every consumer's code path calls those methods unconditionally and needs them to be safe. The dashboard-tile context is the opposite shape: consumers check `!!dashboardTile` to decide whether to dispatch the event at all. A no-op shim with a dummy `tileId: ""` would be semantically wrong â€” dispatching `sourceTileId: ""` to `DashboardView` from a chat-side renderer would pollute the brushed-filter telemetry. Returning null forces the caller to consciously opt in on a non-null.
    - **The two click paths (`grid.toggleFilter` + `dispatchCrossFilter`) are independent on purpose.** They serve different consumers: `ChartGrid` is the explorer's multi-chart in-context filter (user clicks Region=North in one chart, the OTHER chart's rows reduce to North); `globalFilters` is the dashboard-wide filter bar (user clicks Region=North in tile A, tile B re-renders with Region=North applied). The user contract of each is different and the storage shapes are different (`ChartGrid` is a single (field, value), `globalFilters` is `Record<column, ChartFilterSelection>`). Don't unify; let both fire when the renderer is inside both containers.
    - **`toFilterValue(unknown)` is purely additive.** The narrower signature was overly conservative â€” chart-mark values flow as `unknown` from typed renderer internals, so every renderer wiring would have to narrow first if `toFilterValue` insisted on a narrow input. Widening the signature has no runtime cost (the function already used `typeof` checks internally) and reduces the surface every WD2-wiring-rest renderer must thread.
    - **Wave shape mirrors W59a â†’ W59b â†’ W60 and WD2 â†’ WD2-wiring-bar.** Each split lands a pure helper first (WD2's `crossFilter.ts`), then a thin call-site that wires the helper into one consumer (WD2-wiring-bar on BarRenderer). The thin-wiring waves stay reviewable; if a wiring decision turns out wrong (e.g. the `interactive` flag should ALSO consult an env flag) we change one line in one file, not 13.

  **Why this design.**
    - **Single MINI call, deterministic anchors.** The user clicks "âœ¦ Re-explain this view" on a dashboard tile expecting a 1â€“3 sentence narration of the slice â€” not a multi-turn investigation. The deterministic anchors from `summarizeFilteredData` carry the load (top / bottom / mean / row count + distinct x-values); the LLM just renders them in prose. MINI tier handles this well; PRIMARY would be 10Ã— the cost for no quality lift.
    - **Pure helpers first, network boundary last.** Five of the six exports are pure functions with no I/O: schemas, `summarizeFilteredData`, `inferConfidenceTier`, `extractInsightCitations`, `buildInsightRegenPrompt`. Only `regenInsightForFilteredView` touches `callLlm`. Tests cover each pure helper directly without spinning real LLM calls; the stub-resolver path tests the wiring + post-processing without leaving Node. This is the same pattern WI2-cache shipped with (pure cache helpers shipped before the network surface) and it keeps both surfaces atomically reviewable.
    - **Response shape matches `InsightRegenEntry`.** The client cache stores `{ text, citations?, regeneratedAt?, confidenceTier? }`. The endpoint returns `{ text, citations?, regeneratedAt, confidenceTier }` â€” same field names, response makes `regeneratedAt` + `confidenceTier` non-optional. The future WI2-wire hook will call `cache.set(buildCacheKey(tileId, hashGlobalFilters(filters)), response)` with no transformation. Designing the response shape FROM the cache shape (rather than the other way around) keeps the contract tight.
    - **No `promptCacheEligibility` wiring (yet).** The master plan's WI2-server note mentions reusing `promptCacheEligibility` to share the system prompt across tiles in the same session. Inspecting the server tree shows no `promptCacheEligibility.ts` source file (only a test fixture name); the prompt-caching path Anthropic exposes via `cache_control` markers isn't yet plumbed through `callLlm` for non-agentic surfaces. Skipping the explicit cache wiring keeps the wave atomic; the byte-stable system prompt naturally benefits from any future caching layer that hashes on `messages[].content`. Out-of-scope footnote below.
    - **Auth gate via the existing `getAuthenticatedEmail` helper.** Mirrors every other write route on this server (`feedbackController.ts`, `dataOpsController.ts`, etc.). 401 on missing email, 400 on schema fail, 500 on regen failure with telemetry-only error body (no internal stack leakage to the client).
    - **`installLlmStub` default handler returns a plain string.** The W18 stub harness passes strings through verbatim â€” the regen path expects `choices[0].message.content` as plain text, not JSON. Earlier handlers in the default map return objects which the stub `JSON.stringify`s; the regen handler is the second plain-string entry (after the prose-style ones). The W18 default-handlers-coverage test now passes because the new purpose has a handler; the W3.11 routing-regression test now passes because the new purpose has an `EXPECTED_CATEGORY` entry. Both tests are intentional gates the wave updates by design.
    - **`citations` is optional on the response, NOT always present as `[]`.** When the model emits no `[pack-id]` tokens, the response omits the field entirely (zod's `.optional()` accepts missing). The client cache's `InsightRegenEntry.citations?` is also optional. Saving the bytes avoids an `[]` rendering ambiguity on the client (an empty array might suggest "we tried and got nothing" while undefined means "no citations claimed"); the contract is "absent means none claimed".
    - **`text` length cap at 1200 chars.** The footer UI is space-constrained; longer prose would either wrap badly or get truncated by CSS. Capping at the response-schema level keeps the contract honest. `normaliseInsightText` slices to 1199 + appends `â€¦` so the cap is visible to the user when it fires.
