- **2026-05-18** — **Wave W59a · `formatMetricCatalog` pure helper (byte-stable manifest).** W56 (semantic schemas), W57 (`inferModel` upload-time auto-population), and W58 (`compileMetricQuery` `{metric, breakdownBy, filters}` → `QueryPlanBody`) have all been merged for two days, but the chain is dormant: the planner has no idea any of it exists. The semantic layer is the single biggest accuracy lever on the [1000x master plan](/Users/tida/.claude/plans/go-through-the-entire-partitioned-yao.md) Workstream 1 (Cube's paired benchmarks: ~66 % data-error reduction, ~300 % accuracy lift across frontier models), and W59 — the planner prompt rewrite — was the next wave. But `planner.ts` is 976 LOC and already heavily edited by recent waves (WW1 added the TOOL_ROUTER_HINT block; WQ2 added EXTERNAL_CLAIM_MARKERS; WW1+WW2 added PLANNER_CONFIDENCE_DIRECTIVE), so a single-wave rewrite risked entangling the prompt restructuring with the manifest-rendering logic. W59a splits the smaller atomic concern out first: a pure helper that takes a `SemanticModel` and renders a byte-stable markdown manifest suitable for inlining into the planner prompt (W59b), the admin UI (W61), and any narrator-side metric-citation block a future quality wave wires up. W59b becomes a thin call site that just stitches the manifest into the existing prompt scaffolding.

  **What landed.**
    - [server/lib/semantic/prompt.ts](server/lib/semantic/prompt.ts) (new, ~175 LOC). Pure function `formatMetricCatalog(model: SemanticModel, opts?: FormatMetricCatalogOptions): string`. No I/O. No date dependencies. Safe in tests and any runtime context.
      - `FormatMetricCatalogOptions = { includeHidden?: boolean; includeSectionHeadings?: boolean }` — both default `false / true` respectively. `includeHidden: true` is for the admin UI which lists everything including `exposed: false` drafts; the planner gets the visible-only catalog.
      - Empty-model branch returns a stable single-line `_(empty — no metrics, dimensions, or hierarchies declared; fall back to raw `execute_query_plan` against the dataset schema)_` block. Hidden-only models (every entry `exposed: false`) collapse to the same empty branch — the planner never sees the empty `### Metrics (0)` heading.
      - Populated models render: `## Semantic catalog (vN — <name>)` header, a single instruction line `Prefer execute_metric_query when the question maps to one of these metrics; fall back to execute_query_plan only when no metric matches.`, then three sections (Metrics / Dimensions / Hierarchies) each with `### <Section> (count)` heading + bullet block per entry. Empty sections render `_(none)_` so the per-section count + structural shape stay byte-stable across models.
      - Exported helpers `formatMetricLines(m)`, `formatDimensionLines(d)`, `formatHierarchyLines(h)` return the canonical line arrays per entry; callers that want to embed a single item in a different context (narrator-side citation expansion, admin tooltip) reuse the same byte shape without re-implementing the formatting rules.
      - Metric block: ` ``- `name` — label`` `, indented `Expression: `<expr>``, `Format: <hint>` (where `currency (INR)` is rendered when `format === "currency"` + `currencyCode` is set; `, N dp` is appended when `decimals` is set), `References: a, b, c` (only when `m.references.length > 0`), `<description (one-line collapsed)>` (only when present).
      - Dimension block: name + label header, `Column: `<col>``, `Kind: <kind>` with `(grain)` suffix when `kind === "temporal" && temporalGrain`, optional one-lined description.
      - Hierarchy block: name + label header, `Levels: country → region → city` (joined with ` → `), optional one-lined description.
      - Internal helpers: `visibleMetrics` / `visibleDimensions` clone-then-filter-then-sort so the caller's model is never mutated. `sortedHierarchies` clone-then-sort (no exposed flag on hierarchies). All sorts are `a.name.localeCompare(b.name)` ascending — stable under repeated identical inputs.
      - `oneLine(s)` collapses any internal whitespace (newlines + repeated spaces) to single spaces and trims, so a description authored as `"line one\n\n  line two"` renders as a single line in the manifest. Prevents accidental re-indentation from corrupting the markdown structure.

      - **Metrics section** (6): single currency metric renders code + label + expression + `currency (INR), 2 dp` hint + references line; alphabetical sort regardless of input order; `exposed: false` hidden by default, shown with `includeHidden: true` (count line bumps from `(1)` to `(2)`); empty references suppresses the References line; non-currency format omits the parenthetical code; multi-line description collapsed to single line.
      - **Dimensions section** (3): categorical kind has no grain suffix; temporal kind renders `(month)` when grain is set; alphabetical sort regardless of input order.
      - **Hierarchies section** (2): levels joined with ` → `; alphabetical sort.
      - **Byte stability** (2): two identical models produce identical output; same model with different input ordering produces identical output (key prompt-cache property — input ordering cannot perturb manifest bytes).
      - **Section heading toggle** (1): `includeSectionHeadings: false` removes `### Metrics` / `### Dimensions` / `### Hierarchies` lines but keeps the entry blocks.
      - **Direct helpers** (3): `formatMetricLines` / `formatDimensionLines` / `formatHierarchyLines` each return the exact expected line array (anchors the byte shape; any future formatting tweak will trip these tests).

  **Why this design.**
    - **Pure-function split.** `prompt.ts` is a 175 LOC pure helper with no `planner.ts` import. W59b's prompt rewrite is then a 1-line call site that asks `formatMetricCatalog(model)` and inlines the result. Splitting the concerns lets us unit-test the rendering exhaustively without standing up a full planner prompt fixture. The W59a/W59b atomic split also keeps `planner.ts` edits to a minimum diff for W59b, which preserves the prompt-cache breakpoints WW1+WW2+WQ2 have already established.
    - **Byte-stability is a prompt-cache contract.** Per Anthropic's 2026 prompt-caching guidance, the 5-minute TTL means cache only helps within a session — and a single character of drift in the cache prefix invalidates it. The manifest sits inside the planner prompt, which is read across every planner LLM call in a session. The sort + filter + line-shape contract here guarantees that two LLM calls in the same session see the exact same manifest bytes whenever the model is unchanged. Tests `produces identical output for two identical models` and `same model in different input order yields identical output` are the canary; if a future PR breaks them, the wave should be rolled back.
    - **Filter `exposed: false` by default.** The schema's `exposed: false` is the data team's way of marking a draft metric in the admin UI — the planner should never see it. Default `includeHidden: false` is the conservative path; admin UI flips it on. Same treatment for dimensions.
    - **Mutation-free.** `[...model.metrics].sort(...)` clones first. The semantic model lives on `ChatDocument.semanticModel` (Cosmos-persisted); mutating it during prompt rendering would be a silent bug.
    - **Hierarchies have no `exposed` flag.** The schema doesn't expose hierarchies as filterable by exposure — they're either declared or not. The helper reflects this: it sorts but does not filter hierarchies. Future schema waves can add a flag without changing the helper's external API (it would just plumb through `visibleHierarchies` mirroring `visibleMetrics`).
    - **`(none)` placeholder vs. dropping the section.** Even when one section is empty the helper renders `### Section (0)\n\n_(none)_\n` rather than skipping the section. This keeps the prompt structurally stable for the LLM — "the catalog has three sections, one is empty" reads better than "the catalog has two sections, where did the third one go?". And it makes the byte-stability contract simpler: same model → same byte structure, no conditional section presence.
    - **Empty-everything is a single line, not three empty sections.** When the model has zero exposed metrics / dimensions / hierarchies, the helper short-circuits to the parenthetical empty marker. The planner sees a clear "no semantic layer applicable here" signal rather than three empty `_(none)_` sections — the planner can fall straight back to `execute_query_plan` against the dataset schema.
    - **Currency hint as `currency (INR)` not `currency, INR`.** Lets `Format: currency (INR), 2 dp` parse left-to-right as `<kind> [<currency-code>] [, <decimals>]`. A future wave could parse this back out of the manifest for round-trip cache key hashing without ambiguity.

The helper does not currently enforce a cap — a pathological model with 200 metrics × 200 dimensions × 50 hierarchies (the schema maxes) could blow the budget. - **2026-05-18** — **Wave WQ3 · citation hover-cards in narrator prose (client).** W22's server-side anti-hallucination gate ([checkEnvelopeCompleteness.ts:120](server/lib/agents/runtime/checkEnvelopeCompleteness.ts)) guarantees that every backtick-wrapped domain-pack id in narrator prose (e.g. `` `marico-haircare-portfolio` ``) is a real pack from the supplied CONTEXT BUNDLE — no hallucinated citations can ship. But on the client, the custom [`MarkdownRenderer`](client/src/components/ui/markdown-renderer.tsx) was rendering those tokens as plain inline text indistinguishable from any other backtick code-span. Users had no visual signal that a particular claim was grounded in a specific domain pack vs. a generic backticked identifier (column name, acronym), so the citation work W22 enforces was effectively invisible. WQ3 surfaces the citations as superscript Radix hover-cards: when the renderer detects a backtick-wrapped pack id, it replaces the inline code with a numbered superscript pill that opens a hover-card showing the pack's humanised label, canonical id, and a pointer to the admin domain-context surface for the full pack body.

  **What landed.**
    - [client/src/lib/citationTokens.ts](client/src/lib/citationTokens.ts) (new, ~95 LOC). Pure detector mirroring W22's server-side `CITATION_TOKEN_RE` byte-for-byte: `` /`([a-z][a-z0-9-]{4,})`/g `` plus the same hyphen-rule heuristic (tokens without a hyphen are NOT treated as citations — filters `Volume_MT`, `Region`, etc.). Exports:
      - `CITATION_TOKEN_RE` — the regex itself, exported so a future test can pin parity with the server source on every change.
      - `extractCitations(text)` — splits a text fragment into alternating `{ type: "text", value } | { type: "citation", packId, raw }` segments. Empty input → empty array. Bytewise roundtrip property: concatenating each segment's `value` / `raw` reproduces the input. Fresh regex per call (avoids global-regex `lastIndex` bleed across invocations).
      - `listCitedPackIds(text)` — unique pack ids in first-occurrence order (for callers that need to enumerate without walking segments twice).
      - `formatCitationLabel(packId)` — humanises kebab-case (`marico-haircare-portfolio` → `Marico Haircare Portfolio`) for the hover-card header.
    - [client/src/components/CitationHoverCard.tsx](client/src/components/CitationHoverCard.tsx) (new, ~60 LOC). React component wrapping the existing Radix [`HoverCard`](client/src/components/ui/hover-card.tsx) primitive (already in the codebase — no new dep). Renders a small `[N]` superscript pill, dotted-underline + primary-tinted, with `data-citation-pack-id` for downstream selector / e2e use. The hover content is metadata-only: humanised label, canonical id (monospace), explanation block pointing users to **Admin · Domain Context** for the full body. `openDelay: 150ms` / `closeDelay: 100ms` matches the rest of the app's hover-card timings.
    - [client/src/components/ui/markdown-renderer.tsx](client/src/components/ui/markdown-renderer.tsx) edit (~70 LOC). Top of `MarkdownRenderer` builds a per-render `citationIndex: Map<packId, 1-based-index>` once via the new `buildCitationIndex` helper — repeated citations of the same pack get the same superscript number across the message. The index is threaded through `parseMarkdownLine` → `parseInlineMarkdown` → a new `renderTextWithCitations` helper. The latter splits plain-text segments via `extractCitations` and replaces each citation segment with a `<CitationHoverCard packId={packId} index={index} />` node. Citations inside `**bold**` and `*italic*` spans are also detected (the bold / italic processing now recursively defers to `parseInlineMarkdown`). When no citations are present, `renderTextWithCitations` short-circuits to return the input text unchanged so byte-stable rendering of citation-free prose is preserved.
    - Stable React keys: citation nodes use `key={`citation-${counter}-${packId}`}` so React's reconciler doesn't churn across re-renders. The `index` (superscript number) is computed from the citation order in the full message, not the index within a span — so a citation that appears in bold AND a re-citation in plain prose both show `[1]` and correlate visually.

  **Why this design.**
    - **Pure helper + thin React wrapper.** Splitting `citationTokens.ts` (pure) from `CitationHoverCard.tsx` (React) lets us unit-test the detector with `node:test` (server harness), no DOM required. The React component is tested implicitly via the integration path. Pinning the regex shape vs. the W22 server regex catches silent drift if either side is refactored.
    - **Reuse the existing Radix HoverCard.** The codebase already wraps `@radix-ui/react-hover-card` at [`client/src/components/ui/hover-card.tsx`](client/src/components/ui/hover-card.tsx). No new dependency, consistent visual treatment with other hover surfaces (e.g. `@/components/ui/tooltip`).
    - **Metadata-only hover content (no snippet body).** The admin endpoint `fetchDomainContextPacks` returns `{ id, title, category, priority, version, approxTokens, enabled }` — no pack body. Exposing the body would require either (a) a new public endpoint (server-side wave), (b) inlining pack snippets into the session envelope (heavier; potentially leaks content of disabled packs), or (c) a per-mount fetch from the client (admin-gated, would 401 for regular users). Atomic scope for WQ3 means surfacing the citation as a visual anchor + explanation; piping the body in is the natural follow-on wave (and the admin-gated approach is what an analytics user actually expects).
    - **Stable per-message numbering.** Computing the index map once at the top of `MarkdownRenderer` keeps numbering stable across re-renders of the same message and lets repeated citations of the same pack share a single visual identity. Computing per-line or per-span would assign `[1]`, `[2]`, `[3]` to three repeats of `marico-haircare-portfolio` — confusing.
    - **Recursion into bold + italic.** Pre-WQ3, citations inside bold (e.g. `**Per `marico-haircare-portfolio`, MT leads.**`) would not have been detected because `parseInlineMarkdown` only ran on non-bold spans. The recursion change is minimal (re-call `parseInlineMarkdown` on the bold/italic inner text instead of dropping it as a plain string) but means citations are always picked up regardless of formatting.
    - **Byte-stable for citation-free prose.** `renderTextWithCitations` returns `[text]` when no citations are present — the existing rendering path for prose without citations is unchanged. Guards against unintended formatting drift on every message; pinned via the test suite's "no citations → single text segment" case.

- **2026-05-18** — **Wave WQ7 · significance score by default on `run_two_segment_compare` + `run_breakdown_ranking`.** Extends the deterministic-floor evidence-carriage chain (WV4 correlation, WV5 significance, WV6 elasticity, WV7 segment-driver) to the two composite-style segment-comparison tools that had no inline statistical evidence on their summaries. Pre-WQ7, the `run_two_segment_compare` summary carried only the aggregated metric values for both segments + a mix ratio (`rate_ratio_A_per_B`), and `run_breakdown_ranking` carried only the ranked top-N rows — both had a meaningful per-row significance test available (the underlying row-level metric values are right there) but neither emitted p or n on the summary. WQ1 was therefore grading every segment-compare finding and every breakdown-ranking "the top region is the leader" finding as `medium / no evidence supplied`. WQ7 runs Welch's t-test by default on each tool's natural cohort split (A vs. B for the compare tool; headline group vs. all-others-combined for the ranking tool) and appends the canonical `(n = N; p = X)` suffix to the success-path summary so WW2's extractor catches the evidence and WQ1's tier classifier grades the finding on real numbers.

  **What landed.**
    - [server/lib/agents/runtime/tools/twoSegmentCompareTool.ts](server/lib/agents/runtime/tools/twoSegmentCompareTool.ts) edit (~50 LOC):
      - New imports: `runSignificanceTest` from [`../../../significanceTests.js`](server/lib/significanceTests.ts), `composeFindingDetail` from [`../formatFindingEvidence.js`](server/lib/agents/runtime/formatFindingEvidence.ts), `type FindingEvidence` from [`../scaleNarrativeByConfidence.js`](server/lib/agents/runtime/scaleNarrativeByConfidence.ts).
      - `aggregateSegment` extended to also return `values: number[]` — the row-level numeric values for the metric column inside the filtered segment, collected during the existing sum / mean / count pass. For `count` mode the values array is empty (skipped downstream). Previously these values were folded into `sum` and discarded; now they ride along for the t-test.
      - New helper `buildSegmentCompareEvidence(aggregation, valuesA, valuesB)` runs `runSignificanceTest({ test: "welch_t", sampleA: valuesA, sampleB: valuesB })` when `aggregation ∈ {mean, sum}` AND both segments have ≥3 finite obs, builds `{ pValue, n: nA + nB }`, returns `composeFindingDetail("", evidence)`. Skips on count aggregation (different test shape; proportion Z-test needs a global baseline the tool doesn't carry) and on insufficient n. All exceptions short-circuit to `""`.
      - Success-path summary concatenates `wq7EvidenceSuffix` at the end: `` `run_two_segment_compare: ${aggregation}(${metricColumn}) for "${A}" vs "${B}" (capped_input=${frame0.length}).\n${sample.slice(0, 4500)}${wq7EvidenceSuffix}` ``. When the suffix is empty (count / insufficient n / failure), summary is byte-identical to pre-WQ7.
    - [server/lib/agents/runtime/tools/breakdownRankingTool.ts](server/lib/agents/runtime/tools/breakdownRankingTool.ts) edit (~70 LOC):
      - Same import block.
      - New helper `buildBreakdownRankingEvidence(frame, breakdownColumn, metricColumn, topGroupLabel, aggregation)` does a second pass over the frame, splitting row-level metric values into the headline (top-ranked) group vs. all other groups combined. Runs `runSignificanceTest({ test: "welch_t", sampleA: topValues, sampleB: otherValues })`. Same skip rules as the compare tool (count / insufficient n). Returns the canonical suffix or `""`. The two-pass design — `aggregate` reduces sum/count per group first, then the helper makes a second values pass for exactly two buckets — keeps memory bounded: capturing values per group would explode on high-cardinality breakdowns (e.g. 5,000 SKUs × N rows of values arrays).
      - Simple-path return only — the composite (rankBy) branch is scope-bounded out. Composite ranking aggregates multiple metrics via a user-provided expression; "is the top group different on the *score*?" is ill-posed for a t-test (the score is a weighted combination, not a sample). Future waves can add a per-metric or per-component test if needed.
      - Headline group label resolved from `trimmed[0][breakdownColumn]` (the post-sort top row); `null` rows fall back to `"(null)"` to match the upstream `aggregate` key normalisation.
      - Success-path summary concatenates `wq7EvidenceSuffix` at the end of the simple-path return only.
    - Both tools share the same `runSignificanceTest` from [significanceTests.ts](server/lib/significanceTests.ts) (F3 / WV5). No new math dependency, no t-CDF / chi-square implementation; the WV5 helper already handles Welch–Satterthwaite df + Hill (1970) survival function for the t-distribution.

  **Why this design.**
    - **Cohort split that matches the tool's natural question.** Two-segment compare is literally about A vs. B — that's the t-test. Breakdown ranking implicitly stages a "leader vs. field" comparison — the headline finding the narrator's prose around the top row implies. Both splits are unambiguous; neither requires the planner to specify the comparison. The tool answers what its name promises.
    - **Welch's t-test on row-level values, not on aggregates.** A t-test on two `sum` values would have df=0 and zero variance — it's not a statistical test. A t-test on the underlying row-level values *is* a test: it asks "do the per-row metric values differ between segments meaningfully relative to within-segment variance?". This is the right question for both `mean` (where it's also the natural test of the aggregate) and `sum` (where it gracefully reframes the sum question as "are per-row magnitudes different?" — usually what the narrator's prose around the top row actually means).
    - **Skip count aggregation.** Count is the only aggregation where the row-level values aren't a meaningful sample (you'd be comparing "1, 1, 1, 1" vs. "1, 1, 1"). The natural test for "are the count proportions different from the global baseline?" is a Z-test for proportions or a chi-square, and both need a baseline the tool doesn't carry. Skipping is correct; future waves can add a `baseline` arg if needed.
    - **Top-vs-rest for breakdown ranking, not top-vs-second.** "Is region X the leader?" is the headline finding; the relevant comparison is "X vs. everything-not-X", not "X vs. the next region". The latter is a noisier signal (sample of one comparison column) that the narrator rarely cares about — the prose is "X leads with $5M, far above the rest" not "X leads with $5M vs Y at $4M".
    - **Two-pass over the frame, not values-per-group.** Capturing `values: number[]` per group inside `aggregate` would mean N × G memory for high-cardinality breakdowns. The second pass for exactly two buckets (top + other) is one extra O(N) scan — typically <1ms on the diagnostic slice cap (~5k rows) — and keeps the existing `aggregate` Map values cheap.
    - **Composite ranking explicitly excluded, asserted in source-inspection tests.** A future refactor could accidentally invoke `buildBreakdownRankingEvidence` from the composite branch; the test `composite (rankBy) path summary is unchanged (WQ7 scope-bounded)` guards against it.
    - **Empty-suffix on skip / failure is the contract.** Mirrors WV4-WV7. Empty `topValues` or `otherValues` (single-group frame) → empty suffix → byte-stable summary. NaN-only values → empty suffix. `runSignificanceTest` returns `ok: false` for nA/nB < 3 → empty suffix. Failure modes are silent and observable only by the absence of the canonical evidence block.

    - **`topN` cap returns nothing in numericPayload's `cappedTo` vs `totalRules`.** Analyst dashboards can show "showing top 50 of 1,247 rules". The cap doesn't truncate metadata.

- **2026-05-16** — **Wave W58 · Semantic-layer compiler.** The wave that makes Workstream 1 actually usable. W56 shipped types, W57 auto-populated the catalog at upload, but no consumer existed.   **What landed.** New [server/lib/semantic/compiler.ts](server/lib/semantic/compiler.ts) (~330 LOC including types + helpers). Two compilation paths based on expression shape:
    - **Simple single-aggregation metrics.** Expression is exactly one `<OP>(<col>)` or `COUNT(*)` or `COUNT(DISTINCT col)`. Emits one [`aggregationEntry`](server/lib/queryPlanExecutor.ts) under the metric's own name. Recognised operators: SUM, AVG, MEAN, MIN, MAX, COUNT, MEDIAN — mapped to the executor's `aggOpSchema` enum (`AVG` and `MEAN` both → `mean`; `COUNT(DISTINCT)` → `count_distinct`).
    - **Composite arithmetic metrics.** Expression like `SUM(gross_sales) - SUM(returns)` or `SUM(value_sales) / SUM(units)`. Compiler extracts every `<OP>(<col>)` occurrence via a global regex, dedupes by `(operation, column)` so `SUM(x) + SUM(x)` emits ONE aggregation, builds auto-aliases like `_sum_gross_sales` / `_sum_returns`, and rewrites the expression with the alias substitutions. The rewritten expression goes into one `computedAggregation` aliased to the metric name. Validates the rewritten expression against the same `ALLOWED_EXPRESSION_CHARS_RE` the executor uses — character mismatch returns a clear error rather than crashing the executor downstream.

  Breakdown dimensions and filter dimensions are looked up in the model's `dimensions` index; unknown names return a clear error listing the available options. Filters forward the `match` modifier (`exact` / `case_insensitive` / `contains`). Sort target can be the metric alias OR a breakdown dimension name; both resolve to the right column in the emitted `sort[]` entry.

  **Why this design.** Considered (a) emitting raw SQL strings instead of `QueryPlanBody` — rejected because that bypasses the existing W5 `contains` filter LIKE compilation, the W1 windowAggregations infrastructure, the W17 envelope-completeness gate, and the verifier's `numericPayload` re-check. Building on `QueryPlanBody` means every existing tool / validator / verifier works unchanged. (b) Letting the compiler emit arbitrary SQL inside `computedAggregations` — rejected because `computedAggregationSchema`'s character-class restriction is a defense-in-depth measure (admin-controlled expressions are auth-controlled but the executor still validates). The "NULLIF / CASE not supported yet" wall is intentional surface area for the future executor-schema widening wave.

  Dedupe of repeated aggregations matters more than it looks: ratio metrics like `SUM(value_sales) / SUM(category_value_sales)` and the inferred Marico FMCG model often pair a numerator with the *same* aggregation appearing in the denominator of a different metric. Dedupe keeps the plan minimal so the executor's grouping pass walks the data once per `(op, col)` instead of N times.
