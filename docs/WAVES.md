- **2026-05-19** — **Wave WI4-cache-key · brush region in the insight-regen cache key.** Closes a silent cache collision that shipped with the WI4 master-plan series: two brushes on the same `(chartId, filters)` but different sub-regions were sharing a single cache slot. The WI2-cache module keyed entries by `(tileId, hashGlobalFilters(filters))` — load-bearing-correct for the WI2 per-tile footer (which has no region dimension), but too coarse for the WI4 panel that fires `useInsightRegen` once per brushed sub-region. Symptom: brush a chart over `[Mar, Apr]`, see the regenerated prose; brush the same chart over `[Sep, Oct]` with no global-filter change, see the panel paint the `[Mar, Apr]` prose back from cache. No fix-up code at the call site was possible because the hook owned the cache key.

  **What landed.**
    - [`client/src/pages/Dashboard/lib/insightRegenCache.ts`](../client/src/pages/Dashboard/lib/insightRegenCache.ts) gains a `BrushRegion` type import from the WI4 foundation module and two new exports. **`hashBrushRegion(region: BrushRegion | undefined): string`** mirrors `hashGlobalFilters`'s contract — byte-stable per kind, deterministic across identical inputs, returns `""` when no region is supplied so non-brush call sites get a back-compat empty third segment. Three fixed serialisations: `numeric` → `n:<start>..<end>`; `temporal` → `t:<startMs>..<endMs>`; `categorical` → `c:<v1>|<v2>|…`. `buildCacheKey` widens to a third optional arg: `buildCacheKey(tileId, filterHash, regionHash?)`. When `regionHash` is undefined or empty, the third segment is **omitted entirely** — the return is byte-identical to the pre-wave 2-arg shape — so the WI2 footer's existing cached entries remain reachable through this widening.
    - [`client/src/pages/Dashboard/hooks/useInsightRegen.ts`](../client/src/pages/Dashboard/hooks/useInsightRegen.ts) accepts a new optional `brushRegion?: BrushRegion` arg on `InsightRegenArgs`. The cache-key `useMemo` recomputes from the triplet `(tileId, hashGlobalFilters(filters), hashBrushRegion(brushRegion))` and the deps array widens to `[tileId, filters, brushRegion]`. The `BrushRegion` type is imported from `../lib/explainSlice` — same direction the panel already crosses, so the import surface stays tidy.
    - [`client/src/pages/Dashboard/Components/ExplainSlicePanel.tsx`](../client/src/pages/Dashboard/Components/ExplainSlicePanel.tsx) threads `brushRegion: event?.region` into the hook call between `filters` and `cache`. Non-brush callers ([`ChartTileBody`](../client/src/pages/Dashboard/Components/ChartTileBody.tsx) for the WI2 footer) keep their 3-key call shape — TypeScript's optional-property inference makes the new field a no-op for them.
    - 17 new tests in [`client/src/pages/Dashboard/lib/insightRegenCache.test.ts`](../client/src/pages/Dashboard/lib/insightRegenCache.test.ts) across three new describe blocks: **hashBrushRegion byte stability** (undefined → `""`, three-kind serialisation pins, categorical-order preservation as a NEGATIVE pin against an accidental sort, three-kinds-with-same-bounds disambiguation, kind-collision pins); **buildCacheKey backwards compat** (2-arg call, 3-arg-empty call, 3-arg-undefined-via-`hashBrushRegion(undefined)`, 3-arg-non-empty appended); **buildCacheKey collision avoidance** (numeric × numeric, temporal × temporal, categorical × categorical all on the same tile + filters → distinct keys + distinct cache slots; region-bearing vs no-region on the same tile + filters → distinct keys; identical regions → shared cache hit). Plus 3 new source-inspection pins in [`client/src/pages/Dashboard/hooks/useInsightRegen.test.ts`](../client/src/pages/Dashboard/hooks/useInsightRegen.test.ts) (`BrushRegion` import pin, hook destructures `brushRegion` pin, widened cacheKey `useMemo` shape pin) and 1 new pin in [`client/src/pages/Dashboard/lib/wi4Wire.test.ts`](../client/src/pages/Dashboard/lib/wi4Wire.test.ts) (`brushRegion: event?.region` thread-through). Two pre-existing pins broadened: the import-list regex in `useInsightRegen.test.ts` now sequences `hashBrushRegion` between `createInsightRegenCache` and `hashGlobalFilters`; the `useInsightRegen` call-shape pin in `wi4Wire.test.ts` now sequences `brushRegion: event?.region` between `filters: event?.filters ?? {}` and `cache: insightRegenCache`. No new test file — the new tests appended to an existing file already in [`server/package.json`](../server/package.json)'s explicit `test` list, so the invariant-#4 ledger is unchanged.

  **Why this design.**
    - **Pure-additive backwards compat over a key-version migration.** The simplest fix would be a required `buildCacheKey(tileId, filterHash, regionHash)` with `""` as the no-region sentinel. That breaks every existing 2-arg call site and any persistence layer that holds a pre-wave key (none today, but the WI2 footer has been emitting these keys for several waves and will keep doing so). Making the third arg optional + treating `""` as "omit the segment" keeps the wave purely additive — the WI2 footer's keys are byte-identical pre- and post-wave, and the empty-string sentinel is the natural output of `hashBrushRegion(undefined)` so the call surface composes cleanly.
    - **`hashBrushRegion` lives alongside `hashGlobalFilters`, not in `explainSlice.ts`.** The hash is a cache concern, not a region-data concern. Pairing it with `hashGlobalFilters` keeps all the byte-stability contracts in one file (so a future maintainer chasing "why does this entry not de-dup" finds both helpers together) and lets the existing test file extend instead of spawning a new one. The trade-off is a new type-only import of `BrushRegion` from `explainSlice` — accepted: `BrushRegion` is the canonical name + the type-only import has no runtime cost.
    - **Categorical value order preserved (not sorted).** A negative test pin catches a future drift to sorting. The semantics: a categorical region's `values` array comes from the renderer's `xs.slice(i0, i1)` band-scale slice, which is itself ordered. The dispatched order is fixed upstream by `distinctOrdered(data, enc.x.accessor)` — same band-scale domain source the renderer uses to position bars/lines — so identical brushes always yield identical value arrays without sorting. Sorting at hash time would obscure a genuine "different-brush-shape" signal (a hypothetical future renderer that emits values in a different order would alias regions that should differ).
    - **Three-segment string key, not a structural key type.** Considered: switch the cache `Map<string, …>` to a `Map<{ tileId, filters, region }, …>` and write a structural comparator. Rejected — would slow `get` from `O(1)` string lookup to `O(n)` structural walk over every cached entry, and would force a `Map → custom` wrapper for the LRU eviction (currently relying on `Map`'s insertion-order semantics for free LRU). Keeping the key a stable string preserves every existing behavior of the LRU + TTL cache: get / set / has all stay `O(1)`, `keys().next().value` for eviction stays the cheapest-oldest, the test surface keeps the same shape.
    - **Cache slot for the WI2 footer + WI4 panel do NOT share by design.** Pre-wave they could (incorrectly) — a footer key `tile_X::filters` collided with a panel key on the same `(tile, filters)` regardless of region. Post-wave they live in distinct slots (footer = 2 segments; panel = 3 segments). This is the correct outcome — the footer regenerates the WHOLE-chart insight, the panel regenerates the BRUSHED-region insight; sharing the same cache slot was the bug. The WI4-wire wave's comment ("a brushed slice can hit a cache slot populated by the per-tile Re-explain if filter hashes match") was aspirational and never true in practice; the footer and panel produce different prose under the hood (the panel narrows rows via `filterRowsByBrushRegion` before POSTing). Documented as a deliberate slot split in the new tests' "region-bearing brush does NOT collide with the no-region (footer) entry" pin.

  **Tests.** All 92 tests across the three affected files green: `insightRegenCache.test.ts` (~38 cases pre-wave + 17 new = 55); `useInsightRegen.test.ts` (~13 cases pre-wave + 3 new); `wi4Wire.test.ts` (~22 cases pre-wave + 1 new). 4253 server-side tests across 1086 suites — 4249 pass, 4 fail (the same pre-existing ECharts treemap/sunburst carry-forwards; unrelated to this wave). Client `npx tsc --noEmit` reports 53 errors, identical to the WI4-wiring-bar baseline.

  **Conventions reinforced.** Backwards-compat invariant on cache key shapes: a key helper that widens its signature for new optional segments must preserve byte-identical output for the no-segment path, so persisted keys from earlier waves remain reachable. Documented in the function's docstring; the new pin set in `insightRegenCache.test.ts` ("2-arg and 3-arg-empty calls produce byte-identical keys (WI2 footer compat)") is the regression guard.

  **Out of scope.** "Re-explain" button in the panel (~40 LOC dormancy debt) — separate wave, mirrors the WI2 footer's bypass-cache button. Citation chips in the panel (~50 LOC) — separate wave, mirrors the WI3 `Sources:` row. Box-2D brush for scatter (~150 LOC) — requires PointRenderer wiring + a 4th `BrushRegion` kind + a `box2d` branch in `hashBrushRegion`; the dispatch table is OPEN for this extension (the `switch` is over a discriminated union, so adding a fourth variant would surface a TypeScript exhaustiveness error pointing at the right place). Persisting cache slots to localStorage (would need a key-version bump on top of the new shape) — deferred until a need surfaces.

- **2026-05-19** — **Wave WI4-wiring-bar · BarRenderer alt-drag → explain-this-slice.** Closes the WI4 widening series at 3 of 3 brush-capable kinds (Line + Area + Bar). BarRenderer had no pre-existing brush mechanics — only per-bar `onClick` handlers powering WD2 cross-filter + WD3 drill-through — so this wave adds the mouse-down / move / up state + a `<rect>` overlay from scratch, mirroring the WI4-wiring-area shape. The wave is intentionally narrow: categorical-only (Bar charts use a band scale on `enc.x.field`, never a temporal axis in the WI4 sense), no brush-to-zoom (Bar charts have no quantitative axis to zoom into — the categorical axis would just rearrange labels). The first orientation-aware wiring in the WI4 family: vertical bars brush horizontally along the x-axis; horizontal bars brush vertically along the y-axis. The categorical-axis math collapses both orientations into a single 1D brush via three derived locals (`isVertical` / `brushAxisSize` / `brushAxisOffset`) so the handlers stay orientation-agnostic.

  **What landed.**
    - [`client/src/lib/charts/visxRenderers/BarRenderer.tsx`](../client/src/lib/charts/visxRenderers/BarRenderer.tsx) imports four helpers from `@/pages/Dashboard/lib/explainSlice` — `BRUSH_MIN_PX`, `dispatchExplainSlice`, `isBrushDrag`, `makeCategoricalRegion`. The `makeTemporalRegion` and `makeNumericRegion` constructors are DELIBERATELY NOT imported because Bar charts have no temporal / numeric x-axis in the WI4 sense (categorical-only). Negative pin in tests catches a future drift.
    - New brush state declared right after the WD2-dim block: `brushStart` / `brushEnd` useState (pixel coords on the categorical axis) + `brushExplainRef` useRef (alt flag, ref because it doesn't drive re-render). The React hook quartet (`useEffect, useMemo, useRef, useState`) was already imported pre-wave for prior renderer concerns; no widening needed.
    - `useEffect(() => { setBrushStart(null); setBrushEnd(null); }, [data])` resets the brush when the underlying data changes — same shape as WI4-wiring-area's data-change reset.
    - Three orientation-aware locals derived once per render:
      ```ts
      const isVertical = orientation === "vertical";
      const brushAxisSize = isVertical ? innerWidth : innerHeight;
      const brushAxisOffset = isVertical ? MARGIN.left : MARGIN.top;
      ```
      The categorical axis is x in vertical mode (uses `innerWidth` / `MARGIN.left`) and y in horizontal mode (uses `innerHeight` / `MARGIN.top`). All three handlers below reference these locals — the orientation discrimination lives in one place, not duplicated per-handler.
    - Three new handlers (`onBrushDown` / `onBrushMove` / `onBrushUp`):
      - `onBrushDown` reads `localPoint(e)`, computes `coord = (isVertical ? pt.x : pt.y) - brushAxisOffset`, guards against out-of-bounds (`coord < 0 || coord > brushAxisSize`), sets both `brushStart` and `brushEnd` to the same coord, captures `e.altKey === true` into `brushExplainRef.current`.
      - `onBrushMove` short-circuits on `brushStart === null` or `!(e.buttons & 1)` (no active drag), computes the same `coord` expression, clamps to `[0, brushAxisSize]` and updates `brushEnd`.
      - `onBrushUp` reads the lo/hi pair (Math.min/max so reverse drags work), gates on `isBrushDrag(brushStart, brushEnd, BRUSH_MIN_PX)`. Click branch (sub-threshold drag) resets state and yields to the per-bar `onClick`. Drag branch gates explain dispatch on `brushExplainRef.current && dashboardTile`. Inside the explain branch, the i0/i1 derivation uses `brushAxisSize` (not innerWidth/innerHeight inline) so the math stays orientation-agnostic: `i0 = Math.max(0, Math.floor((lo / brushAxisSize) * xValues.length))`, `i1 = Math.min(xValues.length, Math.ceil((hi / brushAxisSize) * xValues.length))`. `xValues` is the renderer's pre-existing `distinctOrdered(data, enc.x.accessor)` memo. `makeCategoricalRegion(xValues.slice(i0, i1))` produces the region; `if (region)` guards against an empty slice constructor return. Dispatch payload is the 5-field `{ chartId, column: enc.x.field, region, sourceTileId, filters: dashboardFilters }` shape — `column = enc.x.field` is symmetry-pinned with the WD2 cross-filter + WD3 drill-through dispatches that already key on the same axis. State + ref reset on all paths.
    - The svg element gains four new handlers gated on `dashboardTile`: `onMouseDown={dashboardTile ? onBrushDown : undefined}`, `onMouseMove`, `onMouseUp`, and a WIDENED `onMouseLeave` that now does TWO things: the pre-existing `hideTooltip()` call AND a brush state clear if `brushStart !== null` (safety net for cursor-leaves-svg-mid-drag). The pre-wave shape was `onMouseLeave={hideTooltip}` — the function-reference form is replaced with an arrow that runs both.
    - Cursor style added to the svg element: `style={brushStart !== null ? { cursor: isVertical ? "ew-resize" : "ns-resize" } : undefined}` — the per-bar `cursor: pointer` style overrides this on the bars themselves, so the svg-level cursor only shows through on the gridlines / axis area, but the brush rect itself provides the dominant visual feedback during a drag.
    - Brush `<rect>` overlay inserted inside `<Group>` right before the existing zero-line. Orientation-aware shape: vertical bars get a full-height column whose horizontal span is the brushed x-range (`x = Math.min(brushStart, brushEnd)`, `y = 0`, `width = Math.abs(brushEnd - brushStart)`, `height = innerHeight`); horizontal bars get a full-width band whose vertical span is the brushed y-range (`x = 0`, `y = Math.min(...)`, `width = innerWidth`, `height = Math.abs(...)`). Same primary tint / dashed stroke styling as LineRenderer + AreaRenderer (primary at 0.1 fill + 0.4 stroke + dashed `3 3`) + `pointerEvents="none"` so the overlay never intercepts the mouseUp / click.
    - Per-bar `onMouseMove` tooltip handler gains a single-line guard at the top: `if (brushStart !== null) return;`. Without this guard, the tooltip would flicker between bars as the cursor drags across the chart, fighting the brush rect for the user's visual attention. The brush rect is sufficient feedback during a drag.
    - Pre-existing per-bar `onClick` handler preserved byte-identical — the WD3 drill-through branch (`if (dashboardTile && isModifierClick(event)) { dispatchDrillThrough({...}); return; }`), the chat/explorer grid branch (`if (grid.inGrid) { grid.toggleFilter({...}); }`), and the dashboard cross-filter branch (`if (dashboardTile) { dispatchCrossFilter({...}); }`) all stay untouched. The brush handlers and per-bar click handlers co-exist: a clean click fires `mouseDown → mouseUp (small distance, reset, return) → click (per-bar dispatch)`; an alt-drag fires `mouseDown → mouseMove* → mouseUp (drag ≥ BRUSH_MIN_PX + alt held → dispatchExplainSlice)` and the browser doesn't fire `click` after a drag.

  **Why this design.**
    - **Categorical-only.** Bar charts always use a band scale on a categorical x-axis (or y-axis in horizontal mode); there's no temporal or quantitative brush target. Importing only the 4 helpers needed (`BRUSH_MIN_PX` / `dispatchExplainSlice` / `isBrushDrag` / `makeCategoricalRegion`) and pinning a NEGATIVE test against the temporal / numeric constructors makes the scope explicit — a future maintainer reading the imports immediately sees that Bar is categorical-by-construction.
    - **No brush-to-zoom.** Zoom on a Bar chart would have to rearrange the categorical axis labels — there's no "zoom into a sub-range of the value axis" gesture that makes sense for bars (the value scale auto-fits to the data). Even if there were, the disambiguation complexity (plain drag = zoom vs alt drag = explain) would surface more confusion than value. Same scope decision as WI4-wiring-area; mirroring it across the categorical family keeps the user model coherent.
    - **Orientation-collapsed via three derived locals.** Three alternatives considered:
      1. **Two parallel handler sets** (one for vertical, one for horizontal) — would duplicate ~50 LOC and any future bug fix would have to land in both copies.
      2. **`if (isVertical) {...} else {...}` branching inside each handler** — would scatter the orientation discrimination across three places, making the handlers harder to read.
      3. **Three locals at the top + handlers reference them** (chosen) — the orientation discrimination lives in one block; handlers read `brushAxisSize` etc. without caring about orientation. Tests pin the locals explicitly so a future "simplification" that inlines them breaks loudly.
    - **`xValues.length` as the brush domain denominator.** `xValues = distinctOrdered(data, enc.x.accessor)` is the renderer's canonical pre-existing source for the band scale's domain. Reading it for the brush math (instead of e.g. `outerScale.domain().length`) keeps the brush in lockstep with the visible categorical axis — if the renderer ever changes the band scale's domain (e.g. to exclude zero-value categories), the brush automatically follows.
    - **Per-bar tooltip suppression during active brush.** Without this guard, the user dragging across the chart sees the tooltip flicker between bars on every mouseMove fired on a bar. The brush rect is the primary visual feedback during a drag — the tooltip just noise. A one-line `if (brushStart !== null) return;` at the top of per-bar onMouseMove is the minimum-invasive fix.
    - **Extended onMouseLeave (not a new handler).** Pre-wave the svg had `onMouseLeave={hideTooltip}`. Adding a SEPARATE onMouseLeave for brush cleanup would conflict (React only allows one). Widening the existing arrow to do both is the minimum-invasive shape; the function-reference form is replaced with an arrow body that runs hideTooltip() AND the brush state clear conditionally on `brushStart !== null`.

  **Why this design.**
    - **Alt-drag for explain (NOT shift / NOT cmd / NOT a separate keyboard shortcut)**. Three modifier-key options were considered:
      1. **Shift-drag**. Conflicts with browser-native text-selection-extends semantics on many platforms; shift is also conventionally reserved for additive multi-select in dashboards (a future WI5 wave might want it for "explain multiple slices in parallel").
      2. **Cmd/ctrl-drag**. Already owned by WD3 drill-through on a single click; layering a brush variant on the same modifier creates a confusing two-distance regime (cmd-tiny-drag = drill, cmd-real-drag = explain) where the user can't predict which fires without precise mouse control. Keeping the modifiers semantically distinct lets the user reason: cmd = drill, alt = explain, plain = zoom.
      3. **Alt-drag** (chosen). Underused on most platforms; option-drag on macOS doesn't conflict with system gestures; aligns with the de-facto "secondary action modifier" convention in design tools (Figma, Sketch use alt for hold-to-show-distance / hold-to-copy on drag).
    - **Two independent refs (`brushModifierRef` + `brushExplainRef`) instead of one combined flag**. The two intents are orthogonal: cmd-tiny-drag fires drill (no brush), alt-real-drag fires explain (brush). A combined flag would force a tri-state enum (`none | cmd | alt`) and an ordering question (cmd + alt held simultaneously → which wins?). Independent refs let the brushUp logic check them in priority order (explain first because it's the "wider" intent — a real drag — vs. drill's tiny-drag click path) without an explicit enum.
    - **Branch BEFORE the zoom path, not AFTER**. Putting the explain branch after the zoom branch would require the zoom branch's `setZoomRange` to short-circuit if `brushExplainRef.current` — extra coupling. Putting it before with an early `return;` keeps the zoom branch's existing structure intact AND makes the new wave a strict additive change (zoom-path code is unchanged byte-for-byte from the WD3-wiring-rest-trend wave).
    - **Reuse the existing zoom math for region computation**. The temporal interpolation (`domMin + (lo / innerWidth) * (domMax - domMin)`) and the categorical index derivation (`Math.max(0, Math.floor((lo / innerWidth) * xs.length))` / `Math.min(xs.length, Math.ceil((hi / innerWidth) * xs.length))`) are byte-identical to the zoom branch. This is intentional: alt-dragging from index 5 to 12 brushes the SAME range that plain-dragging would zoom to. The user's mental model — "I select what I want, then choose the verb (zoom or explain)" — survives the wave.
    - **`xs.slice(i0, i1)` end-exclusive (not `i1 + 1`)**. The existing zoom branch uses `i1` as the ceiling index of `xs.slice(Math.max(0, Math.floor(zoomRange[0])), Math.min(xs.length, Math.ceil(zoomRange[1]) + 1))`. The brush region path uses `xs.slice(i0, i1)` — semantically the same SET of values because `i1` is already `Math.ceil(...)` (one past the last index). This pins the brush region and the zoom range to identical value sets.

  **Why this design.**
    - **Pure-fn module mirroring [`drillThrough.ts`](../client/src/pages/Dashboard/lib/drillThrough.ts)** — the WI4 brief explicitly said "mirror crossFilter.ts / drillThrough.ts shape". The three modules now form a coherent family: each exports an `EVENT` name, an `Event` interface, a `dispatch*` SSR-safe function, and a small set of pure helpers. Renderers and the receiver DashboardView both read all three in parallel, so structural symmetry beats per-feature ergonomic tuning.
    - **Discriminated union over generic `{ start, end }`** — three alternatives considered:
      1. **Single `{ start: unknown; end: unknown; values?: string[] }`** — degenerate type information; the consumer has to type-narrow on every read. The brushed region's KIND is load-bearing (numeric vs temporal vs categorical) — different filter predicates apply per kind — so encoding that in the type is the right call.
      2. **Per-renderer event types** (`NumericBrushEvent` / `TemporalBrushEvent` / `CategoricalBrushEvent`) — too narrow. The receiver dispatches a single panel UI regardless of region kind; three event types would force three listeners.
      3. **Discriminated union on `kind`** (chosen). One event listener, one `filterRowsByBrushRegion` call, kind-aware predicates inside. Future region kinds (e.g. `box-2d` for scatter brushes) add a `kind` member without breaking existing consumers.
    - **Millis-since-epoch on `temporal`, not `Date`** — the receiver hashes the region for cache keys and debouncing; `Date` objects break `JSON.stringify` symmetry (`new Date(0)` → `"1970-01-01T00:00:00.000Z"` is fine, but cyclical references / non-serialisable methods around `Date` complicate downstream piping). Millis is a primitive — trivially serialisable, trivially hashable.
    - **Constructor helpers (not direct unionconstruction)** — `makeNumericRegion` / `makeTemporalRegion` / `makeCategoricalRegion` enforce normalisation AND zero-width rejection in ONE place. Without them, every per-renderer wiring would re-implement `Math.min(a, b)` / `Math.max(a, b)` and the zero-width guard, drifting over time. The helpers are tiny enough that the consolidation doesn't add real overhead.
    - **`filterRowsByBrushRegion` as part of the foundation, not the panel** — the brief positioned slice-filtering as the panel's job. But the predicate logic is pure data-transformation with no UI dependency, AND symmetry with `crossFilter`'s `isCrossFilterActive` / `applyCrossFilter` argues for co-locating predicates with their event definitions. WI4-panel will compose `applyChartFilters(rows, filters)` → `filterRowsByBrushRegion(…, region)`; the pure helper here keeps that composition trivial.
    - **Why `Number(null) === 0` matters** — caught by tests. A row with `{ x: null }` looks like a missing observation, but `Number(null)` silently returns `0`, which falls inside any `[0, N]` brush range and corrupts the slice. The fix is a 4-character `v === null` short-circuit before coercion. Pinning the test against this regression locks in the behavior across future refactors.

Parallel coordinates, choropleth (stub), and gauge stay unwired — they have no categorical filter target, same invariant as KPI / Radar / Regression on the visx side.

  **Why this design.**
    - **One generic prop on EChartsBase, not 5 specialty wrappers.** The natural alternative would be to introduce a per-renderer `EChartsCrossFilterWrapper` that injects `useDashboardTileContext` + `dispatchCrossFilter`. But that doubles the renderer count and risks divergence between marks. Putting the prop on EChartsBase keeps the base shell mark-agnostic (it just forwards the click) and pushes the `params`-shape decoding into each renderer where the encoding-specific knowledge already lives.
    - **Ref-tracking instead of off-then-on.** ECharts' mount-effect runs once with empty deps (the existing pattern), so a closure over `onChartClick` would capture the initial value forever. Two fixes: (1) add `[onChartClick]` to the mount-effect deps — but then a callback identity change would re-create the entire ECharts instance, losing animation state and re-running the canvas init; or (2) bind once with a ref that's updated on every render — single bind, no re-init, always-fresh callback. Option 2 is the standard React idiom for this "external system needs the latest callback" shape.
    - **`inst.on` after `setOption`, not before.** ECharts validates event subscriptions against the chart's series types at bind time. Calling `inst.on('click', ...)` before `setOption` is a no-op for the eventual click events. Pinned by test via source-index ordering (`indexOf("inst.setOption") < indexOf("inst.on")`).
    - **Leaf-only dispatch on hierarchical marks (treemap / sunburst).** A treemap with a `groupCh` shows two levels — parent groups and leaf items. A click on a parent group could plausibly dispatch on the group's field, but then the dispatch column flips between two clicks of the same mark, which is hostile UX. Skipping parent clicks and only dispatching on leaves means every dispatch from a treemap targets the same column — `labelCh.field`. Two-column dispatch (mirroring the RectRenderer row+col approach from WD2-wiring-rest-rect) is a follow-on if a use case appears.
    - **Source-only dispatch on sankey.** The natural alternative is "click a node, dispatch on the column the node belongs to" — but ECharts' params shape doesn't carry that information directly; it would need a side-table lookup. In practice every sankey use case at Marico's scale has same-dimension source / target (e.g. `Channel` → `Channel`, `Brand` → `Brand`), so dispatching on `sourceCh.field` covers both endpoints. Cross-column sankeys (e.g. `Brand` → `Channel`) would need the richer decode — flagged in the comment.
    - **Candlestick reads `xs[dataIndex]` not `params.value`.** ECharts' candlestick params.value is the 4-tuple `[open, close, low, high]`. None of those are the categorical key (the bar's x-axis label). The renderer already built an `xs` array for the x-axis category data — the same array the chart uses for axis labels. Using `dataIndex` to look up the matching `xs` entry is the canonical way to get the categorical key from an ECharts axis-bound series.
    - **Calendar dispatches ISO date string verbatim.** `applyCrossFilter` already handles date columns as categorical selections (discrete value lists). The date string `yyyy-mm-dd` is the canonical wire format the renderer built; round-tripping it through `Date.parse` + re-format would risk a TZ-offset bug. Pass the string straight through.
    - **`onChartClick={dashboardTile ? handler : undefined}` for every renderer.** A `useCallback(() => {...}, [dashboardTile, ...])` always returns a function — even when `dashboardTile` is null. Without the conditional ternary, the EChartsBase mount effect would bind `inst.on('click', ...)` unconditionally, which means a click in a chat / explorer / share-preview surface would invoke the handler, hit the `if (!dashboardTile) return` guard, and no-op. That works but is wasteful — the cleaner shape is to pass `undefined` when there's no consumer, so the bind doesn't happen at all. Mirrors the visx renderers' `interactive` flag pattern.

  **Out of scope.**
    - **Parallel coordinates dispatch.** Each parallel-axis line crosses every dim; clicking the line could plausibly dispatch on the highest-variance dim, but that's heuristic and the user can't see what'll happen before clicking. Deferred until a concrete UX motivates a per-line dispatch design. Test pins the absence to prevent regression.
    - **Choropleth dispatch.** The renderer is a stub — it falls back to an info card explaining that geographic data registration is required. When `echarts.registerMap('world', geoJson)` lands in a future wave + the renderer becomes real, dispatching on the region field will be a straightforward addition.
    - **Gauge dispatch.** A gauge is a single-value KPI; there's no categorical click target. Same invariant as KpiRenderer on the visx side. Test pins the absence.
    - **Two-column dispatch on treemap / sunburst parent rings.** Mirrors what RectRenderer does for row+col on heatmaps (WD2-wiring-rest-rect). For treemap / sunburst this would dispatch on the group field when the parent ring is clicked and on the label field when the leaf is clicked. Two clicks of the same mark dispatching on different columns is borderline-hostile UX, but a small follow-on could carry both columns and dispatch both (toggle both) per click. Defer until a use case demands it.
    - **Edge clicks on sankey.** An edge connects a source node to a target node; dispatching on both columns at once is the natural two-column shape (same as the heatmap pattern). But sankeys with same-dimension source / target would dispatch the same column twice. Without a use case for cross-dimension sankeys, deferred.
    - **`isCrossFilterActive` opacity dimming on ECharts marks.** The visx-side dormancy-debt item (renderers dim non-matching marks when a cross-filter is active) applies equally to ECharts. ECharts series have a `select` / `emphasis` mechanism that could carry the dim effect via the dataItem's `itemStyle.opacity`. Atomic follow-on, ~80 LOC across the 5 wired renderers.
