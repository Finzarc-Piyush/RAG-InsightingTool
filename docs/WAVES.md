  **Why this design.**
    - **Direct `fetch + useState` rather than TanStack Query.** The session brief proposed `useQuery({ enabled: tabOpen })` because that's the natural shape for "fetch when the tab opens", but the host component uses direct `useEffect + useState` throughout (the existing detail-fetch effect at line 633-670 is the same pattern). Introducing TanStack Query mid-component would be a stylistic discontinuity without a behavioural payoff ŌĆö the audit log is fetched at most once per page visit (gated by `historyOpen`) and refreshed on save-bump. A future broader-scope refactor that converts the whole admin viewer to TanStack Query is a separate consideration; that refactor would also restructure the PATCH path's optimistic-update flow, which is out of scope here.
    - **`historyOpen` closed by default.** Open by default would fetch the audit log on every detail-page mount even though most visits are "look up something, leave" without consulting history. Closed-by-default is opt-in: the fetch happens only when the admin explicitly clicks the trigger. The closed state is on the component (not URL-synced) because the deep-link use case ("share me the open history view") is speculative ŌĆö can promote to a `?history=open` toggle later if a user-driven need appears, but adding two more places that read/write search params now would just be friction.
    - **Per-row `reverting: number | null` state, not a global boolean.** A list of up to 10 entries with a single `reverting: boolean` would disable every row's Revert button while one is in flight. With the per-index flag, all other rows remain readable AND queue-able (a re-entry guard on the handler still prevents the second click from launching a second revert, but the admin can read the others while waiting). The server's `withSessionWriteLock` (invariant #9) makes any concurrent attempt safe ŌĆö only one mutation lands per session at a time; the others see fresh state on retry. Tradeoff acknowledged: a Revert click while another is in flight is silently dropped by the handler's guard rather than queued; this is fine because the admin almost always wants to inspect the result of the first revert before deciding the second.
    - **Effect dep on `lastUpdatedAt`, not `data`.** React's dep-array compares by reference; `data?.lastUpdatedAt` is a number (or undefined), comparable by value, and changes exactly on save or revert. Using `data` directly would re-trigger the effect on every `setData` even when only an optimistic toggle bumped ŌĆö and the optimistic-toggle path on the PATCH inside `persistModel` rebuilds the `data` object reference but keeps `lastUpdatedAt` until the server response. Extracting `lastUpdatedAt` to a local variable before the effect spares the re-render storm and reads naturally as "re-fetch when the doc has been written".
    - **Effect cleanup flag matches the existing pattern.** The host component already uses `let cancelled = false; ... return () => { cancelled = true; };` in the detail-fetch effect. Repeating the pattern in the history effect keeps two effects in this file stylistically identical so a future reader doesn't trip on "wait, why does this one use AbortController and that one uses a flag?". `fetch()` itself is unaborted on cancel ŌĆö the host detail fetch doesn't abort either, and a 200 KB audit-log response that lands after navigation just gets garbage-collected without the setState landing. Adding `AbortController` for the audit-log fetch but not the detail fetch would create an inconsistency without a measurable benefit.
    - **Revert success path reuses the existing PATCH-success shape.** Both PATCH and revert produce `{ sessionId, lastUpdatedAt, model }` (W61-audit-revert deliberately reuses `PatchSemanticModelResponse`'s envelope so the client's success handler doesn't fork). The revert handler does the same `setData({ ...data, lastUpdatedAt: res.lastUpdatedAt, model: res.model })` as `persistModel`, and the `lastUpdatedAt` bump rolls into the same effect that PATCH success rolls into ŌĆö meaning the audit log refreshes automatically with the newly-appended (the just-overwritten) entry at index 0. The W61-audit-revert wave's audit-write-on-revert design is what makes this work: the just-overwritten model becomes the new newest entry, and "undo this revert" is naturally `auditEntryIndex: 0` on the next click.
    - **Disabled-gate triple `saving || reverting !== null || historyLoading`.** Three sources can race: an in-flight PATCH from the existing save UI (saving=true), an in-flight revert (revertingŌēĀnull), and an in-flight audit-log refetch (historyLoading=true). Disabling all Revert buttons during any of the three prevents a second mutation against a stale read or against an in-progress write. The cost is mild UI friction (Revert is disabled during the brief window between save-success and the audit-log-refetch landing), but that's correct: clicking Revert during that window would target an index against the pre-save audit log shape.
    - **`<button>` for the trigger, not the page-existing `<Card><header>` shape.** Other cards on the page (`Metrics`, `Dimensions`, `Hierarchies`) use a non-interactive `<header>` because there's no toggle. The Audit-history card needs the whole header row to be clickable for the collapse toggle; a `<button>` element is the semantically-correct interactive role for that, and `Collapsible asChild` lets Radix forward the trigger props onto the `<button>`. Tailwind classes match the other cards' header padding / border / typography so visually the cards read as a family.
    - **Test-IDs everywhere.** Each branch (`trigger`, `loading`, `error`, `empty`, `list`, `revert-error`, `revert-${idx}`) gets a stable `data-testid` so a future smoke test can drive the surface without text-scraping. The cost is ~80 chars of attribute markup; the benefit is that the surface stays mechanically queryable even if the copy changes.

  **Why this design.**
    - **UTC timestamp formatting.** Audit log forensics often involve multiple admins reconciling who-did-what-when across timezones. `2026-05-20 14:23 UTC` reads identically for everyone; a locale-aware formatter would let alice in NYC and bob in Bangalore both see `"14:23"` and disagree on what wall-clock moment that was. A dedicated test pins that the output never includes a comma, an ISO-8601 `T` between date and time, or a seconds component ŌĆö catches a future "let's just call `toISOString()`" refactor that would flip to `2026-05-20T14:23:45.000Z` and break the audit-log row's visual rhythm.
    - **`saveNumber = total - indexFromNewest`.** The buffer is newest-first (W61-audit-log convention). The most recent entry (index 0) is `Save M of M` because it was the M-th save chronologically. Walking back through indices: index 1 ŌåÆ Save M-1, etc., down to index M-1 ŌåÆ Save 1 (the oldest retained entry ŌĆö the W57-inferred original or whatever was at the tail when the buffer first filled). A dedicated test walks all 5 indices of a 5-entry buffer to pin the mapping, and a second test pins the cap-at-10 boundaries (`Save 10 of 10` at index 0, `Save 1 of 10` at index 9).
    - **Two-field `{ headline, subhead }` over a single string.** The history-tab UI renders the headline at base-font and the subhead muted (smaller, lower-contrast); pre-splitting at the helper boundary spares the JSX from string-split-and-style at every render. A future re-style that decides to put both on one line (e.g. for compact mode) just concatenates `headline + " ŌĆö " + subhead` at the call site rather than re-running the formatter.
    - **`priorVersion` in the subhead, not the headline.** Two adjacent audit entries can share `savedBy` (alice edited twice in a row) AND share the same `savedAt` minute (consecutive saves often land within the same minute). The version is what differentiates them, and it matches the revert mental model ("revert to the state of version 2"). The version is also the W64 compiled-query cache key, so `was v2` in the audit log corresponds directly to the cache entry that revert would restore. Putting it in the subhead (not the headline) keeps the headline scannable while the disambiguator stays available.
    - **`buildRevertConfirmation` includes the undo-safety note.** Load-bearing for the "undo this revert works" semantic ŌĆö the prompt should set the admin's expectation that reverting is itself reversible (and the W61-audit-revert wave's audit-write-on-revert is what makes that true). The prompt anchors on `priorVersion` + `savedBy` so admins double-check the target before clicking through; a dedicated test pins both anchors plus the undo-safety wording.
    - **`unknown` savedBy passthrough.** The server stamps `"unknown"` when `getAuthenticatedEmail` returns null (the failure mode for the auth helper). The summary shouldn't munge that value ŌĆö admins should see the raw attribution-failure marker so they know something wasn't recorded. A test pins that `"unknown ┬Ę "` appears in the subhead verbatim.
    - **Local interface mirror, not type re-import.** The server's `SemanticModelAuditEntry` lives in `server/lib/semantic/semanticModelAuditLog.ts` which the client can't import across the runtime boundary. A local `AdminSemanticModelAuditEntry` interface in `admin.ts` is the established W61 pattern (mirrors how `SemanticModel` is inline-imported via `import("@/shared/schema")`). Runtime drift between this mirror and the server surfaces as field-undefined access at TanStack Query call sites rather than a compile error, but the W61 pattern accepts that tradeoff because the schema is small and stable.

  **Why this design.**
    - **Skip W61-source-bump on revert.** The bumper's semantic is "admin edited this entry ŌåÆ bump `source` to `user`". A revert's semantic is "restore as-was" ŌĆö every per-entry `source` field should come verbatim from the snapshot. Running the bumper would stamp every entry that differs between snapshot and current as `"user"` (wrong ŌĆö e.g. reverting from a `source: "user"` admin-edited state back to the original `source: "auto"` W57-inferred snapshot would re-stamp the entry as `"user"`, hiding the revert from the source-filter chips and the planner's source-weighting). The model-level `updatedBy` still stamps the reverting admin (preserves attribution at the model level) but the per-entry `source` comes verbatim from `entry.priorModel`. The "source-preservation" test pins this with a fixture where current is `source: "user"` and the snapshot is `source: "auto"` ŌĆö the test would fail if a future refactor wired the bumper into the revert path.
    - **Bump version monotonically.** Reverting to "the state of version 3" makes the live model `current+1` with the contents of v3, NOT version 3 again. The version field is load-bearing for the W64 compiled-query cache (which keys on it) and for the W61-audit-log's `priorVersion` (which records the version at the moment of save). A non-monotonic version would break both: the cache would serve stale compiled queries, and the audit-log entries would have ambiguous `priorVersion` values (two entries with `priorVersion: 3` from different points in time). The "happy path" test pins `body.model.version === 3` for a revert applied to a v2 doc; the round-trip test extends the pin to `version === 4` after a second revert.
    - **Write the audit-log entry on revert.** Two reasons. (a) Undoing the revert (admin reverts, then realises that was wrong, and reverts again) needs the just-overwritten state preserved. (b) Forensic record ŌĆö "alice@example.com reverted on 2026-05-20" is a saveable event in its own right; the audit log should capture state transitions, not just "edits". The round-trip test pins this directly: revert v2 ŌåÆ v1 (audit log captures v2 as the new newest entry), then revert again with `auditEntryIndex: 0` (which now targets the v2 entry ŌåÆ undoes the revert with content === v2 and version === 4).
    - **Bound-check inside the lock.** A multi-admin race could have admin A growing the buffer (via PATCH or revert) while admin B is composing a revert request; reading the buffer length before the lock would let B's request target a now-stale index. Inside the lock the buffer length is authoritative ŌĆö `parsed.data.auditEntryIndex >= log.length` uses the just-loaded `doc.semanticModelAuditLog`, not a stale client view. Conservative: 404 `audit_entry_not_found` if the race produces an out-of-bounds index, and the client retries with the fresh history fetch.
    - **404 not 400 for `audit_entry_not_found`.** The body parsed cleanly (the schema accepted a nonnegative int); the index is out of bounds because the buffer doesn't have that many entries ŌĆö a resource-not-found condition, not a malformed request. 404 mirrors `session_not_found` / `semantic_model_not_inferred` which are also "the thing you asked for doesn't exist" conditions. The body echoes back the offending index so the client can show "you tried to revert to entry 5 but only 2 entries exist" without a separate state lookup.
    - **`auditEntryIndex: 0` semantics ŌĆö "undo my last save".** The buffer is newest-first (W61-audit-log convention) so index 0 is the most recent prior model. This matches the natural mental model of an undo button: "undo my most recent change". A future history-tab UI's "Undo" affordance is just a `POST { auditEntryIndex: 0 }`.
    - **Reusing the W61-audit-log convention for the audit-write on revert.** The same `appendSemanticModelAuditEntry` helper is called from the revert path with `{ savedAt, savedBy: reverting-admin, priorVersion: current.version, priorModel: doc.semanticModel }`. The cap-at-10 logic applies uniformly ŌĆö if a revert pushes the buffer to 11 entries (revert from a buffer at cap), the oldest is dropped. The cap behavior is the same regardless of whether the new entry came from PATCH or revert; a unified semantics keeps the buffer's invariants tight (always Ōēż10, always newest-first, no special-case entries).
    - **Cap-at-10 edge case: reverting to the oldest entry.** If `auditEntryIndex` is 9 (the oldest at cap) and the buffer is at cap (length 10), the new entry from the revert grows the buffer to 11 ŌåÆ trim to 10 ŌåÆ the targeted entry at index 9 is dropped from the buffer. This is fine because we already extracted its `priorModel` before the prepend. Subsequent reverts target indices 0ŌĆō9 of the new buffer (which always includes the just-overwritten state at index 0, so "undo this revert" remains possible).
    - **Response envelope matches W61-save's.** Both endpoints produce `{ sessionId, lastUpdatedAt, model }`. A future client `useMutation` for the W61-audit-history-ui's revert button can reuse the same success-handler that the existing PATCH mutation uses ŌĆö invalidate the detail query, invalidate the audit-log query, optionally show a toast. A different envelope would force a duplicate success-handler.

Adding a separate `_reverter` injectable would duplicate surface for no behavioral benefit ŌĆö the read+write pair is the same Cosmos doc and the same Cosmos update.

  **Why this design.**
    - **Snapshot the *prior* model, not the next.** Considered: write the next model as the audit entry so the buffer reads as "history of saves you've made". Rejected ŌĆö the next model already lives on `doc.semanticModel` itself; storing it again would be redundant. The buffer's role is to remember what was just *lost*: a revert UI rewinds to "the state before save N" by walking back N entries and re-PATCH-ing the `priorModel` of entry N. Storing the next model would force the revert UI to walk forward through the buffer (less natural) and would not capture the very-first state because there's no "save 0" entry.
    - **Cap at 10.** SemanticModel snapshots are moderate-sized (~5ŌĆō50 KB depending on dataset width ŌĆö a 30-metric / 20-dimension / 5-hierarchy model JSON-serialises around 10 KB; a 100+ metric MMM-style model approaches 50 KB). 10 entries ├Ś 50 KB worst-case = ~500 KB of audit overhead per doc, comfortably under the 2 MB Cosmos document size limit. The deep-history use case (revert to the 11th-most-recent save) is rare enough that paying for unbounded growth across months of saves isn't worth it ŌĆö an admin who has made >10 saves has almost certainly settled on a stable model. The const is exported so a future wave that genuinely needs more history can raise it in one place + the [pin test](../server/tests/semanticModelAuditLogW61AuditLog.test.ts) catches the change in the same diff.
    - **Newest-first ordering.** Matches the MRU convention already used by [WI6 `insightHistory`](../client/src/pages/Dashboard/lib/insightHistory.ts) (per-tile insight history with newest at index 0), the sidebar Recent Sessions list, and the prior W61-filter-persist wave's URL-param semantics.     - **Returns a fresh array reference even when under cap.** Considered: when `next.length <= max`, return the constructed `[entry, ...(prior ?? [])]` array directly (which is already fresh); when `next.length > max`, return `next.slice(0, max)` (also fresh). Both branches return fresh arrays ŌĆö there's no "return the input reference" optimization analogous to the [W61-source-filter `filter === "all"` case](../client/src/pages/Admin/lib/semanticModelSourceFilter.ts) because every call here is a state change (append), so there's no "default-case-is-no-op" path.     - **`SemanticModelAuditEntry.priorModel` is a reference, not a deep clone.** The controller does `priorModel: doc.semanticModel` and then `doc.semanticModel = nextModel` ŌĆö the reference is preserved before the field-overwrite assigns a different reference.   **Why this design.**
    - **`replaceState` rather than `pushState`.** Considered: each chip click pushes a new history entry so the browser back button could step through filter states.     - **Pathname + hash reconstruction not full `window.location.href` rewrite.** The latter would force a full URL parse-and-replace (and could subtly alter the URL's encoding if the browser normalises differently); reconstructing from the three known components (`pathname` / `search` / `hash`) is byte-exact and cheaper.

    - **Fall-back to `"all"` rather than throw on invalid URL.** The URL is admin-controlled and admins mistype; throwing would crash the page-mount and lock the admin out.     - **Short `filter` param name not `source_filter`.** Admins frequently copy URLs into chat / docs / GitHub issues; a verbose param reads as noise. The param's meaning is unambiguous in context (no other filter on this page) so the short name doesn't cost clarity.
    - **Case-sensitive validation.** Considered: lower-case the incoming value before validation so `?filter=USER` works.   **Why this design.**
    - **Always-editable inputs over click-to-edit popovers.** Considered: each cell renders as styled read-only text by default; clicking it swaps in an input with `autoFocus`; blur or commit returns to read-only. Rejected ŌĆö the dance adds state (`editingFieldId`), keyboard-trap risk (Escape during the swap), and an extra click per correction. Admins fixing inference gaps spend their time tabbing through many fields per session; always-editable lets them flow without a context switch per cell. The visual distinction "this is editable" comes from the input's own styling (border, focus ring) which already signals affordance.
    - **Save-on-blur over save-on-Enter or per-row Save button.** Considered: an explicit Save button per row that batches edits + a discard-changes button. Rejected ŌĆö the parent's `persistModel` already debounces by routing through `withSessionWriteLock(sessionId, ...)` (invariant #9), so multiple sequential field saves serialize cleanly without a batch button. Save-on-Enter alone leaves a "ghost dirty" state if the admin tabs out without explicitly committing. Save-on-blur matches Google Docs / Notion semantics and pairs with `isMeaningfulChange` to skip no-op PATCHes.
    - **Client-side validators that mirror but don't share zod schemas.** Considered: import `semanticMetricSchema.shape.label` and re-derive the bounds at runtime. Rejected ŌĆö the validators are 96 LOC of pure-function bounds checks; if the schema bounds change, a drift here just routes the save to the server's authoritative rejection (a 400 with the zod issues array surfacing in the existing "Save failed: {message}" banner). The validators' job is to keep obviously-bad inputs from round-tripping at all, not to be a second source of truth.
    - **Keyword-guard via padded uppercase scan instead of regex.** Considered: `/\b(SELECT|FROM|JOIN|...)\b/i` over the trimmed string. Rejected ŌĆö JavaScript `\b` matches at underscores which would let `_FROM_` slip through; a literal regex with `(?<![A-Za-z0-9_])` lookbehind works but the engine's lookbehind support varies by runtime. The padded-uppercase form (`" ${trimmed.toUpperCase()} "`.includes(" FROM ")) is byte-exact, runs in O(n) per keyword, and avoids regex edge cases.
    - **`isMeaningfulChange` based on trimmed equality.** Considered: deep equality on raw strings. Rejected ŌĆö admins frequently add or trim trailing spaces while typing; firing a PATCH on every whitespace-only edit would burn write throughput and pollute the W64 cache key (version bump per save). The trimmed-equality form means leading/trailing whitespace is normalised by the save itself (the handler does `draft.trim()` before calling `onSave`).
    - **`emptyToUndef` helper for the description's `.optional()` shape.** A literal empty string is a valid `z.string().max(1000).optional()` value (`.optional()` only adds `undefined` to the union; it doesn't strip empty strings), but persisting `""` instead of `undefined` makes the doc shape drift: a freshly-inferred model would carry `description: undefined` for missing descriptions and a manually-cleared one would carry `""`, which are semantically the same but serialise to different bytes (and a future "no description = use the auto-inferred one" feature would break in subtle ways). The helper normalises both into `undefined`.

    - [`registerTools.ts`](../server/lib/agents/runtime/tools/registerTools.ts) WV4 block (the `run_correlation` success path) now imports `bucketCorrelationR`, calls it on `strongest.correlation` (the headline correlation already used for `R┬▓ = r┬▓` and `nPairs`), and assigns the result to `evidence.effectMagnitude` under `if (bucket) { evidence.effectMagnitude = bucket; }`.   **Why this design.**
    - **Cohen's conventions on |r|, not on R┬▓.** Considered: bucket on R┬▓ since R┬▓ is what the prose already carries and what the WQ1 classifier reads. Rejected ŌĆö Cohen's published thresholds for correlation effect sizes are stated on |r|, not R┬▓; bucketing on |r| matches the academic convention and means the same r value buckets identically whether the tool is reporting r or R┬▓. R┬▓ is monotonic in |r| so the bucket transitions still fire at the right places, but pinning to the |r| boundaries gives clean numerical values (0.1 / 0.3 / 0.5) instead of R┬▓ boundaries (0.01 / 0.09 / 0.25).
    - **Bucket via `null` on out-of-range r rather than throwing or defaulting.** Considered: throw on `|r| > 1` (it shouldn't happen with the Pearson formula but defense in depth) or default to `"negligible"`. Rejected ŌĆö `null` lets the caller decide whether to skip emitting the bucket entirely; a default would silently mislabel pathological inputs. The truthy guard at the call site (`if (bucket) { ... }`) means a `null` bucket cleanly degrades to "no `effectMagnitude` emitted", which is the conservative behaviour.
    - **Helper lives in `correlationMath.ts`, not at the tool call site.** Considered: inline the 4-line ladder inside the WV4 block of `registerTools.ts`. Rejected ŌĆö `correlationMath.ts` is the natural neighbour to `calculateCorrelations` (same module, same purity contract, same test path through `correlationMath`-only tests). Future callers (a hypothetical `run_correlation_pair` tool, a deck-planner correlation block) can reuse the helper without copy-pasting the threshold ladder.
    - **No change to `analyzeCorrelations` itself.** The bucket is derived at the tool boundary, not pushed into the `CorrelationResult` interface. Adding `effectBucket?: ...` to `CorrelationResult` was considered and rejected ŌĆö the result struct is consumed by chart rendering / insight prose / fallback paths that don't need the bucket; the bucket is only load-bearing on the WV4 evidence emit. Keeping the field at the call site avoids a wide-blast schema migration.
    - **Sign-invariance pinned explicitly.** A negative r (anti-correlation) and its positive twin describe equally strong relationships ŌĆö they just point opposite directions. The `Math.abs()` inside the helper is load-bearing for this; pinning it in a sign-invariance test (`-0.05 ŌåÆ "negligible"`, `-1 ŌåÆ "large"`) makes a future refactor that drops the abs() break loudly.

Elasticity's coefficient ╬▓ is interpreted as percent-change-per-percent-change with no universal "large vs small" cutoff in the literature (industry-specific ŌĆö FMCG own-price elasticity of `-2.0` is "huge", `-0.5` is "moderate"); segment-driver's feature importances are already relative to the model fit and don't have an absolute scale. Both deferred ŌĆö each warrants its own design wave when the use case demands it.
    - **Bucket exposure in `CorrelationResult` for chart consumers.** The current chart-rendering path consumes the raw r and renders its own annotations (correlation strength chips on scatter charts, ranked-correlation bars on driver charts); pushing the categorical bucket through to those surfaces is a separate UX wave.
    - **Per-correlation-pair bucket vs strongest-only.** The WV4 emit currently uses only the strongest correlation's bucket. A correlation tool that returns N pairs could emit a histogram of bucket counts ("3 large, 5 medium, 2 small, 12 negligible"). Deferred ŌĆö the strongest-only emit matches the existing `n` / `R┬▓` emit pattern, and bucket histograms are a richer surface that would benefit from its own narrator-prompt design.

- **2026-05-20** ŌĆö **Wave WQ8 ┬Ę effect-size as fifth FindingEvidence field ŌĆö distinguish significant-but-negligible from real-and-large.** Closes a long-standing gap in the WQ1 confidence classifier: a finding could pass `assessConfidence` as HIGH with strong `n` / `pValue` / `rSquared` / `ciRelativeWidth` even when the underlying Cohen's d or Cram├®r's V was vanishingly small ŌĆö "statistically significant but practically negligible" findings were getting graded as load-bearing evidence. The WV5 significance-test tool has emitted a categorical `effect_magnitude: "negligible" | "small" | "medium" | "large"` column on its result table since the F3 wave, but the bucket never propagated through to the WQ1 classifier ŌĆö it sat at the tool boundary as user-visible prose without changing the deterministic-floor evidence routing. This wave plumbs the bucket through the existing three-file WV/WW/WQ trinity (formatter ŌåÆ extractor ŌåÆ classifier) so a "significant but negligible" finding is now a hard-fail LOW signal in `assessConfidence`, overriding p Ōēż 0.05 with strong n / CI / R┬▓. The mental model: p-value answers "is it noise?"; effect-size answers "does it matter?" ŌĆö both must clear the bar for a HIGH-tier finding.

  **What landed.**

The classifier `assessConfidence` is updated in four places: (i) `hasAnyEvidence` includes `effectMagnitude !== undefined` so a finding carrying only an effect bucket no longer falls through to the "no statistical evidence supplied" default; (ii) the LOW hard-fail block adds `evidence.effectMagnitude === "negligible"` with reason `"negligible effect size"` ŌĆö this is the load-bearing wedge that downgrades "significant but trivial" findings; (iii) the HIGH all-conditions gate now also requires `effectMagnitude` to be `undefined` / `"medium"` / `"large"`, and reasons surface `"large effect size"` or `"medium effect size"` when present; (iv) the MEDIUM catch-all surfaces `"small effect size"` / `"medium effect size"` as a reason when the bucket fails the HIGH gate by itself.

  **Why this design.**
    - **A new field on `FindingEvidence` rather than overloading the pre-existing `magnitude?: number`.** Considered: repurpose the unused `magnitude: number` field that's been sitting on `FindingEvidence` since WQ1 ("carried through for completeness") to hold the categorical bucket. Rejected ŌĆö the field is a number, not a string union, and changing its type would silently break any caller that's been reading `evidence.magnitude` as a numeric (a grep showed no current readers, but the WQ1 doc string explicitly reserves it for "headline effect-size magnitude post-normalisation"). A new field keeps the existing semantic slot intact and lets a future wave wire the numeric channel in without colliding.
    - **`negligible` as a LOW hard-fail signal, not a tier downgrade.** Considered: have `negligible` only weaken HIGH-tier findings to MEDIUM (the conservative read). Rejected ŌĆö the "significant but trivial" anti-pattern is precisely the failure mode this wave exists to catch. A finding that lands in MEDIUM with `n = 5000, p < 0.001, effect = negligible` would still earn the narrator's MEDIUM-tier verbosity budget (3 sentences) plus a soft hedge phrase ŌĆö that's exactly the overclaim shape the verifier's WV3 short-circuit is trying to stop. Pushing all the way to LOW means the narrator's 2-sentence budget + the explicit "treat as directional" hedge are forced, which is the right framing for a finding that's statistically real but practically immaterial.
    - **`medium` effect contributes to HIGH, not just MEDIUM.** Considered: only `large` clears the HIGH gate; `medium` lives in MEDIUM. Rejected ŌĆö Cohen's standard cutoffs for "medium" (d Ōēź 0.5) and Cram├®r's V (Ōēź 0.3) already represent a substantively meaningful effect; demanding `large` (d Ōēź 0.8 / V Ōēź 0.5) would make the HIGH bar unreachable for many real-world findings even with strong p / n / CI. The HIGH bar stays achievable at `medium` effect; the `large` reason is additive when present.
    - **`small` effect drops to MEDIUM, not LOW.** Considered: treat `small` as a downgrade signal symmetric with the `negligible` hard-fail. Rejected ŌĆö `small` effect (d Ōłł [0.2, 0.5)) is the boundary case where the finding may still be actionable in the right context (high-leverage decisions on large populations where a small mean shift = meaningful dollar impact). Letting it land in MEDIUM with an explicit `"small effect size"` reason gives the narrator the option to hedge appropriately without forcing the floor.
    - **Optional trailing `effect = X` token in WV5 regexes rather than required.** Considered: pin the new token as required to lock in the contract. Rejected ŌĆö the WV5 tests run the **real** significance-test math against real data, so the emitted bucket depends on what `bucketCohensD` / `bucketCramersV` produce. The 30-vs-30 +10 dataset in the welch_t fixture emits `large` today, but a future threshold tweak could shift it to `medium`. Making the token optional keeps the test invariant on "the suffix is well-formed", not on a specific bucket value that depends on indirect math.
