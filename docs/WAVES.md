  **Why this design.**
    - **Why a thin sibling helper rather than inline kickoffs in the host.** Three lines of `Promise.all`-style kickoff code would have been simpler to read in isolation, but the host is already 2,398 LOC of god-object — adding three more closure-captured promises inline would have made the pre-classify block harder to reason about, and the helper is unit-testable in isolation while inline code can only be tested through the full `processStreamChat` integration surface. The helper file is 40 LOC with JSDoc; the host delta is small enough that the inline-vs-helper tradeoff comes out narrowly in favour of helper-extraction for testability rather than readability.
    - **Why generic over `<S, Q, D>` rather than typed to the specific call-site shapes.** Three reasons: (i) the helper's contract is structural (three thunks, three promises, two `.catch` slots) and doesn't depend on the specific shapes — typing to `SchemaBinding` / `QueryParserResult` / `DomainCacheEntry` would import three types just to re-export them in the result; (ii) the call site preserves the structural cast at the await site (`(await kickoff.parsedQuery) as Record<string, unknown> | null`) because the W32 widening of `parsedQueryForLoad` is a host-side concern, not a kickoff-side one; (iii) the generic shape makes the helper trivially reusable if a future wave needs the same parallel-kickoff pattern for a different operation triple (e.g. a hypothetical post-classify or post-retrieve parallel block).
    - **Why `.catch(() => null)` rather than `.catch(err => { console.warn(err); return null })`.** The pre-wave inline code at the domain-context site logged a `console.warn` on failure (`B4 · domain context load for mode classifier failed: <msg>`). Moving the catch into the kickoff helper drops the warn, which is a small observability regression. Considered options: (a) thread an `onError?: (err) => void` callback through the helper's deps (rejected as over-engineering — the failure modes for both parser and domain-context are already non-fatal-by-design); (b) re-add the warn at the await site (rejected because the catch-and-warn shape is wrong — the failure is intentional non-fatal, not an alertable signal); (c) drop the warn silently (selected — the prior warn was noise on a known non-fatal degradation path, and any future need for the signal can route through the agentLog/SSE instead of console.warn).
    - **Why kick off the three operations at line 893 specifically.** Two constraints: (a) the kickoff needs all three deps' inputs in scope — `message` (param) / `chatDocument.dataSummary` (param destructure) / `processingChatHistory` (computed at line 886-889); line 893 is the first point where all are available; (b) the kickoff must precede the first SSE emit at line 893 because thunk side-effects (the actual LLM calls + memoised disk read for domainContext) should be synchronously initiated before the user sees the first "Mapping columns" thinking step — otherwise an unlucky scheduler could let the SSE wire-emit beat the underlying work-kickoff, defeating the purpose. Placing the kickoff one line above the first thinking-step emit hits both constraints.
    - **Why NOT also parallelize `extractAndPersistUserHierarchies` (line 1133) or `analyzeChatWithColumns` (line 929).** `analyzeChatWithColumns` depends on `schemaBinding.canonicalColumns` (via `columnHintsForIntent` at line 917-919) — no parallelization opportunity. `extractAndPersistUserHierarchies` is genuinely independent (only consumes `sessionId` / `username` / `message` / `chatDocument.sessionAnalysisContext`, all available at line 893) but it emits a `session_context_updated` SSE row whose original-position gating would add complexity (the kickoff would need to buffer the SSE emit until the original line 1141 position to preserve on-wire ordering). Deferred to a follow-on wave if profiling proves the ~300-500 ms `extract` LLM call is a meaningful slice of the remaining critical path; today's wave keeps the SSE-ordering invariant trivially preservable.
    - **Why NOT a pre-extract refactor before parallelization.** The host is 2,398 LOC, which is firmly past the React-host 1,500-LOC pre-extract threshold codified in W61-host-extract — but that convention is React-host-specific (sub-component carving is a different operation from server-side service decomposition). A `chatStream.service.ts` decomposition is its own waveline (likely needs three or four ~600-LOC sub-modules — request-validation / cache-lookup / pre-classify / agent-loop-dispatch / post-turn-persist), and the WS2 parallelization is orthogonal to that decomposition. Pre-extracting would have inflated the wave from ~150 LOC to ~1,000+ LOC with no latency-floor improvement until both landed; shipping WS2 standalone realises the floor savings immediately and leaves the decomposition for a dedicated wave when the scope is justified.
    - **Why no integration test against `processStreamChat`.** The kickoff helper is the load-bearing piece — the host-side hoist is a thin three-line delta (one kickoff const + three `await kickoff.X` replacements) that typecheck-baseline-preservation + the existing `tests/chatStreamModeContract.test.ts` + `tests/chatStreamScalarPivot.test.ts` already cover for structural regressions. A full integration test would have needed module mocks for `bindSchemaColumnsForAgentic` + `analyzeChatWithColumns` + `parseUserQuery` + `loadEnabledDomainContext` + `classifyMode` + the SSE wire + the chat-doc DB layer — roughly 200 LOC of stub plumbing for marginal additional coverage. The helper's seven isolated tests catch every refactor-relevant contract violation: concurrent kickoff timing, error semantics (three permutations × propagate-vs-catch), value-shape pass-through, and once-per-kickoff memoisation.

  **Why this design.**
    - **Why these specific thresholds (0.5 and 1.5).** The 0.5 boundary matches `interpretElasticity`'s "highly inelastic" cutoff (the function returns `"highly inelastic"` for |β| < 0.5); the 1.5 boundary sits between the "elastic" and "highly elastic" thresholds (1.1 and 2.0 in the existing five-step ladder). Mapping the five existing buckets onto the four-bucket effectMagnitude taxonomy collapses three middle buckets ("inelastic" + "unit elastic" + most of "elastic") into "medium" — the boundary is set at 1.5 rather than at 1.1 or 2.0 because 1.5 is conceptually "clearly elastic but not yet huge", and the 1.1 unit-elastic boundary is too sensitive to noise (a fit of |β| = 1.08 vs 1.12 shouldn't flip the magnitude bucket). The single-step boundary at 0.5 is unambiguously "very inelastic" → "small".
    - **Why not significant → "negligible" rather than null.** The bucket function returns `"negligible"` for the not-significant case rather than `null` because the WQ1 confidence classifier reads `effectMagnitude` to downgrade narrative; a `null` would leave the field undefined and WQ1 wouldn't see any signal at all. `"negligible"` is the load-bearing "we have no evidence of effect" token — same shape as WV4-bucket's correlation handling, where r ≈ 0 also buckets to "negligible".
    - **Why positive β → null rather than a bucket.** A positive elasticity coefficient is sign-anomalous (Giffen-good shape or data issue per the existing `interpretElasticity` warning); the magnitude is meaningless without first resolving the sign anomaly. Shipping a "large" for a positive coefficient would mislead the WQ1 classifier into HIGH confidence on what's actually a data-quality problem. `null` causes the field to be omitted entirely from the suffix — the n + R² + interpretation text still surface the fit, and the absence of the magnitude is itself a signal.
    - **Why no industry-specific override (FMCG vs durable-goods).** The brief flagged FMCG own-price elasticity of -2.0 as "huge" and -0.5 as "moderate" — slightly different cut-points than the universal ladder I shipped (-2.0 = "large" under both ladders, but -0.5 = "small" universal vs "moderate" / "medium" FMCG). I deliberately did NOT layer industry-specific cut-points because the universal ladder is approximately right for FMCG (off by one bucket at the -0.5 boundary) AND the cross-vertical use case for this tool is broader than FMCG (durable goods, services, B2B SaaS pricing all have different elasticity conventions). If a per-vertical override surfaces as a real use case, the bucket function can take an optional context arg (`bucketPriceElasticity(β, sig, { vertical?: "fmcg" | "durable" | … })`) without breaking the current call sites.
    - **Why widen the existing WV6 tests in-place rather than ship a new wave with the magnitude.** The WV6 base wave's contract was "the canonical suffix carries n + R²"; the bucket adds a third field. Loosening the regex with `[^)]*` is the canonical "tolerant of future widening" shape — it accepts the current suffix plus any future evidence field. A new test file would have duplicated coverage; the in-place change is the same pattern WV4-bucket used.
    - **Why ship the bucket as a separate wave from WV6 rather than bundling.** The brief listed this explicitly as a "WV6 follow-on" because the WV4-bucket precedent took the same approach (correlation got its base WV4 wave + a separate WV4-bucket wave); shipping the bucket as a follow-on keeps each wave at a clean ~80-200 LOC scope and lets the WW2 extractor + WQ1 classifier validations land before the bucket arrives. With the framework already in place from WV4-bucket + WQ8, this WV6-bucket wave is purely about choosing thresholds + wiring.
    - **Why no jsdom mount-test for end-to-end.** The end-to-end coverage is via `runPriceElasticity` (pure-Node function with no DOM) emitting summaries that get regex-matched + extracted; no UI surface to mount. The four end-to-end tests in suite 3 exercise the full path from price/quantity data → fit → bucket → suffix → extractor → recovered field. This is more thorough than a UI mount would be.
    - **Why we ship a degenerate `β = 0 + significant → small` rather than special-casing.** A truly-zero coefficient that's also flagged significant is degenerate (the t-stat denominator should explode), but the bucket function follows the ladder: `|0| < 0.5` → small. The alternative (returning null on β = 0) would add a special case for a state that shouldn't reach the bucket in practice; following the ladder is the right shape if it ever does.

  **Why this design.**
    - **Why one global + per-section overrides shape (`{ global, overrides }`) rather than three independent filter states.** Three independent states would lose the synced-global default that serves the dominant workflow ("show me what I edited applied across all three cards"). A naïve admin who just wants to filter everything by User would need three clicks; the W61-source-filter shipped specifically because that's an unacceptable cost. The hybrid shape preserves the one-click default + adds shift-click for the advanced case.
    - **Why plain-click clears ALL overrides (not just the clicked section's override).** The simpler "clear only this section's override on plain-click" rule sounds more "local" but creates a confusing exit problem: an admin who shift-clicked Metrics→User, then shift-clicked Dimensions→Auto, can't easily get back to fully-synced mode — they'd have to plain-click in BOTH cards. The "clear all on any plain click" rule gives a single predictable exit from override mode: plain-click any chip anywhere → fully synced + new global. The trade-off is that shift-clicking two sections to scope them both then plain-clicking a third dramatically resets, but the wins in discoverability + escape-hatch reliability are larger.
    - **Why a single `applyChipClick` reducer with a `modifier: boolean` branch rather than two named functions.** The chip row's only branching axis is whether the modifier key was held. A single entry point keeps the call site simple (`setFilters(prev => applyChipClick(prev, "metrics", "user", e.shiftKey))`); two named functions (`setGlobalFilter` + `setSectionOverride`) would force the call site to do the branching, which is exactly the kind of boilerplate that the pure-reducer pattern eliminates.
    - **Why URL persistence stays global-only (overrides not in URL).** Per-section overrides are session-local opt-in advanced state — most admins won't use shift-click. A share-link that captures overrides would surprise the recipient ("why is Metrics filtered to Auto when I clicked the share link?"); the cleaner intent is "share my synced view", which is what the global-only URL persistence captures. Overrides also don't survive accidental reload — but a reload-survival admin can always re-shift-click in a few seconds.
    - **Why the `(overridden)` hint is muted text rather than a strong visual cue.** The override state is informational; the chip itself already shows the active filter value. A bold or coloured indicator would compete with the chips visually. The italic muted-foreground span at the end of the chip row reads as "secondary metadata" — discoverable without competing.
    - **Why shift-key (not cmd/ctrl/alt).** Shift is the conventional "modifier for second-tier action" across most web UIs (shift-click in file managers extends selection; shift-click in tab managers opens in new window; shift-click in chart libraries enters multi-select brush). Cmd/ctrl is reserved for OS-level actions (open-in-new-tab on links, cmd-click on Mac); alt-click is sometimes used but feels keyboard-modifier-heavy. Shift is the most discoverable modifier for "scope this action to one card".
    - **Why `Object.freeze` the `NO_OVERRIDES` sentinel.** Identity stability for free across renders — a downstream `useMemo([..., sectionFilters.overrides, ...])` would invalidate on every render if `NO_OVERRIDES` were a fresh literal. Frozen sentinel means three card renders' `overrides` reads point at the same object reference until an override is set. Doesn't matter for current render perf but is a load-bearing invariant for any future memoised consumer of the overrides shape.
    - **Why `isSectionOverridden` returns true even when override value equals global.** The "intent of override" is what matters — an admin who shift-clicks Metrics→"user" when global is already "user" is signalling "I want this section scoped, independent of global". A future global change wouldn't propagate to that section. Surfacing the override hint in that case is the load-bearing signal. The alternative (value-equality check) would silently strip the override which feels magical.
    - **Why no separate "Reset section" or "Clear override" button.** Plain-click any chip in any card is the canonical exit. Adding a "Reset" button per card would duplicate the affordance + invite the question "what does this reset do, exactly?". The chip itself is already the affordance.
    - **Why no jsdom mount-test for the component.** Same rationale as W61-hierarchy-edit + W61-add-client + W61-edit-column-refs — the SourceFilterChips and card wrappers are simple wrappers around Radix primitives + the pure helpers (which ARE pinned by 19 unit tests). The shift-key capture is one line of code that mounts trivially via inline event handler; mounting jsdom for the click would add brittle dependency on Radix portal internals for marginal coverage gain.

Per-mutation save wins because (a) every other inline-edit cell in the host (`EditableText` / `EditableSelect`) saves on blur/select, so per-mutation save preserves the inline-edit semantics admins are already trained on; (b) the W61-save endpoint is fast enough (~150–300 ms typical) that the latency cost of N saves vs 1 save is invisible at admin tag-list cardinalities (typically 2–5 refs per metric); (c) a draft buffer would force a new "discard changes" affordance + a "save the draft" button which would mismatch the rest of the page's auto-save model; (d) version bumps + audit log entries per mutation give admins finer-grained undo via the W61-audit-history-tab — an admin who removes the wrong tag can revert just that mutation, not lose everything in the draft. The trade-off is that a network blip mid-edit could leave the tag list partially updated, but the optimistic-update-rollback in `persistModel` handles that case identically to single-cell edits.
    - **Why `null` (not `[]`) for the no-schema branch.** The server's `projectDatasetSchema` (W61-detail-schema) already normalises empty-columns to `null` so the wire payload distinguishes "session has no `dataSummary`" (null) from "session has dataSummary but it's the empty array" (also null today — but a future enrichment could split them). The card receives `ReadonlyArray<string> | null` rather than `ReadonlyArray<string>` so the consumer can branch on availability explicitly rather than checking `length === 0`. An empty-array prop would force the card to render an empty dropdown which would be UX-degraded vs the free-text fallback. The `null` discriminant also makes the schema-aware-or-free-text branch line explicit in the JSX (`if (datasetColumns === null) return <EditableText … />;`) rather than buried in a length check.
    - **Why show the orphan value in the picker dropdown as a `(not in dataset)` italic option.** Stale state is real: an admin uploads a fresh dataset where the previous "region" column has been renamed to "geo_region"; the model's existing dimensions still point at "region"; the picker should let the admin see the orphan AND pick a valid replacement in one interaction. Hiding the orphan would mean the admin sees "region" as the dimension's current value but the dropdown doesn't contain it — confusing and forces a workaround. Surfacing it in the dropdown with the muted italic `(not in dataset)` hint is the cleanest signal of drift; the admin can read the warning and select a replacement column without leaving the cell.
    - **Why alphabetise column lists in both dropdown options AND tag display.** Predictable scan order — admins looking for a specific column find it in O(log N) eye-saccades rather than O(N). The dataset's raw column order (CSV header order) doesn't carry meaning to the admin reviewing the semantic model; alphabetical is the dominant sort affordance for "find a known string in a list" UX. Symmetric: both the column picker dropdown AND the tag-list chips are alphabetised, so the same column visually appears in the same relative position whether it's currently referenced or selectable to add (which reduces the cognitive load of moving between the two surfaces).
    - **Why dedupe in BOTH the add-from-select path AND the free-text add path.** The dropdown filters its options against `values.includes(c)` so already-referenced columns don't appear in the dropdown — primary defense. The free-text add path (null `datasetColumns` fallback) doesn't have a curated list to filter, so it dedupes via `if (values.includes(trimmed)) { setAddError("Already in references"); return; }` because an admin could type the same column name twice and the validator alone wouldn't catch it (the schema doesn't forbid duplicates but the W58 compiler would treat them as redundant references — semantically a bug). Two defenses for the same invariant is the right shape.
    - **Why no new server-side validation for orphan column refs.** The W58 compiler ([`server/lib/semantic/compiler.ts`](../server/lib/semantic/compiler.ts)) is the authoritative orphan-column lint surface; W61-save / W61-add-server intentionally don't pre-validate column refs because doing so would force the controller to read `dataSummary.columns` to validate the model on every save, coupling the per-session column list to every write. The picker UI's `(not in dataset)` italic hint is the soft pre-validation surface; the W58 compiler's lint catches the hard case at query time. The wire shape stays clean.

  **Why this design.**
    - **Why `<Dialog>` not `<AlertDialog>`.** AlertDialog is Radix's confirmation-only primitive (single warning + Cancel/Confirm pair); Dialog is the canonical form-modal that supports multi-input editing. Hierarchies have 2–8 editable level rows — Dialog is the right semantic primitive. Same rationale as W61-add-client.
    - **Why the modal owns the draft, not the host.** Draft state is mount-scoped and lives only for the duration of the edit. Host ownership would force a `clearDraft` step on every close + a stale-draft concern if the admin closes and re-opens for a different hierarchy quickly. An effect keyed on `[hierarchy]` re-seeds the draft so per-row inputs are always seeded from the current model state on open. Same precedent as W61-add-client.
    - **Why route through the existing W61-save PATCH path, not a dedicated `/hierarchies/:name` endpoint.** Hierarchies are 2–8 entries × maybe 8 hierarchies = ≤64 level strings per model. The full SemanticModel payload is small (~5–50 KB for typical models); the marginal cost of replacing the whole model vs sending just the changed hierarchy is negligible. A dedicated endpoint would mean (a) a new audit-write code path + (b) a new W61-cache-invalidate hook call site + (c) a new test surface — all for no perf benefit. The brief flagged this trade-off explicitly ("no dedicated endpoint per the brief; cheap wholesale-replace fits hierarchies' density").
    - **Why `editingHierarchy: SemanticHierarchy | null` not `{ open: boolean, hierarchy: SemanticHierarchy | null }`.** The non-null IS the open signal. Two-prop shape invites the invalid state `open=true && hierarchy=null` that the modal would have to defend against. Single-discriminant shape mirrors W61-add-client's `addOpen: kind | null` and W61-delete-client's `pendingDelete: { kind, name } | null`.
    - **Why validation runs only after first submit attempt (validationActive flag).** Keystroke validation would show a red border on the level input the moment the admin types `R` (PascalCase rejected) before they've finished typing `Region` (also rejected, but mid-keystroke). Submit-time validation matches the W61-add-client behaviour. The exception: per-row errors stay visible after the first submit attempt so the admin can fix them iteratively without re-clicking submit per error.
    - **Why per-row duplicate-detection runs AFTER per-level format checks.** A level that's malformed (e.g. PascalCase) AND duplicate of another malformed one should surface the format error first — the admin needs to rename to snake_case, after which the duplicate (if still present) becomes the next actionable signal. Surfacing both at once would be noisy and the duplicate message would be misleading because the admin can't tell which one is "the duplicate" without the format being correct.
    - **Why `editLevelsSubmitting: string | null` (name) not `boolean`.** The string holds the hierarchy name being edited, which lets the row's per-mutation gate (`disabled={editLevelsPending}` shared across the icon buttons) coexist with the in-flight signal without conflating "any edit is in flight" with "this specific row is the one being edited". A future widening that lets multiple hierarchies edit in parallel would still be type-safe; a boolean would force a breaking change to `string[] | null`.
    - **Why we don't currently include a "validate against datasetSchema" check.** Hierarchy levels reference SemanticDimensions by `name`, not raw dataset columns. The W61-detail-schema wave (shipped earlier this session) exposes `datasetSchema.columns` but those are dataset column names, not dimension names. A future enhancement could populate a picker from `model.dimensions.map(d => d.name)` for typed selection, but the current free-text input with snake_case validation is sufficient for hierarchies whose dimensions exist OR whose dimensions are about to be added. The W58 compiler's orphan-reference lint catches the hard case at query time.
    - **Why no jsdom mount-test for the component.** The component's branches are all wrappers around the pure helpers (which ARE pinned by 43 unit tests) + Radix Dialog primitives + per-row state mutations through the helpers. Mounting jsdom for the form-submit flow would add brittle dependency on Radix portal internals for marginal coverage gain. Same rationale as W61-add-client. A future Playwright smoke is the right vehicle for the end-to-end click-through.
    - **Why the Edit2 icon goes BEFORE the delete button in the actions column.** Non-destructive read-first ordering. Most admin actions are edit-not-delete, so the more common action sits first visually. Mirrors the W61-add-client `+ Add` button's positioning (non-destructive primary action) and the existing button order across the W61 family.

  **Why this design.**

  **Why this design.**
    - **Why an observability hook + registry rather than just `console.log` lines at each call site.** Three reasons. First, a single grep token — ops can grep `\\[semantic-model-invalidate\\]` and see every model-version change across the four controllers in one query; scattered `console.log` lines would each need their own grep. Second, the contract pin — the controller integration tests assert "every mutation path calls the hook exactly once on success and never on any error path", which gives the W61 sub-wave family (W61-edit-column / W61-edit-references / W61-hierarchy-edit are upcoming) a tests canary if a new mutation path is added without the hook call. Third, the registry — future W64 cache-key work will register a `cache.clear()` callback via `registerInvalidator(fn)` and the existing four call sites + tests just work; no need to chase every controller again. The triple win justifies the small module (~100 LOC) over scattered logs.
    - **Why the hook fires INSIDE the write lock, not outside.** Future invalidators (W64 in-process cache.clear callbacks) need to be serialized with the write they invalidate against — if the hook fires outside the lock, a concurrent reader could observe the post-write doc state via Cosmos but read a stale cached entry that hasn't been cleared yet (the cache.clear would race the next read). Inside the lock, the sequence is `_updater succeeds → hook fires → cache.clear runs → lock releases → readers proceed with fresh state`. The price is that the hook extends the lock-hold window — but the hook is fire-and-forget, synchronous, and listeners are bounded (`console.log` + counter increment + future `cache.clear`), so the extension is microseconds, not milliseconds.
    - **Why the hook is `(event) => void`, not `(event) => Promise<void>`.** Async listeners would force the controller to `await` them, which would either (a) extend the lock-hold window unboundedly (every cache invalidator could become a remote network call) or (b) require a "timeout-then-skip" pattern that masks slow invalidators. The synchronous shape forces every invalidator to be cheap by design — `console.log` + `cache.clear()` is the budget; anything more expensive belongs in a separate fire-and-forget background queue. A future asynchronous invalidator that needs more time should subscribe via a sibling registry (an `IO_INVALIDATOR_REGISTRY` analog) rather than widen this one.
    - **Why the hook fires AFTER `_updater(doc)` succeeds, not before.** If `_updater` throws (Cosmos outage, document-too-large, optimistic-concurrency conflict at the per-document `_etag` level — single-instance correctness per invariant #9 means we don't currently use `ifMatch`, but a future multi-instance world might), the catch block emits the 5xx response and the hook MUST NOT have fired — otherwise a downstream cache invalidates against a write that never landed, and the cache then misses on a stale-on-disk model that's still correct. The "500 updater throw does NOT fire" controller test pins this explicitly.
    - **Why a listener that throws is logged but doesn't propagate.** A registry that propagates means one buggy invalidator can break the write path — concretely: a future cache-clear callback that fails because the cache is in a corrupt state would surface as a 5xx on every mutation, even though the mutation itself succeeded. The shape "log the error, skip the listener, continue with the next" preserves write-path resilience and pushes the failure into ops logs where it's actionable but not catastrophic. Tests pin both halves: subsequent listeners still fire after a throw, and the caller sees normal return.
    - **Why `priorVersion: nextVersion - 1` not `doc.semanticModel.version ?? 0` at the call site.** Subtle ordering: by the time the hook fires (after `_updater(doc)`), the line `doc.semanticModel = nextModel;` has already executed, so `doc.semanticModel.version` is `nextVersion`, not the prior. Reading it would emit `priorVersion === nextVersion`, breaking the monotone-chain test. The arithmetic form `nextVersion - 1` is unambiguous because `nextVersion` is a local declared at the top of the lock callback and is never reassigned. A future refactor that captures `priorVersion` as a local at the top of each function would be marginally cleaner but would touch four functions for zero behavioural change; the canonical `nextVersion - 1` form is intentional minimal-churn.
    - **Why the module exposes `__clearInvalidatorsForTesting` in addition to per-test unsubscribe.** Module-private state survives across tests in the `node --test` runner (no fresh `import` per test). Without a clear shim, a test that registers a listener and then crashes / aborts before unsubscribe leaves the listener wired into the next test. `__clearInvalidatorsForTesting` is the belt-and-braces reset; per-test code can also use unsubscribe defensively, but the central clear is the safety net.
    - **Why two test files, not one.** Module-level tests (`semanticModelInvalidateW61CacheInvalidate.test.ts`) pin the hook's pure-function contract — registry semantics, log shape, throw isolation. Controller-level tests (`adminSemanticModelInvalidateW61CacheInvalidate.test.ts`) pin the integration contract — every mutation path fires the hook, error paths do not. Combining them would force one file to import both the hook module AND the controller fakes, doubling the test-fake surface for every test; splitting matches the W61-references-dashboards (scanner module + controller endpoint) precedent.
    - **Why we don't expose the event payload through a custom log format helper.** The single `console.log(`[semantic-model-invalidate] sessionId=${event.sessionId} ...`)` inline template literal is the simplest shape — a `formatInvalidationLog(event)` helper would be one indirection per call with no benefit (the template literal is already grep-stable). A future structured-logging refactor (JSON logs across the server) would centralize this anyway; for now the inline template literal is fine.
    - **Why we skip exposing `__forceFireInvalidatorsForTesting`.** Some test harnesses use a "force-fire" shim that triggers all registered listeners with a mock event. We don't need it — the unit tests drive `onSemanticModelVersionBumped` directly, and the integration tests drive real controller handlers. A force-fire shim would just be a thinner wrapper around `onSemanticModelVersionBumped` and tests should call the canonical entry-point.

  **Why this design.**
    - **Why a separate scanner module instead of widening `semanticModelReferences.ts`.** The chat-doc walker takes `ReadonlyArray<unknown>` (charts) and returns `{ chartCount, totalOccurrences }`. The dashboard walker takes `ReadonlyArray<unknown>` (dashboards) and returns a dashboard-level + tile-level aggregate — different inputs, different output shape. Folding both into one function would mean a discriminated return type and a kind parameter; two pure modules read more directly. The chat-doc walker is also potentially consumed by other call sites (a future blob-stored-chartReferences mode would re-use it); the dashboard walker is unique to the references-endpoint and benefits from staying isolated.
    - **Why prefer `sheets[].charts[]` and fall back to `dashboard.charts[]` only when sheets is absent/empty.** Modern dashboards (post the DR series) keep `dashboard.charts[]` in sync as the union of all sheet charts — see [`dashboard.model.ts`](../server/models/dashboard.model.ts) `createDashboardFromSpec`, `patchDashboard`, `addChartToDashboard`. Walking both arrays would double-count every tile. Pre-sheets dashboards (legacy) only populated `dashboard.charts[]` and had `sheets` absent or empty; falling back preserves coverage for those rows. A dedicated test (`a dashboard whose only matching tile is on the legacy charts[] fallback contributes once`) pins the sheets-prefer-over-legacy invariant explicitly.
    - **Why "tile" rather than "chart" in the dashboard-side counts.** A single underlying ChartSpec object can be referenced from multiple sheets (the chat-flow agent re-uses chart specs across the Summary and Evidence sheets). For the destructive-op warning the admin needs to know "how many positions on dashboards will lose data," not "how many distinct chart objects exist" — each position is a tile, and removing the metric breaks each position independently. "Tile" also matches the dashboard surface's UX terminology (a sheet is a grid of tiles).
    - **Why a new `_dashboardListerForUser` injectable instead of overloading `_detailFetcher`.** The dashboards container is a different Cosmos resource (partitioned by username, not sessionId). The W61-references-endpoint and W61-audit-history-api re-use `_detailFetcher` because they read the same chat-doc; this wave reads a different document type from a different container, so the injectable boundary follows the container boundary. Introducing a dedicated lister codifies the **"one injectable per Cosmos container"** convention: chat docs use `_detailFetcher`; dashboards use `_dashboardListerForUser`; future shared-dashboards / past-analyses / shared-analyses reference-walk waves get their own injectables rather than reaching into existing ones. This keeps the test-fake surface proportional to actual read paths and prevents one injectable from accumulating multi-container fetch logic.
    - **Why `ReadonlyArray<unknown>` as the lister return type, not `Dashboard[]`.** The scanner's defensive guards (`typeof === "object"`, `Array.isArray`) apply uniformly — a malformed Cosmos row that doesn't match the runtime shape is silently skipped by the scanner rather than parsed eagerly here. The production `getUserDashboards` returns `Dashboard[]` (via an unsafe runtime cast inside the function), but the typed return at the lister-injectable boundary would force eager parsing that could throw on a malformed row mid-iteration. `ReadonlyArray<unknown>` defers the discrimination to the scanner where it's already handled defensively.
    - **Why the empty-username short-circuit.** A chat doc with no `username` (system-test sessions, or pre-username-field-introduction docs) used to slip through and would call `getUserDashboards("")` which would query the empty-string partition (returning nothing, but still costing a Cosmos round-trip and a `console.error` from the eventual catch). The explicit short-circuit returns `[]` immediately, saves the round-trip, and makes the no-username case visible in the test surface (a dedicated test asserts `listerCalled === false` for the empty-username branch).
    - **Why we widen the existing envelope rather than introduce a new endpoint.** Single round-trip per delete confirmation is load-bearing UX — the W61-delete-client modal opens, fires one fetch, renders the warning. A separate `/admin/semantic-models/:sessionId/dashboard-references?entry=<name>` endpoint would force two round-trips per delete and would also force the client to combine two responses into a single warning. The widened envelope keeps the API surface flat and the client logic linear. The trade-off — every references query now also hits the dashboards container — is acceptable because (a) most sessions have <10 dashboards owned by the user, (b) the dashboards container is partitioned by username so the query is single-partition, and (c) the W61-delete-client modal only opens on an explicit user action, not on every page render.
    - **Why two dashboard fields (`dashboardCount` + `dashboardTileCount`), not just one.** The admin needs both to size the blast radius. `dashboardCount` answers "how many distinct dashboards will be affected" (1 dashboard with 10 tiles is one rebuild; 10 dashboards with 1 tile each is ten owner conversations). `dashboardTileCount` answers "how many positions need to be fixed" (the absolute work surface). Surfacing only one would hide the bipolar signal: a single dashboard with many tiles vs many dashboards with few tiles look identical with one count. The combined-clause copy `"M tiles across K dashboards"` makes both visible in one sentence.
    - **Why we skip `dashboardTotalOccurrences` (the analog to in-chat `totalOccurrences`).** The in-chat subhead surfaces field-position-occurrence-density inside a chart (a metric bound to both x AND tooltip on a single chart contributes 2 occurrences). For dashboards, the actionable unit at the admin's level is the tile (the dashboard position) not the field-position-within-a-tile. If the admin needs that detail, they'd open the dashboard view; surfacing it in the confirmation copy would be more noise than signal. The scanner therefore doesn't compute `dashboardTotalOccurrences` and the envelope doesn't expose it; YAGNI applies.
    - **Why the verb agreement follows the LAST enumerated count, not the first or a fixed count.** `"1 chart and 2 tiles ... reference it"` reads naturally (the verb agrees with `2 tiles`, the noun nearest to the verb). `"1 chart and 2 tiles ... references it"` reads as ungrammatical (third-person-singular `references` agreeing with the distant `1 chart` is jarring). Mid-Atlantic-English convention is the verb agrees with the closest subject. The same rule produces the right output for the dashboard-only case (`"5 tiles across 2 dashboards ... reference it"` — last count is `2 dashboards`, plural) and the singular cases (`"1 chart and 1 tile across 1 dashboard ... references it"` — last count is `1 dashboard`, singular).
    - **Why we widen `buildDeleteReferencesWarning` in place rather than introduce a new helper.** The modal calls one helper per render, and the helper's return shape (`{ headline, subhead } | null`) is stable — the consumer is unchanged. A new helper would mean two helpers with overlapping concerns and the modal would need a branching layer to pick between them. Widening preserves the single-source-of-truth invariant for the copy.
    - **Why the chart-side subhead is preserved (not unified into a combined subhead).** The chart-side subhead surfaces when `totalOccurrences > chartCount` — exactly the case where one chart uses the entry in multiple field positions (heavy-usage signal). The dashboard side doesn't have an equivalent signal because we don't track field-positions-within-tiles. Combining them would require us to either (a) compute the tile-side analog (rejected for YAGNI), or (b) drop the chart-side signal (regression). The cleanest design keeps the chart-side subhead and accepts that dashboards have a flatter signal model.
    - **Why we update the existing W61-references-endpoint test file in place rather than create a new wave-specific endpoint test file.** The existing file's 4 200-path tests assert the envelope shape via `assert.deepEqual`; the envelope widened, so those assertions had to change either way. Adding the two new fields to the existing tests + a dedicated W61-references-dashboards test file for the dashboard-specific scenarios is the cleanest split: the existing file pins the chart-side semantics (still load-bearing); the new file pins the dashboard-side semantics. The alternative — leaving the existing tests asserting partial bodies — would have been an anti-pattern (tests that don't assert the full envelope are weaker).

  **Why this design.**
    - **Why `<Dialog>` not `<AlertDialog>` for the modal.** AlertDialog is for "confirm this destructive action" UX (single primary message + Cancel/Confirm buttons). Dialog is Radix's canonical form-modal — supports multiple inputs and a primary submit button. The W61-delete-client used AlertDialog because the body was a single warning paragraph; this wave's body has 4-7 inputs depending on the kind. Using the wrong primitive would lose semantic correctness (screen readers announce the two differently) and would also lose the multi-field form-validation patterns Dialog supports.
    - **Why `NameAlreadyExistsError` instance not error-message string-matching.** HTTP 409 carries structured JSON `{ error: "name_already_exists", sessionId, kind, name }`. Parsing the body in the helper and re-throwing a typed Error subclass lets the host distinguish via `instanceof` rather than fragile regex on `err.message`. The pattern keeps the host's error-handling branches type-narrowed — `instanceof NameAlreadyExistsError` proves `err.kind` and `err.entryName` are defined, no `?? "unknown"` fallbacks needed. Future i18n changes to the error message don't break the branch logic.
    - **Why the modal owns the draft (parent owns the mutation).** Mirrors W61-delete-client. Draft state is mount-scoped — it lives for the duration of the modal and is GC'd on close; parent ownership would force a `clearDraft` step on every close + a stale-draft concern when admin opens for one kind then re-opens for another quickly. The mutation needs to update parent `data` (the model) on success, which the modal can't do on its own; parent-owned mutation + signalling close via `onOpenChange(false)` is the proven shape from delete.
    - **Why `addOpen: AdminSemanticModelEntryKind | null` not `{ open: boolean, kind: AdminSemanticModelEntryKind | null }`.** The non-null IS the open signal. Two-prop shape would invite the invalid state `open=true && kind=null` that the modal would have to defend against. Single-discriminant shape is the same pattern as W61-delete-client's `pendingDelete: { kind, name } | null`.
    - **Why validate at submit (not on every keystroke).** Keystroke validation would show a red border on the name field the moment the admin types `M` (PascalCase rejected) before they've finished typing `Monthly_Revenue` (also rejected, but the admin's mid-keystroke). Submit-time validation matches the W61-edit-text pattern (validate-on-blur is roughly equivalent to validate-on-submit-in-a-modal). The exception: name-collision after submit, which is rendered immediately because the admin has finished typing and seen the failure.
    - **Why run every validator at submit (not short-circuit on first failure).** If the admin submits a form with three invalid fields, they should see all three errors at once — fixing one and re-submitting only to discover the next is a worse UX than seeing all three the first time. Hence the `results = [validateField(...), ...]; results.every(Boolean)` pattern rather than chained `&&`.
    - **Why `+ Add` buttons per-card, not a single page-level dropdown.** Locality. An admin reading the Dimensions table thinks "I need to add a dimension here". A page-top dropdown forces a kind-pick step before the form opens; per-card buttons skip that step. The button label echoes the card heading (`Metrics` card → `+ Add metric` button) for visual scan-ability.
    - **Why `<Button variant="outline" size="sm">` for the Add buttons.** Add is non-destructive but also non-primary at the page level (the page's primary surface is the existing entries — the admin is reviewing, not authoring). Outline reads as "secondary action" — the same visual weight as the SourceFilterChips it sits next to. Default-variant would steal visual focus from the table content; ghost-variant would under-signal the action's significance.
    - **Why we mirror `validateName` server-side regex exactly.** Saves a round-trip — typing a PascalCase identifier surfaces inline immediately rather than after a 400 round-trip. The server's `safeParse` is authoritative; drift just degrades UX (one extra round-trip) without correctness damage.
    - **Why hierarchy levels are a textarea (one per line) not a tag-input or chip-list.** The tag-input libraries (react-select multi, downshift, etc.) are heavier and require additional dependencies; the textarea-with-parser shape is good enough for this density (most hierarchies have 2-4 levels). A future W61-hierarchy-edit wave could swap in a richer reorderable UI with drag-and-drop; defer until use case demands.
    - **Why the kind-change handler in DimensionFields resets `temporalGrain` to `__auto__`.** Switching `kind` from `"temporal"` to e.g. `"categorical"` would leave a stale grain value in the draft. On switch BACK to `"temporal"`, the prior grain would re-surface — confusing because the admin's intent during the categorical detour was unrelated. Resetting to `__auto__` on each kind change keeps the form state aligned with the visible UI.
    - **Why we don't pre-fill the `references[]` field on metric add.** The schema's `.default([])` populates an empty array if the field is absent, which is the right default for a brand-new metric (the admin doesn't yet know which columns the expression touches; the W58 compiler's orphan-reference lint surfaces the gap on first execute). A future W61-add-references-field could add a tag-input for the columns; for now empty is fine.
    - **Why parent `addError` + `addNameCollision` are two separate slots.** The collision is field-scoped (rendered under the name field by `NameField`); the generic error is form-scoped (rendered at the bottom of the dialog body). Combining them would force the modal to either render the typed-409 message twice (once under name, once at the bottom) or surface generic errors in an incorrect location.

  **Why this design.**
    - **Why a dedicated POST rather than re-using W61-save's PATCH.** The PATCH path replaces the wholesale model — this forces the client to (a) round-trip the entire model body (potentially ~5–50 KB depending on session) for what's logically a single-entry append, and (b) re-implement name-uniqueness validation locally before sending (the server's `safeParse` would accept duplicate names because the schema doesn't enforce within-collection uniqueness). The typed POST takes just the new entry as the body, the server validates uniqueness inside the lock against the authoritative state (not a stale client view), and the response carries the full new model so the client mutation success handler is identical to save / revert / delete. The four-endpoint family now reads as a coherent set: save = wholesale replace, add = append one, delete = remove one, revert = restore prior.
    - **Why 409 Conflict for `name_already_exists`, not 404.** HTTP 409 semantically means "the request conflicts with the current state of the resource" — exactly the case here (we're trying to create an entry that already exists by name within the same kind). The W61-revert / W61-delete endpoints use 404 for `*_not_found` because the resource is absent; for `name_already_exists` the conflict semantic is collision, not absence. The client can distinguish 409 from 4xx generally and render a "this name is already taken" inline message rather than the generic "something went wrong" copy. The 409 envelope includes `{ sessionId, kind, name }` echo so the client doesn't need to track which request triggered the collision.
    - **Why cross-kind name collisions are explicitly allowed.** A metric named "revenue" and a dimension named "revenue" are different resources — the planner addresses them via `{ kind, name }` not `{ name }` alone, and the SemanticModel schema doesn't forbid the collision (each kind has its own collection in the model). Requiring `:kind` in the URL gives the uniqueness check the right scope: same-kind collisions are rejected with 409, cross-kind collisions are allowed. A dedicated test pins this behaviour (`metric "revenue"` exists; `POST /entries/dimension` with `body.name === "revenue"` → 200, both coexist in the model).
    - **Why we skip the source-bumper on add.** Same precedent as W61-delete-server: survivors are byte-identical to their prior selves (we're appending, not editing existing entries), so the bumper's content-hash diff would match them as unchanged and preserve their prior `source`. Running it would be a no-op for the existing entries. For the new entry, the bumper's lookup against `priorByName` returns undefined (the new name doesn't yet exist in the prior model — we just verified that with the uniqueness check) so the bumper's "new entry" branch returns the entry unchanged with its client-sent `source` preserved. The schema's `.default("user")` on each source field populates new entries by default; if the client sends `"domain"` (e.g., importing from a domain pack) it's preserved verbatim. Skipping the bumper call entirely is clearer about intent — the documentation comment explains the no-op equivalence so a future reader doesn't waste time wondering why W61-add doesn't bump while W61-save does.
    - **Why audit-write BEFORE the append (not after).** Same precedent as W61-save / W61-audit-revert / W61-delete-server. The buffer's role is to remember what was just changed; if the audit-write happened after the mutation, a crash between the two would lose the snapshot. Inside `withSessionWriteLock` the pair is atomic at the per-session level (single-instance correctness per invariant #9). A future "undo this add via revert" walks the audit log, finds the pre-add snapshot, and restores it — wiping the just-added entry. The audit-write-before-mutation pattern is the same shape across all four W61 endpoints; the consistency makes the four-endpoint family interchangeable from the audit-trail perspective.
    - **Why bump `version` on add (we're not changing existing entries).** W64's compiled-query cache keys on `semanticModel.version`. Adding a new metric — even one with a snake_case name that no existing chart references — could affect compilation of future queries: the planner might suggest the new metric in subsequent turns, the W58 compiler might emit lint warnings for orphan references against the new entry's `references[]` columns, etc. Bumping makes the cache miss explicit so the next planner round consults the fresh model. The cost of an unnecessary cache miss (one extra compilation) is much smaller than the cost of stale compilation against an out-of-date model.
    - **Why we reuse the existing `_detailFetcher` / `_updater` injectables.** Same Cosmos doc read+write path as save / revert / delete — fetch the chat document by sessionId, mutate `semanticModel` + `semanticModelAuditLog` + `lastUpdatedAt`, write the whole doc back. A new pair of add-specific injectables would just be the same functions under different names and would double the test-fake surface for no behavioural benefit.
    - **Why the kind-appropriate body schema is picked via ternary, not a `Record<kind, schema>` lookup.** A `Record<"metric"|"dimension"|"hierarchy", z.ZodSchema>` would type-check fine, but the three zod schemas have different inferred entry types (`SemanticMetric` vs `SemanticDimension` vs `SemanticHierarchy`) — the ternary preserves the per-kind type narrowing inside the `if (parsedKind.data === "metric")` blocks where we cast `parsedBody.data` to the concrete kind type. A `Record` lookup would resolve to a union type that the `.some((m) => m.name === entry.name)` line would have to type-narrow again. The ternary is uglier but reads more directly as "pick the right schema for this kind".

  **Why this design.**
    - **Why a controlled `<AlertDialog>` rather than `window.confirm`.** The downstream-references count is async — the modal needs to render `"Checking for downstream chart references…"` while the round-trip is in flight. `window.confirm` is synchronous and can't show intermediate states; the W61-audit-revert path uses `window.confirm` because its only context is the entry already in hand, but the delete path is references-count-first. The async dependency dictates the UI primitive.
    - **Why the modal owns the references fetch, not the parent.** References state is mount-scoped — it lives for the duration of the modal and is garbage-collected on close. Keeping the fetch / loading / error state inside the component lets React's effect cleanup handle the teardown automatically. Putting the references in the parent would force a `clearReferences` step on every close + a stale-fetch guard when the admin re-opens for a different entry quickly. The audit-history-tab's references-fetch-in-parent shape is appropriate THERE because the parent re-fetches on every `data.lastUpdatedAt` bump (save / revert grow the buffer); for delete-confirm the references count is per-modal-instance and doesn't outlive it.
    - **Why the parent still owns the delete mutation.** Success updates the parent's `data` (the model). The modal can't update parent state on its own; it would have to call back to a parent callback regardless. Putting the mutation in the parent + signalling close via `onOpenChange(false)` is the simpler shape. The parent's `deleting` flag disables both modal buttons while the DELETE is in flight; the `e.preventDefault()` on the action button keeps Radix from closing the modal early so the admin sees the result.
    - **Why a trailing Actions column rather than an inline button next to each entry name.** Visual scan-ability. An "Actions" column reads as a destructive-op landing strip; an inline delete button next to the entry name would compete with the SourceBadge chip and the EditableText label inputs for the eye. The trailing column is the same convention Radix uses for table actions. The `<th><span className="sr-only">Actions</span></th>` keeps the column accessible without consuming a visual header slot (the icon-only buttons in the body cells provide the visual cue).
    - **Why `Trash2` (lucide) rather than `X` or a text-only "Delete" button.** `Trash2` reads as "delete this entry"; `X` reads as "close this modal / dismiss this card". The icon-only button (no text label) keeps the trailing column narrow so the existing edit cells don't get squeezed; the `aria-label="Delete metric ${m.name}"` provides screen-reader coverage. Lucide is already a project dependency from W61-audit-history-tab's `ChevronDown`.

Reusing the type keeps the host's success handler shape symmetric across all three operations: same `setData({ ...data, lastUpdatedAt: res.lastUpdatedAt, model: res.model })` lands on every successful mutation, and the audit-history-tab's `useEffect` keyed on `lastUpdatedAt` re-fetches the buffer automatically.
    - **Why a separate sibling component for the modal, not inlined in the host.** The host was at 1,131 LOC post-W61-detail-extract. Inlining the modal (~270 LOC) would push it to ~1,400 LOC.     - **Why `e.preventDefault()` on `<AlertDialogAction>`.** Radix's default behaviour closes the dialog when the action is clicked, but we want the modal to stay open with `deleting=true` so the destructive button can show `"Deleting…"` and the parent can surface a success / failure outcome inline. Without `preventDefault`, a transient network failure would close the modal and lose the error visibility — the admin would see the page state unchanged with no breadcrumb explaining why.

Suppressing the reassurance on the error branch would leave the admin to wonder if the failed precheck also implies a broken audit-log path; explicit reassurance side-steps that doubt.

The subhead surfaces precisely when one chart uses the entry multiple times (e.g. the entry is bound to both `x` and `tooltip` channels): `"1 chart that references it"` + `"2 references total across the affected chart"` tells the admin "heavy usage in one chart" without forcing them to compute the divergence.   **Why this design.**
    - **Why path params for `:kind` and `:name`, not query / body.** Both are part of the resource identity — "which entry are we deleting?" DELETE has no body in idiomatic HTTP; query params are for filters, not identifiers. Path identifies the resource; the three-segment shape (`/entries/:kind/:name`) reads naturally as a hierarchical resource path.
    - **Why explicit `:kind`, not auto-scanning all three collections for the name.** Name collisions across kinds are legal — the semantic model schema doesn't forbid a metric and a dimension sharing a name. A fuzzy "find this name anywhere and delete it" would silently delete the wrong thing when a collision exists. Requiring `:kind` in the URL makes the operation unambiguous; a dedicated test (`kind=dimension` + `name=alpha` when only a metric named `alpha` exists → 404 `entry_not_found`) pins the no-fuzzy contract.
    - **Why audit-write BEFORE delete (not after).** The buffer's role is to remember what was just lost. If the audit-write happened after the delete, a crash between the two would lose forensics. Inside `withSessionWriteLock` the pair is atomic at the per-session level (single-instance correctness only; multi-instance horizontal scaling still needs Cosmos `ifMatch` ETag). Mirrors the W61-audit-log + W61-audit-revert precedent.
    - **Why bump `version` even though we removed something.** The version is W64's compiled-query cache key. Removing a metric invalidates any cached compilation that referenced it; bumping makes the cache miss explicit and ensures the next planner round consults the fresh model.
    - **Why we do NOT run `bumpMetricsSource` / `bumpDimensionsSource` / `bumpHierarchiesSource` on the survivors.** The surviving entries are byte-identical to their prior selves; the bumper's content-hash diff would match them as unchanged and preserve their prior source. Running it would be a no-op for unchanged entries; skipping is semantically equivalent and clearer about intent. A success-path test pins this: a metric with `source: "user"` and another with `source: "domain"` keep those sources verbatim after a third metric is deleted (no spurious bump-to-user on the survivors).
    - **Why the audit-log write happens unconditionally on success.** A delete IS a state change; preserving the just-deleted entry in the buffer is required for "undo this delete via revert" to work. The W61-audit-revert wave's revert endpoint walks the buffer and restores `priorModel`; this wave's delete entry produces an audit entry whose `priorModel.metrics` (or `.dimensions` / `.hierarchies`) includes the deleted entry, so a future revert call can put it back.
    - **Why the `DeleteResult` discriminated union sentinel.** Throwing from inside the `withSessionWriteLock` callback would mark prior chained callers as failed (the lock serialises per-session writes); returning a sentinel from the callback and mapping it to an HTTP status in the outer handler keeps the per-session error semantics clean. Same pattern as `PatchResult` (W61-save) and `RevertResult` (W61-audit-revert).
    - **Why `removed` is set then `void removed`.** The early `return { kind: "entry_not_found" }` in each branch guarantees the flag is true on every success path; asserting it would be dead code. Keeping the variable (with the `void` to silence the unused-variable lint) makes the per-branch shape read consistently — a future fourth kind would set `removed = true` in the same way. The cost is one line; the benefit is symmetric branches.
