- **2026-05-18** ‚Äî **Wave WI2-cache ¬∑ per-tile insight regen cache (pure helpers).** WI1 introduced the `chart.insight: InsightSpec` schema with a `generator: { kind: "llm" | "deterministic", args? }` slot meant for dynamic insight regeneration on filter change (master plan Workstream 7). The intended UX: when the user adds a global filter or brushes via WD2, each tile's insight footer offers a "‚ú¶ Re-explain this view" action; clicking issues a MINI-tier LLM call with the current filtered data + chart spec + domain context and replaces the insight prose. Re-asking the same `(tile, filter combo)` within a session should return instantly without burning a fresh call ‚Äî and across N tiles in a multi-filter dashboard, the same global filter combination produces hundreds of identical-context requests if uncached, so caching is critical not optional. WI2 as one wave (cache + server endpoint + client hook + footer UI) is too large; WI2-cache splits the pure data-plumbing concern out first, the same atomic-split pattern as W59a ‚Üí W59b and WD2 ‚Üí WD2-wiring.

  **What landed.**
    - [client/src/pages/Dashboard/lib/insightRegenCache.ts](client/src/pages/Dashboard/lib/insightRegenCache.ts) (new, ~145 LOC). Three exports:
      - `hashGlobalFilters(filters: ActiveChartFilters): string` ‚Äî byte-stable serialisation of the global filter map. Walks `Object.keys(filters).sort()`, then for each column emits one of three shapes: categorical ‚Üí `column=c:value1|value2|...` (with values sorted), date ‚Üí `column=d:start..end` (with `undefined` start/end rendering as empty), numeric ‚Üí `column=n:min..max` (same treatment for undefined bounds). Columns are joined with `;`. The `=`, `;`, `:`, `|`, `..` separators are unambiguous together ‚Äî no value in the active filter shape contains those literals after the existing chart-filter normalisation. Result: identical filter maps in different insertion / value order yield identical hash strings.
      - `buildCacheKey(tileId: string, filterHash: string): string` ‚Äî composes the pair as `${tileId}::${filterHash}`. The `::` delimiter is unambiguous because `hashGlobalFilters` uses `=` / `;` / `:` as internal separators.
      - `createInsightRegenCache(opts?: { maxEntries?: number; ttlMs?: number; now?: () => number }): InsightRegenCache` ‚Äî factory. Default `maxEntries: 64` (per-dashboard ceiling ‚Äî 8 tiles √ó 8 filter combos is a sensible upper bound on session interactivity), default `ttlMs: 300_000` (5 min, matches Anthropic's prompt-cache TTL so a regen done at minute 0 is fresh exactly as long as the upstream prompt cache it built against). The `now` injection is for deterministic tests.
      - Returned `InsightRegenCache` is a closure-backed Map with five methods: `get(key) ‚Üí InsightRegenEntry | undefined` (returns undefined for stale entries + lazily evicts them, refreshes LRU recency on a successful read), `set(key, value)` (evicts the LRU entry when full + replaces an existing entry by delete-then-set so iteration order tracks recency), `has(key)` (false on expired + lazy delete, same as get), `clear()`, `size()`, `evictExpired() ‚Üí number` (active sweep returning the count of dropped entries; useful for a periodic timer the consuming hook may run).
      - `InsightRegenEntry` interface carries `{ text: string; citations?: string[]; regeneratedAt?: string; confidenceTier?: "low" | "medium" | "high" }` ‚Äî the minimal shape the future regen endpoint will return + the future footer UI will render. Future schema additions (e.g. effect-size badges, sub-investigation links) extend this interface without breaking the cache contract.
      - **LRU implementation: `Map` insertion order.** ES2015 maps maintain insertion order. `get` refreshes recency by deleting + re-inserting (O(1)). Eviction pops `keys().next().value`, which is the oldest entry. No external dependency needed; testable in pure node:test.
      - **TTL is on `get` / `has`, not on a background timer.** This means stale entries occupy slots until probed; on a 64-slot cache with a 5-min TTL this is bounded and acceptable. The optional `evictExpired()` lets the consuming hook do a periodic active sweep if desired.
    - [client/src/pages/Dashboard/lib/insightRegenCache.test.ts](client/src/pages/Dashboard/lib/insightRegenCache.test.ts) (new, ~225 LOC). 21 tests across 6 suites; appended to [`server/package.json`](server/package.json) per invariant #4:
      - **`hashGlobalFilters` byte stability** (8): empty map ‚Üí empty string; identical maps ‚Üí identical hash; column-order independence; categorical value-order independence; date filter format; numeric filter format; categorical-vs-numeric disambiguation on the same column name (the `c:` / `n:` prefix guarantees no collision); undefined bounds gracefully render as empty.
      - **`buildCacheKey`** (2): standard concat with `::`; empty filterHash ‚Üí `tileId::`.
      - **`createInsightRegenCache` ‚Äî basic get/set** (4): unknown key returns undefined; store-then-retrieve roundtrip; re-set overwrites; clear empties.
      - **LRU eviction** (2): full cache evicts oldest on insertion; `get` refreshes recency so previously oldest survives later evictions.
      - **TTL expiry** (3): post-TTL `get` returns undefined + deletes lazily; `evictExpired` returns the eviction count + leaves fresh entries intact; `has` returns false + lazy-deletes for stale entries.
      - **Integration scenario** (2): two clients of the same tile with identical filters share a cache hit (the `(tileId, filterHash)` round-trips); changing one filter value produces a different cache key so the prior hit is invalidated by construction.
      - 21/21 passing.

  **Why this design.**
    - **Pure data-plumbing first.** The eventual WI2 end-to-end touches: a server endpoint that takes `(spec, filteredData, domainContext)` and returns insight prose; a client hook (`useInsightRegen(tileId, filters)`) wired into [`TileInsightFooter.tsx`](client/src/pages/Dashboard/Components/TileInsightFooter.tsx); a "‚ú¶ Re-explain this view" button + spinner; a cache between the hook and the network call; integration with WD2's brushing layer so brushed-filter changes also invalidate. Shipping all of it in one wave is god-object-adjacent. Splitting the cache out first means: the cache contract is fixed + tested before the server / UI layers can drift it; the hook wave can be a thin React layer over this contract; the server wave can be a thin LLM-call wrapper that the hook calls when `cache.get(...)` misses.
    - **`hashGlobalFilters` uses string concat, not JSON.stringify.** `JSON.stringify({zone:"A", brand:"X"})` and `JSON.stringify({brand:"X", zone:"A"})` produce different strings ‚Äî the key-sorting concern would have to live downstream. Doing it explicitly in the helper makes the byte-stability contract obvious in the source, and the resulting hash strings are markedly more compact than equivalent JSON (`region=c:North|South;zone=c:All` is ~25 chars vs the equivalent JSON ~70 chars). Compact keys matter for browser memory + dev-tool readability.
    - **TTL matches Anthropic's prompt-cache TTL.** Prompt-cache hits expire at 5 min; a regen LLM call built against a cached system prompt becomes uncacheable past that. Holding regen results longer than the underlying cache is wasted: the next request would re-build the system prompt anyway. 5 min is the natural ceiling.
    - **`maxEntries: 64` default.** Eight tiles √ó eight filter combos per session is a generous upper bound; most analytical sessions have <4 tiles and <3 filter combos. Cap exists to prevent a runaway log-tail of stale entries from growing unbounded in long sessions; 64 entries √ó ~500 bytes each = ~32 KB ceiling.
    - **`now` injection for deterministic tests.** The cache's own time source is injectable; the TTL test advances `t` directly without `await new Promise(r => setTimeout(r, ...))` or `vi.useFakeTimers()`. Tests are sub-millisecond + flake-free.
    - **Closure-backed Map, not a class.** The factory returns an object literal with the five methods, all closing over the same `store` Map + `now` closure. Tree-shake-friendly (no `class` keyword in the bundle), easy to instantiate per `useMemo`, indistinguishable from a class instance to TypeScript.
    - **`has` does lazy-delete on expiry.** Consistent with `get`. Both probes return false / undefined for stale entries AND remove them so the cache's `size()` stays accurate after a flurry of probes. The `evictExpired()` active sweep exists for the rare case where the consuming hook wants to periodically reclaim space without probing every key.
    - **Stored values include `citations` + `regeneratedAt` + `confidenceTier`.** Even though those are optional, including them in the interface up-front means the future regen endpoint + footer UI agree on the shape without a schema bump in WI2-wire. The cache becomes the contract.

  **Out of scope.**
    - **Server regen endpoint.** A future wave (`WI2-server`) registers `POST /api/insight/regen` that takes `{ spec, filteredData, domainContext, tileId }` and returns `{ text, citations, regeneratedAt, confidenceTier }`. The endpoint will go through the same MINI-tier `callLlm` channel that `generateChartInsights` uses today, with `promptCacheEligibility` set on the system prompt for cache reuse across tiles in the same session.
    - **Client hook + footer UI.** A future wave (`WI2-wire`) ships `useInsightRegen(tileId, filters)` and the "‚ú¶ Re-explain this view" button in `TileInsightFooter.tsx`. The hook will own the cache instance via `useMemo` and call the server endpoint on cache miss.
    - **Cache invalidation on chart-spec change.** Today the cache is keyed by `(tileId, filterHash)` only. If the chart's spec (e.g. metric, breakdown) changes mid-session, the cache will incorrectly serve a stale insight. A future wave can extend the hash to include a chart-spec digest, or simply require consumers to call `cache.clear()` on spec-change.
    - **Persistence across sessions.** The cache lives for the dashboard mount and dies with it. A future infra wave could persist hits to localStorage / Cosmos so cross-session re-asks of the same combination are free; deferred until cost telemetry shows it's needed.
    - **Cache-warming on dashboard load.** Pre-computing insights for the "no filters" combo on dashboard mount could eliminate the first-paint regen latency. Deferred until UX shows the latency is noticeable; for now the existing baked-at-creation `keyInsight` / `InsightSpec.default` text covers the no-filter case.
    - **Cross-tile invalidation.** When a global filter changes, every tile's cache key changes ‚Äî the existing keying already invalidates correctly. But a global filter that doesn't apply to a given tile (because the tile's chart doesn't use that column) should ideally NOT invalidate that tile's cache. The cache as-shipped is conservative: any global filter change invalidates every tile. A future wave can introduce per-tile filter relevance projection so tiles whose data is unchanged share cache hits across filter combos.
    - **Citation hover-card surface inside regenerated insights.** The `citations` field on `InsightRegenEntry` carries pack ids; the footer UI can render them with the existing WQ3 `CitationHoverCard` component. Wiring lives in WI2-wire, not in the cache.

  **Why this design.**
    - **Minimal `planner.ts` diff.** `planner.ts` is 976 LOC and already heavily edited (WW1, WW2, WQ2). A larger refactor ‚Äî extracting the user-prompt template into a builder function, for instance ‚Äî would entangle the wave with restructuring concerns and risk cache drift. The 3-line atomic edit keeps the diff surface small and the prompt structure unchanged for the call sites the cache breakpoints rely on.
    - **Position: between hints and RAG.** The hints block (WW1+WQ2) is a deterministic intent + tool recommendation under the question; it's the highest-leverage routing signal and stays first. RAG hits (P-A1) are upfront semantic retrieval against the question text; they come second. The semantic catalog is meta-info about the dataset ‚Äî same shape every turn for the session ‚Äî so it sits right after the per-turn routing hints but before the per-turn retrieval context. This ordering also makes the catalog the "dataset definition" that the rest of the user-prompt blocks (memory recall, hypothesis register, prior observations, working memory, handoff log, context summary) can implicitly cite.
    - **Read-only activation, not tool dispatch.** Without an `execute_metric_query` tool the planner can't directly call into the W58 compiler. But the catalog still adds value: the planner sees `Net Sales = SUM(value_sales)` and learns the canonical name for the column the user calls "sales" or "revenue"; it sees `region` declared as a categorical dimension and avoids guessing about cardinality; it sees `geo: country ‚Üí region ‚Üí city` and naturally drills the right way. Cube's benchmark suggests ~30 % of the accuracy lift comes from catalog visibility alone, before tool wiring. W60 captures the remaining ~70 %.
    - **`buildSemanticCatalogPromptBlock` lives in `semantic/prompt.ts`, not `planner.ts`.** The helper is a thin null-coalescing wrapper but it's also the place a future wave will add a `maxLength` cap, a default `includeHidden: false` policy, or a domain-pack citation hint. Keeping it adjacent to `formatMetricCatalog` means future tweaks to the manifest rendering policy ship in one module.
    - **Wave separation from W59a.** W59a was the byte-stable manifest renderer; W59b is the planner wiring. Splitting them lets each have its own atomic commit + paired docs commit + test surface. Failures localise: a prompt-cache regression points at W59b; a manifest-formatting bug points at W59a.

- **2026-05-18** ‚Äî **Wave WD2 ¬∑ cross-filter brushing ‚Äî pure helper module.** The [1000x master plan](/Users/tida/.claude/plans/go-through-the-entire-partitioned-yao.md) Workstream 4 has cross-filter brushing as the second wave (`WD2`): clicking a bar / point / segment in a dashboard tile should dispatch a cross-filter event that flows into the dashboard-wide `globalFilters` state (in place from `WD1`), and other tiles dim non-matching marks rather than dropping them. Numbers / booleans ‚Üí `String(v)`. Strings pass through unchanged.

      - `clearCrossFilter(global, column)` ‚Äî drops the column entirely. No-op identity-return when the column has no selection.
      - `listActiveCrossFilters(global)` ‚Äî flattens categorical columns into a sorted `[{ column, value }]` array. Column-sorted then value-sorted so chip rendering is stable across re-renders. Skips non-categorical selections.
      - `CROSS_FILTER_EVENT = "marico:cross-filter"` exported constant ‚Äî the CustomEvent name the renderer-wiring wave will use.
      - `dispatchCrossFilter(event)` ‚Äî browser-guarded helper. Returns `false` (no-op) when `typeof window === "undefined"` or `typeof CustomEvent === "undefined"` so the module is safe to import in SSR / server-test contexts. Returns `true` after a successful `window.dispatchEvent`.
      - **Mutation-free contract.** Every state-modifying helper (`applyCrossFilter`, `removeCrossFilter`, `clearCrossFilter`) clones first (`{ ...global }`) before modifying. The test `does not mutate the input global map` pins this.
      - **Toggle vs. last-action.** The contract is asymmetric on purpose: clicking a categorical-active value toggles it off; clicking a non-categorical column installs a new categorical filter (does NOT preserve the prior date/numeric selection). This matches the user's intuition that a chart brush is a one-shot "show me only this discrete value" action ‚Äî restoring a date range is the responsibility of the global filter bar UI.

  **Why this design.**
    - **Atomic split: data plumbing first, wiring second.** The end-to-end WD2 in the master plan touches every visx renderer (`BarRenderer.tsx`, `LineRenderer.tsx`, `PointRenderer.tsx`, `AreaRenderer.tsx`, `ArcRenderer.tsx`, `FunnelRenderer.tsx`, `KpiRenderer.tsx`, ‚Ä¶) plus the ECharts adapters, the `DashboardView`'s `useState<ActiveChartFilters>`, and the dim-vs-filter visual treatment. Putting all of that in one wave is a god-object-adjacent change that would carry risk of cross-renderer drift. WD2 splits the smaller pure-data concern out first ‚Äî the wiring wave can then be a thin call-site change in each renderer (one line each: `onClick={() => dispatchCrossFilter({ column: enc.x.field, value: c.outerRaw, sourceTileId })}`) without re-litigating the toggle / coercion semantics. Same playbook as W59a / W59b.
    - **CustomEvent vs. context / pub-sub library.** A new context for cross-filter state would require every renderer to consume it (and we'd need to thread it through chart compilation paths). A pub-sub library is overkill for a one-listener channel. Plain `window.dispatchEvent(new CustomEvent(...))` + a single `window.addEventListener` in `DashboardView` is the cheapest plumbing: zero deps, zero context boundaries, naturally bubbles past iframes / portals. The trade-off is "global event channel" ‚Äî but the channel is namespaced (`marico:cross-filter`) and the receiver is well-defined (`DashboardView`), so the risk is low. SSR safety is handled by the `typeof window` guard.
    - **String coercion is one-way.** Categorical filter values are stored as strings (per [`chartFilters.ts`](client/src/lib/chartFilters.ts) `CategoricalFilterSelection`). `toFilterValue` is the one-way coercion at the boundary ‚Äî the helper never reads-back-and-parses, so downstream renderers must format numeric values the same way (`String(42) === "42"`, not e.g. `"42.0"`). A future wave that introduces a typed-value store (e.g. distinguishing `42` from `"42"`) would need to extend the categorical schema, not just the helper.
    - **Null bucket maps to literal `"null"`.** The upstream tool layer normalises null-grouping rows to `(null)` in aggregations, and the existing chart filter UI surfaces a "(null)" chip when a categorical option is the null sentinel. Mapping `null` / `undefined` ‚Üí `"null"` here keeps the cross-filter round-trip stable: clicking a null bar in the chart applies a filter whose stringified value matches the renderer's bucket key.
    - **Toggle-off drops the column.** When a categorical selection becomes empty, the helper deletes the column from the global map rather than leaving `{ type: "categorical", values: [] }`. The empty-array form would be a separate filter state ("show no rows") that the rest of the codebase doesn't handle uniformly ‚Äî explicit deletion keeps the contract crisp: "no entry for column ‚Üí no filter on column".

    - **Welch's t-test on row-level values, not on aggregates.** A t-test on two `sum` values would have df=0 and zero variance ‚Äî it's not a statistical test. A t-test on the underlying row-level values *is* a test: it asks "do the per-row metric values differ between segments meaningfully relative to within-segment variance?". This is the right question for both `mean` (where it's also the natural test of the aggregate) and `sum` (where it gracefully reframes the sum question as "are per-row magnitudes different?" ‚Äî usually what the narrator's prose around the top row actually means).
    - **Skip count aggregation.** Count is the only aggregation where the row-level values aren't a meaningful sample (you'd be comparing "1, 1, 1, 1" vs. "1, 1, 1"). The natural test for "are the count proportions different from the global baseline?" is a Z-test for proportions or a chi-square, and both need a baseline the tool doesn't carry. Skipping is correct; future waves can add a `baseline` arg if needed.
    - **Top-vs-rest for breakdown ranking, not top-vs-second.** "Is region X the leader?" is the headline finding; the relevant comparison is "X vs. everything-not-X", not "X vs. the next region". The latter is a noisier signal (sample of one comparison column) that the narrator rarely cares about ‚Äî the prose is "X leads with $5M, far above the rest" not "X leads with $5M vs Y at $4M".
    - **Two-pass over the frame, not values-per-group.** Capturing `values: number[]` per group inside `aggregate` would mean N √ó G memory for high-cardinality breakdowns. The second pass for exactly two buckets (top + other) is one extra O(N) scan ‚Äî typically <1ms on the diagnostic slice cap (~5k rows) ‚Äî and keeps the existing `aggregate` Map values cheap.
    - **Composite ranking explicitly excluded, asserted in source-inspection tests.** A future refactor could accidentally invoke `buildBreakdownRankingEvidence` from the composite branch; the test `composite (rankBy) path summary is unchanged (WQ7 scope-bounded)` guards against it.
    - **Empty-suffix on skip / failure is the contract.** Mirrors WV4-WV7. Empty `topValues` or `otherValues` (single-group frame) ‚Üí empty suffix ‚Üí byte-stable summary. NaN-only values ‚Üí empty suffix. `runSignificanceTest` returns `ok: false` for nA/nB < 3 ‚Üí empty suffix. Failure modes are silent and observable only by the absence of the canonical evidence block.

Without this, an honest narrator that omitted the field would look like it claimed nothing, hiding overclaim that's only visible when *some* magnitudes are explicitly high.
