  **Why this design.**
    - **Why bind to `xPx(tooltipData.xRaw)` (the snapped x) rather than `tooltipLeft` (the raw cursor pixel).** The tooltip already snaps to the nearest data point via the existing `onMouseMove` handler's nearest-x scan â€” that snap is what makes the tooltip's row values meaningful (the user sees Y values at a discrete X, not at an interpolated cursor position). The cross-hair MUST follow the same snap so the indicator visually aligns with the tooltip's row values. If the cross-hair tracked `tooltipLeft` (the raw `pt.x` from `localPoint(e)`), the indicator would slide smoothly across the chart while the tooltip's title jumped between discrete buckets â€” a visual contradiction that would erode the user's trust in either the tooltip OR the cross-hair. The negative test for this binding (`const cx = xPx(tooltipData.xRaw);` pinned exactly) is the load-bearing invariant â€” any refactor that "simplifies" by using `tooltipLeft` re-introduces the failure.
    - **Why three guards (`tooltipOpen && tooltipData && brushStart === null`) rather than just `tooltipOpen`.** `tooltipOpen` alone is theoretically sufficient (you can't have a snap without an open tooltip), but the `tooltipData` guard is required for TypeScript narrowing (the IIFE reads `tooltipData.xRaw`, which would be `undefined` without the narrow). The `brushStart === null` guard handles a subtle visual conflict: during an active brush drag, the existing brush rectangle renders a dashed border at `hsl(var(--primary))` â€” if the cross-hair drew on top of (or under) that rectangle, the two dashed shapes would visually compete and the user would lose the brush-region affordance. The clean separation is "brush owns the visual during drag, cross-hair owns it during pure hover" â€” gated by `brushStart === null` which IS the "no active drag" condition. The triple-guard is pinned by the conjunction test in the new test file.
    - **Why dashed `strokeDasharray="3 3"` at 0.45 opacity rather than solid or higher-contrast.** The cross-hair is a HINT, not a DIVIDER. A solid stroke at full opacity (e.g., `hsl(var(--foreground))` at 1.0) would over-assert against the data lines underneath â€” the data is the figure, the indicator is the ground. The chosen style (muted-foreground at 45% opacity, dashed pattern matching the brush rect's own dash style for stylistic consistency, 1px width â€” the thinnest visible) is the smallest visual footprint that still reads as "this is the hovered x". The dashed-not-solid invariant is the visual contract pin in the test file; the exact opacity / color / width values are intentionally not pinned so future visual-design tweaks can adjust without test churn.
    - **Why placed UNDER the data lines (rendered earlier in the Group) rather than as the topmost overlay.** Standard financial-chart layering: the cross-hair is a passive indicator the user reads at a glance; the data is the figure they study. Putting the indicator above the lines would visually break each line where the cross-hair crossed it, creating a 1px gap at the hovered x on every series â€” distracting. Placing it under the lines means the cross-hair only shows in the gaps BETWEEN series strokes (and above the grid background) â€” exactly where the eye expects a hint, never where the eye is reading data. This is the inverse of the brush rectangle's layering choice (the brush rect sits below the lines too, for the same reason â€” the user wants to see the data through the brush region, not behind it).
    - **Why no echarts-side cross-hair config in this wave.** The Explore-subagent's report noted echarts has native `axisPointer.type: 'cross'` available with ~3-5 LOC per renderer, no new state needed. But the 8 echarts renderers (Treemap / Sunburst / Sankey / Calendar / Candlestick / Choropleth / Gauge / Parallel) are NOT temporal-x renderers â€” they're hierarchical / categorical / map / radial marks where the "snap to a discrete x bucket" mental model doesn't apply. A cross-hair on a Treemap or a Sunburst is a category error; on a Sankey it would highlight a column irrelevant to the underlying flow lookup. The 8 visx trend renderers ARE the right targets; LineRenderer is the natural first-instance for the pattern. Future waves can mirror this exact shape onto AreaRenderer / BarRenderer / ComboRenderer / PointRenderer / etc., one at a time. If a specific echarts-side use case ever surfaces (e.g. Candlestick with a temporal x deserving cross-hair), THAT becomes a separate-scope wave with the axisPointer config approach.

    - **Why thread `sheetId` end-to-end as an opt-in additive field rather than tighten the resolver into an always-scoped shape.** Three reasons.   **Why this design.**
    - **Additive `sheetId` chosen over widening chartId to `chart-<sheetId>-N`.** The widening shape is conceptually cleaner â€” single field carries full chart identity â€” but the cost analysis runs against it on three axes. (a) **Test surface:** widening breaks every existing test that pins literal `"chart-N"` values, including `dashboardDrillThroughWD3Server.test.ts` lines 260-296 (the multi-sheet lookup tests that already encode the collision as expected behaviour), `dashboardDrillThroughTelemetryWD3Telemetry.test.ts` controller cases at lines 64-218, the equivalent block in `dashboardExplainSliceTelemetryWI4Telemetry.test.ts`, and the `dashboardTemplates.test.ts` literal pins. The Explore subagent that mapped the surface for this wave estimated ~30 test-line updates across those files. (b) **Cosmos historical rows:** widening creates a mixed-format aggregation surface â€” pre-wave rows have `chartId: "chart-0"`, post-wave rows have `chartId: "chart-<sheetId>-0"`. Any analytics query that tries to span the wave boundary has to parse both formats or back-fill the old rows by walking every dashboard's then-current sheet structure (which itself drifts as users edit dashboards). (c) **Renderer / server resolution churn:** widening propagates through every renderer that emits `dashboardTile.tileId` (15 visx + ECharts renderers per the Explore report), the server's `findChartByTileId` regex parser, and the `ExplainSlicePanel` chart resolution at DashboardView lines 1144-1150 â€” all of which would need to learn the new format. **Additive `sheetId` gives consumers the same (dashboardId, sheetId, chartId) tuple for disambiguation while preserving chartId's stable per-sheet local identity** â€” no test rewrites, no Cosmos format break, no renderer changes, no server-resolution churn. The chartId remains a local identifier within a sheet; the sheetId disambiguates across sheets. That's exactly what observability needs and nothing more.
    - **Single consolidated test file vs per-endpoint files.** The two prior waves shipped per-endpoint server test files (`dashboardDrillThroughTelemetryWD3Telemetry.test.ts` + `dashboardExplainSliceTelemetryWI4Telemetry.test.ts`) because each wave introduced a NEW endpoint and the test file mirrored the route file 1:1. This wave widens BOTH existing endpoints in lockstep; consolidating the new tests into a single `dashboardWdWiSheetIdTelemetry.test.ts` puts the doublet-drift guards (the two source-inspection tests that pin both endpoints had the same widening) in one place where they're maximally legible. A future single-endpoint wave on either route would still get its own per-endpoint file; this consolidation is specifically about the doublet-widening shape.
    - **Explicit-omit invariant (`...(sheetId ? { sheetId } : {})` rather than `sheetId,`).** Setting `metadata: { sheetId, ... }` when `sheetId === undefined` would write `metadata.sheetId === undefined` to Cosmos. JSON serialisation drops the key â€” but the `recordUsageEvent` writer might or might not pass through the SDK's serialiser at every layer; the conditional spread guarantees the key is omitted at the object-construction layer, before any serialisation step. Cosmos distinguishes `{}` from `{sheetId: null}` on the wire; the conditional spread also pre-empts a future refactor where a callsite accidentally passes `sheetId: null` (which would survive a `?? undefined` collapse but fail the `.optional()` parse â€” except the parse runs BEFORE the controller, so the recorder would see a validated-undefined that then propagates to Cosmos as a present-undefined). The source-inspection test pins the spread pattern at both controller sites so a refactor can't silently regress.
    - **Both effects' deps arrays widen to `[dashboard.id, activeSheetId]`.** The listener handler closes over `activeSheetId` for the telemetry payload; ESLint exhaustive-deps would flag a missed dep, but more importantly the listener needs to re-bind when `activeSheetId` changes so subsequent event firings capture the new value. The re-bind is cheap (`removeEventListener` + `addEventListener` on `window`) and runs once per sheet switch; on a multi-sheet dashboard the user-perceived cost is zero. Without the dep, the listener would close over the initial `activeSheetId` (typically `null` on first mount) and every subsequent click on a sheet other than the initial active sheet would record telemetry with `sheetId: undefined` even though the user is clearly on a different sheet â€” which would silently revert the entire observability benefit of this wave.
    - **Why third instance doesn't promote a fourth convention.** Last session's WD3-telemetry â†’ WI4-telemetry doublet codified three conventions on the second-instance promotion threshold (route-level-recorder-seam, pii-safe-discriminant-tag-on-observability-events, strict-zod-on-observability-endpoints). This wave is the third instance of all three patterns (the recorder seam is shared, the sheetId field is a safe identifier with no PII concern, the strict zod stays strict). No new pattern emerged that hasn't been seen before â€” this is a pure additive widening on top of established conventions. The "additive optional field on a strict observability endpoint" shape is itself a soft pattern with one instance now; a future wave that adds a different optional field to an observability endpoint would promote it.

    - **Why co-locate the WI4 route handler in the same `telemetry.ts` rather than ship a sibling `telemetryExplainSlice.ts` module.** Three reasons: (i) the route-level recorder seam (`recorder` mutable local + `__set...ForTesting` / `__reset...ForTesting` setters) is now shared between both controllers, which is the second-instance promotion that codifies the pattern as a documented convention â€” a sibling module would have duplicated the seam (DRY violation) OR exported it from one module and imported into the other (load-order coupling + import-cycle risk between sibling route files); (ii) both endpoints serve the dashboard-click-intent telemetry family conceptually, so the module-level docstring + named JSDoc block can describe both in one place; (iii) future dashboard click-intent telemetry events (WD2 cross-filter sample, dashboard-exit, etc.) will land in the same module on the same precedent.     - **Why `regionKind` as a strict z.enum() rather than a free-form `string.min(1)` like WD3's `valueType`.** The WD3 `valueType` field carries the JS `typeof` tag, which has 8 well-defined possible values but is conceptually open (a future JS spec extension could add a ninth token); free-form `string.min(1)` is the right shape for an open-but-strongly-typed token. The WI4 `regionKind` field carries the `BrushRegion` discriminant from [`explainSlice.ts`](../client/src/pages/Dashboard/lib/explainSlice.ts), which is closed by the renderer-side type union â€” there are exactly four valid values and any other token represents a renderer-side wiring bug or an upstream injection attempt. `z.enum(["numeric", "temporal", "categorical", "box2d"])` is the load-bearing defense: a malformed renderer can't smuggle non-discriminant tokens through, and the schema-layer rejection catches the bug before it hits Cosmos. This is also a cleaner observability story â€” Cosmos aggregation queries on `regionKind` can use exact-string comparison rather than enum-validation-after-the-fact.
    - **Why fire BETWEEN validation and `setExplainSliceEvent(detail)`, not BEFORE validation.** Same answer as WD3-telemetry's matching question: invalid events represent renderer bugs (or stale CustomEvents from prior mounts), not successful slice-explanations; firing before validation would skew the metrics. The validation gate is the canonical "did the user successfully brush a slice" boundary, and the telemetry fire belongs strictly on the success side of that gate. The 7-line inline comment block (a touch longer than the matching WD3 block because it also documents the discriminant-vs-typeof distinction) preserves the rationale at the call site.

The "WI4 Â· " naming-prefix gives the visual separation in test-runner output without splitting the file.

  **Why this design.**
    - **Why a dedicated clientâ†’server endpoint rather than firing the recordUsageEvent inline from an existing server route.** The drill-through is a client-side action: the renderer dispatches a window CustomEvent, the DashboardView listener captures it, and the side-sheet opens. There's no server-side touchpoint in the existing flow (the sheet's row fetch happens lazily via the WD3-server-fetch endpoint when the sheet body mounts â€” but that endpoint is the data fetch, not the observability hook; conflating "user drilled into chart X" with "client successfully fetched rows for chart X" would obscure two different signals). Adding a dedicated telemetry endpoint keeps the observability concern orthogonal: fires once per validated event regardless of whether the data fetch succeeds; fires from the listener handler, not from React rendering, so re-renders don't double-count; and the endpoint stays single-purpose (one event type per route, easy to extend per-event with custom validation if the schema needs widening).
    - **Why the four-field payload shape (chartId / column / valueType / dashboardId?) instead of a richer envelope.** Each field is load-bearing for its observability question: `chartId` answers "which chart got drilled into" (per-tile heatmap of clicks); `column` answers "which dataset column was the drill axis" (cross-dashboard heatmap of which columns are most queried); `valueType` answers "what was the shape of the clicked mark's value" without sending the value itself (a `typeof` tag, never the raw string/number â€” load-bearing PII contract); `dashboardId` answers "across which dashboards" (allows per-dashboard aggregation). Fields deliberately NOT included: the raw `value` (PII risk â€” column values can be user names, customer IDs, sales figures, etc.; sending only `typeof` preserves the observability signal without the leak); `sourceTileId` (the click came from a tile but the tile id is a UI artefact, not a stable observability key â€” chart id is the durable identifier); `filters` snapshot (the active filter context at click time is interesting but adds payload bytes for a question we can answer from the chart spec + the dashboard config aggregate; deferred to a follow-on wave if the question surfaces); `extraPins[]` (RectRenderer heatmaps fire multi-pin events; the primary pin's column already carries the observability signal, and `extraPins[]` would balloon payload size for a marginal question; defer).
    - **Why `typeof detail.value` as the canonical PII-safe value-shape token.** The JS `typeof` operator returns one of `"string" | "number" | "boolean" | "object" | "undefined" | "symbol" | "function" | "bigint"` â€” six of those would actually appear in drill-through values (`symbol` / `function` / `bigint` never reach the client from the server). Each tells a story: `"string"` = categorical column (region, product name), `"number"` = numeric column (sales, units, week index), `"boolean"` = flag column (active, on-promo), `"object"` = a Date object (the renderer dispatches Date instances for temporal columns), `"undefined"` = a null/missing slot the user clicked through. Aggregating by `valueType` across many drill-through events reveals the distribution of click intent without leaking any individual click's value. Alternatives considered and rejected: (a) coerce via the server's existing `toFilterValue` canonicaliser and send the bucket name â€” adds runtime dependency on the server-side canonical-form module on the client, breaks the W61-cross-runtime-boundary local-mirror convention, no observability benefit; (b) hash the raw value via SHA-256 and send the prefix â€” uniqueness preserved for aggregation but the hash leaks information (two same-value clicks emit the same hash; a small-cardinality column reveals its value distribution); (c) send a length-bucketed string (`"short" | "medium" | "long"`) â€” useless for numeric columns and confuses temporal observability. `typeof` is the smallest informative token.
    - **Why fire BETWEEN validation and `setDrillThroughEvent(detail)`, not BEFORE validation.** Firing before validation would mean an invalid event (chartId not a string, column missing, or both) still emits telemetry. Two reasons that's wrong: (a) the validation is the gate that decides whether the user successfully drilled â€” an invalid event represents a misbehaving renderer (or a stale CustomEvent fired by a previous mount), and counting it as a drill-through skews the metrics; (b) the telemetry payload requires `chartId: string` + `column: string` â€” if validation fails, those guarantees don't hold, and the type-checker would either reject the call or force a runtime-cast that could send `chartId: undefined` to the server (which the server's zod min(1) check would 400). Firing after validation, before setState, means: only successful drill-throughs are counted; both fields are guaranteed non-empty strings; and the telemetry fires even if the sheet mount fails downstream (the click intent was successful â€” sheet-mount failure is a separate failure mode).
    - **Why a route-level test seam (`__setUsageEventRecorderForTesting`) rather than a model-level one.** The model-level mock would have to replace `recordUsageEvent` on `usageEventModel.recordUsageEvent` â€” but Node's ESM module bindings are immutable read-only references from the importer's perspective, so reassigning the named export from outside the module fails with `TypeError: Cannot assign to read only property`. The route-level seam works because the route module imports `recordUsageEvent` once at module load, then captures it in the mutable `let recorder` local; tests can swap the local without violating ESM semantics. The pattern generalises to "inject the model writer on the route layer instead of the model layer" â€” first instance of this; the W61 admin-controller-setter precedent is the closest analogue but those setters live on controller exports rather than route exports. Would promote to a documented convention on a second instance.
    - **Why `credentials: "include"` rather than `"same-origin"`.** The dashboard view runs on the SPA at port 3000 in dev, but POSTs to the API at port 3002 â€” different origins. `"same-origin"` would fail to attach auth cookies in dev; `"include"` works in both dev (cross-origin via CORS) and prod (same origin behind the load-balancer). The server's CORS config already allows credentials from the SPA origin per the existing route surface. No security widening â€” the endpoint is auth-gated, so missing cookies â†’ 401, not a silent telemetry leak.

That's ~150 LOC of jsdom-mount plumbing for marginal additional coverage given that (a) the client helper is unit-tested in isolation with 7 tests pinning the fetch contract, (b) the server route is unit-tested in isolation with 15 tests pinning the body shape + auth + recording, and (c) the listener's previous behavior is covered by the existing `drillThrough.test.ts` + `wd3WiringEcharts.test.ts` suite (which would catch a regression in the sheet-open path).   **Why this design.**
    - **Why one global + per-section overrides shape (`{ global, overrides }`) rather than three independent filter states.** Three independent states would lose the synced-global default that serves the dominant workflow ("show me what I edited applied across all three cards"). A naÃ¯ve admin who just wants to filter everything by User would need three clicks; the W61-source-filter shipped specifically because that's an unacceptable cost. The hybrid shape preserves the one-click default + adds shift-click for the advanced case.
    - **Why plain-click clears ALL overrides (not just the clicked section's override).** The simpler "clear only this section's override on plain-click" rule sounds more "local" but creates a confusing exit problem: an admin who shift-clicked Metricsâ†’User, then shift-clicked Dimensionsâ†’Auto, can't easily get back to fully-synced mode â€” they'd have to plain-click in BOTH cards. The "clear all on any plain click" rule gives a single predictable exit from override mode: plain-click any chip anywhere â†’ fully synced + new global. The trade-off is that shift-clicking two sections to scope them both then plain-clicking a third dramatically resets, but the wins in discoverability + escape-hatch reliability are larger.
    - **Why a single `applyChipClick` reducer with a `modifier: boolean` branch rather than two named functions.** The chip row's only branching axis is whether the modifier key was held. A single entry point keeps the call site simple (`setFilters(prev => applyChipClick(prev, "metrics", "user", e.shiftKey))`); two named functions (`setGlobalFilter` + `setSectionOverride`) would force the call site to do the branching, which is exactly the kind of boilerplate that the pure-reducer pattern eliminates.
    - **Why URL persistence stays global-only (overrides not in URL).** Per-section overrides are session-local opt-in advanced state â€” most admins won't use shift-click. A share-link that captures overrides would surprise the recipient ("why is Metrics filtered to Auto when I clicked the share link?"); the cleaner intent is "share my synced view", which is what the global-only URL persistence captures. Overrides also don't survive accidental reload â€” but a reload-survival admin can always re-shift-click in a few seconds.
    - **Why the `(overridden)` hint is muted text rather than a strong visual cue.** The override state is informational; the chip itself already shows the active filter value. A bold or coloured indicator would compete with the chips visually. The italic muted-foreground span at the end of the chip row reads as "secondary metadata" â€” discoverable without competing.
    - **Why shift-key (not cmd/ctrl/alt).** Shift is the conventional "modifier for second-tier action" across most web UIs (shift-click in file managers extends selection; shift-click in tab managers opens in new window; shift-click in chart libraries enters multi-select brush). Cmd/ctrl is reserved for OS-level actions (open-in-new-tab on links, cmd-click on Mac); alt-click is sometimes used but feels keyboard-modifier-heavy. Shift is the most discoverable modifier for "scope this action to one card".
    - **Why `Object.freeze` the `NO_OVERRIDES` sentinel.** Identity stability for free across renders â€” a downstream `useMemo([..., sectionFilters.overrides, ...])` would invalidate on every render if `NO_OVERRIDES` were a fresh literal. Frozen sentinel means three card renders' `overrides` reads point at the same object reference until an override is set. Doesn't matter for current render perf but is a load-bearing invariant for any future memoised consumer of the overrides shape.

The alternative (value-equality check) would silently strip the override which feels magical.
    - **Why no separate "Reset section" or "Clear override" button.** Plain-click any chip in any card is the canonical exit. Adding a "Reset" button per card would duplicate the affordance + invite the question "what does this reset do, exactly?". The chip itself is already the affordance.

  **Why this design.**
    - **Why `<Dialog>` not `<AlertDialog>`.** AlertDialog is Radix's confirmation-only primitive (single warning + Cancel/Confirm pair); Dialog is the canonical form-modal that supports multi-input editing. Hierarchies have 2â€“8 editable level rows â€” Dialog is the right semantic primitive. Same rationale as W61-add-client.
    - **Why the modal owns the draft, not the host.** Draft state is mount-scoped and lives only for the duration of the edit. Host ownership would force a `clearDraft` step on every close + a stale-draft concern if the admin closes and re-opens for a different hierarchy quickly. An effect keyed on `[hierarchy]` re-seeds the draft so per-row inputs are always seeded from the current model state on open. Same precedent as W61-add-client.
    - **Why route through the existing W61-save PATCH path, not a dedicated `/hierarchies/:name` endpoint.** Hierarchies are 2â€“8 entries Ã— maybe 8 hierarchies = â‰¤64 level strings per model. The full SemanticModel payload is small (~5â€“50 KB for typical models); the marginal cost of replacing the whole model vs sending just the changed hierarchy is negligible. A dedicated endpoint would mean (a) a new audit-write code path + (b) a new W61-cache-invalidate hook call site + (c) a new test surface â€” all for no perf benefit. The brief flagged this trade-off explicitly ("no dedicated endpoint per the brief; cheap wholesale-replace fits hierarchies' density").
    - **Why `editingHierarchy: SemanticHierarchy | null` not `{ open: boolean, hierarchy: SemanticHierarchy | null }`.** The non-null IS the open signal. Two-prop shape invites the invalid state `open=true && hierarchy=null` that the modal would have to defend against. Single-discriminant shape mirrors W61-add-client's `addOpen: kind | null` and W61-delete-client's `pendingDelete: { kind, name } | null`.
    - **Why validation runs only after first submit attempt (validationActive flag).** Keystroke validation would show a red border on the level input the moment the admin types `R` (PascalCase rejected) before they've finished typing `Region` (also rejected, but mid-keystroke). Submit-time validation matches the W61-add-client behaviour. The exception: per-row errors stay visible after the first submit attempt so the admin can fix them iteratively without re-clicking submit per error.
    - **Why per-row duplicate-detection runs AFTER per-level format checks.** A level that's malformed (e.g. PascalCase) AND duplicate of another malformed one should surface the format error first â€” the admin needs to rename to snake_case, after which the duplicate (if still present) becomes the next actionable signal. A future widening that lets multiple hierarchies edit in parallel would still be type-safe; a boolean would force a breaking change to `string[] | null`.

The W61-detail-schema wave (shipped earlier this session) exposes `datasetSchema.columns` but those are dataset column names, not dimension names.   **Why this design.**
    - **Why an observability hook + registry rather than just `console.log` lines at each call site.** Three reasons. First, a single grep token â€” ops can grep `\\[semantic-model-invalidate\\]` and see every model-version change across the four controllers in one query; scattered `console.log` lines would each need their own grep. Second, the contract pin â€” the controller integration tests assert "every mutation path calls the hook exactly once on success and never on any error path", which gives the W61 sub-wave family (W61-edit-column / W61-edit-references / W61-hierarchy-edit are upcoming) a tests canary if a new mutation path is added without the hook call. Third, the registry â€” future W64 cache-key work will register a `cache.clear()` callback via `registerInvalidator(fn)` and the existing four call sites + tests just work; no need to chase every controller again. The triple win justifies the small module (~100 LOC) over scattered logs.
    - **Why the hook fires INSIDE the write lock, not outside.** Future invalidators (W64 in-process cache.clear callbacks) need to be serialized with the write they invalidate against â€” if the hook fires outside the lock, a concurrent reader could observe the post-write doc state via Cosmos but read a stale cached entry that hasn't been cleared yet (the cache.clear would race the next read). Inside the lock, the sequence is `_updater succeeds â†’ hook fires â†’ cache.clear runs â†’ lock releases â†’ readers proceed with fresh state`. The price is that the hook extends the lock-hold window â€” but the hook is fire-and-forget, synchronous, and listeners are bounded (`console.log` + counter increment + future `cache.clear`), so the extension is microseconds, not milliseconds.
    - **Why the hook is `(event) => void`, not `(event) => Promise<void>`.** Async listeners would force the controller to `await` them, which would either (a) extend the lock-hold window unboundedly (every cache invalidator could become a remote network call) or (b) require a "timeout-then-skip" pattern that masks slow invalidators. The synchronous shape forces every invalidator to be cheap by design â€” `console.log` + `cache.clear()` is the budget; anything more expensive belongs in a separate fire-and-forget background queue. A future asynchronous invalidator that needs more time should subscribe via a sibling registry (an `IO_INVALIDATOR_REGISTRY` analog) rather than widen this one.
    - **Why the hook fires AFTER `_updater(doc)` succeeds, not before.** If `_updater` throws (Cosmos outage, document-too-large, optimistic-concurrency conflict at the per-document `_etag` level â€” single-instance correctness per invariant #9 means we don't currently use `ifMatch`, but a future multi-instance world might), the catch block emits the 5xx response and the hook MUST NOT have fired â€” otherwise a downstream cache invalidates against a write that never landed, and the cache then misses on a stale-on-disk model that's still correct. The "500 updater throw does NOT fire" controller test pins this explicitly.
    - **Why a listener that throws is logged but doesn't propagate.** A registry that propagates means one buggy invalidator can break the write path â€” concretely: a future cache-clear callback that fails because the cache is in a corrupt state would surface as a 5xx on every mutation, even though the mutation itself succeeded. The shape "log the error, skip the listener, continue with the next" preserves write-path resilience and pushes the failure into ops logs where it's actionable but not catastrophic. Tests pin both halves: subsequent listeners still fire after a throw, and the caller sees normal return.
    - **Why `priorVersion: nextVersion - 1` not `doc.semanticModel.version ?? 0` at the call site.** Subtle ordering: by the time the hook fires (after `_updater(doc)`), the line `doc.semanticModel = nextModel;` has already executed, so `doc.semanticModel.version` is `nextVersion`, not the prior. Reading it would emit `priorVersion === nextVersion`, breaking the monotone-chain test. The arithmetic form `nextVersion - 1` is unambiguous because `nextVersion` is a local declared at the top of the lock callback and is never reassigned. A future refactor that captures `priorVersion` as a local at the top of each function would be marginally cleaner but would touch four functions for zero behavioural change; the canonical `nextVersion - 1` form is intentional minimal-churn.
    - **Why the module exposes `__clearInvalidatorsForTesting` in addition to per-test unsubscribe.** Module-private state survives across tests in the `node --test` runner (no fresh `import` per test). Without a clear shim, a test that registers a listener and then crashes / aborts before unsubscribe leaves the listener wired into the next test. `__clearInvalidatorsForTesting` is the belt-and-braces reset; per-test code can also use unsubscribe defensively, but the central clear is the safety net.
    - **Why two test files, not one.** Module-level tests (`semanticModelInvalidateW61CacheInvalidate.test.ts`) pin the hook's pure-function contract â€” registry semantics, log shape, throw isolation. Controller-level tests (`adminSemanticModelInvalidateW61CacheInvalidate.test.ts`) pin the integration contract â€” every mutation path fires the hook, error paths do not. Combining them would force one file to import both the hook module AND the controller fakes, doubling the test-fake surface for every test; splitting matches the W61-references-dashboards (scanner module + controller endpoint) precedent.
    - **Why we don't expose the event payload through a custom log format helper.** The single `console.log(`[semantic-model-invalidate] sessionId=${event.sessionId} ...`)` inline template literal is the simplest shape â€” a `formatInvalidationLog(event)` helper would be one indirection per call with no benefit (the template literal is already grep-stable). A future structured-logging refactor (JSON logs across the server) would centralize this anyway; for now the inline template literal is fine.
    - **Why we skip exposing `__forceFireInvalidatorsForTesting`.** Some test harnesses use a "force-fire" shim that triggers all registered listeners with a mock event. We don't need it â€” the unit tests drive `onSemanticModelVersionBumped` directly, and the integration tests drive real controller handlers. A force-fire shim would just be a thinner wrapper around `onSemanticModelVersionBumped` and tests should call the canonical entry-point.

  **Conventions added.**
    - **Version-bump invalidator hook.** [`docs/conventions/version-bump-invalidator-hook.md`](../docs/conventions/version-bump-invalidator-hook.md) (new). A future widening would breach the payload-leak canary (test #2's strict-equality assertion) so any addition has to update that test deliberately.
    - **Async invalidators / Promise-returning listeners.** The synchronous shape forces every invalidator to be cheap. A future use case for async (e.g. evicting an entry from a remote distributed cache) belongs in a sibling registry (`asyncInvalidators`) rather than widening this one, so the synchronous fast path stays predictable.
    - **Cross-session invalidation events.** The hook fires once per session per mutation; a future "invalidate every session" use case (e.g., a global semantic-model schema migration) would need a separate registry or a wildcard event shape. Deferred until use case warrants.
    - **In-browser verification.** N/A â€” the wave is server-only and the new tests are exhaustive. The hook's log token is consumed only by ops greps and the registry is consumed only by future cache code, neither of which has a browser surface.
