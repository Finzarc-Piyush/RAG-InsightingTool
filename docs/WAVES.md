- **2026-05-19** — **Wave WD2-dim-foundation · `dashboardTileContext` carries optional `filters` for the upcoming dim-non-matching-marks family.** The WD2-wiring family (10 visx renderers + 5 ECharts renderers, finalised earlier today by WD2-wiring-echarts) closed the *input* half of the dashboard cross-filter loop: clicking a mark dispatches `CROSS_FILTER_EVENT`, `DashboardView` applies the toggle, and the dashboard-wide filter map updates. The *visual feedback* half — "filtered tiles dim non-matching marks instead of removing them" per the [1000x master plan](/Users/tida/.claude/plans/go-through-the-entire-partitioned-yao.md) Workstream 4 UX — has been blocked by a structural mismatch: each renderer needs to read the active filter for its `enc.x.field` (or equivalent encoding field) so it can call the already-shipped [`isCrossFilterActive(global, column, value)`](../client/src/pages/Dashboard/lib/crossFilter.ts) helper and decide if the mark it's about to render matches. But the `useDashboardTileContext()` hook only exposes `{ tileId }`, so renderers had no `ActiveChartFilters` map to call the helper against. The pre-WD2-dim options were (a) thread a new `filters` prop through every renderer's call site — a wide-blast-radius change on a god-object-adjacent ChartRenderer surface; (b) read `globalFilters` off a dashboard React context — there isn't one; the filter map lives in `DashboardView`'s `useState` and is forwarded as a prop; or (c) extend the existing `dashboardTileContext` (already mounted at the chart-tile boundary, already read by every wired renderer) with an optional filters slot — single touch point, single import. This wave ships option (c).

  **Why this design.** The split-first approach (foundation alone, ~50 LOC, no renderer touched) was chosen over a fat WD2-dim-foundation+bar wave for two reasons. First, the foundation extension is type-system-load-bearing across at least 6 follow-on waves (WD2-dim-bar / -cat / -rect / -trend / -point / -echarts) — getting it wrong forces re-litigating the foundation inside every follow-on. Splitting first lets the type shape land + bake + get verified before any renderer wave is started. Second, the foundation is contract-only and so trivially atomic that landing it alone keeps the next renderer wave focused on a single concern (the dim factor math + the opacity composition) rather than mixing context plumbing with rendering semantics. The spread-conditional pattern (`undefined → omit`) was preferred over `filters: filters ?? {}` because the upcoming dim-renderer guard wants to distinguish "rendered inside a dashboard tile but no filters yet" (provider mounted with `filters={undefined}` while DashboardView's filter map is still empty) from "rendered outside any dashboard tile" (the `useDashboardTileContext()` returns `null`). The current shape conflates these two cases at the *value* level — both produce an object with `value.filters === undefined` — but the `null`-vs-non-null at the *context* level is preserved (a dashboard-mounted provider always returns a non-null value, even with no filters), so renderers can still discriminate "dashboard tile context active" via `!!dashboardTile`. Alternative considered: a separate `DashboardFiltersProvider` distinct from the tile provider. Rejected because (a) it would require every renderer to consume *two* contexts, doubling the touch points; (b) the filter snapshot is conceptually tile-scoped — the same filter map applies to every mark inside a given tile — so co-locating it with the tile context matches the data shape.

  **Why this design.**
    - **One generic prop on EChartsBase, not 5 specialty wrappers.** The natural alternative would be to introduce a per-renderer `EChartsCrossFilterWrapper` that injects `useDashboardTileContext` + `dispatchCrossFilter`. But that doubles the renderer count and risks divergence between marks. Putting the prop on EChartsBase keeps the base shell mark-agnostic (it just forwards the click) and pushes the `params`-shape decoding into each renderer where the encoding-specific knowledge already lives.
    - **Ref-tracking instead of off-then-on.** ECharts' mount-effect runs once with empty deps (the existing pattern), so a closure over `onChartClick` would capture the initial value forever. Two fixes: (1) add `[onChartClick]` to the mount-effect deps — but then a callback identity change would re-create the entire ECharts instance, losing animation state and re-running the canvas init; or (2) bind once with a ref that's updated on every render — single bind, no re-init, always-fresh callback. Option 2 is the standard React idiom for this "external system needs the latest callback" shape.
    - **`inst.on` after `setOption`, not before.** ECharts validates event subscriptions against the chart's series types at bind time. Calling `inst.on('click', ...)` before `setOption` is a no-op for the eventual click events. Pinned by test via source-index ordering (`indexOf("inst.setOption") < indexOf("inst.on")`).
    - **Leaf-only dispatch on hierarchical marks (treemap / sunburst).** A treemap with a `groupCh` shows two levels — parent groups and leaf items. A click on a parent group could plausibly dispatch on the group's field, but then the dispatch column flips between two clicks of the same mark, which is hostile UX. Skipping parent clicks and only dispatching on leaves means every dispatch from a treemap targets the same column — `labelCh.field`. Two-column dispatch (mirroring the RectRenderer row+col approach from WD2-wiring-rest-rect) is a follow-on if a use case appears.
    - **Source-only dispatch on sankey.** The natural alternative is "click a node, dispatch on the column the node belongs to" — but ECharts' params shape doesn't carry that information directly; it would need a side-table lookup. In practice every sankey use case at Marico's scale has same-dimension source / target (e.g. `Channel` → `Channel`, `Brand` → `Brand`), so dispatching on `sourceCh.field` covers both endpoints. Cross-column sankeys (e.g. `Brand` → `Channel`) would need the richer decode — flagged in the comment.
    - **Candlestick reads `xs[dataIndex]` not `params.value`.** ECharts' candlestick params.value is the 4-tuple `[open, close, low, high]`. None of those are the categorical key (the bar's x-axis label). The renderer already built an `xs` array for the x-axis category data — the same array the chart uses for axis labels. Using `dataIndex` to look up the matching `xs` entry is the canonical way to get the categorical key from an ECharts axis-bound series.
    - **Calendar dispatches ISO date string verbatim.** `applyCrossFilter` already handles date columns as categorical selections (discrete value lists). The date string `yyyy-mm-dd` is the canonical wire format the renderer built; round-tripping it through `Date.parse` + re-format would risk a TZ-offset bug. Pass the string straight through.
    - **`onChartClick={dashboardTile ? handler : undefined}` for every renderer.** A `useCallback(() => {...}, [dashboardTile, ...])` always returns a function — even when `dashboardTile` is null. Without the conditional ternary, the EChartsBase mount effect would bind `inst.on('click', ...)` unconditionally, which means a click in a chat / explorer / share-preview surface would invoke the handler, hit the `if (!dashboardTile) return` guard, and no-op. That works but is wasteful — the cleaner shape is to pass `undefined` when there's no consumer, so the bind doesn't happen at all. Mirrors the visx renderers' `interactive` flag pattern.

  **Why this design.**
    - **Surface the citations array, don't replace the inline rendering.** The `MarkdownRenderer` WQ3 integration is the canonical inline-citation surface — it works for narrator prose in the chat, for dashboard insights, and now (transitively, without code change) for WI2-server-generated regen text. Removing or replacing that path would force every consumer to re-implement the inline hover-card logic. The Sources row is *additive*: a separate, discoverable list for users who don't visually parse the `[1]` superscripts. Two surfaces for two reading patterns.
    - **Reuse `CitationHoverCard` rather than build a new chip component.** `CitationHoverCard` already encodes the "this is a verifiable citation" visual language across the app — the dotted-underline `[N]` superscript, the Radix HoverCard popover with pack label / canonical id / explanatory note. Adopting it in the Sources row keeps the visual identity consistent: a user who sees `[1]` in narrator prose recognises the same `[1]` in a tile insight, and hovers expecting the same metadata. Building a new chip would diverge the language without adding capability.
    - **Truthy AND length-gate avoids the empty array.** `InsightRegenEntry.citations` is `string[] | undefined` on the schema. The server may return an empty array when `extractInsightCitations(text)` finds no backtick tokens (the W22 hyphen rule filters generic spans). Without the length check, the row would render with just the "Sources:" label and no chips — visually broken. The `&& length > 0` lights up only when there's something to show.
    - **`flex flex-wrap` for long citation lists.** A regen response can cite up to ~4 packs in practice (the WI2-server prompt is biased toward conciseness, but doesn't cap citations). Four `[1] [2] [3] [4]` superscripts fit in a tile footer of typical width; six or seven would overflow without wrap. `flex-wrap` keeps the visual stable across citation counts without imposing a hard cap.
    - **1-based index passed explicitly via `i + 1`.** `CitationHoverCard` renders `[{index}]`, so passing `i + 1` (instead of `i`) gives users `[1] [2] [3]` rather than `[0] [1] [2]`. The 1-based convention matches the existing `MarkdownRenderer`-driven inline rendering (which uses `buildCitationIndex` returning `map.size + 1`).
    - **Place between metadata and action row.** The visual flow on a regen-populated footer is: prose → "Updated 2 min ago · medium confidence" → sources → action. Reading the prose primes the reader, the metadata says "fresh and trustworthy", the sources qualify the trust ("grounded in haircare-portfolio + kpi-glossary"), and only then the action button invites further interaction. Reordering would break the rhythm.
    - **Key by `packId`, not by `i`.** The citations array is expected to be already deduped by the server-side `extractInsightCitations` (it walks the prose left-to-right and adds each pack id to a set). Keying by `packId` is stable across re-renders even if the array order changes; keying by index would cause React to re-mount HoverCards on reorder, losing any open hover state. Both approaches work for the current usage but `packId` keying is the future-proof choice.

  **Why this design.**
    - **Pure helper sidesteps the ChartShim refactor.** The previous wave's "Out of scope" warned that exposing filteredData might require either a new prop on ChartShim that the renderers call back into, or a `getFilteredData` accessor exposed on a ref.   **Why this design.**
    - **Dynamic context at call time, NOT instantiation time.** The chart spec, filtered data, and domain context can change with every user interaction — every filter add, every brush, every legend toggle. Threading them through `useEffect` deps would either over-fire the network on every render (an expensive `[filteredData]` dep array would force a deep-equality memo in every caller) or under-fire (sloppy comparison would miss valid updates).   **Why this design.**
    - **Context, not props.** `PremiumChart` is a generic dispatcher with a fixed `VisxRendererProps` shape (`{ spec, data, width, height, ariaLabel? }`). Threading `sourceTileId` through every renderer would (a) widen the dispatcher's prop shape, (b) require touching every renderer in this single wave, (c) re-litigate the "what props are universal" question that PremiumChart already settled. A small React context that the tile body wraps the dispatcher in keeps the dispatcher's prop shape stable and makes BarRenderer's wiring a 3-line addition. Future WD2-wiring-rest waves each get a similarly thin addition (`useDashboardTileContext` + `dispatchCrossFilter` + a one-line onClick branch).
    - **`useDashboardTileContext` returns `null` outside a provider — no fallback shim.** The existing ChartGrid context's `useChartGrid()` returns a no-op shim (`{ filter: null, setFilter: () => {}, toggleFilter: () => {}, applyFilter: rows => rows, inGrid: false }`) because every consumer's code path calls those methods unconditionally and needs them to be safe. The dashboard-tile context is the opposite shape: consumers check `!!dashboardTile` to decide whether to dispatch the event at all. A no-op shim with a dummy `tileId: ""` would be semantically wrong — dispatching `sourceTileId: ""` to `DashboardView` from a chat-side renderer would pollute the brushed-filter telemetry. Returning null forces the caller to consciously opt in on a non-null.
    - **The two click paths (`grid.toggleFilter` + `dispatchCrossFilter`) are independent on purpose.** They serve different consumers: `ChartGrid` is the explorer's multi-chart in-context filter (user clicks Region=North in one chart, the OTHER chart's rows reduce to North); `globalFilters` is the dashboard-wide filter bar (user clicks Region=North in tile A, tile B re-renders with Region=North applied). The user contract of each is different and the storage shapes are different (`ChartGrid` is a single (field, value), `globalFilters` is `Record<column, ChartFilterSelection>`). Don't unify; let both fire when the renderer is inside both containers.
    - **`toFilterValue(unknown)` is purely additive.** The narrower signature was overly conservative — chart-mark values flow as `unknown` from typed renderer internals, so every renderer wiring would have to narrow first if `toFilterValue` insisted on a narrow input. Widening the signature has no runtime cost (the function already used `typeof` checks internally) and reduces the surface every WD2-wiring-rest renderer must thread.
    - **Wave shape mirrors W59a → W59b → W60 and WD2 → WD2-wiring-bar.** Each split lands a pure helper first (WD2's `crossFilter.ts`), then a thin call-site that wires the helper into one consumer (WD2-wiring-bar on BarRenderer). The thin-wiring waves stay reviewable; if a wiring decision turns out wrong (e.g. the `interactive` flag should ALSO consult an env flag) we change one line in one file, not 13.

  **Why this design.**
    - **Dispatch through `execute_query_plan`, not a parallel executor.** The W58 compiler already emits a `QueryPlanBody`; re-implementing DuckDB execution + the QL6 hard-fail rules + the temporal-facet promotion + the aggregation-without-groupBy verifier would duplicate ~400 LOC of executor logic and create a divergence risk. The model is *always* the session's `ChatDocument.semanticModel`.

  **What landed.**
    - [client/src/pages/Dashboard/lib/insightRegenCache.ts](client/src/pages/Dashboard/lib/insightRegenCache.ts) (new, ~145 LOC). Three exports:
      - `hashGlobalFilters(filters: ActiveChartFilters): string` — byte-stable serialisation of the global filter map. Walks `Object.keys(filters).sort()`, then for each column emits one of three shapes: categorical → `column=c:value1|value2|...` (with values sorted), date → `column=d:start..end` (with `undefined` start/end rendering as empty), numeric → `column=n:min..max` (same treatment for undefined bounds). The `=`, `;`, `:`, `|`, `..` separators are unambiguous together — no value in the active filter shape contains those literals after the existing chart-filter normalisation. Result: identical filter maps in different insertion / value order yield identical hash strings.
      - `buildCacheKey(tileId: string, filterHash: string): string` — composes the pair as `${tileId}::${filterHash}`. The `::` delimiter is unambiguous because `hashGlobalFilters` uses `=` / `;` / `:` as internal separators.

The `now` injection is for deterministic tests.
      - Returned `InsightRegenCache` is a closure-backed Map with five methods: `get(key) → InsightRegenEntry | undefined` (returns undefined for stale entries + lazily evicts them, refreshes LRU recency on a successful read), `set(key, value)` (evicts the LRU entry when full + replaces an existing entry by delete-then-set so iteration order tracks recency), `has(key)` (false on expired + lazy delete, same as get), `clear()`, `size()`, `evictExpired() → number` (active sweep returning the count of dropped entries; useful for a periodic timer the consuming hook may run).

Future schema additions (e.g. effect-size badges, sub-investigation links) extend this interface without breaking the cache contract.

  **Why this design.**
    - **Pure data-plumbing first.** The eventual WI2 end-to-end touches: a server endpoint that takes `(spec, filteredData, domainContext)` and returns insight prose; a client hook (`useInsightRegen(tileId, filters)`) wired into [`TileInsightFooter.tsx`](client/src/pages/Dashboard/Components/TileInsightFooter.tsx); a "✦ Re-explain this view" button + spinner; a cache between the hook and the network call; integration with WD2's brushing layer so brushed-filter changes also invalidate. Shipping all of it in one wave is god-object-adjacent. Splitting the cache out first means: the cache contract is fixed + tested before the server / UI layers can drift it; the hook wave can be a thin React layer over this contract; the server wave can be a thin LLM-call wrapper that the hook calls when `cache.get(...)` misses.
    - **`hashGlobalFilters` uses string concat, not JSON.stringify.** `JSON.stringify({zone:"A", brand:"X"})` and `JSON.stringify({brand:"X", zone:"A"})` produce different strings — the key-sorting concern would have to live downstream. Doing it explicitly in the helper makes the byte-stability contract obvious in the source, and the resulting hash strings are markedly more compact than equivalent JSON (`region=c:North|South;zone=c:All` is ~25 chars vs the equivalent JSON ~70 chars). Compact keys matter for browser memory + dev-tool readability.
    - **TTL matches Anthropic's prompt-cache TTL.** Prompt-cache hits expire at 5 min; a regen LLM call built against a cached system prompt becomes uncacheable past that. Holding regen results longer than the underlying cache is wasted: the next request would re-build the system prompt anyway. 5 min is the natural ceiling.
    - **`maxEntries: 64` default.** Eight tiles × eight filter combos per session is a generous upper bound; most analytical sessions have <4 tiles and <3 filter combos. Cap exists to prevent a runaway log-tail of stale entries from growing unbounded in long sessions; 64 entries × ~500 bytes each = ~32 KB ceiling.
    - **`now` injection for deterministic tests.** The cache's own time source is injectable; the TTL test advances `t` directly without `await new Promise(r => setTimeout(r, ...))` or `vi.useFakeTimers()`. Tests are sub-millisecond + flake-free.
    - **Closure-backed Map, not a class.** The factory returns an object literal with the five methods, all closing over the same `store` Map + `now` closure. Tree-shake-friendly (no `class` keyword in the bundle), easy to instantiate per `useMemo`, indistinguishable from a class instance to TypeScript.
    - **`has` does lazy-delete on expiry.** Consistent with `get`. Both probes return false / undefined for stale entries AND remove them so the cache's `size()` stays accurate after a flurry of probes. The `evictExpired()` active sweep exists for the rare case where the consuming hook wants to periodically reclaim space without probing every key.
    - **Stored values include `citations` + `regeneratedAt` + `confidenceTier`.** Even though those are optional, including them in the interface up-front means the future regen endpoint + footer UI agree on the shape without a schema bump in WI2-wire. The cache becomes the contract.

  **Why this design.**
    - **Cohort split that matches the tool's natural question.** Two-segment compare is literally about A vs. B — that's the t-test. Breakdown ranking implicitly stages a "leader vs. field" comparison — the headline finding the narrator's prose around the top row implies. Both splits are unambiguous; neither requires the planner to specify the comparison. The tool answers what its name promises.
    - **Welch's t-test on row-level values, not on aggregates.** A t-test on two `sum` values would have df=0 and zero variance — it's not a statistical test. A t-test on the underlying row-level values *is* a test: it asks "do the per-row metric values differ between segments meaningfully relative to within-segment variance?". This is the right question for both `mean` (where it's also the natural test of the aggregate) and `sum` (where it gracefully reframes the sum question as "are per-row magnitudes different?" — usually what the narrator's prose around the top row actually means).
    - **Skip count aggregation.** Count is the only aggregation where the row-level values aren't a meaningful sample (you'd be comparing "1, 1, 1, 1" vs. "1, 1, 1"). The natural test for "are the count proportions different from the global baseline?" is a Z-test for proportions or a chi-square, and both need a baseline the tool doesn't carry. Skipping is correct; future waves can add a `baseline` arg if needed.
    - **Top-vs-rest for breakdown ranking, not top-vs-second.** "Is region X the leader?" is the headline finding; the relevant comparison is "X vs. everything-not-X", not "X vs. the next region". The latter is a noisier signal (sample of one comparison column) that the narrator rarely cares about — the prose is "X leads with $5M, far above the rest" not "X leads with $5M vs Y at $4M".
    - **Two-pass over the frame, not values-per-group.** Capturing `values: number[]` per group inside `aggregate` would mean N × G memory for high-cardinality breakdowns. The second pass for exactly two buckets (top + other) is one extra O(N) scan — typically <1ms on the diagnostic slice cap (~5k rows) — and keeps the existing `aggregate` Map values cheap.
    - **Composite ranking explicitly excluded, asserted in source-inspection tests.** A future refactor could accidentally invoke `buildBreakdownRankingEvidence` from the composite branch; the test `composite (rankBy) path summary is unchanged (WQ7 scope-bounded)` guards against it.
    - **Empty-suffix on skip / failure is the contract.** Mirrors WV4-WV7. Empty `topValues` or `otherValues` (single-group frame) → empty suffix → byte-stable summary. NaN-only values → empty suffix. `runSignificanceTest` returns `ok: false` for nA/nB < 3 → empty suffix. Failure modes are silent and observable only by the absence of the canonical evidence block.

WW2's extractor matches the first `R² = X` / `n = N` it sees in the prose — which, with WV7, is the correlation branch's canonical suffix anchored at the end of that line.

  **Why this design.**
    - **Pick the strongest correlation, not all of them.** The correlation branch may produce many `topCorrelations` (the analyzer caps at 20 here); appending evidence for each would inflate the summary and the WW2 extractor only reads the first match. The headline result is the strongest correlation (sorted by |r| descending — the same sort `analyzeCorrelations` applies internally for chart selection), which is also the correlation the narrator builds prose around. Future waves with structured `Finding.evidence` per-correlation would carry per-pair tiers without changing this baseline.
    - **Mirror WV4 exactly, not WV5/WV6.** WV5 introduced a `buildEvidenceSuffix(pValue, effectiveN)` helper because three sub-tests shared the same shape with one variant (effective n). WV6 needed similar logic for the per-group branch. WV7 has only one call site — one helper variable, no shared abstraction needed. Direct inline assignment matches WV4's pattern and keeps the wave footprint to ~24 LOC. If a fifth tool ever needs the same shape (n + R² from the strongest of N correlations), promoting this to a helper is a trivial follow-up.
    - **The correlation branch's text, not the joined summary.** The suffix lives inside the correlation branch's `text` field, not appended to the final summary after the join. This means it's structurally adjacent to the data it describes (`...for **${outcome}**.${wv7EvidenceSuffix}`), which keeps the prose readable for both humans and the WW2 extractor. Putting it at the very end of the joined summary would have placed it after a breakdown table or a benchmark line, where the regex could miss the n / R² context.
    - **Empty-evidence guard already in `composeFindingDetail`.** When `evidence` is `{}` (no fields), the formatter returns `""` rather than `" ()"`. This was the load-bearing WV2 design decision — every tool migration depends on it. Tested explicitly here (`empty topCorrelations → empty suffix → branch text unchanged from pre-WV7 shape`) so future formatter refactors that break the empty-evidence contract surface immediately on this branch's tests, not as a silent regression in production prose.
    - **Defensive guards on `Number.isFinite` + range.** Same policy as WV2/WV4/WV5/WV6. NaN or out-of-range silently drops the offending field. Worst case: pre-WV7 baseline (finding tiers as medium / no evidence). No throws, no error paths.
    - **Composite-tool migration template confirmed.** WV7 is the first migration of a composite tool (not a direct `tools/<name>Tool.ts` registration). The pattern transfers: import the formatter + type, destructure the analyzer's plumbed-through field, build the suffix, concat onto the text that names the headline metric. Future composite migrations (if any tool grows that calls `runSignificanceTest` or `runPriceElasticity` internally) can follow WV7's import-path adjustment + same body.
