- **2026-05-20** — **Wave W61-audit-log · prior-model audit trail written to the ChatDocument on every admin PATCH.** Closes the rollback / forensics gap that the W61 polish trail had left open through the trilogy + edit-text + edit-enums + source-bump + source-badge + source-filter + filter-persist arc — every admin save up to now permanently overwrote the prior model and `doc.semanticModel` was the only surviving copy. An admin who edited a metric's expression in error had no way to recover the pre-edit form short of reloading the dataset and re-running the W57 inference pipeline (which would also blow away every *good* correction made in the same session). With this wave each save snapshots the prior model into a capped newest-first ring buffer on the parent doc — `doc.semanticModelAuditLog?: SemanticModelAuditEntry[]` — so a future `W61-audit-revert` UI can walk the buffer and re-PATCH a chosen prior version. Server-only wave (zero client code touched, no schema change because [`ChatDocument`](../server/models/chat.model.ts) is a plain TS interface per the W57 precedent, not a zod schema).

  **What landed.**
    - New pure module [`server/lib/semantic/semanticModelAuditLog.ts`](../server/lib/semantic/semanticModelAuditLog.ts) (~80 LOC) exports `SemanticModelAuditEntry` interface (`savedAt: number` ms-epoch, `savedBy: string` admin email or `"unknown"`, `priorVersion: number` from the prior model's `version`, `priorModel: SemanticModel` full snapshot), `SEMANTIC_MODEL_AUDIT_LOG_MAX_ENTRIES = 10` const (the ring-buffer cap), and `appendSemanticModelAuditEntry(prior, entry, max = SEMANTIC_MODEL_AUDIT_LOG_MAX_ENTRIES): SemanticModelAuditEntry[]` (prepends newest-first, caps at `max`, returns a fresh array, accepts `undefined` for `prior` to handle the first-save-ever case where the field doesn't exist on the doc yet). Implementation is a one-liner: `const next = [entry, ...(prior ?? [])]; return next.length > max ? next.slice(0, max) : next;` — input is treated as `readonly` and never mutated.
    - [`server/models/chat.model.ts`](../server/models/chat.model.ts) gains `semanticModelAuditLog?: SemanticModelAuditEntry[]` on the `ChatDocument` interface, sibling to the existing `semanticModel?: SemanticModel` field (lines 137–148 with full doc-comment). Imported via `import type { SemanticModelAuditEntry } from "../lib/semantic/semanticModelAuditLog.js"` (type-only import keeps the runtime module dep light; the cycle-free path is `chat.model.ts → semanticModelAuditLog.ts → shared/schema.ts`).
    - [`server/controllers/adminSemanticModelController.ts`](../server/controllers/adminSemanticModelController.ts) imports `appendSemanticModelAuditEntry` and wires the audit-write into the `withSessionWriteLock` callback inside `patchSemanticModel` (the existing per-session mutex per invariant #9 — the prior W61-source-bump wave established the pattern of nesting per-entry bookkeeping inside the lock; the audit-write fits the same shape). A single `const savedAt = Date.now()` capture is taken once and used for **both** the audit entry's `savedAt` field and `doc.lastUpdatedAt` so a future history-tab UI can correlate `lastUpdatedAt` to a specific audit entry without clock-skew confusion (prior code did `doc.lastUpdatedAt = Date.now()` directly — behaviorally equivalent because both happen within microseconds inside the lock, but the explicit reuse documents the intent). The audit-write happens **before** the `doc.semanticModel = nextModel` overwrite so the snapshot captures the actual prior state, not the next state by accident: `doc.semanticModelAuditLog = appendSemanticModelAuditEntry(doc.semanticModelAuditLog, { savedAt, savedBy: updatedBy, priorVersion: doc.semanticModel.version ?? 0, priorModel: doc.semanticModel });` — note the `?? 0` defensive coalesce for the rare pre-W57 doc that somehow reached the PATCH path without a `version` field (the route's pre-check at line 218 guarantees `doc.semanticModel` is defined, but `version` is technically optional in the zod schema).
    - [`server/tests/semanticModelAuditLogW61AuditLog.test.ts`](../server/tests/semanticModelAuditLogW61AuditLog.test.ts) (~390 LOC) appended to [`server/package.json`](../server/package.json)'s explicit test list per invariant #4 (right after `semanticModelSourceBumpW61SourceBump.test.ts`). 13 tests across two groups: 10 pure-module tests + 3 controller-integration tests.

  **Why this design.**
    - **Snapshot the *prior* model, not the next.** Considered: write the next model as the audit entry so the buffer reads as "history of saves you've made". Rejected — the next model already lives on `doc.semanticModel` itself; storing it again would be redundant. The buffer's role is to remember what was just *lost*: a revert UI rewinds to "the state before save N" by walking back N entries and re-PATCH-ing the `priorModel` of entry N. Storing the next model would force the revert UI to walk forward through the buffer (less natural) and would not capture the very-first state because there's no "save 0" entry.
    - **Cap at 10.** SemanticModel snapshots are moderate-sized (~5–50 KB depending on dataset width — a 30-metric / 20-dimension / 5-hierarchy model JSON-serialises around 10 KB; a 100+ metric MMM-style model approaches 50 KB). 10 entries × 50 KB worst-case = ~500 KB of audit overhead per doc, comfortably under the 2 MB Cosmos document size limit. The deep-history use case (revert to the 11th-most-recent save) is rare enough that paying for unbounded growth across months of saves isn't worth it — an admin who has made >10 saves has almost certainly settled on a stable model. The const is exported so a future wave that genuinely needs more history can raise it in one place + the [pin test](../server/tests/semanticModelAuditLogW61AuditLog.test.ts) catches the change in the same diff.
    - **Newest-first ordering.** Matches the MRU convention already used by [WI6 `insightHistory`](../client/src/pages/Dashboard/lib/insightHistory.ts) (per-tile insight history with newest at index 0), the sidebar Recent Sessions list, and the prior W61-filter-persist wave's URL-param semantics. A future history-tab UI renders the buffer as a list without needing to invert; the most recent prior is always `[0]`.
    - **Pure-function append + cap, no class.** Considered: an `AuditLogRingBuffer` class with `append()` / `revert(index)` / `prune()` methods. Rejected for two reasons: (a) the cap-and-prepend semantics are a one-liner — adding a class shell adds ceremony without information; (b) the prior W61 polish trail's pure-module precedent ([`semanticModelSourceBump.ts`](../server/lib/semantic/semanticModelSourceBump.ts) / [`semanticModelEditValidation.ts`](../client/src/pages/Admin/lib/semanticModelEditValidation.ts) / [`semanticModelSourceBadge.ts`](../client/src/pages/Admin/lib/semanticModelSourceBadge.ts) / [`semanticModelSourceFilter.ts`](../client/src/pages/Admin/lib/semanticModelSourceFilter.ts) / [`semanticModelFilterUrlSync.ts`](../client/src/pages/Admin/lib/semanticModelFilterUrlSync.ts)) is clean pure-function + frozen-table prose; a class would be the only OO surface in the polish trail and would jar against the established style.
    - **`undefined` accepted for the `prior` argument.** First save on a session means `doc.semanticModelAuditLog` is `undefined` (the field is `?:` optional). The function inlines the `?? []` rather than forcing callers to do it, because every caller is the same controller path and the alternative would either duplicate the coalesce at the call site or force every caller to pass `[]`. The pure-module tests pin this with a dedicated case (`appendSemanticModelAuditEntry(undefined, makeEntry(1))` → `[entry]`).
    - **`max` param exposed in the public signature.** Defaults to `SEMANTIC_MODEL_AUDIT_LOG_MAX_ENTRIES` so production callers don't pass it. Exposed for testability — exercising the cap mechanics in tests by writing 11 fresh entries against the default would burn 11 fixture constructions; passing `max: 3` lets a single test cover the cap with 5 entries. The `max = 0` degenerate case is also pinned (legal but rare — append-then-immediately-trim produces an empty array; load-bearing because it confirms the cap is applied AFTER prepend, not before).
    - **Returns a fresh array reference even when under cap.** Considered: when `next.length <= max`, return the constructed `[entry, ...(prior ?? [])]` array directly (which is already fresh); when `next.length > max`, return `next.slice(0, max)` (also fresh). Both branches return fresh arrays — there's no "return the input reference" optimization analogous to the [W61-source-filter `filter === "all"` case](../client/src/pages/Admin/lib/semanticModelSourceFilter.ts) because every call here is a state change (append), so there's no "default-case-is-no-op" path. A test pins the fresh-reference contract because a future "defensive" refactor that returned the input reference unchanged would silently break callers that expect to assign the result back into the field without aliasing the prior buffer.
    - **`SemanticModelAuditEntry.priorModel` is a reference, not a deep clone.** The controller does `priorModel: doc.semanticModel` and then `doc.semanticModel = nextModel` — the reference is preserved before the field-overwrite assigns a different reference. The prior model's POJO is never mutated by any code path in the runtime (every save constructs a fresh model object via `{...parsed.data, ...overrides}` and assigns it whole-cloth; no in-place mutation occurs). Adding `structuredClone(doc.semanticModel)` would defend against a future in-place mutation refactor but would cost ~5–50 KB of allocation per save for a hypothetical scenario that no existing code triggers. Skip the defensive clone; rely on the replace-don't-mutate convention. The integration tests pin the actual prior-vs-next semantics — if a future refactor mutates the prior model in place, the `assert.equal(entry.priorModel.metrics[0].label, "Prior label")` pin in the "first save writes a 1-entry audit log" test would fail.
    - **Audit-write inside the existing `withSessionWriteLock`, not a new lock.** Per invariant #9 the unified per-session mutex is `withSessionWriteLock`; adding a separate lock for the audit write would (a) violate the invariant and (b) open a race window where the audit entry is written without the corresponding model overwrite committing (and vice versa). The audit-write piggybacks on the existing critical section so the entry-and-overwrite pair is atomic at the per-session level. Single-instance correctness only; multi-instance horizontal scaling still needs Cosmos `ifMatch` ETag (out of scope for this wave per the existing lock semantics).
    - **`savedBy` stamps the email at save time, not a backref to a user table.** `getAuthenticatedEmail(req)` returns the same `updatedBy` that the new `nextModel.updatedBy` carries — they always match for a given save. Stamping the email directly into the audit entry (rather than e.g. an `updatedById` FK) means the entry is self-describing even if the future user-management surface evolves; an admin's email rotation later (e.g. `alice@old.com` → `alice@new.com`) preserves the historical attribution.
    - **No new GET endpoint to fetch the audit log separately.** The detail endpoint `GET /admin/semantic-models/:sessionId` already returns the full `ChatDocument` shape internally and selectively projects to `AdminSemanticModelDetailResponse` (which currently excludes `semanticModelAuditLog`). A future history-tab wave will either (a) widen `AdminSemanticModelDetailResponse` to optionally include the log (`?includeAuditLog=true` to keep the default response light) or (b) add a dedicated endpoint. Out of scope here — this wave is the write-path only; the read-path is the next wave's surface.
    - **Why field name `semanticModelAuditLog` not `auditLog`.** A bare `auditLog` reads like cross-cutting audit (filter changes, message replays, dataset edits, etc.) — naming the field after its specific content makes the scope obvious and reserves the bare name for a future cross-cutting audit field if one is ever needed. The future history-tab UI's hook reads `doc.semanticModelAuditLog` which matches the field's role in the data model. The single test that pins this naming (the integration test's `currentDoc.semanticModelAuditLog!.length === 1` assertion) would surface a future rename as a TypeScript error.

- **2026-05-20** — **Wave W61-filter-persist · URL-state persistence of the W61-source-filter active chip via `?filter=X` query param.** Layers share-link + reload-survival on top of the just-shipped W61-source-filter chip row. The prior wave's `useState`-only filter state reset to `"all"` on every page reload, and a copy-pasted URL lost the active chip — the admin who clicked User, scrolled to find an issue, then accidentally reloaded the page lost their position. With this wave the URL becomes `…/admin/semantic-models/<sessionId>?filter=user` on chip click, so a reload reads it back, and a shared link opens the same filtered view for a teammate. Pure-client wave (zero server code touched, zero schema change).

  **What landed.**
    - New pure module [`client/src/pages/Admin/lib/semanticModelFilterUrlSync.ts`](../client/src/pages/Admin/lib/semanticModelFilterUrlSync.ts) exports `FILTER_PARAM_NAME = "filter"` (the URL param name, deliberately terse — 5 chars vs. e.g. `source_filter` so URLs read clean when admins paste them into chat / docs / GitHub issues; the param's meaning is unambiguous in context because no other filter on this page), `readFilterFromSearch(search): SemanticEntryFilter` (parses a `window.location.search` string with optional leading `?`, validates the value against a `ReadonlySet<SemanticEntryFilter>` for O(1) check; falls back to `"all"` for missing / invalid / empty / wrong-case), and `writeFilterToSearch(search, filter): string` (returns the updated search string with the param set; for `filter === "all"` REMOVES the param entirely so the URL stays clean at the default state). Both functions are string-in / string-out — no direct `window.location` reads — so they're trivially testable in `node:test` without a jsdom mount. Return values never include the leading `?` so the caller decides whether to prepend it (`"?" + search` for `history.replaceState`, or empty string for "no query params at all") and the empty-case stays unambiguous.
    - Validation is case-sensitive: `?filter=USER` does NOT match `"user"` and falls back to default rather than silently flipping to an unintended state. URL params are byte-stable in the wild because admins copy URLs from browsers which preserve case; lenient case-insensitive matching would let a mistyped URL look like it worked while silently routing to the wrong filter. The `ReadonlySet` declaration mirrors `SOURCE_FILTER_ORDER` from [`./semanticModelSourceFilter.js`](../client/src/pages/Admin/lib/semanticModelSourceFilter.ts) — a test in the suite pins this membership against the four-value vocabulary so a future widening lands as a TypeScript error in the helper module rather than silently allowing an unknown param.
    - [`AdminSemanticModelDetail.tsx`](../client/src/pages/Admin/AdminSemanticModelDetail.tsx) wires them in via a lazy `useState` initializer (`useState<SemanticEntryFilter>(() => typeof window === "undefined" ? "all" : readFilterFromSearch(window.location.search))` — the `typeof window === "undefined"` guard is for SSR / non-window environments; the admin route is client-only today but a future Next.js / static-prerender migration shouldn't crash) and a new `useEffect` keyed on `[sourceFilter]` that calls `window.history.replaceState(window.history.state, "", nextUrl)` where `nextUrl = window.location.pathname + (nextSearch ? "?" + nextSearch : "") + window.location.hash`. The pathname + hash reconstruction means a future deep-link feature that stashes hash anchors (e.g. `#metric-revenue`) won't get clobbered.
    - The `SOURCE_FILTER_ALL` import is removed from the page (no longer needed — the initializer reads from the URL helper directly and the default-fall-back happens inside `readFilterFromSearch`). The `useState`'s explicit `SemanticEntryFilter` type annotation keeps the inferred type narrow.

  **Why this design.**
    - **`replaceState` rather than `pushState`.** Considered: each chip click pushes a new history entry so the browser back button could step through filter states. Rejected — admins expect back to navigate to the previous page (the admin index, in this case), not toggle through chip states. Stacking five chip clicks would require five back-presses to leave the page, which is broken behaviour. The single history entry stays at the most-recently-clicked filter so the URL reflects the current view AND the back button does what admins expect.
    - **Pre-existing `window.history.state` as the first arg to `replaceState`.** The first arg is the state object the browser exposes via `popstate`; passing the existing state means a future deep-link feature that stashes nav metadata there won't get clobbered by the filter-sync rewrite. Passing `null` would silently wipe any pre-existing state from a parent route's `pushState` call.
    - **Pathname + hash reconstruction not full `window.location.href` rewrite.** The latter would force a full URL parse-and-replace (and could subtly alter the URL's encoding if the browser normalises differently); reconstructing from the three known components (`pathname` / `search` / `hash`) is byte-exact and cheaper.
    - **Lazy `useState` initializer, not `useEffect`-on-mount.** Considered: `useState("all")` plus a `useEffect` that reads from URL on mount and calls `setSourceFilter`. Rejected — that approach renders once with `"all"` then re-renders with the URL value, which causes a visible chip-flicker on page load if the URL has a non-default filter. The lazy initializer reads the URL on the first render so the first paint already shows the correct chip.
    - **`typeof window === "undefined"` guard.** Pure defense against future SSR / prerender — the admin route is client-only today but the guard costs nothing and saves a confusing crash if someone migrates the project to Next.js.
    - **Fall-back to `"all"` rather than throw on invalid URL.** The URL is admin-controlled and admins mistype; throwing would crash the page-mount and lock the admin out. Falling back to the safe default gracefully degrades while the URL still shows the wrong value (the next chip click will rewrite it cleanly).
    - **Short `filter` param name not `source_filter`.** Admins frequently copy URLs into chat / docs / GitHub issues; a verbose param reads as noise. The param's meaning is unambiguous in context (no other filter on this page) so the short name doesn't cost clarity.
    - **Case-sensitive validation.** Considered: lower-case the incoming value before validation so `?filter=USER` works. Rejected — `URL` capitalisation is byte-stable in real use (browsers preserve case in `location.search`); a lenient match would mean an admin typing `USER` thinks they're applying a filter when they are, while a different scenario with `User` (title-case from a casually-typed link) would also work — both feel correct until a future use case requires the byte-stable form (e.g., URL hashing for analytics) and the leniency masks a real drift.
    - **Remove the param entirely for `"all"`, not write `?filter=all`.** The default state IS no filter; `?filter=all` would be redundant byte-noise and would clutter the URL admins are copying. The absence of the param means the same thing (`readFilterFromSearch` falls back to `"all"` on missing). The `URLSearchParams.delete` call is the standard idiom.
    - **Return strings without a leading `?`.** The caller's `history.replaceState` call wants to control the leading-? semantics — empty string means "no query params" and "?" + value means "these params". Returning a leading `?` from the helper would make the empty-case ambiguous (was the URL meant to have an empty `?` or no `?`?). The pin in tests catches a future "helpful" refactor that adds the prefix.

  **Why this design.**
    - **Replace-whole-model PATCH body rather than partial-patch.** Considered: a JSON Patch–style body that diffs only the changed field (`{ op: "replace", path: "/metrics/0/exposed", value: false }`). Rejected — admin edits are small and the client already holds the prior model in memory; sending the whole thing keeps the diff logic out of the server and lets `semanticModelSchema.parse` validate the payload uniformly. JSON Patch would also break the server's "validate the whole shape" invariant in subtle ways (a malformed metric appended via patch wouldn't trigger the full-model parse).
    - **Server overrides version + updatedAt + updatedBy regardless of client-sent values.** A client-set version would let an admin accidentally bump backwards (set it to 1) or jump forward (set to 9999), and the W64 compiled-query cache reads this version as its cache key; a stable monotonic bump per save is load-bearing. The server-side stamp is enforced by overwriting after `parsed.data` is constructed.
    - **`PatchResult` discriminated union over throwing inside the lock.** `withSessionWriteLock` chains promises per session; an inner throw rejects the chained promise that future callers `await`, which the existing implementation already swallows in its `catch {}` — but the inner result is still lost. Returning a typed result via the union keeps the 404 paths first-class and makes the controller's HTTP-mapping layer one clean switch.
    - **`withSessionWriteLock` over a global mutex.** Per invariant #9 every RMW on a `ChatDocument` MUST acquire `withSessionWriteLock(sessionId, fn)` — a global mutex would serialize unrelated sessions and slow the admin's batch-edit flow. The per-session lock is the existing contract; the admin path slots into it cleanly.
    - **Single `saving` flag over per-row state.** Considered: each row could have its own `saving` state so two simultaneous toggles each show their own spinner. Rejected — `withSessionWriteLock` already serializes the writes on the same session; firing two toggles in parallel would mean the second sees a stale optimistic view of the first's result and the round-trip ordering becomes hard to reason about. A single flag disables every toggle during a save, forces a clean "one save at a time" UX, and matches the actual server-side serialization.

A future wave could introduce per-tool effect-size buckets (e.g., a correlation `r` → bucket via Cohen's conventions: `|r| < 0.1` → negligible, `< 0.3` → small, `< 0.5` → medium, else large) but that's a separate scoping question per tool and was outside the WQ8 budget.
    - **Surfacing `effectMagnitude` in the narrator FINDING_CONFIDENCE prompt block.** [`buildNarratorConfidenceBlock`](../server/lib/agents/runtime/narratorHintsBlock.ts) summarises each finding's tier + extracted reasons; the WQ8 reasons strings (`"large effect size"`, `"negligible effect size"`, etc.) flow through automatically because the reasons array is the rendering surface, but the prompt block doesn't currently expose the raw bucket as a separate field. A future polish wave could add a `magnitudeBucket: <bucket>` line per finding for narrator framing — deferred until the narrator demonstrably under-uses the existing reasons.
    - **Backfilling the WV1 verifier-side `detectConfidenceOverclaims` to use `effectMagnitude` independently of the tier.** Today the WV1 detector compares **claimed** vs **actual** tier counts; the WQ8 change feeds the actual-side tier distribution via the classifier. A future wave could surface per-finding overclaim flags like `narrator_claims_large_effect_but_blackboard_evidence_is_small` directly off the bucket without rolling it into a tier. Deferred — the tier-based overclaim check is already the load-bearing surface.
    - **Migrating `FindingEvidence.magnitude` (the numeric channel) to a structured `{ value, bucket }` shape.** The two fields (`magnitude: number` + `effectMagnitude: bucket`) now sit side-by-side without coordination. A cleaner long-run shape would be a single `effectSize?: { value: number; bucket: "negligible" | ... }`. Deferred — the migration is mechanical but touches every caller of `FindingEvidence`, and the current two-field layout is correct and non-breaking.
