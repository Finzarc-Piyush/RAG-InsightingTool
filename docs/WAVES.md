- **2026-05-20** — **Wave W61-edit-enums · enum-select editing for `format` / `kind` / `temporalGrain` + `currencyCode`.** Second polish wave on top of the W61 trilogy + W61-edit-text foundation: admins can now correct enum-shaped inference gaps on the semantic model inline. The most common case the data team needs is a temporal column whose `kind` was inferred as `categorical` because the values are string month names ("Jan", "Feb", …) rather than ISO dates — two clicks now flip it to `temporal` + `temporalGrain: "month"`. Similarly a percent metric inferred as `format: "number"` (because the dataset stores `1.0–100.0` not `0.0–1.0`) flips with one select. Currency-code text edit appears conditionally when `format === "currency"` so the planner manifest's `currency (INR)` hint renders correctly. The trilogy → edit-text → edit-enums sequence now closes every common-admin correction except adds / deletes / hierarchy reordering.

  **What landed.**
    - [`AdminSemanticModelDetail.tsx`](../client/src/pages/Admin/AdminSemanticModelDetail.tsx) gains a new `EditableSelect<T extends string>` sub-component sibling to the `EditableText` from W61-edit-text. Unlike `EditableText` there's no draft / validation step — every option is by-construction valid because the option list is byte-locked to the zod enum at the source. Wraps Radix `<Select>` with `onValueChange={(v) => onSave(v as T)}` firing the parent's optimistic-update-and-PATCH flow directly. Standard Radix trigger / content / item composition; `h-8 text-sm` styling matches the row density of `EditableText`. The `value` prop is typed `T | undefined`; Radix accepts `undefined` and renders the placeholder, used for cases where the underlying field is `.optional()` and unset.
    - Three frozen option-list constants declared with the `as const satisfies ReadonlyArray<{ value: …; label: string }>` pattern so a typo against the zod enum becomes a TypeScript error at the declaration site rather than a runtime "Save failed: invalid_semantic_model" with a confusing zod issue message: `METRIC_FORMAT_OPTIONS` (5 entries — `number` / `percent` / `currency` / `ratio` / `duration`), `DIMENSION_KIND_OPTIONS` (4 entries — `categorical` / `temporal` / `numeric_binned` / `geo`), `TEMPORAL_GRAIN_OPTIONS` (6 entries — `__auto__` sentinel plus `day` / `week` / `month` / `quarter` / `year`). The `__auto__` constant is exported alongside the option list so the parent handler's sentinel check (`nextGrain === TEMPORAL_GRAIN_AUTO ? undefined : ...`) stays in lockstep.
    - `MetricRow` widens with two new handler props: `onEditFormat(next)` and `onEditCurrencyCode(next)`. The format cell becomes an `EditableSelect` over `METRIC_FORMAT_OPTIONS` plus a conditional `EditableText` (monospace, `placeholder="USD / INR / EUR"`) for `currencyCode` rendered only when `m.format === "currency"`. The pre-existing read-only `decimals` hint is preserved as a sibling line below.
    - `DimensionRow` widens with `onEditKind(next)` and `onEditTemporalGrain(next)`. The kind cell becomes an `EditableSelect` over `DIMENSION_KIND_OPTIONS` plus a conditional `EditableSelect` over `TEMPORAL_GRAIN_OPTIONS` rendered only when `d.kind === "temporal"`. The temporal-grain select's `value` prop maps `undefined` → `TEMPORAL_GRAIN_AUTO` so Radix has a concrete string to display ("Auto (let agent derive)").
    - Three new parent handlers funnel through the existing `patchMetric` / `patchDimension` narrow handlers from W61-edit-text: `handleMetricFormatChange(metricName, nextFormat)` (clears `currencyCode` when `nextFormat !== "currency"` — a stale `"USD"` value on a `format: "percent"` metric would mislead the planner manifest's `currency (USD)` hint), `handleDimensionKindChange(dimensionName, nextKind)` (clears `temporalGrain` when `nextKind !== "temporal"` — a `"month"` grain on a `categorical` dimension is semantically nonsensical), and `handleTemporalGrainChange(dimensionName, nextGrain)` (maps the `__auto__` sentinel back to `undefined` so the agent's `temporalFacetColumns` derivation continues to derive the grain from data when the admin hasn't explicitly pinned one).
    - [`semanticModelEditValidation.ts`](../client/src/pages/Admin/lib/semanticModelEditValidation.ts) gains `validateCurrencyCode(value)`: returns `null` for empty / whitespace-only (optional field — the empty string is a legitimate "not set yet" state while the admin is mid-edit), validates `/^[A-Z]{3}$/` after trim otherwise. Does NOT enforce the format-coupling (i.e. "if format=currency then currencyCode required") because `EditableText` validates fields in isolation; the server's `semanticModelSchema.safeParse` is authoritative on cross-field constraints and rejects a `currency` format paired with an empty `currencyCode` at PATCH time with a zod issue path that surfaces in the existing "Save failed: {message}" banner.

  **Why this design.**
    - **`as const satisfies ReadonlyArray<{ value: T; label: string }>` for the option lists.** Considered: a runtime `if (!OPTIONS.find(o => o.value === schemaValue)) throw` cross-check at app boot. Rejected — TypeScript's `satisfies` keyword does exactly this at compile time without runtime cost, makes the typo a red squiggle in the IDE, and the `as const` keeps the literal-narrowing so the `value: T` flows through `EditableSelect`'s `onSave: (next: T) => void` without a runtime cast.
    - **Save-on-select with no validation step (vs `EditableText`'s blur-then-validate).** A select's value is constrained to the option list by construction; there's no analogue to "admin typed an invalid expression". Skipping the draft/validation dance keeps the component small (~30 LOC) and lets the parent's optimistic-update-and-PATCH flow fire the moment the admin picks an option.
    - **`__auto__` sentinel for the temporal-grain undefined case.** Radix `<Select>` rejects `value={""}` (empty string is reserved for the "no selection / show placeholder" state) and doesn't have a built-in "clear selection" affordance. The sentinel keeps Radix happy AND lets the admin explicitly opt back into auto-derivation after pinning a grain by mistake. The mapping is contained to a 3-line handler at the save boundary; the schema sees only `undefined | "day" | "week" | ...` so no zod surprise.
    - **Clear `currencyCode` on format-change-away-from-currency / `temporalGrain` on kind-change-away-from-temporal.** Considered: leave the orphan fields in place so toggling currency-then-back-to-currency preserves the prior code. Rejected — the orphan fields surface in the planner manifest (W59's `formatMetricCatalog` reads currencyCode unconditionally when it's defined) and would render `Net Sales (USD)` on a metric that's now `format: "percent"`. The "preserve on round-trip" UX win is small (admin can type the 3-char code in 2 seconds); the misleading-manifest cost is the planner downstream of every wrong-shape correction.
    - **`validateCurrencyCode` empty-accepts even though the schema makes it required-with-format-currency.** The field is `.optional()` at the schema level; the cross-field constraint is enforced by the server at PATCH time. Making the client validator reject empty would block the admin's natural "I'm switching to currency, give me a second to type the code" workflow — they'd hit a red border before typing the first letter. The server's authoritative rejection (with a clear zod issue path) is the right place for that check.
    - **Conditional render rather than always-rendered-but-disabled.** A disabled `currencyCode` input on a non-currency metric would clutter the cell visually + need explanatory text ("only applies when format = currency"). The conditional render is cleaner and the admin's mental model is "if I see the currency-code field, I need to fill it" which is correct.

- **2026-05-20** — **Wave W61-edit-text · inline text editing for metric / dimension fields.** First polish wave on top of the just-closed W61 trilogy: admins can now edit metric `label` / `description` / `expression` and dimension `label` / `description` directly in the per-session viewer with save-on-blur semantics. The trilogy shipped the read-and-toggle loop (W61-list index → W61-detail viewer → W61-save PATCH with exposed-flag affordance); this wave widens the W61-save endpoint's actual usefulness — the most common admin correction after the data team spots an inference gap is "this metric's expression should be `SUM` not `AVG`" or "this dimension's label should be 'Region (geo)' not the raw column name", both read-only after W61-save. Pure-text scope: enum selects (`kind` / `format` / `temporalGrain` / `currencyCode`) and structural edits (add / delete / hierarchy) deferred to follow-on waves.

  **What landed.**
    - New pure module [`client/src/pages/Admin/lib/semanticModelEditValidation.ts`](../client/src/pages/Admin/lib/semanticModelEditValidation.ts) ships four exports: `validateLabel(value): string | null` (mirrors `semanticMetricSchema.label.min(1).max(120)` — empty / whitespace-only / >120 char rejections), `validateDescription(value): string | null` (mirrors the `.max(1000)` cap; empty allowed because the field is `.optional()`), `validateExpression(value): string | null` (the interesting one — see below), and `isMeaningfulChange(prior, next): boolean` (returns `false` when trimmed values are equal — used by the save-on-blur handler to skip a no-op PATCH when the admin blurs without making a change). Each validator returns `null` for valid or a short error message suitable for inline display under the input.
    - `validateExpression` is a cheap obvious-broken filter ahead of the W58 compiler's authoritative parse. Rejects: empty / whitespace-only, length over 2000, semicolons (multi-statement guard — without it an admin could paste `SUM(x); DROP TABLE users` and the back-half would be a separate Cosmos-stored statement the compiler ignores; the schema regex doesn't catch this since the field is a free-text string), SQL comments (`--` or `/*` — they hide content from the W58 validation, e.g. `SUM(x) -- )` could mask a paren imbalance), and top-level keywords `SELECT` / `FROM` / `JOIN` / `WHERE` / `GROUP BY` / `ORDER BY` / `UNION` matched via a padded-space scan (`SUM(selector_count)` is NOT rejected even though "SELECT" appears as a substring — the padded `" ${trimmed.toUpperCase()} "` form requires whitespace boundaries). The W58 compiler still does the authoritative parse and column-binding check; this is the cheap "admin pasted a full SELECT statement from a notebook" catch. Bounds (LABEL_MAX = 120, DESCRIPTION_MAX = 1000, EXPRESSION_MAX = 2000) mirror [`semanticMetricSchema`](../server/shared/schema.ts) / [`semanticDimensionSchema`](../server/shared/schema.ts); drift here just round-trips a save attempt to the server's authoritative rejection, not a correctness bug.
    - [`AdminSemanticModelDetail.tsx`](../client/src/pages/Admin/AdminSemanticModelDetail.tsx) gains a new `EditableText` sub-component sibling to the `ExposedToggle` from W61-save. Props: `{ value, onSave, validate, disabled, ariaLabel, multiline?, monospace?, placeholder? }`. Local `draft` state syncs from the `value` prop on every change via `useEffect([value])` — when the server's authoritative reply lands (with a normalised whitespace or version bump), the field re-syncs. `onBlur` semantics: if validation failed, the draft resets to the last-known-good prop value (visible red border briefly cleared); if `!isMeaningfulChange(value, draft)` the draft resets without firing the save; otherwise `onSave(draft.trim())` triggers the parent's optimistic-update-and-PATCH flow. `Enter` blurs (single-line only — Enter in a multiline textarea inserts a newline). `Escape` discards the draft and blurs. Renders either an `<Input>` (single-line, `h-8`) or a `<Textarea>` (multiline, `min-h-[60px]`) based on the `multiline` prop; both wear a `font-mono text-xs` class when `monospace` is true (for the expression cell). Validation errors render below the input with `role="alert"` for a11y; the input also gets `aria-invalid` when an error is present.
    - `MetricRow` widens to accept three additional handler props: `onEditLabel(next)`, `onEditDescription(next)`, `onEditExpression(next)`. The label / description / expression cells now render `EditableText` in place of the previous read-only spans. Cell layout reorganised so the snake-case `m.name` identifier sits above the editable label (the name itself is NOT editable — it's the planner-facing canonical id), with the description's textarea stacked below. `DimensionRow` widens symmetrically with `onEditLabel(next)` + `onEditDescription(next)` — no expression edit since dimensions don't have an `expression` field. The `column` / `kind` / `temporalGrain` cells stay read-only this wave.
    - Parent component gains two new narrow handlers: `patchMetric(metricName, patch: Partial<SemanticMetric>)` and `patchDimension(dimensionName, patch: Partial<SemanticDimension>)`. Both build the `nextModel` by mapping over the current model's arrays — other entries pass through identity-preserved; only the edited row's patched fields change. Both call the existing `persistModel` helper from W61-save which handles the optimistic update → PATCH → reconcile-with-server-stamps → roll-back-on-error flow uniformly. A small `emptyToUndef(s): string | undefined` helper trims the input and returns `undefined` for empty values so the round-trip matches the zod schema's `.optional()` shape (empty strings would re-emit on the server and subtly drift the doc shape).

  **Why this design.**
    - **Always-editable inputs over click-to-edit popovers.** Considered: each cell renders as styled read-only text by default; clicking it swaps in an input with `autoFocus`; blur or commit returns to read-only. Rejected — the dance adds state (`editingFieldId`), keyboard-trap risk (Escape during the swap), and an extra click per correction. Admins fixing inference gaps spend their time tabbing through many fields per session; always-editable lets them flow without a context switch per cell. The visual distinction "this is editable" comes from the input's own styling (border, focus ring) which already signals affordance.
    - **Save-on-blur over save-on-Enter or per-row Save button.** Considered: an explicit Save button per row that batches edits + a discard-changes button. Rejected — the parent's `persistModel` already debounces by routing through `withSessionWriteLock(sessionId, ...)` (invariant #9), so multiple sequential field saves serialize cleanly without a batch button. Save-on-Enter alone leaves a "ghost dirty" state if the admin tabs out without explicitly committing. Save-on-blur matches Google Docs / Notion semantics and pairs with `isMeaningfulChange` to skip no-op PATCHes.
    - **Client-side validators that mirror but don't share zod schemas.** Considered: import `semanticMetricSchema.shape.label` and re-derive the bounds at runtime. Rejected — the validators are 96 LOC of pure-function bounds checks; if the schema bounds change, a drift here just routes the save to the server's authoritative rejection (a 400 with the zod issues array surfacing in the existing "Save failed: {message}" banner). The validators' job is to keep obviously-bad inputs from round-tripping at all, not to be a second source of truth.
    - **Keyword-guard via padded uppercase scan instead of regex.** Considered: `/\b(SELECT|FROM|JOIN|...)\b/i` over the trimmed string. Rejected — JavaScript `\b` matches at underscores which would let `_FROM_` slip through; a literal regex with `(?<![A-Za-z0-9_])` lookbehind works but the engine's lookbehind support varies by runtime. The padded-uppercase form (`" ${trimmed.toUpperCase()} "`.includes(" FROM ")) is byte-exact, runs in O(n) per keyword, and avoids regex edge cases.
    - **`isMeaningfulChange` based on trimmed equality.** Considered: deep equality on raw strings. Rejected — admins frequently add or trim trailing spaces while typing; firing a PATCH on every whitespace-only edit would burn write throughput and pollute the W64 cache key (version bump per save). The trimmed-equality form means leading/trailing whitespace is normalised by the save itself (the handler does `draft.trim()` before calling `onSave`).
    - **`emptyToUndef` helper for the description's `.optional()` shape.** A literal empty string is a valid `z.string().max(1000).optional()` value (`.optional()` only adds `undefined` to the union; it doesn't strip empty strings), but persisting `""` instead of `undefined` makes the doc shape drift: a freshly-inferred model would carry `description: undefined` for missing descriptions and a manually-cleared one would carry `""`, which are semantically the same but serialise to different bytes (and a future "no description = use the auto-inferred one" feature would break in subtle ways). The helper normalises both into `undefined`.

  **Out of scope.**
    - **Enum selects.** `kind` (categorical / temporal / numeric_binned / geo), `format` (number / percent / currency / ratio / duration), `temporalGrain` (day / week / month / quarter / year) are zod enums; the inline edit affordance needs a `<Select>` component (Radix select primitive is already in `client/src/components/ui/select.tsx`) plus a conditional render for `currencyCode` when `format === "currency"` and `temporalGrain` when `kind === "temporal"`. - **2026-05-20** — **Wave W61-save · PATCH endpoint + exposed-toggle edit affordance.** Third and final W61 sub-wave: closes the trilogy by adding the write path on top of W61-list's index and W61-detail's viewer. Admins can now toggle each metric / dimension's `exposed` flag inline; the change auto-saves via PATCH with server-stamped version bump and `updatedAt` / `updatedBy`. The trilogy as a whole opens the data-team's review-and-correct loop on the W57-inferred semantic model: review a session's inferred metrics (W61-list / W61-detail), spot an inference miss (a `total_revenue` field tagged `AVG` instead of `SUM`, a `month_name` column inferred as `categorical` instead of `temporal`), and one-click-toggle to fix it before the planner sees the wrong shape. Future polish waves can add the text-field inline edit (label / description / expression / kind) on top of this foundation.

  **What landed.**
    - [`adminSemanticModelController.ts`](../server/controllers/adminSemanticModelController.ts) gains `patchSemanticModel` — gated by `isAdminRequest` (403 → `admin_required`), validates `req.params.sessionId` (400 → `missing_session_id` on empty / whitespace), runs `semanticModelSchema.safeParse(req.body)` (400 → `invalid_semantic_model` with the first 10 zod issues mapped to `{ path, message }` for client display). On success, the RMW goes through [`withSessionWriteLock(sessionId, ...)`](../server/lib/sessionWriteLock.ts) per invariant #9 so a concurrent chat-turn-end persist on the same session can't race-corrupt `messages[]`. Inside the lock: load via injectable `_detailFetcher`, validate again (404 modes split: `session_not_found` vs `semantic_model_not_inferred` — same shape as the W61-detail GET), build `nextModel` with `version: (prior.version ?? 0) + 1`, `updatedAt: new Date().toISOString()`, `updatedBy: getAuthenticatedEmail(req) ?? "unknown"`. Server-set fields override any client-sent values for those keys; the client cannot bump version backwards or spoof `updatedBy`. The injected `_updater` (defaults to `updateChatDocument`) persists; the result is returned via a typed `PatchResult` discriminated union (`{ kind: "ok" | "session_not_found" | "semantic_model_not_inferred" }`) rather than throwing — throwing inside the lock would mark prior callers' chained work as failed and surface as 500 to them.
    - New `__setSemanticModelUpdaterForTesting(fn | null)` shim completes the trio (lister / detailFetcher / updater) so every Cosmos boundary is injectable in tests. The route test runs the full PATCH path including the lock without booting Cosmos.
    - Route `PATCH /admin/semantic-models/:sessionId → patchSemanticModel` registered in [`routes/admin.ts`](../server/routes/admin.ts) directly under the W61-detail GET route. The three `:sessionId` operations (GET detail / PATCH save / future DELETE) share a namespace.
    - [`lib/api/admin.ts`](../client/src/lib/api/admin.ts) gains `patchSemanticModel(sessionId, model)` + `PatchSemanticModelResponse` interface (`{ sessionId, lastUpdatedAt, model }`). Uses the existing `adminHeaders()` MSAL+userEmail helper + adds `Content-Type: application/json`.
    - [`AdminSemanticModelDetail.tsx`](../client/src/pages/Admin/AdminSemanticModelDetail.tsx) gains a new pure `ExposedToggle` sub-component wrapping Radix `<Switch>` with `aria-label="Toggle {label} exposed"` for a11y. `MetricRow` and `DimensionRow` are widened to accept `saving: boolean` + `onToggleExposed: (next: boolean) => void` props and render the toggle in place of the previous chip-only status indicator. A single shared `saving` flag (rather than per-row) disables every toggle while a PATCH is in flight — cheaper than per-row state and matches "auto-save on toggle" UX (the user shouldn't fire two PATCHes in parallel because the second would race the first's response into the cache). The page handler `persistModel(nextModel)` does optimistic update first (the toggle responds instantly), awaits the PATCH, reconciles with the server's authoritative view (the server's version bump + `updatedAt` / `updatedBy` need to overwrite the optimistic update), and rolls back on failure. Two new banners above the tables: a "Saving…" Card (primary-styled, shown during PATCH) and a "Save failed: {message}. Change rolled back; try again." Card (destructive-styled, shown after a failed PATCH).
    - `handleToggleMetricExposed(metricName, next)` and `handleToggleDimensionExposed(dimensionName, next)` build the `nextModel` by mapping over the current model's arrays — the array's other entries pass through identity-preserved; only the toggled row's `exposed` flag flips. Hierarchies are not editable in this wave (no `exposed` field exists on `SemanticHierarchy`).

  **Why this design.**
    - **Replace-whole-model PATCH body rather than partial-patch.** Considered: a JSON Patch–style body that diffs only the changed field (`{ op: "replace", path: "/metrics/0/exposed", value: false }`). Rejected — admin edits are small and the client already holds the prior model in memory; sending the whole thing keeps the diff logic out of the server and lets `semanticModelSchema.parse` validate the payload uniformly. JSON Patch would also break the server's "validate the whole shape" invariant in subtle ways (a malformed metric appended via patch wouldn't trigger the full-model parse).
    - **Server overrides version + updatedAt + updatedBy regardless of client-sent values.** A client-set version would let an admin accidentally bump backwards (set it to 1) or jump forward (set to 9999), and the W64 compiled-query cache reads this version as its cache key; a stable monotonic bump per save is load-bearing. The server-side stamp is enforced by overwriting after `parsed.data` is constructed.
    - **`PatchResult` discriminated union over throwing inside the lock.** `withSessionWriteLock` chains promises per session; an inner throw rejects the chained promise that future callers `await`, which the existing implementation already swallows in its `catch {}` — but the inner result is still lost. Returning a typed result via the union keeps the 404 paths first-class and makes the controller's HTTP-mapping layer one clean switch.
    - **`withSessionWriteLock` over a global mutex.** Per invariant #9 every RMW on a `ChatDocument` MUST acquire `withSessionWriteLock(sessionId, fn)` — a global mutex would serialize unrelated sessions and slow the admin's batch-edit flow. The per-session lock is the existing contract; the admin path slots into it cleanly.
    - **Single `saving` flag over per-row state.** Considered: each row could have its own `saving` state so two simultaneous toggles each show their own spinner. Rejected — `withSessionWriteLock` already serializes the writes on the same session; firing two toggles in parallel would mean the second sees a stale optimistic view of the first's result and the round-trip ordering becomes hard to reason about. A single flag disables every toggle during a save, forces a clean "one save at a time" UX, and matches the actual server-side serialization.

  **Why this design.**
    - **Pure-additive backwards compat over a key-version migration.** The simplest fix would be a required `buildCacheKey(tileId, filterHash, regionHash)` with `""` as the no-region sentinel. That breaks every existing 2-arg call site and any persistence layer that holds a pre-wave key (none today, but the WI2 footer has been emitting these keys for several waves and will keep doing so). Making the third arg optional + treating `""` as "omit the segment" keeps the wave purely additive — the WI2 footer's keys are byte-identical pre- and post-wave, and the empty-string sentinel is the natural output of `hashBrushRegion(undefined)` so the call surface composes cleanly.
    - **`hashBrushRegion` lives alongside `hashGlobalFilters`, not in `explainSlice.ts`.** The hash is a cache concern, not a region-data concern. Pairing it with `hashGlobalFilters` keeps all the byte-stability contracts in one file (so a future maintainer chasing "why does this entry not de-dup" finds both helpers together) and lets the existing test file extend instead of spawning a new one. The trade-off is a new type-only import of `BrushRegion` from `explainSlice` — accepted: `BrushRegion` is the canonical name + the type-only import has no runtime cost.
    - **Categorical value order preserved (not sorted).** A negative test pin catches a future drift to sorting. The semantics: a categorical region's `values` array comes from the renderer's `xs.slice(i0, i1)` band-scale slice, which is itself ordered. The dispatched order is fixed upstream by `distinctOrdered(data, enc.x.accessor)` — same band-scale domain source the renderer uses to position bars/lines — so identical brushes always yield identical value arrays without sorting. Sorting at hash time would obscure a genuine "different-brush-shape" signal (a hypothetical future renderer that emits values in a different order would alias regions that should differ).
    - **Three-segment string key, not a structural key type.** Considered: switch the cache `Map<string, …>` to a `Map<{ tileId, filters, region }, …>` and write a structural comparator. Rejected — would slow `get` from `O(1)` string lookup to `O(n)` structural walk over every cached entry, and would force a `Map → custom` wrapper for the LRU eviction (currently relying on `Map`'s insertion-order semantics for free LRU). Keeping the key a stable string preserves every existing behavior of the LRU + TTL cache: get / set / has all stay `O(1)`, `keys().next().value` for eviction stays the cheapest-oldest, the test surface keeps the same shape.
    - **Cache slot for the WI2 footer + WI4 panel do NOT share by design.** Pre-wave they could (incorrectly) — a footer key `tile_X::filters` collided with a panel key on the same `(tile, filters)` regardless of region. Post-wave they live in distinct slots (footer = 2 segments; panel = 3 segments). This is the correct outcome — the footer regenerates the WHOLE-chart insight, the panel regenerates the BRUSHED-region insight; sharing the same cache slot was the bug. The WI4-wire wave's comment ("a brushed slice can hit a cache slot populated by the per-tile Re-explain if filter hashes match") was aspirational and never true in practice; the footer and panel produce different prose under the hood (the panel narrows rows via `filterRowsByBrushRegion` before POSTing). Documented as a deliberate slot split in the new tests' "region-bearing brush does NOT collide with the no-region (footer) entry" pin.

  **Why this design.**
    - **Reuse the WI2 `/api/insight/regen` endpoint, NOT a new endpoint.** The WI4 panel's regen call is structurally identical to the WI2 per-tile regen call (same spec, same row narrowing, same MINI-tier LLM, same response shape). Two alternatives considered:
      1. **New endpoint `/api/insight/regen-slice`** — would duplicate everything WI2 ships (auth, schema, telemetry, prompt building, post-processing) for no payload-shape difference. The brush is just an additional `applyChartFilters`-style filter applied client-side before the request fires.
      2. **Reuse `/api/insight/regen`** (chosen) — zero server changes; the client narrows rows BEFORE sending so the server sees the same input shape it sees from per-tile regen.
    - **`useEffect`-driven auto-fire (NOT user-clicked button).** The user just brushed a slice — making them click a second "Explain" button to fire the regen would feel like an unnecessary step. The brush IS the intent. The hook's `seqRef` already guards against stale-resolve races, so a second brush while the first is resolving is safe.
    - **`IDLE_TILE_ID = "__wi4_idle__"` (NOT a runtime-conditional hook call).** React hook rules forbid conditional hook calls; the only way to keep the panel functional in both `event !== null` and `event === null` states is to always call `useInsightRegen`. The idle tileId never matches a real tile id (the `__` suffix is reserved), so the cache slot never collides.
    - **`regen` omitted from `useEffect` deps, with `eslint-disable` comment.** The hook captures `regenerate` via `useRef` internally — the function reference is stable across renders within an instance. Including it in deps would create a render loop (the effect fires → setState in the hook → re-render → effect re-fires → …). The eslint-disable comment is intentional + documented; without the comment the lint rule would auto-suggest a "fix" that breaks the design.
    - **Inline IIFE for chart resolution (NOT a `useMemo` at the DashboardView level).** The resolution math is 5 lines and runs ONLY when the panel re-renders. Lifting it into a memo would force a new render dependency on `explainSliceEvent` + `activeSheet` AND a separate `let explainChart = …` line elsewhere — more code for no perf win. Co-locating the IIFE at the prop pass-through keeps the resolution + the consumer next to each other.
    - **Two-stage narrowing (NOT a single combined filter).** Splitting `applyChartFilters` (handles categorical / numeric / date filters from the global filter map) and `filterRowsByBrushRegion` (handles the brushed sub-domain) lets each helper stay focused on its concern. A future widening that adds e.g. a per-tile filter to the WI4 pipeline drops in as a third filter call without restructuring. Same separation-of-concerns precedent as `applyChartFilters` itself (which is already split per filter type in `chartFilters.ts`).
    - **Cache sharing with WI2 per-tile regen.** Same `insightRegenCache` instance, same `(tileId, filterHash)` key shape. A user who clicks "Re-explain" on a tile, then brushes the same tile with no global filter changes, hits the cache (the brushed tileId === the per-tile tileId). The brushed slice doesn't usually match the per-tile shape (different row count), but the cache key uses tile + filters, not rows — so a hit is possible. This is a free cross-surface win and not a contract risk (the cached entry's prose IS describing the right rows because filters are part of the key).
