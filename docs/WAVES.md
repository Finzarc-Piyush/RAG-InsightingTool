- **2026-07-01** — **Wave W-SEM · LLM-driven per-column semantic typing (Data Summary + whole analysis).** A brand-manager's wide Channel-P&L upload made the Data Summary panel confidently wrong on nearly every column: `Year`=26 averaged, `fy_month_number` summed, single-date `Month` shown "Daily / weekly", margins summed to >100%, 100%-blank columns surfaced as normal categoricals, and `Volume(KL)`/`MRP Value` with absurd-looking mean/sum. - **2026-06-30** — **Wave W-NEWANALYSIS · clicking "New analysis" wedged the app on an infinite "Loading analysis" spinner.** From an open chat at `/analysis/:SID`, clicking the header **"New analysis"** button (or the bare "New chat" path) restored the same session URL and stranded the user on the loader forever — a fresh upload could never be started. Deterministic, every click.
  - **Root cause.** Home's "notify parent of my sessionId" effect ([`Home.tsx`](../client/src/pages/Home/Home.tsx)) was `useEffect(() => onSessionChange(sessionId, fileName), [sessionId, fileName, onSessionChange])`. App's `onSessionChange` is `handleSessionChange` ([`App.tsx`](../client/src/App.tsx)) — an **inline, unmemoized closure recreated on every `Router` render**. `handleUploadNew` strips the URL to `/analysis` (so `urlSessionId`→null), bumps `resetTrigger`, and nulls `loadedSessionData` — all batched into one App render that hands Home a **new callback identity**. The notify effect re-fired purely from that identity churn, while its render-snapshot `sessionId` was **still the OLD SID** (the sibling `resetState` effect's `setSessionId(null)` is only *scheduled* — invisible to another effect in the same commit). It reported the stale SID back to App, whose self-mint guard (which only checked `sessionId && !urlSessionId && location.startsWith('/analysis')`) could not tell a **fresh upload mint** from a **stale echo**, so it `setLocation('/analysis/SID')` — **restoring the just-stripped URL** — and set `selfMintedSessionRef=SID`. That ref then made App's rehydration effect **skip the recovery fetch**, so `loadedSessionData` stayed `null`, `useSessionLoader` never repopulated `sessionId`, and `isResumingSession = !sessionId && (!!loadedSessionData || !!urlSessionId)` = `!null && (false || !!SID)` = **`true` permanently**. An unrecoverable fixed point.
  - **What landed.** Two layers, each a small pure/testable seam. **Consumer fix (the keystone):** new hook [`useNotifySessionChange`](../client/src/pages/Home/modules/useNotifySessionChange.ts) holds `onSessionChange` in a ref (freshened by a no-dep effect declared *before* the notify effect, so the ref is always current when read) and gates the notify on `[sessionId, fileName]` **only** — parent callback-identity churn no longer fires it, so the stale-SID replay can't happen. Wired in [`Home.tsx`](../client/src/pages/Home/Home.tsx) replacing the inline effect, exported from [`modules/index.ts`](../client/src/pages/Home/modules/index.ts). **Source-side belt-and-suspenders:** new pure predicate [`shouldSelfMintSessionUrl`](../client/src/lib/shouldSelfMintSessionUrl.ts) — App now tracks the last-reported sessionId in `lastReportedSessionIdRef` and self-mints only on a genuine `prevReported !== reported` transition (null→real), so even a stale echo from *any* future trigger can never re-mint a just-stripped URL.
  - **Why this design.** The bug is the classic React footgun — an effect depending on an unstable function prop. Fixing at the **consumer** (Home stops reacting to identity churn) is more robust than memoizing the source: an adversarial three-agent trace (confirmed end-to-end at 0.9/0.9/0.95) noted that `useCallback`-ing `handleSessionChange` alone would also break *this* bug but stays fragile — any future dep change reintroduces churn, and *every* App handler (`handleNewChat`, `handleNavigate`, `handleLoadSession`, `handleUploadNew`) is an unmemoized closure.   **Why this design.**
    - **Why bind to `xPx(tooltipData.xRaw)` (the snapped x) rather than `tooltipLeft` (the raw cursor pixel).** The tooltip already snaps to the nearest data point via the existing `onMouseMove` handler's nearest-x scan — that snap is what makes the tooltip's row values meaningful (the user sees Y values at a discrete X, not at an interpolated cursor position). The cross-hair MUST follow the same snap so the indicator visually aligns with the tooltip's row values. If the cross-hair tracked `tooltipLeft` (the raw `pt.x` from `localPoint(e)`), the indicator would slide smoothly across the chart while the tooltip's title jumped between discrete buckets — a visual contradiction that would erode the user's trust in either the tooltip OR the cross-hair. The negative test for this binding (`const cx = xPx(tooltipData.xRaw);` pinned exactly) is the load-bearing invariant — any refactor that "simplifies" by using `tooltipLeft` re-introduces the failure.
    - **Why three guards (`tooltipOpen && tooltipData && brushStart === null`) rather than just `tooltipOpen`.** `tooltipOpen` alone is theoretically sufficient (you can't have a snap without an open tooltip), but the `tooltipData` guard is required for TypeScript narrowing (the IIFE reads `tooltipData.xRaw`, which would be `undefined` without the narrow). The `brushStart === null` guard handles a subtle visual conflict: during an active brush drag, the existing brush rectangle renders a dashed border at `hsl(var(--primary))` — if the cross-hair drew on top of (or under) that rectangle, the two dashed shapes would visually compete and the user would lose the brush-region affordance. The clean separation is "brush owns the visual during drag, cross-hair owns it during pure hover" — gated by `brushStart === null` which IS the "no active drag" condition. The triple-guard is pinned by the conjunction test in the new test file.
    - **Why dashed `strokeDasharray="3 3"` at 0.45 opacity rather than solid or higher-contrast.** The cross-hair is a HINT, not a DIVIDER. A solid stroke at full opacity (e.g., `hsl(var(--foreground))` at 1.0) would over-assert against the data lines underneath — the data is the figure, the indicator is the ground. The chosen style (muted-foreground at 45% opacity, dashed pattern matching the brush rect's own dash style for stylistic consistency, 1px width — the thinnest visible) is the smallest visual footprint that still reads as "this is the hovered x". The dashed-not-solid invariant is the visual contract pin in the test file; the exact opacity / color / width values are intentionally not pinned so future visual-design tweaks can adjust without test churn.
    - **Why placed UNDER the data lines (rendered earlier in the Group) rather than as the topmost overlay.** Standard financial-chart layering: the cross-hair is a passive indicator the user reads at a glance; the data is the figure they study. Putting the indicator above the lines would visually break each line where the cross-hair crossed it, creating a 1px gap at the hovered x on every series — distracting. Placing it under the lines means the cross-hair only shows in the gaps BETWEEN series strokes (and above the grid background) — exactly where the eye expects a hint, never where the eye is reading data. This is the inverse of the brush rectangle's layering choice (the brush rect sits below the lines too, for the same reason — the user wants to see the data through the brush region, not behind it).
    - **Why no echarts-side cross-hair config in this wave.** The Explore-subagent's report noted echarts has native `axisPointer.type: 'cross'` available with ~3-5 LOC per renderer, no new state needed. The 8 visx trend renderers ARE the right targets; LineRenderer is the natural first-instance for the pattern. Future waves can mirror this exact shape onto AreaRenderer / BarRenderer / ComboRenderer / PointRenderer / etc., one at a time.   **Why this design.**
    - **Why thread `sheetId` end-to-end as an opt-in additive field rather than tighten the resolver into an always-scoped shape.** Three reasons. (i) **Shareable URLs:** the dashboard drill-through is accessible via a shareable URL that carries `chartId` + `column` + `value` but not `sheetId` — users who shared the link before this wave shipped, or users who built their own GET URLs against the documented endpoint, would suddenly see `chart_not_found` if scoped lookup became mandatory. Backwards-compat is the load-bearing reason for the legacy-walk fallback. (ii) **Single-sheet dashboards:** the majority of real dashboards in production have one sheet; scoping is a no-op there (the legacy walk hits the only sheet first), so requiring `sheetId` everywhere would have been pure ceremony with no observability benefit on the dominant case. (iii) **Renderer surface:** the renderers that dispatch `DrillThroughEvent` don't know which sheet they're rendered into — they read `dashboardTile.tileId` from React context, not `sheetId`. Plumbing `sheetId` through the 15-renderer surface (visx + ECharts) would have meant 15+ file touches and a context-widening of `DashboardTileProvider` to carry sheet identity. Instead, injecting at the DashboardView listener layer (where `activeSheetId` is already in scope from the prior telemetry wave's deps widening) is a single 14-LOC delta that captures the same intent. The listener is the canonical "received a click, decide what to do" layer in the dispatch architecture; sheet-context injection naturally lives there.
    - **Why the predictable-failure invariant on stale sheetId (return null, NOT fall back to the legacy walk).** A `sheetId` that doesn't match any sheet in the dashboard could happen two ways: (a) the user shared a URL after deleting the sheet — the linked sheet is gone, the chart they shared no longer exists in any meaningful sense, and silently resolving to a different sheet's `chart-0` would render rows for a chart the user never intended; (b) a hand-crafted URL with a wrong `sheetId` value — pure user error, and silent fallback would obscure the bug. In both cases, returning null (which the resolver maps to `chart_not_found` → controller maps to 404) is the correct behaviour because it surfaces "this chart doesn't exist anymore" to the user / caller. The legacy-walk fallback is reserved strictly for `sheetId === undefined` (the explicit "no scoping requested" signal); any provided-but-invalid `sheetId` is treated as a hard miss. The source-inspection test in the new test file pins this branch shape via an explicit negative-match check — a future "be helpful" edit that adds a fallback inside the `if (sheetId !== undefined)` branch would fail the regex check.
    - **Why inject at click time on the event detail, NOT at render time as a separate prop.** Two alternatives were considered. **Alternative A (chosen):** capture `activeSheetId` at click time, inject onto the event detail via the listener's `setDrillThroughEvent(activeSheetId ? { ...detail, sheetId: activeSheetId } : detail)`, propagate through the hook's `event.sheetId`. **Alternative B:** pass `activeSheetId` as a separate prop to `<DrillThroughSheet>` → `useDrillThroughRows({ dashboardId, event, sheetId: activeSheetId })`. Alternative A wins because the Radix Sheet is modal (it overlays the dashboard with a backdrop), so user navigation between sheets while the panel is open requires closing first — but defensive-design considerations still matter: if the user opens drill-through panel on Sheet 1, and somehow the dashboard re-renders with a different `activeSheetId` (e.g. a re-mount triggered by a parent state change, an external navigation event, a future feature that allows sheet-switching from inside the panel), Alternative B would silently re-resolve against the new sheet's chart-0 (which is a DIFFERENT chart). Alternative A captures the user's intent at the moment of the click — the resolution context is frozen until the panel closes and clears the event. The DRILL_THROUGH_EVENT contract becomes "this event represents what the user clicked on, including the sheet they were on when they clicked"; resolution is a pure function of the event payload. This is also why we kept the conditional-spread shape (`activeSheetId ? { ...detail, sheetId } : detail`) instead of unconditional injection — the null-activeSheetId branch (degenerate first-mount case before any sheets exist) returns the pre-wave shape verbatim, so the test regex's negative-pin against unconditional injection holds.   **Conventions added.** None codified outright. Two soft patterns shipped with one instance each (would promote on a second instance):
    1. **"Predictable-failure on optional disambiguator"** — when an optional disambiguating field is provided but doesn't match any known entity (here: `sheetId` provided but no sheet has that id), return null / 404 rather than silently falling back to the legacy unscoped path. The fallback is reserved for the `undefined` case (the explicit "no scoping requested" signal). The motivation is that a hand-crafted-but-wrong identifier represents a real bug somewhere upstream (stale share-link, user typo, malformed client request), and silent fallback obscures it. The convention would generalise to any future optional disambiguator on any endpoint — e.g. if a future wave adds optional `versionId` to a versioned-entity fetch endpoint, the same predictable-failure invariant applies.
    2. **"Click-time context capture on event detail, NOT render-time prop"** — when a UI surface receives a captured event and needs to resolve against a context value that could drift (active sheet, active mode, active filter snapshot), the listener captures the context AT CLICK TIME by spreading it onto the event detail before storing in state. The receiving panel then resolves against the captured value, NOT against the current state's value. Frozen-at-click-time mental model: the event represents what the user clicked on, including all relevant context, and the panel renders that intent regardless of subsequent state changes. First instance is the prior WD3-WI4-sheetId-telemetry wave (which also captured `activeSheetId` at listener time for the telemetry payload); this wave is the same shape but on the resolution side (the captured context now drives server-side resolution, not just observability). Second-instance promotion would happen on a future event-resolution wave that captures a different context value (e.g. capturing the active filter snapshot at brush time, capturing the brush mode discriminant at dispatch time).

  **Why this design.**
    - **Additive `sheetId` chosen over widening chartId to `chart-<sheetId>-N`.** The widening shape is conceptually cleaner — single field carries full chart identity — but the cost analysis runs against it on three axes. (a) **Test surface:** widening breaks every existing test that pins literal `"chart-N"` values, including `dashboardDrillThroughWD3Server.test.ts` lines 260-296 (the multi-sheet lookup tests that already encode the collision as expected behaviour), `dashboardDrillThroughTelemetryWD3Telemetry.test.ts` controller cases at lines 64-218, the equivalent block in `dashboardExplainSliceTelemetryWI4Telemetry.test.ts`, and the `dashboardTemplates.test.ts` literal pins. The Explore subagent that mapped the surface for this wave estimated ~30 test-line updates across those files. (b) **Cosmos historical rows:** widening creates a mixed-format aggregation surface — pre-wave rows have `chartId: "chart-0"`, post-wave rows have `chartId: "chart-<sheetId>-0"`. Any analytics query that tries to span the wave boundary has to parse both formats or back-fill the old rows by walking every dashboard's then-current sheet structure (which itself drifts as users edit dashboards). (c) **Renderer / server resolution churn:** widening propagates through every renderer that emits `dashboardTile.tileId` (15 visx + ECharts renderers per the Explore report), the server's `findChartByTileId` regex parser, and the `ExplainSlicePanel` chart resolution at DashboardView lines 1144-1150 — all of which would need to learn the new format. **Additive `sheetId` gives consumers the same (dashboardId, sheetId, chartId) tuple for disambiguation while preserving chartId's stable per-sheet local identity** — no test rewrites, no Cosmos format break, no renderer changes, no server-resolution churn. The chartId remains a local identifier within a sheet; the sheetId disambiguates across sheets. That's exactly what observability needs and nothing more.
    - **Single consolidated test file vs per-endpoint files.** The two prior waves shipped per-endpoint server test files (`dashboardDrillThroughTelemetryWD3Telemetry.test.ts` + `dashboardExplainSliceTelemetryWI4Telemetry.test.ts`) because each wave introduced a NEW endpoint and the test file mirrored the route file 1:1. This wave widens BOTH existing endpoints in lockstep; consolidating the new tests into a single `dashboardWdWiSheetIdTelemetry.test.ts` puts the doublet-drift guards (the two source-inspection tests that pin both endpoints had the same widening) in one place where they're maximally legible. A future single-endpoint wave on either route would still get its own per-endpoint file; this consolidation is specifically about the doublet-widening shape.
    - **Explicit-omit invariant (`...(sheetId ? { sheetId } : {})` rather than `sheetId,`).** Setting `metadata: { sheetId, ... }` when `sheetId === undefined` would write `metadata.sheetId === undefined` to Cosmos. JSON serialisation drops the key — but the `recordUsageEvent` writer might or might not pass through the SDK's serialiser at every layer; the conditional spread guarantees the key is omitted at the object-construction layer, before any serialisation step. Cosmos distinguishes `{}` from `{sheetId: null}` on the wire; the conditional spread also pre-empts a future refactor where a callsite accidentally passes `sheetId: null` (which would survive a `?? undefined` collapse but fail the `.optional()` parse — except the parse runs BEFORE the controller, so the recorder would see a validated-undefined that then propagates to Cosmos as a present-undefined). The source-inspection test pins the spread pattern at both controller sites so a refactor can't silently regress.
    - **Both effects' deps arrays widen to `[dashboard.id, activeSheetId]`.** The listener handler closes over `activeSheetId` for the telemetry payload; ESLint exhaustive-deps would flag a missed dep, but more importantly the listener needs to re-bind when `activeSheetId` changes so subsequent event firings capture the new value. The re-bind is cheap (`removeEventListener` + `addEventListener` on `window`) and runs once per sheet switch; on a multi-sheet dashboard the user-perceived cost is zero. Without the dep, the listener would close over the initial `activeSheetId` (typically `null` on first mount) and every subsequent click on a sheet other than the initial active sheet would record telemetry with `sheetId: undefined` even though the user is clearly on a different sheet — which would silently revert the entire observability benefit of this wave.
    - **Why third instance doesn't promote a fourth convention.** Last session's WD3-telemetry → WI4-telemetry doublet codified three conventions on the second-instance promotion threshold (route-level-recorder-seam, pii-safe-discriminant-tag-on-observability-events, strict-zod-on-observability-endpoints). This wave is the third instance of all three patterns (the recorder seam is shared, the sheetId field is a safe identifier with no PII concern, the strict zod stays strict). No new pattern emerged that hasn't been seen before — this is a pure additive widening on top of established conventions. The "additive optional field on a strict observability endpoint" shape is itself a soft pattern with one instance now; a future wave that adds a different optional field to an observability endpoint would promote it.

  **Why this design.**
    - **Why co-locate the WI4 route handler in the same `telemetry.ts` rather than ship a sibling `telemetryExplainSlice.ts` module.** Three reasons: (i) the route-level recorder seam (`recorder` mutable local + `__set...ForTesting` / `__reset...ForTesting` setters) is now shared between both controllers, which is the second-instance promotion that codifies the pattern as a documented convention — a sibling module would have duplicated the seam (DRY violation) OR exported it from one module and imported into the other (load-order coupling + import-cycle risk between sibling route files); (ii) both endpoints serve the dashboard-click-intent telemetry family conceptually, so the module-level docstring + named JSDoc block can describe both in one place; (iii) future dashboard click-intent telemetry events (WD2 cross-filter sample, dashboard-exit, etc.) will land in the same module on the same precedent. Co-location avoids the "one route module per endpoint" anti-pattern that would have shipped 4+ telemetry modules within a few waves.
    - **Why `regionKind` as a strict z.enum() rather than a free-form `string.min(1)` like WD3's `valueType`.** The WD3 `valueType` field carries the JS `typeof` tag, which has 8 well-defined possible values but is conceptually open (a future JS spec extension could add a ninth token); free-form `string.min(1)` is the right shape for an open-but-strongly-typed token. The WI4 `regionKind` field carries the `BrushRegion` discriminant from [`explainSlice.ts`](../client/src/pages/Dashboard/lib/explainSlice.ts), which is closed by the renderer-side type union — there are exactly four valid values and any other token represents a renderer-side wiring bug or an upstream injection attempt. `z.enum(["numeric", "temporal", "categorical", "box2d"])` is the load-bearing defense: a malformed renderer can't smuggle non-discriminant tokens through, and the schema-layer rejection catches the bug before it hits Cosmos. This is also a cleaner observability story — Cosmos aggregation queries on `regionKind` can use exact-string comparison rather than enum-validation-after-the-fact.
    - **Why fire BETWEEN validation and `setExplainSliceEvent(detail)`, not BEFORE validation.** Same answer as WD3-telemetry's matching question: invalid events represent renderer bugs (or stale CustomEvents from prior mounts), not successful slice-explanations; firing before validation would skew the metrics. The validation gate is the canonical "did the user successfully brush a slice" boundary, and the telemetry fire belongs strictly on the success side of that gate. The 7-line inline comment block (a touch longer than the matching WD3 block because it also documents the discriminant-vs-typeof distinction) preserves the rationale at the call site.
    - **Why widen the existing `client/src/lib/telemetry.test.ts` rather than create a sibling `telemetry.explainSlice.test.ts`.** The helper module is the same (`client/src/lib/telemetry.ts` exports both helpers); the test file should mirror the helper module shape, so widening in-place preserves the single-module-single-test-file pattern. A sibling test file would have meant a new `server/package.json` entry, more files to traverse, and would have anchored the false intuition that "every wave needs a new file." The "WI4 · " naming-prefix gives the visual separation in test-runner output without splitting the file.
    - **Why the four `regionKind` values are spread across the test suite rather than enumerated in one test.** Enumerating all four in a single test would have hidden the value-specific behavior under a `.forEach(...)` — the regression footprint would be one fail-N-pass count, but the bisect signal would be muddier. Spreading the four values across distinct named tests (one each: `numeric` in canonical POST + missing-column case; `temporal` in omits-dashboardId; `categorical` in swallow-rejection; `box2d` in SSR-safe + omits-dashboardId-controller) means a future per-value regression (e.g. a `box2d`-specific server-side bug after a renderer change) bisects to its specific case directly. Suite 1's empty/missing tests don't need to cover all four — they're testing the schema's required-fields behavior, not the enum-value behavior; suite 1's two enum-validation tests (one valid, one invalid) lock the schema-layer contract.

  **Why this design.**
    - **Why a dedicated client→server endpoint rather than firing the recordUsageEvent inline from an existing server route.** The drill-through is a client-side action: the renderer dispatches a window CustomEvent, the DashboardView listener captures it, and the side-sheet opens. There's no server-side touchpoint in the existing flow (the sheet's row fetch happens lazily via the WD3-server-fetch endpoint when the sheet body mounts — but that endpoint is the data fetch, not the observability hook; conflating "user drilled into chart X" with "client successfully fetched rows for chart X" would obscure two different signals). Adding a dedicated telemetry endpoint keeps the observability concern orthogonal: fires once per validated event regardless of whether the data fetch succeeds; fires from the listener handler, not from React rendering, so re-renders don't double-count; and the endpoint stays single-purpose (one event type per route, easy to extend per-event with custom validation if the schema needs widening).
    - **Why the four-field payload shape (chartId / column / valueType / dashboardId?) instead of a richer envelope.** Each field is load-bearing for its observability question: `chartId` answers "which chart got drilled into" (per-tile heatmap of clicks); `column` answers "which dataset column was the drill axis" (cross-dashboard heatmap of which columns are most queried); `valueType` answers "what was the shape of the clicked mark's value" without sending the value itself (a `typeof` tag, never the raw string/number — load-bearing PII contract); `dashboardId` answers "across which dashboards" (allows per-dashboard aggregation). Fields deliberately NOT included: the raw `value` (PII risk — column values can be user names, customer IDs, sales figures, etc.; sending only `typeof` preserves the observability signal without the leak); `sourceTileId` (the click came from a tile but the tile id is a UI artefact, not a stable observability key — chart id is the durable identifier); `filters` snapshot (the active filter context at click time is interesting but adds payload bytes for a question we can answer from the chart spec + the dashboard config aggregate; deferred to a follow-on wave if the question surfaces); `extraPins[]` (RectRenderer heatmaps fire multi-pin events; the primary pin's column already carries the observability signal, and `extraPins[]` would balloon payload size for a marginal question; defer).
    - **Why `typeof detail.value` as the canonical PII-safe value-shape token.** The JS `typeof` operator returns one of `"string" | "number" | "boolean" | "object" | "undefined" | "symbol" | "function" | "bigint"` — six of those would actually appear in drill-through values (`symbol` / `function` / `bigint` never reach the client from the server). Each tells a story: `"string"` = categorical column (region, product name), `"number"` = numeric column (sales, units, week index), `"boolean"` = flag column (active, on-promo), `"object"` = a Date object (the renderer dispatches Date instances for temporal columns), `"undefined"` = a null/missing slot the user clicked through. Aggregating by `valueType` across many drill-through events reveals the distribution of click intent without leaking any individual click's value. Alternatives considered and rejected: (a) coerce via the server's existing `toFilterValue` canonicaliser and send the bucket name — adds runtime dependency on the server-side canonical-form module on the client, breaks the W61-cross-runtime-boundary local-mirror convention, no observability benefit; (b) hash the raw value via SHA-256 and send the prefix — uniqueness preserved for aggregation but the hash leaks information (two same-value clicks emit the same hash; a small-cardinality column reveals its value distribution); (c) send a length-bucketed string (`"short" | "medium" | "long"`) — useless for numeric columns and confuses temporal observability. `typeof` is the smallest informative token.
    - **Why fire BETWEEN validation and `setDrillThroughEvent(detail)`, not BEFORE validation.** Firing before validation would mean an invalid event (chartId not a string, column missing, or both) still emits telemetry. Two reasons that's wrong: (a) the validation is the gate that decides whether the user successfully drilled — an invalid event represents a misbehaving renderer (or a stale CustomEvent fired by a previous mount), and counting it as a drill-through skews the metrics; (b) the telemetry payload requires `chartId: string` + `column: string` — if validation fails, those guarantees don't hold, and the type-checker would either reject the call or force a runtime-cast that could send `chartId: undefined` to the server (which the server's zod min(1) check would 400). Firing after validation, before setState, means: only successful drill-throughs are counted; both fields are guaranteed non-empty strings; and the telemetry fires even if the sheet mount fails downstream (the click intent was successful — sheet-mount failure is a separate failure mode).
    - **Why a route-level test seam (`__setUsageEventRecorderForTesting`) rather than a model-level one.** The model-level mock would have to replace `recordUsageEvent` on `usageEventModel.recordUsageEvent` — but Node's ESM module bindings are immutable read-only references from the importer's perspective, so reassigning the named export from outside the module fails with `TypeError: Cannot assign to read only property`. The route-level seam works because the route module imports `recordUsageEvent` once at module load, then captures it in the mutable `let recorder` local; tests can swap the local without violating ESM semantics. The pattern generalises to "inject the model writer on the route layer instead of the model layer" — first instance of this; the W61 admin-controller-setter precedent is the closest analogue but those setters live on controller exports rather than route exports. Would promote to a documented convention on a second instance.
    - **Why `credentials: "include"` rather than `"same-origin"`.** The dashboard view runs on the SPA at port 3000 in dev, but POSTs to the API at port 3002 — different origins. `"same-origin"` would fail to attach auth cookies in dev; `"include"` works in both dev (cross-origin via CORS) and prod (same origin behind the load-balancer). The server's CORS config already allows credentials from the SPA origin per the existing route surface. No security widening — the endpoint is auth-gated, so missing cookies → 401, not a silent telemetry leak.
    - **Why no integration test against DashboardView.** The listener wiring is a 7-line delta inside an existing `useEffect`; testing it would require mounting `<DashboardView>` with a mock `DashboardData` prop, dispatching a fake `CustomEvent`, mocking `fetch`, and asserting on the recorded body. That's ~150 LOC of jsdom-mount plumbing for marginal additional coverage given that (a) the client helper is unit-tested in isolation with 7 tests pinning the fetch contract, (b) the server route is unit-tested in isolation with 15 tests pinning the body shape + auth + recording, and (c) the listener's previous behavior is covered by the existing `drillThrough.test.ts` + `wd3WiringEcharts.test.ts` suite (which would catch a regression in the sheet-open path). The 11-LOC listener delta is small enough that source review + the two isolated test suites give adequate coverage; a future regression in the wiring would surface as "telemetry never fires in prod" which the Cosmos aggregator would catch within hours.
