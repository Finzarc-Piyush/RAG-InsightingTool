- **2026-05-20** — **Wave WI5 · per-tile "Try this" recommendation chips under the regen prose.** Closes Workstream 7 to 5 of 6 — the recommendations infra was already partially built in [`AnalysisSummaryPanel.tsx`](../client/src/pages/Dashboard/Components/AnalysisSummaryPanel.tsx) at the *whole-analysis* scope (LLM-generated, grouped by horizon: now / this-quarter / strategic). WI5 ships the per-tile counterpart: lightweight, deterministic, single-click CTAs derived synchronously from the tile's current spec + filtered rows + active filters. No LLM call, no server round-trip — the rules are simple enough to run on every render. The user reported the feature as "When the tile's data shows an actionable finding (bottom-quartile region, anomalous spike), surface a 'Try this' CTA right under the insight footer"; this wave is the first deterministic rule set scoping recommendations down to "this tile" rather than "this analysis".

  **What landed.**
    - New pure module [`client/src/pages/Dashboard/lib/tileRecommendations.ts`](../client/src/pages/Dashboard/lib/tileRecommendations.ts). Exports a `TileRecommendation` discriminated union with three kinds — `filter-bottom`, `filter-top`, `clear-filters` — and a pure `deriveTileRecommendations(spec, rows, activeFilters): TileRecommendation[]` function that returns up to `MAX_TILE_RECOMMENDATIONS = 3` recs. Pinned thresholds: `FILTER_BOTTOM_RATIO = 0.5` ("less than half the median"), `FILTER_TOP_RATIO = 2.0` ("at least twice the median"). Helper `aggregateByX(rows, xField, yField)` groups rows by their x-value and sums y per bucket; `median(values)` returns the median of a numeric array (mutation-free). `isValueAlreadyFiltered(activeFilters, column, value)` checks whether the column has an existing categorical filter pinning the value — numeric / date filters on the same column do NOT count (they narrow ranges without pinning a specific categorical value). The `TileRecommendationSpec` interface is the minimum read-surface (`type, x, y, aggregate?, seriesColumn?`) — deliberately not re-exporting `InsightChartSpecLite` from `useInsightRegen` so a tile with no regen wiring can still surface recommendations. `TileRecommendationRow = Record<string, string | number | boolean | null>` is the strict subset of `InsightRegenRow` the aggregator consumes.
    - **Three deterministic rules, in source order**:
      - **filter-bottom** fires when the lowest-aggregated x-bucket's total is `< median × FILTER_BOTTOM_RATIO`. Surface: `"Focus lowest: <value>"`. CTA: clicking pins a single-value categorical filter on the column (`{ type: "categorical", values: [value] }`).
      - **filter-top** is the mirror rule: highest bucket's total `> median × FILTER_TOP_RATIO`. Surface: `"Focus highest: <value>"`.
      - **clear-filters** fires when at least one entry in `activeFilters` is non-undefined. Always slotted LAST so value-driven recs render first. Surface: `"Clear filters"`. CTA: `onFiltersChange({})`.
    - **Chart-type gating**. Both filter rules only fire for `bar` and `pie` types — these are the chart kinds with naturally categorical x-encodings where each value maps to one mark / slice. `line` / `area` / `scatter` / `heatmap` have continuous or 2D x-encodings where "filter to the bottom bucket" doesn't have a clean meaning, so they short-circuit. Test pins all six chart types.
    - **Minimum-bucket guard**. Filter rules require `≥3 distinct x buckets` (a median over fewer datapoints is meaningless). With 2 buckets the rule short-circuits — pinned in tests.
    - **Aggregator hygiene**. Rows with null / undefined / empty-string x are dropped (`continue` at the top of the loop). Rows with null / undefined / empty-string y are ALSO dropped BEFORE the `Number()` coercion — this is load-bearing: without the y-drop, `Number(null) === 0` and `Number("") === 0` would silently create phantom buckets at total = 0, shifting the median down and firing spurious filter-bottom recs. Caught during test development when the "skips rows whose y is non-numeric" test asserted on F (a null-y row) and the rule fired on F instead of the intended A. Matches the null-drop convention in `filterRowsByBrushRegion` in `explainSlice.ts`. Non-finite y (NaN, ±Infinity from `Number("not-a-number")`) is also dropped.
    - **Recommendation id shape**. Each rec carries a stable `id` encoding `kind + column + value` (`filter-bottom:region:A` / `filter-top:region:D` / `clear-filters`). Used as the React `key`; uniqueness verified in tests (two recs on the same tile never collide).
    - [`TileInsightFooter.tsx`](../client/src/pages/Dashboard/Components/TileInsightFooter.tsx) gains two new optional props — `recommendations?: TileRecommendation[]` and `onRecommendationClick?: (rec: TileRecommendation) => void`. Default-off: when omitted / empty, the footer renders byte-identically to the pre-WI5 shape. New "Try this:" chip row inserted between the Sources row and the action row (so the section order is: prose → metadata → sources → try-this → actions). Each chip is a `Button variant="outline" size="sm" h-6 px-2 text-[11px]` with `aria-label={rec.label}` for screen readers; `e.stopPropagation()` on the click prevents the parent footer chevron from toggling the collapse state. Visual row style is `mt-1 flex flex-wrap items-center gap-1 text-[11px] text-muted-foreground` — parallels the WI3 Sources row exactly so a future visual tweak can land in both rows together.
    - [`ChartTileBody.tsx`](../client/src/pages/Dashboard/Components/ChartTileBody.tsx) imports `deriveTileRecommendations` + `TileRecommendation`, computes recommendations via `useMemo(() => deriveTileRecommendations(specLite, filteredRows, filters ?? {}), [specLite, filteredRows, filters])` — same memoisation dependencies as the chart itself reads, so chip changes are pinned to genuine state shifts. `handleRecommendationClick` is a `useCallback` over `[filters, onFiltersChange]` that routes each kind: `filter-bottom` / `filter-top` clobbers any prior categorical filter on the column with `{ ...filters, [rec.column]: { type: "categorical", values: [rec.value] } }` (safe because the derive function's `isValueAlreadyFiltered` guard already filters out no-op pins); `clear-filters` calls `onFiltersChange({})`. Passed to `<TileInsightFooter>` alongside the existing regen surface.
    - 30 new behavioural tests across 12 suites in [`tileRecommendations.test.ts`](../client/src/pages/Dashboard/lib/tileRecommendations.test.ts). Real import + runtime assertions, NOT source-inspection — the helper has no React / DOM dependency, so the tests exercise it directly (same convention as [`explainSlice.test.ts`](../client/src/pages/Dashboard/lib/explainSlice.test.ts) and [`drillThrough.test.ts`](../client/src/pages/Dashboard/lib/drillThrough.test.ts)). Coverage:
      - Constants pinned (`MAX_TILE_RECOMMENDATIONS = 3`, `FILTER_BOTTOM_RATIO = 0.5`, `FILTER_TOP_RATIO = 2.0`) so renames break loudly.
      - Empty rows + balanced rows → empty list.
      - Asymmetric distributions: bottom-only fires `filter-bottom`; top-only fires `filter-top`; both → both.
      - Threshold negative cases: bottom within `median × 0.5` doesn't fire; top within `median × 2.0` doesn't fire.
      - Each chart type tested independently — `bar` + `pie` fire; `line` + `area` + `scatter` + `heatmap` don't.
      - `< 3` bucket short-circuit + "all rows share same x" edge case.
      - Already-filtered values are NOT re-surfaced (categorical filter pins them); numeric / date filters on the same column do NOT count as pinning (a `numeric` filter on `region` doesn't block `filter-bottom` from firing on a `region` value).
      - `clear-filters` appears when any filter is non-undefined; doesn't appear when activeFilters is `{}` or has only `undefined` values; always slotted last.
      - MAX cap respected at the slice boundary.
      - Aggregator hygiene: null / empty-string x rows skipped; null / NaN / non-numeric y rows skipped; duplicate x values summed (matches sum-aggregator semantics).
      - Rec ids encode `kind + column + value` so the React `key` never collides.
    - Test file appended to [`server/package.json`](../server/package.json)'s explicit `test` list per invariant #4 (right after `wi4WiringPoint.test.ts`).

  **Why this design.**
    - **Deterministic rules in the client, not LLM-generated.** Considered: extend `InsightRegenEntry` with a `recommendations` field and have the server `/api/insight/regen` endpoint emit "Try this" CTAs alongside the regen prose. Rejected — the LLM-generated recommendations live at the whole-analysis scope already (`DashboardAnswerEnvelope.recommendations`); duplicating that surface per-tile would either spam the dashboard or require careful deduplication. The user brief framed WI5 as deterministic CTAs ("filter to bottom quartile" / "anomalous spike"), which maps cleanly to a synchronous client function with no server round-trip, no LLM latency, no token cost. The chip semantics are mechanical (apply a filter), not advisory, so prose-style rationale isn't needed.
    - **Three rules now, not seven.** Considered shipping more rules in the first cut: outlier-focus (z-score > 2 for scatter), brush-to-peak (spike > 2x median for trend), drill-into-bucket (categorical → drill-through), compare-to-baseline (always-on toggle). Rejected — keeping the rule set small in the opener wave preserves the budget for behavioural-test coverage per rule and avoids combinatorial explosion in test setup. Each future rule joins the `TileRecommendation` discriminated union without breaking existing call sites — the consumer's switch on `rec.kind` is the only place that needs new code.
    - **Recommendation derivation in the parent, not inside the footer.** Considered: have `TileInsightFooter` accept the raw spec + rows + filters and derive recommendations itself. Rejected — the footer's pre-WI5 interface is `insight: string` + ids + regen; widening it to know about the chart spec would create a coupling between the footer (a presentational component) and the chart-data pipeline (the parent's concern). Deriving in `ChartTileBody` keeps the footer purely presentational and lets the parent decide WHEN to re-derive (the `useMemo` dependency array).
    - **Median anchor, not absolute thresholds.** Considered: fire `filter-bottom` when a bucket is < some absolute floor (10 units? 100 units?). Rejected — the dataset's scale isn't known in advance (sales in lakhs vs revenue in crores vs share-points in %). A median-anchored ratio is scale-invariant: a "much smaller than the typical bucket" judgment works at any scale. The 0.5 / 2.0 ratios were chosen as loud-enough thresholds that don't fire on mild dips/spikes (an 80/100 bucket pair has a 0.8 ratio, well above the 0.5 floor).
    - **Already-filtered-value guard inside the helper, not at click time.** Considered: derive all rule-matching recs in the helper, then have the consumer's click handler check "is this value already pinned?" and skip. Rejected — the user would see "Focus lowest: A" appear in the chip row even when A is already pinned; clicking would be a no-op, hostile UX. The guard inside `deriveTileRecommendations` ensures the chip never appears when the click would be a no-op.
    - **Sum semantics for the aggregator, not mean / count.** Considered using the chart spec's declared `aggregate` field. Rejected — three of the five chart spec aggregators (`sum`, `mean`, `count`) produce different scales but preserve relative ordering of buckets, and the rule is median-anchored (relative). Sticking to `sum` regardless of the declared aggregator keeps the helper simple AND aligned with what bar / pie tiles render by default. A future polish wave could read `spec.aggregate` if a counterexample appears.
    - **Tests as behavioural, not source-inspection.** The WI4 family used source-inspection tests because each renderer wave wires the helper into a 1500-LOC ChartRenderer-adjacent file that's expensive to behaviourally test (it depends on visx + React + browser APIs). The `tileRecommendations` module is a pure function with no dependencies, so behavioural tests are strictly better: they catch real logic bugs, not just regex drift. Both styles are valid — pick the one that matches the unit under test.
    - **`text-[11px]` chip size matches the Sources row, not the Re-explain button.** Considered using the WI2 footer button's shape (`size="sm" h-7 text-[11px]`). Rejected — the Re-explain button is a single-action CTA per footer (one button max), so it deserves a slightly larger affordance. The Try-this chips can be up to 3 per footer; `h-6` (one row pixel shorter than `h-7`) keeps the chip row visually distinct from the action row and prevents the footer from feeling button-heavy.

  **Conventions reinforced.** The "pure-function helper module + behavioural tests" pattern shipped earlier in WI4-foundation, WD2-foundation, WD3-foundation now has a fourth instance — `tileRecommendations.ts` follows the same shape (no React import, types + constants + pure functions, ESM `.js` import paths, test file is `*.test.ts` colocated with the source). The "consumer derives via `useMemo`, click-handler routes by kind via `useCallback`" pattern in `ChartTileBody` parallels the WI4 panel's `useInsightRegen` integration shape.

  **Out of scope.**
    - **Server-side / LLM-generated tile recommendations.** The deterministic rules cover the majority of "obvious next click" cases; an LLM-grade recommendation surface would be a separate workstream that consumes the regen response. The `TileRecommendation` discriminated union is intentionally extensible — a future `kind: "narrative"` variant carrying server-generated prose could land without changing the helper signature.
    - **Outlier / anomaly rules for scatter + trend.** A z-score-based outlier-focus for scatter and a 2x-median anomaly-brush for trend are the natural follow-on rules. Each is a separate rule with its own threshold constant + test suite; deferred to a WI5-followon wave if the user requests breadth over depth.
    - **Recommendation analytics / dedup.** No telemetry on which chips are clicked vs ignored; no per-session dedup (a user who repeatedly opens the same dashboard sees the same rec set every time). A future polish wave could add `analytics?: { fired_at: ISOString; clicked_at?: ISOString }` to the rec shape — `id` is already stable enough to be an analytics primary key.
    - **Multi-column / multi-value recs.** Each rec currently pins ONE column / ONE value. A "Focus top-3" rec that pins the three highest values is a small extension (`values: string[]`), deferred until a use case appears.
    - **Animation on chip apply.** Clicking a chip applies the filter mutation immediately; the chip animates only via the standard Button hover state. A future polish could add a fade-out on the clicked chip + a fade-in on the resulting filter chip in the dashboard filter bar.
    - **Custom rec ordering / priority.** Recs append in source order (filter-bottom, then filter-top, then clear-filters). A future polish could surface a score per rec (e.g. how far the bucket is from the median in terms of standard deviations) and sort by descending score so the most-actionable recs always come first.

- **2026-05-20** — **Wave WI4-wiring-point · PointRenderer 2D brush mechanics + box2d explain dispatch.** Closes the WI4 widening series at 4 of 4 brush-capable kinds (Line + Area + Bar + Point) — alt-drag on any chart inside a dashboard tile now opens the ExplainSlicePanel with regenerated insight prose for the brushed sub-region, scatter included. PointRenderer was the last hold-out: it had NO pre-existing brush mechanics — only per-point onClick handlers powering WD2 cross-filter + WD3 drill-through, both gated on `colorCh` — so this wave adds mouse-down / move / up state + a 2D rectangular overlay from scratch. Scatter is the only chart kind with TWO independently-continuous axes, so the brush captures coordinates on both dimensions; the WI4-foundation-box2d wave shipped earlier today provided the load-bearing `box2d` region variant + `makeBox2dRegion` constructor + `b2:` cache-hash branch that make this wave a clean call-site swap.

  **What landed.**
    - [`client/src/lib/charts/visxRenderers/PointRenderer.tsx`](../client/src/lib/charts/visxRenderers/PointRenderer.tsx) imports the four WI4-foundation helpers from `@/pages/Dashboard/lib/explainSlice`: `BRUSH_MIN_PX`, `dispatchExplainSlice`, `isBrushDrag`, `makeBox2dRegion`. The three 1D constructors (`makeNumericRegion` / `makeTemporalRegion` / `makeCategoricalRegion`) are DELIBERATELY NOT imported — scatter is box2d-only, and a negative pin in the tests catches a future drift that would downgrade the brush to 1D. React import widens from `useMemo, useRef` to `useEffect, useMemo, useRef, useState` for the brush state + ref + data-reset effect.
    - **Two new useState hooks for 2D pixel coords**: `brushStart` and `brushEnd` as `{ x: number; y: number } | null`. Distinct from the 1D brushes (Line / Area / Bar) whose `brushStart` is a bare `number` — the shape carries `x` AND `y` because the scatter brush is a 2D rectangle. `brushExplainRef = useRef<boolean>(false)` captures `e.altKey === true` at brushDown — ref (not state) because the alt flag doesn't drive a re-render.
    - `useEffect(() => { setBrushStart(null); setBrushEnd(null); }, [data])` resets brush state when the underlying data changes — same shape as the 1D wirings.
    - **Three new handlers gated on `dashboardTile`**:
      - `onBrushDown` reads `localPoint(e)`, computes `(x, y)` in inner-plot space (`pt.x - MARGIN.left` and `pt.y - MARGIN.top`), guards BOTH axes' bounds (`x < 0 || x > innerWidth` AND `y < 0 || y > innerHeight`), captures `e.altKey === true` into `brushExplainRef.current`, and sets `brushStart` AND `brushEnd` to the same `{x, y}` pair.
      - `onBrushMove` short-circuits on `brushStart === null` and on `!(e.buttons & 1)` (no held button) — same gating shape as the 1D brushes' move handlers. Reads `localPoint(e)`, clamps `pt.x - MARGIN.left` and `pt.y - MARGIN.top` independently to `[0, innerWidth]` and `[0, innerHeight]` via `Math.max(0, Math.min(...))` on each axis, updates `brushEnd`.
      - `onBrushUp`'s click-vs-drag gate requires **BOTH axes** to cross `BRUSH_MIN_PX`: `isBrushDrag(brushStart.x, brushEnd.x, BRUSH_MIN_PX) && isBrushDrag(brushStart.y, brushEnd.y, BRUSH_MIN_PX)`. A 100×3 sliver is still a click in the y-dimension; a 3×100 sliver is a click in the x-dimension. Click branch resets state and yields to the per-point onClick (when the cursor lands on a point). Drag branch gates on `brushExplainRef.current && dashboardTile`, inverts pixel coords via `xScale.invert(brushStart.x)` / `xScale.invert(brushEnd.x)` / `yScale.invert(brushStart.y)` / `yScale.invert(brushEnd.y)`, and builds the region via `makeBox2dRegion(x1Data, x2Data, y1Data, y2Data, yCh.field)`. `yCh.field` rides on the region's `yColumn` (the constructor pattern established in WI4-foundation-box2d). yScale's `range = [innerHeight, 0]` means low-pixel-y inverts to high-data-y; the constructor normalises so `xMin / yMin / xMax / yMax` come out correctly regardless of drag direction. Dispatch shape is the 5-field `{ chartId, column: xCh.field, region, sourceTileId, filters: dashboardFilters }` (same as the 1D brushes' WI4 dispatches), guarded by `if (region)` so a null constructor return (zero-area) doesn't fire a malformed event. Ref + state reset on all paths.
    - **SVG element gains `onMouseDown` / `onMouseMove` / `onMouseUp`** (gated on `dashboardTile`) + a **widened `onMouseLeave` (renamed `onSvgMouseLeave`)** that does BOTH `hideTooltip()` AND a brush state clear if `brushStart !== null` (safety net for cursor-leaves-svg-mid-drag). svg cursor toggles to **`crosshair`** during an active brush — the canonical 2D-rect-selection affordance, distinct from the 1D brushes' `ew-resize` / `ns-resize`. The pre-existing per-point `cursor: pointer` style overrides the svg-level cursor on the points themselves, so the crosshair shows through on the gridlines / empty plot area.
    - **Brush `<rect>` overlay inserted inside `<Group>`** right after the zero-line: anchored at `(Math.min(brushStart.x, brushEnd.x), Math.min(brushStart.y, brushEnd.y))` with `width = Math.abs(brushEnd.x - brushStart.x)` and `height = Math.abs(brushEnd.y - brushStart.y)` so reverse drags render correctly. Same primary tint / dashed stroke styling as the 1D brushes (primary at 0.1 fill + 0.4 stroke + dashed `3 3`) + `pointerEvents="none"` so the overlay never intercepts mouse events — load-bearing for per-point click pass-through (a cmd-click on a point UNDER the overlay still reaches the point's onClick → WD3 drill).
    - **Per-point `onMouseMove` tooltip handler gains a single-line guard** `if (brushStart !== null) return;` at the top — during a brush drag the cursor passes over many points and the tooltip would flicker. Same guard shape as BarRenderer's onMouseMove.
    - Pre-existing WD2 cross-filter + WD3 drill-through per-point onClick handlers preserved byte-identical. The brush (SVG-level) and per-point click (point-level) handle different event paths and never conflict in practice: a clean click on a point fires mousedown → mouseup on the point → click on the point, with the SVG's mouseup (`onBrushUp`) hitting the click branch (sub-threshold drag → reset and return) before the point's click handler fires.

Test file appended to [`server/package.json`](../server/package.json)'s explicit `test` list per invariant #4 (right after `wi4Citations.test.ts`).

  **Why this design.**
    - **2D brush state as `{x, y}` pairs, not two parallel scalar states.** Considered: `brushStartX` / `brushStartY` / `brushEndX` / `brushEndY` (four parallel useState hooks). Rejected — the four would update together in every handler, doubling re-render churn and inviting drift where one of the four gets stale (e.g. a hypothetical future handler that updates Y but forgets X). The `{x, y}` pair shape keeps the 2D coord atomic at the React state level: one setState updates both dimensions, the overlay reads `brushStart.x` / `brushStart.y` cleanly, and a null check (`brushStart !== null`) covers both axes simultaneously.
    - **`yColumn = yCh.field` passed to the constructor, NOT widened on the dispatch event.** The WI4-foundation-box2d wave deliberately put `yColumn` on the region itself so the dispatch event's `column` field stays "x-axis column" for all four kinds — this wave consumes that contract directly. Considered: widen `ExplainSliceEvent` with an optional `yColumn?: string` field and have PointRenderer dispatch with it. Rejected — would diverge from the byte-for-byte `DrillThroughEvent` shape that the three click-intent events all mirror, and would force the panel's `filterRowsByBrushRegion` call to widen too. Letting the region carry the y-axis metadata is the encapsulation that keeps the dispatch surface tight.
    - **Both axes must cross BRUSH_MIN_PX for a drag (Option B), not "either axis crosses" (Option A) or Euclidean distance (Option C).** Option A would let a 6×0.5 drag (one dim over threshold, one under) count as a drag — but `makeBox2dRegion` would then return a malformed-feeling region with a 0.5-pixel-wide y-band that filters basically no rows; the panel would render an essentially empty slice. Option C (Euclidean `sqrt(dx² + dy²) >= 6`) would let a 5×5 diagonal drag (≈7.07 px) count as a drag, but a 5×5 brush is barely larger than the click target — likely a jitter, not an intentional rectangle. Option B (both axes >= 6) requires a meaningful rectangle in both dimensions and aligns with user intent for a "selection box" gesture. The constructor's zero-area rejection is a downstream guard for the truly degenerate cases (xMin === xMax or yMin === yMax); this gate is the click-vs-drag split at the brush-up level.
    - **`crosshair` cursor (not `ew-resize` + `ns-resize` combined or `nwse-resize`).** Considered: switch cursor based on which axis the drag is dominant on, OR use a corner-arrow cursor (`nwse-resize`) suggesting a resize handle. Rejected — `crosshair` is the universal "you're selecting a region" affordance across image editors / spreadsheet selection / map drawing tools; users see it and understand "drag to select a 2D region". The 1D brushes use `ew-resize` because they're explicitly 1D widening interactions; scatter's 2D selection is a different gesture and deserves a different affordance.
    - **SVG-level brush handlers + per-point click handlers co-exist via event bubbling.** The mouseDown on a point bubbles to the SVG (so `onBrushDown` captures); the mouseUp on a point bubbles to the SVG (so `onBrushUp` fires with a sub-threshold drag → click branch → reset and return). The browser then fires `click` on the point → the per-point onClick (WD2 cross-filter / WD3 drill on cmd-click) runs as if no brush had happened. A drag from one point to another point fires no click event (the browser only fires click when mousedown and mouseup target the same element) — so the brush handles it cleanly without per-point onClick noise. The `pointerEvents="none"` on the brush rect is the lynchpin: without it, the rect would absorb mouseUp during the latter half of a drag and the brush would never close.
    - **Click-vs-drag gate inside `onBrushUp`, NOT inside `onBrushMove` or via a click-deferral timer.** Considered: track whether the cursor has moved >= BRUSH_MIN_PX on either axis during `onBrushMove` and stash a `hasMoved` flag for `onBrushUp` to read. Rejected — adds a third ref / state without solving a real problem: the `onBrushUp` time is already in possession of both `brushStart` and `brushEnd`, the gate is a single comparison, no benefit to splitting the logic. The 1D brushes (Area / Bar) use this same shape — gate at brushUp time.

  **Why this design.**
    - **Pure-additive backwards compat over a key-version migration.** The simplest fix would be a required `buildCacheKey(tileId, filterHash, regionHash)` with `""` as the no-region sentinel. That breaks every existing 2-arg call site and any persistence layer that holds a pre-wave key (none today, but the WI2 footer has been emitting these keys for several waves and will keep doing so). Making the third arg optional + treating `""` as "omit the segment" keeps the wave purely additive — the WI2 footer's keys are byte-identical pre- and post-wave, and the empty-string sentinel is the natural output of `hashBrushRegion(undefined)` so the call surface composes cleanly.
    - **`hashBrushRegion` lives alongside `hashGlobalFilters`, not in `explainSlice.ts`.** The hash is a cache concern, not a region-data concern. Pairing it with `hashGlobalFilters` keeps all the byte-stability contracts in one file (so a future maintainer chasing "why does this entry not de-dup" finds both helpers together) and lets the existing test file extend instead of spawning a new one. The trade-off is a new type-only import of `BrushRegion` from `explainSlice` — accepted: `BrushRegion` is the canonical name + the type-only import has no runtime cost.
    - **Categorical value order preserved (not sorted).** A negative test pin catches a future drift to sorting. The semantics: a categorical region's `values` array comes from the renderer's `xs.slice(i0, i1)` band-scale slice, which is itself ordered. The dispatched order is fixed upstream by `distinctOrdered(data, enc.x.accessor)` — same band-scale domain source the renderer uses to position bars/lines — so identical brushes always yield identical value arrays without sorting. Sorting at hash time would obscure a genuine "different-brush-shape" signal (a hypothetical future renderer that emits values in a different order would alias regions that should differ).
    - **Three-segment string key, not a structural key type.** Considered: switch the cache `Map<string, …>` to a `Map<{ tileId, filters, region }, …>` and write a structural comparator. Rejected — would slow `get` from `O(1)` string lookup to `O(n)` structural walk over every cached entry, and would force a `Map → custom` wrapper for the LRU eviction (currently relying on `Map`'s insertion-order semantics for free LRU). Keeping the key a stable string preserves every existing behavior of the LRU + TTL cache: get / set / has all stay `O(1)`, `keys().next().value` for eviction stays the cheapest-oldest, the test surface keeps the same shape.
    - **Cache slot for the WI2 footer + WI4 panel do NOT share by design.** Pre-wave they could (incorrectly) — a footer key `tile_X::filters` collided with a panel key on the same `(tile, filters)` regardless of region. Post-wave they live in distinct slots (footer = 2 segments; panel = 3 segments). This is the correct outcome — the footer regenerates the WHOLE-chart insight, the panel regenerates the BRUSHED-region insight; sharing the same cache slot was the bug. The WI4-wire wave's comment ("a brushed slice can hit a cache slot populated by the per-tile Re-explain if filter hashes match") was aspirational and never true in practice; the footer and panel produce different prose under the hood (the panel narrows rows via `filterRowsByBrushRegion` before POSTing). Documented as a deliberate slot split in the new tests' "region-bearing brush does NOT collide with the no-region (footer) entry" pin.

  **Why this design.**
    - **Reuse the WI2 `/api/insight/regen` endpoint, NOT a new endpoint.** The WI4 panel's regen call is structurally identical to the WI2 per-tile regen call (same spec, same row narrowing, same MINI-tier LLM, same response shape). Two alternatives considered:
      1. **New endpoint `/api/insight/regen-slice`** — would duplicate everything WI2 ships (auth, schema, telemetry, prompt building, post-processing) for no payload-shape difference. The brush is just an additional `applyChartFilters`-style filter applied client-side before the request fires.
      2. **Reuse `/api/insight/regen`** (chosen) — zero server changes; the client narrows rows BEFORE sending so the server sees the same input shape it sees from per-tile regen.
    - **`useEffect`-driven auto-fire (NOT user-clicked button).** The user just brushed a slice — making them click a second "Explain" button to fire the regen would feel like an unnecessary step. The brush IS the intent. The hook's `seqRef` already guards against stale-resolve races, so a second brush while the first is resolving is safe.
    - **`IDLE_TILE_ID = "__wi4_idle__"` (NOT a runtime-conditional hook call).** React hook rules forbid conditional hook calls; the only way to keep the panel functional in both `event !== null` and `event === null` states is to always call `useInsightRegen`. The idle tileId never matches a real tile id (the `__` suffix is reserved), so the cache slot never collides.
    - **`regen` omitted from `useEffect` deps, with `eslint-disable` comment.** The hook captures `regenerate` via `useRef` internally — the function reference is stable across renders within an instance. Including it in deps would create a render loop (the effect fires → setState in the hook → re-render → effect re-fires → …). The eslint-disable comment is intentional + documented; without the comment the lint rule would auto-suggest a "fix" that breaks the design.
    - **Inline IIFE for chart resolution (NOT a `useMemo` at the DashboardView level).** The resolution math is 5 lines and runs ONLY when the panel re-renders. Lifting it into a memo would force a new render dependency on `explainSliceEvent` + `activeSheet` AND a separate `let explainChart = …` line elsewhere — more code for no perf win. Co-locating the IIFE at the prop pass-through keeps the resolution + the consumer next to each other.
    - **Two-stage narrowing (NOT a single combined filter).** Splitting `applyChartFilters` (handles categorical / numeric / date filters from the global filter map) and `filterRowsByBrushRegion` (handles the brushed sub-domain) lets each helper stay focused on its concern. A future widening that adds e.g. a per-tile filter to the WI4 pipeline drops in as a third filter call without restructuring. Same separation-of-concerns precedent as `applyChartFilters` itself (which is already split per filter type in `chartFilters.ts`).
    - **Cache sharing with WI2 per-tile regen.** Same `insightRegenCache` instance, same `(tileId, filterHash)` key shape. A user who clicks "Re-explain" on a tile, then brushes the same tile with no global filter changes, hits the cache (the brushed tileId === the per-tile tileId). The brushed slice doesn't usually match the per-tile shape (different row count), but the cache key uses tile + filters, not rows — so a hit is possible. This is a free cross-surface win and not a contract risk (the cached entry's prose IS describing the right rows because filters are part of the key).
