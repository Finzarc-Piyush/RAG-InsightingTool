  **Why this design.**
    - **Why a dedicated POST rather than re-using W61-save's PATCH.** The PATCH path replaces the wholesale model — this forces the client to (a) round-trip the entire model body (potentially ~5–50 KB depending on session) for what's logically a single-entry append, and (b) re-implement name-uniqueness validation locally before sending (the server's `safeParse` would accept duplicate names because the schema doesn't enforce within-collection uniqueness). The typed POST takes just the new entry as the body, the server validates uniqueness inside the lock against the authoritative state (not a stale client view), and the response carries the full new model so the client mutation success handler is identical to save / revert / delete. The four-endpoint family now reads as a coherent set: save = wholesale replace, add = append one, delete = remove one, revert = restore prior.
    - **Why 409 Conflict for `name_already_exists`, not 404.** HTTP 409 semantically means "the request conflicts with the current state of the resource" — exactly the case here (we're trying to create an entry that already exists by name within the same kind). The W61-revert / W61-delete endpoints use 404 for `*_not_found` because the resource is absent; for `name_already_exists` the conflict semantic is collision, not absence. The client can distinguish 409 from 4xx generally and render a "this name is already taken" inline message rather than the generic "something went wrong" copy. The 409 envelope includes `{ sessionId, kind, name }` echo so the client doesn't need to track which request triggered the collision.
    - **Why cross-kind name collisions are explicitly allowed.** A metric named "revenue" and a dimension named "revenue" are different resources — the planner addresses them via `{ kind, name }` not `{ name }` alone, and the SemanticModel schema doesn't forbid the collision (each kind has its own collection in the model). Requiring `:kind` in the URL gives the uniqueness check the right scope: same-kind collisions are rejected with 409, cross-kind collisions are allowed. A dedicated test pins this behaviour (`metric "revenue"` exists; `POST /entries/dimension` with `body.name === "revenue"` → 200, both coexist in the model).
    - **Why we skip the source-bumper on add.** Same precedent as W61-delete-server: survivors are byte-identical to their prior selves (we're appending, not editing existing entries), so the bumper's content-hash diff would match them as unchanged and preserve their prior `source`. Running it would be a no-op for the existing entries. For the new entry, the bumper's lookup against `priorByName` returns undefined (the new name doesn't yet exist in the prior model — we just verified that with the uniqueness check) so the bumper's "new entry" branch returns the entry unchanged with its client-sent `source` preserved. The schema's `.default("user")` on each source field populates new entries by default; if the client sends `"domain"` (e.g., importing from a domain pack) it's preserved verbatim. Skipping the bumper call entirely is clearer about intent — the documentation comment explains the no-op equivalence so a future reader doesn't waste time wondering why W61-add doesn't bump while W61-save does.
    - **Why audit-write BEFORE the append (not after).** Same precedent as W61-save / W61-audit-revert / W61-delete-server. The buffer's role is to remember what was just changed; if the audit-write happened after the mutation, a crash between the two would lose the snapshot. Inside `withSessionWriteLock` the pair is atomic at the per-session level (single-instance correctness per invariant #9). A future "undo this add via revert" walks the audit log, finds the pre-add snapshot, and restores it — wiping the just-added entry. The audit-write-before-mutation pattern is the same shape across all four W61 endpoints; the consistency makes the four-endpoint family interchangeable from the audit-trail perspective.
    - **Why bump `version` on add (we're not changing existing entries).** W64's compiled-query cache keys on `semanticModel.version`. Adding a new metric — even one with a snake_case name that no existing chart references — could affect compilation of future queries: the planner might suggest the new metric in subsequent turns, the W58 compiler might emit lint warnings for orphan references against the new entry's `references[]` columns, etc. Bumping makes the cache miss explicit so the next planner round consults the fresh model. The cost of an unnecessary cache miss (one extra compilation) is much smaller than the cost of stale compilation against an out-of-date model.
    - **Why we reuse the existing `_detailFetcher` / `_updater` injectables.** Same Cosmos doc read+write path as save / revert / delete — fetch the chat document by sessionId, mutate `semanticModel` + `semanticModelAuditLog` + `lastUpdatedAt`, write the whole doc back. A new pair of add-specific injectables would just be the same functions under different names and would double the test-fake surface for no behavioural benefit.
    - **Why the kind-appropriate body schema is picked via ternary, not a `Record<kind, schema>` lookup.** A `Record<"metric"|"dimension"|"hierarchy", z.ZodSchema>` would type-check fine, but the three zod schemas have different inferred entry types (`SemanticMetric` vs `SemanticDimension` vs `SemanticHierarchy`) — the ternary preserves the per-kind type narrowing inside the `if (parsedKind.data === "metric")` blocks where we cast `parsedBody.data` to the concrete kind type. A `Record` lookup would resolve to a union type that the `.some((m) => m.name === entry.name)` line would have to type-narrow again. The ternary is uglier but reads more directly as "pick the right schema for this kind".

  **Why this design.**
    - **Why a controlled `<AlertDialog>` rather than `window.confirm`.** The downstream-references count is async — the modal needs to render `"Checking for downstream chart references…"` while the round-trip is in flight. `window.confirm` is synchronous and can't show intermediate states; the W61-audit-revert path uses `window.confirm` because its only context is the entry already in hand, but the delete path is references-count-first. The async dependency dictates the UI primitive.
    - **Why the modal owns the references fetch, not the parent.** References state is mount-scoped — it lives for the duration of the modal and is garbage-collected on close. Keeping the fetch / loading / error state inside the component lets React's effect cleanup handle the teardown automatically. Putting the references in the parent would force a `clearReferences` step on every close + a stale-fetch guard when the admin re-opens for a different entry quickly.   **Why this design.**
    - **Why path params for `:kind` and `:name`, not query / body.** Both are part of the resource identity — "which entry are we deleting?" DELETE has no body in idiomatic HTTP; query params are for filters, not identifiers. Path identifies the resource; the three-segment shape (`/entries/:kind/:name`) reads naturally as a hierarchical resource path.
    - **Why explicit `:kind`, not auto-scanning all three collections for the name.** Name collisions across kinds are legal — the semantic model schema doesn't forbid a metric and a dimension sharing a name. A fuzzy "find this name anywhere and delete it" would silently delete the wrong thing when a collision exists. Requiring `:kind` in the URL makes the operation unambiguous; a dedicated test (`kind=dimension` + `name=alpha` when only a metric named `alpha` exists → 404 `entry_not_found`) pins the no-fuzzy contract.
    - **Why audit-write BEFORE delete (not after).** The buffer's role is to remember what was just lost. If the audit-write happened after the delete, a crash between the two would lose forensics. Inside `withSessionWriteLock` the pair is atomic at the per-session level (single-instance correctness only; multi-instance horizontal scaling still needs Cosmos `ifMatch` ETag). Mirrors the W61-audit-log + W61-audit-revert precedent.
    - **Why bump `version` even though we removed something.** The version is W64's compiled-query cache key. Removing a metric invalidates any cached compilation that referenced it; bumping makes the cache miss explicit and ensures the next planner round consults the fresh model.
    - **Why we do NOT run `bumpMetricsSource` / `bumpDimensionsSource` / `bumpHierarchiesSource` on the survivors.** The surviving entries are byte-identical to their prior selves; the bumper's content-hash diff would match them as unchanged and preserve their prior source. Running it would be a no-op for unchanged entries; skipping is semantically equivalent and clearer about intent. A success-path test pins this: a metric with `source: "user"` and another with `source: "domain"` keep those sources verbatim after a third metric is deleted (no spurious bump-to-user on the survivors).
    - **Why the audit-log write happens unconditionally on success.** A delete IS a state change; preserving the just-deleted entry in the buffer is required for "undo this delete via revert" to work. The W61-audit-revert wave's revert endpoint walks the buffer and restores `priorModel`; this wave's delete entry produces an audit entry whose `priorModel.metrics` (or `.dimensions` / `.hierarchies`) includes the deleted entry, so a future revert call can put it back.
    - **Why the `DeleteResult` discriminated union sentinel.** Throwing from inside the `withSessionWriteLock` callback would mark prior chained callers as failed (the lock serialises per-session writes); returning a sentinel from the callback and mapping it to an HTTP status in the outer handler keeps the per-session error semantics clean. Same pattern as `PatchResult` (W61-save) and `RevertResult` (W61-audit-revert).
    - **Why `removed` is set then `void removed`.** The early `return { kind: "entry_not_found" }` in each branch guarantees the flag is true on every success path; asserting it would be dead code. Keeping the variable (with the `void` to silence the unused-variable lint) makes the per-branch shape read consistently — a future fourth kind would set `removed = true` in the same way. The cost is one line; the benefit is symmetric branches.

  **Why this design.**
    - **Why a GET, not a POST.** This is an idempotent read — the scan walks the doc's persisted chart list and returns a count; nothing mutates. GET semantics let the client cache the response via standard HTTP rules, let the admin paste the URL to share "this metric is used N places" without a body, and let DevTools / curl exercise the endpoint without crafting JSON. POST would imply state change.
    - **Why a query param (`?entry=foo`), not a path param (`/.../references/foo`).** Entry names can in principle contain characters that are awkward in path positions (slashes most obviously; `metric.label` strings can have anything). `metric.name` itself is snake_case in practice, but the endpoint accepts any string the admin might type. Query-param is the canonical Express shape for an idempotent read filter; the path identifies the resource (the session), and the query is the filter on it. A future variant that wants a different filter (e.g. by entry kind) lands as another query param without re-shaping the path.
    - **Why an array `?entry=a&entry=b` produces a 400 (not "pick first" / "scan all").** Express's `qs` parser turns repeated keys into a string array; the handler's `typeof entryRaw === "string"` narrow rejects arrays. Picking the first (or scanning all) would silently mask a malformed client request — better to 400 so the client surfaces the bug immediately. A future bulk-scan endpoint (`?entries=a,b,c`) is a separate shape that opts into multi-entry semantics explicitly.
    - **Why walk `doc.charts[]` and NOT `doc.messages[].charts[]`.** [`chat.model.ts:55`](../server/models/chat.model.ts) describes `doc.charts: ChartSpec[]` as "All charts generated for this chat". The chat-model save path (around lines 713-743) dedup-merges per-message charts into the top-level list, so `doc.charts[]` is the authoritative full list. Walking both would double-count any chart that appears in both arrays.
    - **Why blob-stored `chartReferences[]` are NOT fetched.** Pulling each blob would turn a free in-memory count into a multi-second multi-request fetch; the cost grows linearly with the session's chart count, and most admin sessions don't have blob-stored charts in the first place (the threshold is large data arrays only). A future enhancement could load blob charts on-demand for sessions that exceeded the threshold; that's its own wave because it changes the endpoint's latency profile.

A separate `_referencesFetcher` would be the same function under a different name and would double the test-fake surface. The W61-audit-history-api wave established the "share the detail fetcher across read endpoints" pattern; this wave continues it.
    - **Why echo the server-trimmed `entry` in the envelope.** `?entry=%20net_sales_value%20` arrives as `" net_sales_value "` and the server trims it before the scan. Echoing the trimmed form (not the raw query value) lets the client compare the response's `entry` against its own state without re-implementing the trim rule.   **Why this design.**
    - **Why exact-identifier-only matching, no substring or fuzzy.** Substring matching would false-positive across similarly-named columns ("sales" matches "sales_volume" AND "net_sales_value" AND "sales_amount"); too noisy to be load-bearing for a destructive delete prompt. The contract is `name === field` with no normalisation, no case-folding, no trim. A future enhancement could add opt-in expression-substring mode behind a flag once a real false-negative surfaces — but the v1 surface stays strict.
    - **Why `transform.filter.expr` and `transform.calculate.expr` are NOT walked.** Those fields are SQL-like strings (e.g. `"net_sales_value > 0"`); substring matching would inherit the noise from above. A test pins the negative behaviour so a future "let's just substring-match expressions" refactor surfaces immediately as a test break. The right path to expression support is a SQL identifier tokenizer + opt-in flag, not silent string contains.
    - **Why empty `name` short-circuits at three layers.** Top-level `countSemanticModelReferences("", ...)` returns zero; each per-spec scanner ALSO guards on empty name. Defense in depth: a malformed semantic-model entry with an empty `name` field would otherwise chart-count every chart that happens to have an empty `seriesColumn` or empty `y2`. The schema's `.max(...)` bounds don't prevent empty strings at the type level — they prevent oversized strings. Empty-name handling is a runtime concern.
    - **Why the top-level function accepts `ReadonlyArray<unknown>`, not `(ChartSpec | ChartSpecV2)[]`.** The caller is the future `getSemanticModelReferences` controller handler; it hands us whatever lives at `doc.charts` — which is typed as `ChartSpec[]` but the runtime values may be v2 specs after the planned migration AND may contain stray null / undefined / primitive slots from older data shapes. The defensive `typeof === "object"` + `value !== null` guard at the top of the loop means a malformed doc doesn't 500 the endpoint; the unreadable chart is silently skipped, which is the right UX for a "how many" question (an unparseable chart can't be destructively-affected by removing a metric we don't know it references).
    - **Why discriminate v1 vs v2 via [`isChartSpecV2`](../server/shared/schema.ts) at runtime rather than at the type level.** The chart storage path types as `ChartSpec[]` (v1) but [`isChartSpecV2`](../server/shared/schema.ts) is the canonical runtime discriminator (`value.version === 2 && typeof value.mark === "string"`). Walking the right field set per shape produces accurate counts; falling back to v1 scanner on a v2 chart would silently undercount (v1 has no `encoding.color.field` to walk). Future-proofs against the v1→v2 migration without needing a rebuild of the scanner.
    - **Why count `totalOccurrences` AND `chartCount`, not just one.** The UI's delete prompt only needs `chartCount` ("removing this metric will break N charts"). `totalOccurrences` is a debug / observability signal — a metric used in 1 chart but referenced 4 times (x + y + facetRow + tooltip) signals a different reliability concern than the same chart count with 1 occurrence each. Both fields are cheap to compute in the same pass; emitting both leaves room for a future detail-level UX without a scanner change.
    - **Why a separate wave from the endpoint rather than bundling.** The brief estimated ~50 LOC for the reference-scan endpoint, which is mostly the controller handler wiring + 403 / 400 / 404 / 200 plumbing. Bundled with the scanner the wave would be ~600 LOC + tests — at the upper edge of the wave budget. Splitting keeps each sub-wave small (~500 LOC scanner + tests this wave; ~150 LOC endpoint + tests next wave), and the scanner's pure-function semantics get the focused test coverage they deserve before the controller layer adds Express plumbing on top.

  **Why this design.**
    - **Direct `fetch + useState` rather than TanStack Query.** The session brief proposed `useQuery({ enabled: tabOpen })` because that's the natural shape for "fetch when the tab opens", but the host component uses direct `useEffect + useState` throughout (the existing detail-fetch effect at line 633-670 is the same pattern). Introducing TanStack Query mid-component would be a stylistic discontinuity without a behavioural payoff — the audit log is fetched at most once per page visit (gated by `historyOpen`) and refreshed on save-bump. A future broader-scope refactor that converts the whole admin viewer to TanStack Query is a separate consideration; that refactor would also restructure the PATCH path's optimistic-update flow, which is out of scope here.
    - **`historyOpen` closed by default.** Open by default would fetch the audit log on every detail-page mount even though most visits are "look up something, leave" without consulting history. Closed-by-default is opt-in: the fetch happens only when the admin explicitly clicks the trigger. The closed state is on the component (not URL-synced) because the deep-link use case ("share me the open history view") is speculative — can promote to a `?history=open` toggle later if a user-driven need appears, but adding two more places that read/write search params now would just be friction.
    - **Per-row `reverting: number | null` state, not a global boolean.** A list of up to 10 entries with a single `reverting: boolean` would disable every row's Revert button while one is in flight. With the per-index flag, all other rows remain readable AND queue-able (a re-entry guard on the handler still prevents the second click from launching a second revert, but the admin can read the others while waiting). The server's `withSessionWriteLock` (invariant #9) makes any concurrent attempt safe — only one mutation lands per session at a time; the others see fresh state on retry. Tradeoff acknowledged: a Revert click while another is in flight is silently dropped by the handler's guard rather than queued; this is fine because the admin almost always wants to inspect the result of the first revert before deciding the second.
    - **Effect dep on `lastUpdatedAt`, not `data`.** React's dep-array compares by reference; `data?.lastUpdatedAt` is a number (or undefined), comparable by value, and changes exactly on save or revert. Using `data` directly would re-trigger the effect on every `setData` even when only an optimistic toggle bumped — and the optimistic-toggle path on the PATCH inside `persistModel` rebuilds the `data` object reference but keeps `lastUpdatedAt` until the server response. Extracting `lastUpdatedAt` to a local variable before the effect spares the re-render storm and reads naturally as "re-fetch when the doc has been written".
    - **Effect cleanup flag matches the existing pattern.** The host component already uses `let cancelled = false; ... return () => { cancelled = true; };` in the detail-fetch effect. Repeating the pattern in the history effect keeps two effects in this file stylistically identical so a future reader doesn't trip on "wait, why does this one use AbortController and that one uses a flag?". `fetch()` itself is unaborted on cancel — the host detail fetch doesn't abort either, and a 200 KB audit-log response that lands after navigation just gets garbage-collected without the setState landing. Adding `AbortController` for the audit-log fetch but not the detail fetch would create an inconsistency without a measurable benefit.
    - **Revert success path reuses the existing PATCH-success shape.** Both PATCH and revert produce `{ sessionId, lastUpdatedAt, model }` (W61-audit-revert deliberately reuses `PatchSemanticModelResponse`'s envelope so the client's success handler doesn't fork). The revert handler does the same `setData({ ...data, lastUpdatedAt: res.lastUpdatedAt, model: res.model })` as `persistModel`, and the `lastUpdatedAt` bump rolls into the same effect that PATCH success rolls into — meaning the audit log refreshes automatically with the newly-appended (the just-overwritten) entry at index 0. The W61-audit-revert wave's audit-write-on-revert design is what makes this work: the just-overwritten model becomes the new newest entry, and "undo this revert" is naturally `auditEntryIndex: 0` on the next click.
    - **Disabled-gate triple `saving || reverting !== null || historyLoading`.** Three sources can race: an in-flight PATCH from the existing save UI (saving=true), an in-flight revert (reverting≠null), and an in-flight audit-log refetch (historyLoading=true). Disabling all Revert buttons during any of the three prevents a second mutation against a stale read or against an in-progress write. The cost is mild UI friction (Revert is disabled during the brief window between save-success and the audit-log-refetch landing), but that's correct: clicking Revert during that window would target an index against the pre-save audit log shape.
    - **`<button>` for the trigger, not the page-existing `<Card><header>` shape.** Other cards on the page (`Metrics`, `Dimensions`, `Hierarchies`) use a non-interactive `<header>` because there's no toggle. The Audit-history card needs the whole header row to be clickable for the collapse toggle; a `<button>` element is the semantically-correct interactive role for that, and `Collapsible asChild` lets Radix forward the trigger props onto the `<button>`. Tailwind classes match the other cards' header padding / border / typography so visually the cards read as a family.
    - **Test-IDs everywhere.** Each branch (`trigger`, `loading`, `error`, `empty`, `list`, `revert-error`, `revert-${idx}`) gets a stable `data-testid` so a future smoke test can drive the surface without text-scraping. The cost is ~80 chars of attribute markup; the benefit is that the surface stays mechanically queryable even if the copy changes.

  **Why this design.**
    - **UTC timestamp formatting.** Audit log forensics often involve multiple admins reconciling who-did-what-when across timezones. `2026-05-20 14:23 UTC` reads identically for everyone; a locale-aware formatter would let alice in NYC and bob in Bangalore both see `"14:23"` and disagree on what wall-clock moment that was. A dedicated test pins that the output never includes a comma, an ISO-8601 `T` between date and time, or a seconds component — catches a future "let's just call `toISOString()`" refactor that would flip to `2026-05-20T14:23:45.000Z` and break the audit-log row's visual rhythm.
    - **`saveNumber = total - indexFromNewest`.** The buffer is newest-first (W61-audit-log convention). The most recent entry (index 0) is `Save M of M` because it was the M-th save chronologically. Walking back through indices: index 1 → Save M-1, etc., down to index M-1 → Save 1 (the oldest retained entry — the W57-inferred original or whatever was at the tail when the buffer first filled). A dedicated test walks all 5 indices of a 5-entry buffer to pin the mapping, and a second test pins the cap-at-10 boundaries (`Save 10 of 10` at index 0, `Save 1 of 10` at index 9).
    - **Two-field `{ headline, subhead }` over a single string.** The history-tab UI renders the headline at base-font and the subhead muted (smaller, lower-contrast); pre-splitting at the helper boundary spares the JSX from string-split-and-style at every render. A future re-style that decides to put both on one line (e.g. for compact mode) just concatenates `headline + " — " + subhead` at the call site rather than re-running the formatter.
    - **`priorVersion` in the subhead, not the headline.** Two adjacent audit entries can share `savedBy` (alice edited twice in a row) AND share the same `savedAt` minute (consecutive saves often land within the same minute). The version is what differentiates them, and it matches the revert mental model ("revert to the state of version 2"). The version is also the W64 compiled-query cache key, so `was v2` in the audit log corresponds directly to the cache entry that revert would restore. Putting it in the subhead (not the headline) keeps the headline scannable while the disambiguator stays available.
    - **`buildRevertConfirmation` includes the undo-safety note.** Load-bearing for the "undo this revert works" semantic — the prompt should set the admin's expectation that reverting is itself reversible (and the W61-audit-revert wave's audit-write-on-revert is what makes that true). The prompt anchors on `priorVersion` + `savedBy` so admins double-check the target before clicking through; a dedicated test pins both anchors plus the undo-safety wording.
    - **`unknown` savedBy passthrough.** The server stamps `"unknown"` when `getAuthenticatedEmail` returns null (the failure mode for the auth helper). The summary shouldn't munge that value — admins should see the raw attribution-failure marker so they know something wasn't recorded. A test pins that `"unknown · "` appears in the subhead verbatim.
    - **Local interface mirror, not type re-import.** The server's `SemanticModelAuditEntry` lives in `server/lib/semantic/semanticModelAuditLog.ts` which the client can't import across the runtime boundary. A local `AdminSemanticModelAuditEntry` interface in `admin.ts` is the established W61 pattern (mirrors how `SemanticModel` is inline-imported via `import("@/shared/schema")`). Runtime drift between this mirror and the server surfaces as field-undefined access at TanStack Query call sites rather than a compile error, but the W61 pattern accepts that tradeoff because the schema is small and stable.

Running the bumper would stamp every entry that differs between snapshot and current as `"user"` (wrong — e.g. reverting from a `source: "user"` admin-edited state back to the original `source: "auto"` W57-inferred snapshot would re-stamp the entry as `"user"`, hiding the revert from the source-filter chips and the planner's source-weighting). The model-level `updatedBy` still stamps the reverting admin (preserves attribution at the model level) but the per-entry `source` comes verbatim from `entry.priorModel`. The "source-preservation" test pins this with a fixture where current is `source: "user"` and the snapshot is `source: "auto"` — the test would fail if a future refactor wired the bumper into the revert path.
    - **Bump version monotonically.** Reverting to "the state of version 3" makes the live model `current+1` with the contents of v3, NOT version 3 again. The version field is load-bearing for the W64 compiled-query cache (which keys on it) and for the W61-audit-log's `priorVersion` (which records the version at the moment of save). A non-monotonic version would break both: the cache would serve stale compiled queries, and the audit-log entries would have ambiguous `priorVersion` values (two entries with `priorVersion: 3` from different points in time). The "happy path" test pins `body.model.version === 3` for a revert applied to a v2 doc; the round-trip test extends the pin to `version === 4` after a second revert.
    - **Write the audit-log entry on revert.** Two reasons. (a) Undoing the revert (admin reverts, then realises that was wrong, and reverts again) needs the just-overwritten state preserved. (b) Forensic record — "alice@example.com reverted on 2026-05-20" is a saveable event in its own right; the audit log should capture state transitions, not just "edits". The round-trip test pins this directly: revert v2 → v1 (audit log captures v2 as the new newest entry), then revert again with `auditEntryIndex: 0` (which now targets the v2 entry → undoes the revert with content === v2 and version === 4).
    - **Bound-check inside the lock.** A multi-admin race could have admin A growing the buffer (via PATCH or revert) while admin B is composing a revert request; reading the buffer length before the lock would let B's request target a now-stale index. Inside the lock the buffer length is authoritative — `parsed.data.auditEntryIndex >= log.length` uses the just-loaded `doc.semanticModelAuditLog`, not a stale client view. Conservative: 404 `audit_entry_not_found` if the race produces an out-of-bounds index, and the client retries with the fresh history fetch.
    - **404 not 400 for `audit_entry_not_found`.** The body parsed cleanly (the schema accepted a nonnegative int); the index is out of bounds because the buffer doesn't have that many entries — a resource-not-found condition, not a malformed request. 404 mirrors `session_not_found` / `semantic_model_not_inferred` which are also "the thing you asked for doesn't exist" conditions. The body echoes back the offending index so the client can show "you tried to revert to entry 5 but only 2 entries exist" without a separate state lookup.
    - **`auditEntryIndex: 0` semantics — "undo my last save".** The buffer is newest-first (W61-audit-log convention) so index 0 is the most recent prior model. This matches the natural mental model of an undo button: "undo my most recent change". A future history-tab UI's "Undo" affordance is just a `POST { auditEntryIndex: 0 }`.
    - **Reusing the W61-audit-log convention for the audit-write on revert.** The same `appendSemanticModelAuditEntry` helper is called from the revert path with `{ savedAt, savedBy: reverting-admin, priorVersion: current.version, priorModel: doc.semanticModel }`. The cap-at-10 logic applies uniformly — if a revert pushes the buffer to 11 entries (revert from a buffer at cap), the oldest is dropped. The cap behavior is the same regardless of whether the new entry came from PATCH or revert; a unified semantics keeps the buffer's invariants tight (always ≤10, always newest-first, no special-case entries).
    - **Cap-at-10 edge case: reverting to the oldest entry.** If `auditEntryIndex` is 9 (the oldest at cap) and the buffer is at cap (length 10), the new entry from the revert grows the buffer to 11 → trim to 10 → the targeted entry at index 9 is dropped from the buffer. This is fine because we already extracted its `priorModel` before the prepend. Subsequent reverts target indices 0–9 of the new buffer (which always includes the just-overwritten state at index 0, so "undo this revert" remains possible).
    - **Response envelope matches W61-save's.** Both endpoints produce `{ sessionId, lastUpdatedAt, model }`. A future client `useMutation` for the W61-audit-history-ui's revert button can reuse the same success-handler that the existing PATCH mutation uses — invalidate the detail query, invalidate the audit-log query, optionally show a toast. A different envelope would force a duplicate success-handler.

Adding a separate `_reverter` injectable would duplicate surface for no behavioral benefit — the read+write pair is the same Cosmos doc and the same Cosmos update.

    - **Dedicated GET endpoint rather than widening `AdminSemanticModelDetailResponse`.** Two alternatives were considered.     - **Return `entries: []` rather than `entries: undefined` for the never-PATCHed case.** The buffer is `?: SemanticModelAuditEntry[]` on the doc (TypeScript-optional) so it's literally `undefined` on a session that's never been PATCHed. A test (`response.entries is byte-equivalent to doc.semanticModelAuditLog`) pins this — a future refactor that introduced a projection would have to update the pin in the same diff.

  **Why this design.**
    - **Snapshot the *prior* model, not the next.** Considered: write the next model as the audit entry so the buffer reads as "history of saves you've made". Storing the next model would force the revert UI to walk forward through the buffer (less natural) and would not capture the very-first state because there's no "save 0" entry.

The deep-history use case (revert to the 11th-most-recent save) is rare enough that paying for unbounded growth across months of saves isn't worth it — an admin who has made >10 saves has almost certainly settled on a stable model. The const is exported so a future wave that genuinely needs more history can raise it in one place + the [pin test](../server/tests/semanticModelAuditLogW61AuditLog.test.ts) catches the change in the same diff.

  **Why this design.**
    - **`replaceState` rather than `pushState`.** Considered: each chip click pushes a new history entry so the browser back button could step through filter states. Rejected — admins expect back to navigate to the previous page (the admin index, in this case), not toggle through chip states. Stacking five chip clicks would require five back-presses to leave the page, which is broken behaviour. The single history entry stays at the most-recently-clicked filter so the URL reflects the current view AND the back button does what admins expect.
    - **Pre-existing `window.history.state` as the first arg to `replaceState`.** The first arg is the state object the browser exposes via `popstate`; passing the existing state means a future deep-link feature that stashes nav metadata there won't get clobbered by the filter-sync rewrite. Passing `null` would silently wipe any pre-existing state from a parent route's `pushState` call.
    - **Pathname + hash reconstruction not full `window.location.href` rewrite.** The latter would force a full URL parse-and-replace (and could subtly alter the URL's encoding if the browser normalises differently); reconstructing from the three known components (`pathname` / `search` / `hash`) is byte-exact and cheaper.
    - **Lazy `useState` initializer, not `useEffect`-on-mount.** Considered: `useState("all")` plus a `useEffect` that reads from URL on mount and calls `setSourceFilter`. Rejected — that approach renders once with `"all"` then re-renders with the URL value, which causes a visible chip-flicker on page load if the URL has a non-default filter. The lazy initializer reads the URL on the first render so the first paint already shows the correct chip.
    - **`typeof window === "undefined"` guard.** Pure defense against future SSR / prerender — the admin route is client-only today but the guard costs nothing and saves a confusing crash if someone migrates the project to Next.js.
    - **Fall-back to `"all"` rather than throw on invalid URL.** The URL is admin-controlled and admins mistype; throwing would crash the page-mount and lock the admin out. Falling back to the safe default gracefully degrades while the URL still shows the wrong value (the next chip click will rewrite it cleanly).
    - **Short `filter` param name not `source_filter`.** Admins frequently copy URLs into chat / docs / GitHub issues; a verbose param reads as noise. The param's meaning is unambiguous in context (no other filter on this page) so the short name doesn't cost clarity.
    - **Case-sensitive validation.** Considered: lower-case the incoming value before validation so `?filter=USER` works. Rejected — `URL` capitalisation is byte-stable in real use (browsers preserve case in `location.search`); a lenient match would mean an admin typing `USER` thinks they're applying a filter when they are, while a different scenario with `User` (title-case from a casually-typed link) would also work — both feel correct until a future use case requires the byte-stable form (e.g., URL hashing for analytics) and the leniency masks a real drift.
    - **Remove the param entirely for `"all"`, not write `?filter=all`.** The default state IS no filter; `?filter=all` would be redundant byte-noise and would clutter the URL admins are copying. The absence of the param means the same thing (`readFilterFromSearch` falls back to `"all"` on missing). The `URLSearchParams.delete` call is the standard idiom.
    - **Return strings without a leading `?`.** The caller's `history.replaceState` call wants to control the leading-? semantics — empty string means "no query params" and "?" + value means "these params". Returning a leading `?` from the helper would make the empty-case ambiguous (was the URL meant to have an empty `?` or no `?`?). The pin in tests catches a future "helpful" refactor that adds the prefix.

    - **`__auto__` sentinel for the temporal-grain undefined case.** Radix `<Select>` rejects `value={""}` (empty string is reserved for the "no selection / show placeholder" state) and doesn't have a built-in "clear selection" affordance. The sentinel keeps Radix happy AND lets the admin explicitly opt back into auto-derivation after pinning a grain by mistake. The "preserve on round-trip" UX win is small (admin can type the 3-char code in 2 seconds); the misleading-manifest cost is the planner downstream of every wrong-shape correction.
    - **`validateCurrencyCode` empty-accepts even though the schema makes it required-with-format-currency.** The field is `.optional()` at the schema level; the cross-field constraint is enforced by the server at PATCH time. Making the client validator reject empty would block the admin's natural "I'm switching to currency, give me a second to type the code" workflow — they'd hit a red border before typing the first letter. The server's authoritative rejection (with a clear zod issue path) is the right place for that check.

    - **Save-on-blur over save-on-Enter or per-row Save button.** Considered: an explicit Save button per row that batches edits + a discard-changes button. Rejected — the parent's `persistModel` already debounces by routing through `withSessionWriteLock(sessionId, ...)` (invariant #9), so multiple sequential field saves serialize cleanly without a batch button.     - **Client-side validators that mirror but don't share zod schemas.** Considered: import `semanticMetricSchema.shape.label` and re-derive the bounds at runtime. Rejected — the validators are 96 LOC of pure-function bounds checks; if the schema bounds change, a drift here just routes the save to the server's authoritative rejection (a 400 with the zod issues array surfacing in the existing "Save failed: {message}" banner).     - **Keyword-guard via padded uppercase scan instead of regex.** Considered: `/\b(SELECT|FROM|JOIN|...)\b/i` over the trimmed string. Rejected — JavaScript `\b` matches at underscores which would let `_FROM_` slip through; a literal regex with `(?<![A-Za-z0-9_])` lookbehind works but the engine's lookbehind support varies by runtime.   **Why this design.**
    - **Replace-whole-model PATCH body rather than partial-patch.** Considered: a JSON Patch–style body that diffs only the changed field (`{ op: "replace", path: "/metrics/0/exposed", value: false }`). Rejected — admin edits are small and the client already holds the prior model in memory; sending the whole thing keeps the diff logic out of the server and lets `semanticModelSchema.parse` validate the payload uniformly. JSON Patch would also break the server's "validate the whole shape" invariant in subtle ways (a malformed metric appended via patch wouldn't trigger the full-model parse).
    - **Server overrides version + updatedAt + updatedBy regardless of client-sent values.** A client-set version would let an admin accidentally bump backwards (set it to 1) or jump forward (set to 9999), and the W64 compiled-query cache reads this version as its cache key; a stable monotonic bump per save is load-bearing. The server-side stamp is enforced by overwriting after `parsed.data` is constructed.
    - **`PatchResult` discriminated union over throwing inside the lock.** `withSessionWriteLock` chains promises per session; an inner throw rejects the chained promise that future callers `await`, which the existing implementation already swallows in its `catch {}` — but the inner result is still lost. Returning a typed result via the union keeps the 404 paths first-class and makes the controller's HTTP-mapping layer one clean switch.
    - **`withSessionWriteLock` over a global mutex.** Per invariant #9 every RMW on a `ChatDocument` MUST acquire `withSessionWriteLock(sessionId, fn)` — a global mutex would serialize unrelated sessions and slow the admin's batch-edit flow. The per-session lock is the existing contract; the admin path slots into it cleanly.
    - **Single `saving` flag over per-row state.** Considered: each row could have its own `saving` state so two simultaneous toggles each show their own spinner. Rejected — `withSessionWriteLock` already serializes the writes on the same session; firing two toggles in parallel would mean the second sees a stale optimistic view of the first's result and the round-trip ordering becomes hard to reason about. A single flag disables every toggle during a save, forces a clean "one save at a time" UX, and matches the actual server-side serialization.

    - **Cohen's conventions on |r|, not on R².** Considered: bucket on R² since R² is what the prose already carries and what the WQ1 classifier reads.
